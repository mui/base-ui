# Handoff: docs-infra React index generation — canary `4c7a7b6` does not reproduce the section-grouped `page.mdx`

## Context

base-ui is migrating generation of the React docs index page
(`docs/src/app/(docs)/react/page.mdx`) **out of a standalone script and into the
docs-infra pipeline**.

- The old approach: a repo-local script `docs/scripts/generateReactIndexPage/index.mjs`
  read `sitemap.data[sectionKey].pages` and emitted a page grouped under four
  `## Section` headings (Overview / Handbook / Components / Utilities), tagging
  `[New]` only when a page's own `page.tags` contained it.
- The new approach: let docs-infra's `transformMarkdownMetadata` (`extractToIndex`)
  and `loadPrecomputedTypes` (`updateParentIndex`) own the index page. The script is
  being deleted.

We are testing this against a canary build to see whether the pipeline reproduces
the intended grouped output. **It does not yet.**

## Package under test

```
@mui/internal-docs-infra: https://pkg.pr.new/mui/mui-public/@mui/internal-docs-infra@4c7a7b6
```

Relevant consumer config (base-ui `docs/next.config.mjs`):

```js
// remark
['@mui/internal-docs-infra/pipeline/transformMarkdownMetadata', {
  titleSuffix: ' · Base UI',
  extractToIndex: {
    include: ['src/app/react'],
    exclude: ['src/app/careers', 'src/app/production-error', 'src/app/test',
              'src/app/experiments', 'src/app/playground'],
    baseDir,
    useVisibleDescription: true,
  },
}]

// types loader
updateParentIndex: { baseDir, onlyUpdateIndexes: true }
```

Docs live under Next.js route-group folders:
`docs/src/app/(docs)/react/(components)/collapsible/page.mdx`, etc.
Sections are the four route groups `(overview)`, `(handbook)`, `(components)`, `(utils)`.

## Expected vs. actual

**Expected** — index grouped by section:

```
# React

## Overview
- Quick start - ([Outline](#quick-start), [Contents](/react/quick-start))
...
## Components
- Accordion - (...)
...
```

**Actual** (clean regenerate: empty the editable region → `rm -rf docs/.next` →
production build) — a single flat list, unordered, `[New]` on every item, route
groups stripped from the paths:

```
# React

[//]: # 'This section is autogenerated, ...'

- Input [New] - ([Outline](#input), [Contents](./input/page.mdx))
- Merge Props [New] - ([Outline](#merge-props), [Contents](./merge-props/page.mdx))
- CSP Provider [New] - ([Outline](#csp-provider), [Contents](./csp-provider/page.mdx))
... (93 items, all [New])
```

`## Overview/Handbook/Components/Utilities` headings appear only in the
detail/outline region lower in the file (which *does* keep `(components)/…` paths) —
never in the main index list.

## Root causes (in docs-infra)

All file references are in the installed package under
`node_modules/@mui/internal-docs-infra/pipeline/`.

### 1. Grouping never triggers for the main index — route groups are stripped from its paths

`syncPageIndex/metadataToMarkdown.mjs` groups only when it can detect a route group
on each page:

```js
// metadataToMarkdown.mjs
export function routeGroupOfPath(path) {
  const firstSegment = path.replace(/^\.\//, '').split('/')[0];
  return firstSegment?.startsWith('(') && firstSegment.endsWith(')') ? firstSegment : undefined;
}
const isGrouped = !!(sections && sections.length > 0);
```

`deriveIndexSections()` (`syncPageIndex/mergeMetadataMarkdown.mjs:9-36`) builds
`sections` from those route groups. But the **main index list** is written with the
route group **stripped** (`./collapsible/page.mdx`, not
`./(components)/collapsible/page.mdx`), so `routeGroupOfPath` returns `undefined`
for every entry, `sections` is empty, `isGrouped` is false, and the output is flat.

The grouping code paths (`## Section` emission in `metadataToMarkdown.mjs:1026-1044`)
are effectively dead for the main index because the paths they receive have already
had the route group removed. (The detail/outline region keeps the `(group)` paths,
which is why headings show up there.)

### 2. `[New]` is applied to every page, with no consumer control

`syncPageIndex/mergeMetadataMarkdown.mjs:148-159`:

```js
// add any new pages that weren't in the existing markdown → automatically add the [New] tag
for (const newPage of newMetadata.pages) {
  if (!addedPaths.has(newPage.path)) {
    pages.push({ ...newPage, tags: newPage.tags ? [...newPage.tags, 'New'] : ['New'] });
    addedPaths.add(newPage.path);
  }
}
```

`[New]` is not read from frontmatter/`tags` and is not date-based — it is appended
whenever a page's `path` is not already present in the existing `page.mdx`. There is
no `newUntil` / threshold / opt-out. From a fresh index (or any path-format change),
**every** page qualifies. Once written it is preserved as user-managed and never
auto-removed.

### 3. Path-format change makes an existing index self-duplicate

Because merging matches pages by exact `path` string, an index generated by an older
version (route-group-stripped paths) does not match the newer route-group paths.
Result: the pipeline treats every page as brand-new, appends a **second** copy of the
list (grouped, all `[New]`) on top of the stale one, and the file ends up with two
`autogenerated` markers and duplicated entries.

### 4. `extractToIndex` and `updateParentIndex` disagree on path format / layout

- `next build` (`extractToIndex` remark) and `pnpm docs:api` (`updateParentIndex`
  types loader) compute paths and layout differently.
- Regenerating from a stub via the **types loader** yields flat per-component `##`
  headings and stripped paths; the **remark** path yields yet another shape.
- Running one after the other compounds the mismatch (see #3).

Note: `metadataToMarkdownAst()` (`metadataToMarkdown.mjs:515-878`) is flat-only and
ignores `sections`; only the string `metadataToMarkdown()` supports grouping. Worth
confirming both entry points route through the grouping-aware renderer.

## No consumer-side fix exists

Full option sets (from the installed `.d.mts`):

- `extractToIndex`: `include`, `exclude`, `markerDir`, `useVisibleDescription`,
  `indexWrapperComponent`, `baseDir`, `onlyUpdateIndexes`, `errorIfOutOfDate`, `cacheDir`.
- `updateParentIndex`: `baseDir`, `onlyUpdateIndexes`, `indexFileName`, `markerDir`,
  `errorIfOutOfDate`, `cacheDir`.

None control grouping, headings, sections, or tags.

## Requested upstream fixes

1. **Preserve the route group on main-index paths** (or otherwise feed
   `deriveIndexSections` the group) so grouping actually triggers and the index
   renders under `## Overview/Handbook/Components/Utilities`. Today the grouping code
   is dead for the main list.
2. **Make `[New]` opt-in / suppressible** (a threshold, a flag, or "only tag pages
   whose own metadata marks them new") instead of blanket-tagging every
   not-yet-present page.
3. **Make `extractToIndex` and `updateParentIndex` produce identical path formats and
   layout** so `build` and `docs:api`/`docs:validate` don't fight each other.

## Reproduction

In a base-ui checkout with the canary installed:

```bash
# reset the index to just its header + marker, then regenerate clean
printf '# React\n\n[//]: # '"'"'This section is autogenerated, ...'"'"'\n' \
  > "docs/src/app/(docs)/react/page.mdx"
rm -rf docs/.next docs/export
pnpm -F docs exec cross-env NODE_ENV=production next build --webpack
# inspect docs/src/app/(docs)/react/page.mdx → flat list, [New] on every item, no ## grouping
```

`pnpm docs:api` (types-loader path) produces a different-but-also-flat result.
