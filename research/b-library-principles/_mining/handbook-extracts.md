## The composition / render-prop model

- "Use the `render` prop to compose a Base UI part with your own React components." — `docs/src/app/(docs)/react/handbook/composition/page.mdx`
- "most triggers render a `<button>` by default" — `docs/src/app/(docs)/react/handbook/composition/page.mdx`
- Contract for custom components passed to `render`: "The custom component must forward the `ref`, and spread all the received props on its underlying DOM node." — `docs/src/app/(docs)/react/handbook/composition/page.mdx`
- "In situations where you need to compose multiple Base UI components with custom React components, `render` props can be nested as deeply as necessary." (canonical example: `Tooltip.Trigger render={<Dialog.Trigger render={<Menu.Trigger render={<MyButton/>}>}/>}`) — `docs/src/app/(docs)/react/handbook/composition/page.mdx`
- "You can also use the `render` prop to override the rendered element of the component." (e.g. `<Menu.Item render={<a href/>}>` to make an item work like a link) — `docs/src/app/(docs)/react/handbook/composition/page.mdx`
- "Each Base UI component renders the most appropriate element by default, and in most cases, rendering a different element is recommended only on a case-by-case basis." — `docs/src/app/(docs)/react/handbook/composition/page.mdx`
- Render *function* form: "If you are working in an extremely performance-sensitive application, you might want to pass a function to the `render` prop instead of a React element." — `docs/src/app/(docs)/react/handbook/composition/page.mdx`
- "Using a function gives you complete control over spreading props and also allows you to render different content based on the component's state." (function signature is `(props, state) => element`) — `docs/src/app/(docs)/react/handbook/composition/page.mdx`
- "When using the function form of the `render` prop, props are not merged automatically. You can use `mergeProps` to combine Base UI's props with your own" — `docs/src/app/(docs)/react/utils/merge-props/page.mdx`
- Intended altitude of `render`: "The `render` prop is primarily designed for composing event handlers and behavioral props. In most cases it should render the same tag as the default element." — `docs/src/app/(docs)/react/utils/use-render/page.mdx`
- Polymorphism caveat: "Using `render` for polymorphism (rendering a different tag) requires more care, as some default props may not be valid on the new element. For example, `type="button"` is only valid on a `<button>`. Since the component can't know what element `render` will produce at render time and before hydration, props like these need an explicit signal. This is why Base UI's Button provides a `nativeButton` prop to control which defaults are applied." — `docs/src/app/(docs)/react/utils/use-render/page.mdx`
- "The `useRender` hook lets you build custom components that provide a `render` prop to override the default rendered element." (i.e. consumers can make *their own* components speak the same composition protocol) — `docs/src/app/(docs)/react/utils/use-render/page.mdx`
- "The callback version of the `render` prop enables more control of how props are spread, and also passes the internal `state` of a component." — `docs/src/app/(docs)/react/utils/use-render/page.mdx`
- Radix migration framing: "Radix UI uses an `asChild` prop, while Base UI uses a `render` prop." `useRender` is presented as the equivalent of Radix's `Slot` for implementing it — `docs/src/app/(docs)/react/utils/use-render/page.mdx`
- Ref merging: `useRender`'s `ref` option holds an array of refs to merge, "letting both parties have access to the underlying DOM element"; in React 19 `React.forwardRef()` is not needed ("the external ref prop is already contained inside `props`"), while React 17/18 require `forwardRef` and adding the forwarded ref to the `ref` array — `docs/src/app/(docs)/react/utils/use-render/page.mdx`

### mergeProps semantics (the prop-merging contract)

- "`mergeProps` helps you combine multiple prop objects (for example, internal props + user props) into a single set of props you can spread onto an element. It behaves like `Object.assign` (rightmost wins) with a few special cases, so common React patterns work as expected." — `docs/src/app/(docs)/react/utils/merge-props/page.mdx`
- For most keys the rightmost value wins; "`ref` is not merged. Only the rightmost ref is kept" — `docs/src/app/(docs)/react/utils/merge-props/page.mdx`
- "`className` values are concatenated right-to-left (rightmost first)" (`mergeProps({className:'a'},{className:'b'})` → `'b a'`) — `docs/src/app/(docs)/react/utils/merge-props/page.mdx`
- "`style` objects are merged, with keys from the rightmost style overwriting earlier ones." — `docs/src/app/(docs)/react/utils/merge-props/page.mdx`
- "Event handlers are merged and executed right-to-left (rightmost first)" — i.e. user handlers (rightmost) run before Base UI's internal ones, which is what makes cancellation possible — `docs/src/app/(docs)/react/utils/merge-props/page.mdx`
- "For React synthetic events, Base UI adds `event.preventBaseUIHandler()`. Calling it prevents Base UI's internal logic from running. This does not call `preventDefault()` or `stopPropagation()`." — `docs/src/app/(docs)/react/utils/merge-props/page.mdx`
- "For non-synthetic events (custom events with primitive/object values), this mechanism isn't available and all handlers always execute." — `docs/src/app/(docs)/react/utils/merge-props/page.mdx`
- Function arguments: "Each argument can be a props object or a function that receives the merged props up to that point (left to right) and returns a props object. … the function's return value completely replaces the accumulated props up to that point. If you want to chain event handlers from the previous props, you must call them manually." — `docs/src/app/(docs)/react/utils/merge-props/page.mdx`
- `mergeProps` accepts up to 5 arguments; `mergePropsN` takes an array and "is slightly less efficient than `mergeProps`, so only use it when you need to merge more than 5 sets of props" — `docs/src/app/(docs)/react/utils/merge-props/page.mdx`
- The `useRender` page's summary of the same utility: "The `mergeProps` function merges two or more sets of React props together. It safely merges three types of props: 1. Event handlers, so that all are invoked 2. `className` strings 3. `style` properties" and "merges objects from left to right, so that subsequent objects' properties in the arguments overwrite previous ones" — `docs/src/app/(docs)/react/utils/use-render/page.mdx`
- `useRender` TypeScript pairing: "`useRender.ComponentProps` for a component's external (public) props. It types the `render` prop and HTML attributes. `useRender.ElementProps` for the element's internal (private) props. It types HTML attributes alone." — `docs/src/app/(docs)/react/utils/use-render/page.mdx`

### Portals & stacking (part of the render model's environment)

- "Base UI uses portals for components that render popups, such as Dialog and Popover." — `docs/src/app/(docs)/react/overview/quick-start/page.mdx`
- Recommended setup: wrap the app in a `.root` element with `isolation: isolate`; "This way, popups always appear above the page contents, and any `z-index` property in your styles won't interfere with them." — `docs/src/app/(docs)/react/overview/quick-start/page.mdx`

## Styling contract

- "Base UI components are unstyled, don't bundle CSS, and are compatible with Tailwind, CSS Modules, CSS-in-JS, or any other styling solution you prefer. You retain total control of your styling layer." — `docs/src/app/(docs)/react/handbook/styling/page.mdx`
- The styling page enumerates exactly four "style hooks": CSS classes, data attributes, CSS variables, and the style prop — `docs/src/app/(docs)/react/handbook/styling/page.mdx`
- "Components that render an HTML element accept a `className` prop to style the element with CSS classes." — `docs/src/app/(docs)/react/handbook/styling/page.mdx`
- `className` **can be a function of state**: "The prop can also be passed a function that takes the component's state as an argument." (e.g. `className={(state) => (state.checked ? 'checked' : 'unchecked')}`) — `docs/src/app/(docs)/react/handbook/styling/page.mdx`
- "Components provide data attributes designed for styling their states." (e.g. Switch's `[data-checked]` / `[data-unchecked]`) — `docs/src/app/(docs)/react/handbook/styling/page.mdx`
- "Components expose CSS variables to aid in styling, often containing dynamic numeric values to be used in sizing or transform calculations." (e.g. Popover Popup's `--available-height` and `--anchor-width`; example: `max-height: var(--available-height)`) — `docs/src/app/(docs)/react/handbook/styling/page.mdx`
- "Check out each component's API reference for a complete list of available data attributes and CSS variables." (i.e. data attributes + CSS variables are documented, per-part, public API) — `docs/src/app/(docs)/react/handbook/styling/page.mdx`
- "Components that render an HTML element accept a `style` prop to style the element with a CSS object." and "The prop also accepts a function that takes the component's state as an argument." — `docs/src/app/(docs)/react/handbook/styling/page.mdx`
- Tailwind stance: "Apply Tailwind CSS classes to each part via the `className` prop." (Tailwind is a first-class, documented path; example uses state-variant classes like `data-popup-open:`, `data-starting-style:`, `data-ending-style:`, `data-highlighted:`, and `origin-(--transform-origin)`) — `docs/src/app/(docs)/react/handbook/styling/page.mdx`
- CSS Modules stance: "Apply custom CSS classes to each part via the `className` prop. Then style those classes in a CSS Modules file." — `docs/src/app/(docs)/react/handbook/styling/page.mdx`
- CSS-in-JS stance: "Wrap each component part and apply styles, then assemble your styled components." (works because parts accept `className`; example uses `@emotion/styled` around `Menu.Trigger` etc.) — `docs/src/app/(docs)/react/handbook/styling/page.mdx`
- Quick-start reiterates neutrality: "There are examples for both Tailwind and CSS Modules below, but since Base UI is unstyled, you can use CSS-in-JS, plain CSS, or any other styling solution you prefer." — `docs/src/app/(docs)/react/overview/quick-start/page.mdx`
- iOS 26+ Safari environment requirement: "Backdrops such as those used by dialogs must use `position: absolute` instead of `position: fixed` to cover the entire visual viewport. For this to work after the page is scrolled … `body { position: relative }`" must be in global styles — `docs/src/app/(docs)/react/overview/quick-start/page.mdx`
- In-repo demo styling rules (internal, but reveal the intended aesthetic contract of docs demos): new public demos should follow the component's `hero` demo styles (CSS Modules and Tailwind variants) and "Do not add custom styling beyond the critical layout styles necessary for new demos"; CSS Modules demos use raw color values from the Tailwind `@theme` block (e.g. `oklch(14.5% 0 0deg)`) instead of `var(--color-neutral-950)`; every `user-select: none` needs `-webkit-user-select: none` for Safari — `AGENTS.md`

## Animation model

- "Base UI components can be animated using CSS transitions, CSS animations, or JavaScript animation libraries. Each component provides a number of data attributes to target its states, as well as a few attributes specifically for animation." — `docs/src/app/(docs)/react/handbook/animation/page.mdx`
- CSS transitions hooks: "`[data-starting-style]` corresponds to the initial style to transition from. `[data-ending-style]` corresponds to the final style to transition to." — `docs/src/app/(docs)/react/handbook/animation/page.mdx`
- "Transitions are recommended over CSS animations, because a transition can be smoothly cancelled midway. For example, if the user closes a popup before it finishes opening, with CSS transitions it will smoothly animate to its closed state without any abrupt changes." — `docs/src/app/(docs)/react/handbook/animation/page.mdx`
- CSS animations hooks: "`[data-open]` corresponds to the style applied when a component becomes visible. `[data-closed]` corresponds to the style applied before a component becomes hidden." (keyframe `scaleIn`/`scaleOut` example) — `docs/src/app/(docs)/react/handbook/animation/page.mdx`
- The canonical transition recipe also uses `transform-origin: var(--transform-origin)` (popup scales from its anchor-relative origin) — `docs/src/app/(docs)/react/handbook/animation/page.mdx`
- "JavaScript animation libraries such as Motion require control of the mounting and unmounting lifecycle of components in order for exit animations to play." — `docs/src/app/(docs)/react/handbook/animation/page.mdx`
- Exit-animation detection: "Base UI relies on `element.getAnimations()` to detect if animations have finished on an element. When using Motion, `opacity` animations are reflected in `element.getAnimations()`, so Base UI automatically waits for the animation finish before unmounting the component. If `opacity` isn't part of your animation (such as in a translating drawer component), you should still animate it using a value close to `1` (such as `opacity: 0.9999`), so that Base UI can detect the animation." — `docs/src/app/(docs)/react/handbook/animation/page.mdx`
- Default mount lifecycle: "Most popup components like Popover, Dialog, Tooltip, and Menu are unmounted from the DOM when they are closed by default." — `docs/src/app/(docs)/react/handbook/animation/page.mdx`
- Motion recipe for unmount-on-close components: control with the `open` prop (so `<AnimatePresence>` can see state), specify `keepMounted` on the `<Portal>` part, and compose `<Popup>` with `motion.div` via the `render` prop — `docs/src/app/(docs)/react/handbook/animation/page.mdx`
- Motion recipe for kept-in-DOM components: "Components that specify `keepMounted` remain rendered in the DOM when they are closed" → use the render-callback form and "Animate the properties based on the `open` state, avoiding `<AnimatePresence>`" — `docs/src/app/(docs)/react/handbook/animation/page.mdx`
- "The Select component is initially unmounted but remains mounted after interaction. To animate it with Motion, a mix of the two previous approaches is needed." — `docs/src/app/(docs)/react/handbook/animation/page.mdx`
- Manual unmounting escape hatch: "For full control, you can manually unmount the component when it's closed once animations have finished using an `actionsRef` passed to the `<Root>`" (calls `actionsRef.current.unmount()` in `onAnimationComplete`) — `docs/src/app/(docs)/react/handbook/animation/page.mdx`

## Forms model

- "Base UI form control components extend the native constraint validation API so you can build forms for collecting user input or providing control over an interface. They also integrate seamlessly with third-party libraries like React Hook Form and TanStack Form." — `docs/src/app/(docs)/react/handbook/forms/page.mdx`
- "Form controls must have an accessible name in order to be recognized by assistive technologies. Use the label strategy below for each control type." — `docs/src/app/(docs)/react/handbook/forms/page.mdx`
- Labeling strategy — input controls: use `<Field.Label>` or a native `<label>` for `Input`, `NumberField`, `OTPField`, `Autocomplete`, `Combobox` (input outside popup), `Checkbox`, `Radio`, `Switch`; `Checkbox`/`Radio`/`Switch` can be implicitly labeled by enclosing them in `<Field.Label>` — `docs/src/app/(docs)/react/handbook/forms/page.mdx`
- Labeling strategy — trigger-based controls: `Combobox` (input inside popup) → `<Combobox.Label>`; `Select` → `<Select.Label>`; `Slider` → `<Slider.Label>` plus "For multi-thumb sliders, also add an `aria-label` on each `<Slider.Thumb>` to distinguish the thumbs." — `docs/src/app/(docs)/react/handbook/forms/page.mdx`
- "If no visible label is rendered, provide `aria-label` on the actual form control." — `docs/src/app/(docs)/react/handbook/forms/page.mdx`
- "`<Field.Description>` automatically assigns an accessible description" — `docs/src/app/(docs)/react/handbook/forms/page.mdx`
- Group labeling: "Compose `<Fieldset>` when a single label applies to multiple controls, such as a range slider with multiple thumbs or a section that combines several inputs. For checkbox and radio groups, keep the group label in `<Fieldset.Legend>` and wrap each option with `<Field.Item>`" — `docs/src/app/(docs)/react/handbook/forms/page.mdx`
- "`<Field.Item>` should enclose each checkbox or radio option so every control has its own label and description" — `docs/src/app/(docs)/react/handbook/forms/page.mdx`
- Fieldset composes over other roots via render (`<Fieldset.Root render={<Slider.Root />}>`, `<Fieldset.Root render={<RadioGroup />}>`) — the render prop is also the forms-composition mechanism — `docs/src/app/(docs)/react/handbook/forms/page.mdx`
- "Pass the `name` prop to `<Field.Root>` to include the wrapped control's value when a parent form is submitted" — `docs/src/app/(docs)/react/handbook/forms/page.mdx`
- Submission: "You can take over form submission using the native `onSubmit`, or custom `onFormSubmit` props"; with `onFormSubmit` "you receive form values as a JavaScript object, with `eventDetails` provided as a second argument. Additionally, `preventDefault()` is automatically called on the native submit event" — `docs/src/app/(docs)/react/handbook/forms/page.mdx`
- Constraint validation: "Base UI form components support native HTML validation attributes for many validation rules" — `required`, `minLength`/`maxLength`, `pattern`, `step` — `docs/src/app/(docs)/react/handbook/forms/page.mdx`
- Native form association: "Base UI form components use a hidden input to participate in native form submission and validation. To anchor the hidden input near a control so the native validation bubble points to the correct area, ensure the component has been given a `name`, and wrap controls in a relatively positioned container for best results." — `docs/src/app/(docs)/react/handbook/forms/page.mdx`
- Custom validation: "You can add custom validation logic by passing a synchronous or asynchronous validation function to the `validate` prop, which runs after native validations have passed." Return an error message string (or array) when invalid, "return `null` when valid" — `docs/src/app/(docs)/react/handbook/forms/page.mdx`
- Validation modes: "`onSubmit` (default) validates all fields when the containing `<Form>` is submitted, afterwards invalid fields revalidate when their value changes. `onBlur` validates the field when focus moves away. `onChange` validates the field when the value changes" — `docs/src/app/(docs)/react/handbook/forms/page.mdx`
- "`validationDebounceTime` can be used to debounce the function in use cases such as asynchronous requests or text fields that validate `onChange`." — `docs/src/app/(docs)/react/handbook/forms/page.mdx`
- Server errors: "You can pass errors returned by (post-submission) server-side validation to the `errors` prop, which will be merged into the client-side field state for display. This should be an object with field names as keys, and an error string or array of strings as the value. Once a field's value changes, any corresponding error in `errors` will be cleared from the field state." — `docs/src/app/(docs)/react/handbook/forms/page.mdx`
- React Server Functions: "When using Server Functions with Form Actions you can return server-side errors from `useActionState` to the `errors` prop." (`<Form action={formAction} errors={state.errors}>`) — `docs/src/app/(docs)/react/handbook/forms/page.mdx`
- Displaying errors: "Use `<Field.Error>` without `children` to automatically display the field's native error message when invalid. The `match` prop can be used to customize the message based on the validity state, and manage internationalization from your application logic" (e.g. `match="valueMissing"`) — `docs/src/app/(docs)/react/handbook/forms/page.mdx`
- React Hook Form integration: use `<Controller>`, forwarding `name`, `field` (`ref`, `value`, `onBlur`, `onChange`→`onValueChange`) and `fieldState` (`invalid`→`invalid`, `isTouched`→`touched`, `isDirty`→`dirty`) onto `<Field.Root>`/`<Field.Control>`; `<Field.Error match={!!error}>` delegates error rendering — `docs/src/app/(docs)/react/handbook/forms/page.mdx`
- "For React Hook Form to focus invalid fields when performing validation, you must ensure that any wrapping components forward the `ref` to the underlying Base UI component. You can typically accomplish this using the `inputRef` prop, or directly as the `ref` for components that render an input element like `<NumberField.Input>`." — `docs/src/app/(docs)/react/handbook/forms/page.mdx`
- TanStack Form integration mirrors the same mapping via `<form.Field>` children render props; "The Base UI `<Form>` component is not needed when using TanStack Form." — `docs/src/app/(docs)/react/handbook/forms/page.mdx`
- (Cross-reference) "Base UI provides components like Form, Input, Field, Fieldset to automatically associate form controls." — `docs/src/app/(docs)/react/overview/accessibility/page.mdx`

## Events, controlled state & behavior customization

(From the Customization handbook page — the behavioral counterpart to the styling contract.)

- "Change events such as `onOpenChange`, `onValueChange`, and `onPressedChange` are custom to Base UI. They can be emitted by various different DOM events, effects, or even during rendering." — `docs/src/app/(docs)/react/handbook/customization/page.mdx`
- Signature convention: every change handler is `(value, eventDetails) => void` — "The `eventDetails` property is passed as a second argument to Base UI event handlers. This enables the event to be customized, and also allows you to conditionally run side effects based on the reason or DOM event that caused the change." — `docs/src/app/(docs)/react/handbook/customization/page.mdx`
- `eventDetails` shape: `{ reason: string; event: Event; cancel(): void; allowPropagation(): void; isCanceled: boolean; isPropagationAllowed: boolean }` — `docs/src/app/(docs)/react/handbook/customization/page.mdx`
- "`reason` is used to determine why the change event occurred, which can be useful to conditionally run certain side effects. Most IDEs show the possible string values after typing `reason === '`." — `docs/src/app/(docs)/react/handbook/customization/page.mdx`
- "`cancel` stops the component from changing its internal state." Canceling "lets you leave the component uncontrolled as its internal state is prevented from updating. This is an alternative to controlling the component with external state and guarding the state updates conditionally." (example: cancel `trigger-press` to keep a tooltip open) — `docs/src/app/(docs)/react/handbook/customization/page.mdx`
- "In most components, pressing the Esc key stops the propagation of the event so parent popups don't close simultaneously. This can also be customized with the `allowPropagation()` method" — `docs/src/app/(docs)/react/handbook/customization/page.mdx`
- "To prevent Base UI from handling a React event like `onClick`, you can use the `preventBaseUIHandler()` method on the event object. … This should be used as an escape hatch in cases where there isn't a prop yet to customize the behavior. In various cases, native events are used instead of React events, so this method has no effect." — `docs/src/app/(docs)/react/handbook/customization/page.mdx`
- "Components are uncontrolled by default, meaning that they manage their own state internally." — `docs/src/app/(docs)/react/handbook/customization/page.mdx`
- "A component can be made controlled by passing external state to a prop, such as `open` or `value`, and the state's setter to its corresponding change handler, such as `onOpenChange` or `onValueChange`." Controlling "also allows you to read the state of the component outside of the root component." — `docs/src/app/(docs)/react/handbook/customization/page.mdx`
- Reason strings follow a stable vocabulary refined at beta.0: "`hover` is now `trigger-hover`; `click` is now `trigger-press`; `focus` is now `trigger-focus`" — `docs/src/app/(docs)/react/overview/releases/v1-0-0-beta-0/page.mdx`

## Security / CSP model (environment contract)

- "Some Base UI components render inline `<style>` or `<script>` tags for functionality such as removing scrollbars or pre-hydration behavior. Under a strict Content Security Policy (CSP), these tags may be blocked unless they include a matching nonce attribute." — `docs/src/app/(docs)/react/utils/csp-provider/page.mdx`
- "`CSPProvider` allows configuring this behavior globally for all Base UI components within its tree." (generate a nonce per request → CSP header via `style-src-elem`/`script-src` → pass same nonce to `CSPProvider`) — `docs/src/app/(docs)/react/utils/csp-provider/page.mdx`
- "You can avoid supplying a `nonce` if you disable inline `<style>` elements entirely and rely on external stylesheets only. The relevant components are `<ScrollArea.Viewport>` and `<Select.Popup>` or `<Select.List>` when `alignItemWithTrigger` is enabled, which inject a style tag to disable native scrollbars." (`disableStyleElements`) — `docs/src/app/(docs)/react/utils/csp-provider/page.mdx`
- "`<script>` tags across all components are opt-in, so they are not affected by this prop and don't have their own disable flag. A `nonce` is required if any component uses inline scripts." — `docs/src/app/(docs)/react/utils/csp-provider/page.mdx`
- Inline style *attributes* are outside CSPProvider's scope; options: relax `style-src-attr`, "Render the affected components only on the client, so that no inline styles are present in the initial HTML", or manually unset inline styles ("Any component can have its inline styles unset, such as `<ScrollArea.Viewport style={{ overflow: undefined }}>`. Note that you'll need to ensure you vet upgrades for any new inline styles added by Base UI components.") — `docs/src/app/(docs)/react/utils/csp-provider/page.mdx`

## SSR / hydration stance

- No dedicated SSR/hydration page exists in overview, handbook, or utils. The stance must be assembled from scattered statements:
- Pre-hydration behavior is a first-class concern: inline `<script>` tags exist "for functionality such as … pre-hydration behavior" — `docs/src/app/(docs)/react/utils/csp-provider/page.mdx`
- Element identity must be knowable "at render time and before hydration": "Since the component can't know what element `render` will produce at render time and before hydration, props like these need an explicit signal" (the `nativeButton` rationale) — `docs/src/app/(docs)/react/utils/use-render/page.mdx`
- Server pre-rendered HTML may contain inline style attributes (CSP `style-src-attr` discussion assumes SSR output) — `docs/src/app/(docs)/react/utils/csp-provider/page.mdx`
- Release-note evidence of active SSR/hydration hardening: "Add `suppressHydrationWarning` to hidden inputs" and "Skip client-only prehydration scripts" — `docs/src/app/(docs)/react/overview/releases/v1-4-0/page.mdx`; "Fix autofocus in SSR environments" — `docs/src/app/(docs)/react/overview/releases/v1-2-0/page.mdx`; "Fix ARIA attributes during SSR" — `docs/src/app/(docs)/react/overview/releases/v1-3-0/page.mdx`
- Select's `placeholder` became required partly for SSR: "Previously, only SSR needed it to prevent a hydration flash, but client-side rendering now also requires it." — `docs/src/app/(docs)/react/overview/releases/v1-0-0-beta-0/page.mdx`
- ID generation moved to React's mechanism: "Use `useId` instead of composite index as fallback value." (breaking change) — `docs/src/app/(docs)/react/overview/releases/v1-0-0-beta-4/page.mdx`
- Internal convention (not public docs, but the enforced house rule): always use `useIsoLayoutEffect` instead of `React.useLayoutEffect` (SSR-safe isomorphic layout effect) — `AGENTS.md`

## TypeScript conventions users should know

- "Base UI uses namespaces to organize types. Every component has two core interfaces: `Props` (such as `Tooltip.Root.Props`) [and] `State` (such as `Tooltip.Root.State`)" — `docs/src/app/(docs)/react/handbook/typescript/page.mdx`
- "When creating wrapping components, you can use the `Props` type to accept all of the underlying Base UI props for the component." (`function MyTooltip(props: Tooltip.Root.Props)`) — `docs/src/app/(docs)/react/handbook/typescript/page.mdx`
- "The `State` type is the internal state of the component. For example, `Positioner` components (such as `<Popover.Positioner>`) have state that describes the position of the element in relation to their anchor." (fields shown: `open`, `side`, `align`, `anchorHidden`) — `docs/src/app/(docs)/react/handbook/typescript/page.mdx`
- Event types: "`ChangeEventDetails` (such as `Combobox.Root.ChangeEventDetails`) is the object passed to change handlers like `onValueChange` and `onOpenChange`. `ChangeEventReason` (such as `Combobox.Root.ChangeEventReason`) is the union of possible reason strings for a change event." — `docs/src/app/(docs)/react/handbook/typescript/page.mdx`
- "Popups have an `actionsRef` prop to access imperative methods. For example, `Menu.Root.Actions` gives access to the shape of the `actionsRef` object prop on `<Menu.Root>`." — `docs/src/app/(docs)/react/handbook/typescript/page.mdx`
- "The `toast` object on `<Toast.Root>` is a complex object with many properties. `Toast.Root.ToastObject` gives access to this interface." — `docs/src/app/(docs)/react/handbook/typescript/page.mdx`
- "Components that have a `render` prop have an extended `React.ComponentProps` type, enhanced with a `render` prop. The `useRender.ComponentProps` type on the function gives access to this interface." — `docs/src/app/(docs)/react/handbook/typescript/page.mdx`
- For custom components: `useRender.ComponentProps<'tag'>` = public props (render + HTML attributes); `useRender.ElementProps<'tag'>` = private element props (HTML attributes only) — `docs/src/app/(docs)/react/utils/use-render/page.mdx`
- Generic handles: "Provide a type argument to `createHandle()` to strongly type the payload." — `docs/src/app/(docs)/react/components/menu/page.mdx`
- Internal convention behind the namespaces (docgen requirement): namespaced types must also be exported at module top level (`export type AlertDialogPopupProps = …; export declare namespace AlertDialogPopup { type Props = AlertDialogPopupProps }`) — `CONTRIBUTING.md`

## Internal engineering & docs-authoring conventions (AGENTS.md / CONTRIBUTING.md)

These are contributor-facing, but they encode principles the library holds itself to:

- Timing/effects discipline: always `useTimeout` (`@base-ui/utils/useTimeout`) over `window.setTimeout`, `useAnimationFrame` over `requestAnimationFrame`, `useStableCallback` over `React.useCallback` for handler/effect-called functions, `useIsoLayoutEffect` over `React.useLayoutEffect` — `AGENTS.md`
- Shadow DOM / multi-realm correctness is a standing requirement: "Always use the shadow DOM-safe utilities for DOM traversal and event targeting: `contains`, `getTarget`, and `activeElement`. Always use the owner utilities `ownerDocument` and `ownerWindow` instead of global `document`/`window` lookups when the code is tied to a DOM node, including realm-sensitive checks such as `instanceof`." — `AGENTS.md`
- Shared behavior lives in parents: "If two components can share logic (such as event handlers), define the logic/handlers in the parent and share it through a context to the child" — `AGENTS.md`
- Error-message contract for public packages: every error must "Say what happened", "Say why it's a problem", "Point toward how to solve it"; prefixed `Base UI:`; include a docs link when applicable; error codes are minified via `pnpm extract-error-codes` into `docs/src/error-codes.json` — `AGENTS.md`
- Test conventions: Vitest only; jsdom by default, Chromium (`it.skipIf(isJSDOM)`) when layout measurement is needed; one assertion per `waitFor()`; tests colocated as `Name.test.tsx` next to source — `AGENTS.md`
- Commit format: "`[scope] Imperative summary`" with scopes mirroring package/component names; "`[all components]`" for broad changes — `AGENTS.md`
- Docs are partially generated: `types.md` files and page-index `page.mdx` outlines are autogenerated (`pnpm docs:*`); page title comes from the `# h1`, description from `<Subtitle>`, `metadata` export sits at the end of each page — `CONTRIBUTING.md`
- Demos are built with `createDemo` / `createDemoWithVariants` and typically ship in two styling variants: CSS Modules and Tailwind — `CONTRIBUTING.md`
- Multi-part vs single-part components is a first-class docs distinction (`createTypes` vs `createMultipleTypes`) — `CONTRIBUTING.md`

## Glossary seeds

Term → definition sketch → source(s).

- **part** → One named subcomponent of a multi-part component (e.g. `Menu.Trigger`, `Switch.Thumb`, `Popover.Popup`); styling and composition are applied "to each part via the `className` prop"; docs distinguish single-part vs multi-part components. → `docs/src/app/(docs)/react/handbook/styling/page.mdx`, `CONTRIBUTING.md`
- **root** → The stateful coordinating part of a component (`Dialog.Root`, `Field.Root`, `Menu.Root`); owns controlled/uncontrolled state (`open`/`onOpenChange`, `value`/`onValueChange`), `actionsRef`, and context for descendant parts; "read the state of the component outside of the root component" implies root = state boundary. → `docs/src/app/(docs)/react/handbook/customization/page.mdx`, `docs/src/app/(docs)/react/handbook/animation/page.mdx`
- **trigger** → The interactive part that opens/controls a popup; "most triggers render a `<button>` by default"; since beta.5 triggers can be *detached* (live anywhere in the tree, connected via a handle) and popovers/menus support multiple triggers; hover/delay props live on the trigger, not the root. → `docs/src/app/(docs)/react/handbook/composition/page.mdx`, `docs/src/app/(docs)/react/overview/releases/v1-0-0-beta-5/page.mdx`
- **portal** → Part that renders the popup subtree elsewhere in the DOM ("Base UI uses portals for components that render popups, such as Dialog and Popover"); carries the `keepMounted` prop. → `docs/src/app/(docs)/react/overview/quick-start/page.mdx`, `docs/src/app/(docs)/react/handbook/animation/page.mdx`
- **positioner** → Part that computes and applies a popup's position relative to its anchor; its `State` exposes `side`, `align`, `anchorHidden`; accepts props like `sideOffset`. → `docs/src/app/(docs)/react/handbook/typescript/page.mdx`, `docs/src/app/(docs)/react/handbook/styling/page.mdx`
- **popup** → The floating element itself (`Menu.Popup`, `Popover.Popup`); target of animation attributes (`data-starting-style`, `data-ending-style`) and CSS variables (`--available-height`, `--anchor-width`, `--transform-origin`); "Most popup components … are unmounted from the DOM when they are closed by default." → `docs/src/app/(docs)/react/handbook/animation/page.mdx`, `docs/src/app/(docs)/react/handbook/styling/page.mdx`
- **backdrop** → Overlay layer behind modal popups (dialogs); on iOS 26+ Safari "Backdrops such as those used by dialogs must use `position: absolute` instead of `position: fixed`"; clicking it is a close reason. → `docs/src/app/(docs)/react/overview/quick-start/page.mdx`, `docs/src/app/(docs)/react/components/dialog/page.mdx`
- **arrow** → Optional pointer part of anchored popups (`Popover.Arrow`, `Tooltip.Arrow`) rendered inside the popup and positioned against the anchor. → `docs/src/app/(docs)/react/components/popover/page.mdx` (anatomy), `docs/src/app/(docs)/react/components/tooltip/page.mdx`
- **viewport** → (1) `Dialog.Viewport` / `AlertDialog.Viewport`: outer scrollable/positioning container for scrollable dialogs; (2) `ScrollArea.Viewport`: the scroll container that injects the scrollbar-hiding style tag. → `docs/src/app/(docs)/react/components/dialog/page.mdx` (lines ~165–173), `docs/src/app/(docs)/react/overview/releases/v1-0-0-beta-5/page.mdx`, `docs/src/app/(docs)/react/utils/csp-provider/page.mdx`
- **anchor** → The element a popup is positioned "in relation to"; surfaces in `--anchor-width` CSS variable and `anchorHidden` positioner state; anchor tracking is configurable (`disableAnchorTracking`, formerly `trackAnchor`). → `docs/src/app/(docs)/react/handbook/typescript/page.mdx`, `docs/src/app/(docs)/react/handbook/styling/page.mdx`, `docs/src/app/(docs)/react/overview/releases/v1-0-0-beta-5/page.mdx`
- **handle** → Object created with `Menu.createHandle()` (and equivalents) that connects a detached trigger to a root rendered elsewhere: "When the trigger and menu content need to live in different parts of the tree … create a `handle` with `Menu.createHandle()` and pass it to both the trigger and the root." → `docs/src/app/(docs)/react/components/menu/page.mdx` (line ~197)
- **payload** → Per-trigger data delivered to a shared popup: "Pass a `payload` prop to each `<Menu.Trigger>` and read it via a function child on `<Menu.Root>`. Provide a type argument to `createHandle()` to strongly type the payload." → `docs/src/app/(docs)/react/components/menu/page.mdx` (lines ~257–258)
- **detached triggers** → Triggers separated from their root's subtree (multiple allowed per popup); introduced as a breaking change moving `openOnHover`/`delay`/`closeDelay` from Root to Trigger. → `docs/src/app/(docs)/react/overview/releases/v1-0-0-beta-5/page.mdx`
- **starting style / ending style** → `[data-starting-style]` = "the initial style to transition from"; `[data-ending-style]` = "the final style to transition to"; the CSS-transition enter/exit hooks. → `docs/src/app/(docs)/react/handbook/animation/page.mdx`
- **open / closed (data attributes)** → `[data-open]` = "style applied when a component becomes visible"; `[data-closed]` = "style applied before a component becomes hidden"; the CSS-keyframe-animation hooks. → `docs/src/app/(docs)/react/handbook/animation/page.mdx`
- **transition status** → Internal mount/open lifecycle that the animation data attributes are mapped from (term used in release notes: "Fix transition status mapping"). → `docs/src/app/(docs)/react/overview/releases/v1-0-0-beta-1/page.mdx`, semantics in `docs/src/app/(docs)/react/handbook/animation/page.mdx`
- **keepMounted** → Portal-level prop keeping a closed popup in the DOM ("Components that specify `keepMounted` remain rendered in the DOM when they are closed"); prerequisite for JS exit animations. → `docs/src/app/(docs)/react/handbook/animation/page.mdx`
- **actionsRef** → Ref to a root's imperative actions object (e.g. `unmount()`); typed as `Menu.Root.Actions` etc.; used for manual unmounting after JS animations. → `docs/src/app/(docs)/react/handbook/animation/page.mdx`, `docs/src/app/(docs)/react/handbook/typescript/page.mdx`
- **render prop** → Base UI's composition primitive (element form or `(props, state) => element` function form) replacing Radix's `asChild`; "primarily designed for composing event handlers and behavioral props." → `docs/src/app/(docs)/react/handbook/composition/page.mdx`, `docs/src/app/(docs)/react/utils/use-render/page.mdx`
- **state (component state)** → The internal state object exposed to `className`/`style`/`render` functions and typed as `Part.State` (e.g. `{ checked }`, `{ open, side, align, anchorHidden }`). → `docs/src/app/(docs)/react/handbook/styling/page.mdx`, `docs/src/app/(docs)/react/handbook/typescript/page.mdx`
- **data attributes** → Per-part attributes "designed for styling their states" (`[data-checked]`, `[data-highlighted]`, `[data-popup-open]`…); enumerated in each component's API reference. → `docs/src/app/(docs)/react/handbook/styling/page.mdx`
- **CSS variables** → Per-part custom properties exposing "dynamic numeric values" (`--available-height`, `--anchor-width`, `--transform-origin`); enumerated per component API reference. → `docs/src/app/(docs)/react/handbook/styling/page.mdx`
- **eventDetails** → Second argument to every Base UI change handler: `{ reason, event, cancel(), allowPropagation(), isCanceled, isPropagationAllowed }`. → `docs/src/app/(docs)/react/handbook/customization/page.mdx`
- **reason** → String on `eventDetails` naming why a change occurred (`trigger-press`, `trigger-hover`, `escape-key`…); typed as `ChangeEventReason`. → `docs/src/app/(docs)/react/handbook/customization/page.mdx`, `docs/src/app/(docs)/react/handbook/typescript/page.mdx`

- **typeahead** → Matching list items by typed characters in list-driven popups (Select, Menu); covered by "alphanumeric keys" keyboard support; behavior refined across releases (skip disabled items, reset on external blur). → `docs/src/app/(docs)/react/overview/accessibility/page.mdx`, `docs/src/app/(docs)/react/overview/releases/v1-6-0/page.mdx`, `docs/src/app/(docs)/react/overview/releases/v1-2-0/page.mdx`

- **validationMode** → When field validation runs: `onSubmit` (default), `onBlur`, `onChange`; paired with `validationDebounceTime`. → `docs/src/app/(docs)/react/handbook/forms/page.mdx`
