# Disclosure cluster — Accordion vs Collapsible vs Tabs

Self-contained decision guidance for the "disclosure & structure" overlap named in `research/a-documentation-standard/taxonomy.md`. Legend: [E] direct evidence + citation, [I] inference (basis stated). Docs paths relative to repo root; issue/PR numbers are mui/base-ui.

**Scope note on how this file was built**: accordion and collapsible briefs are being written by a parallel agent in this session; this cluster note is deliberately self-contained from the three components' own docs pages so it doesn't depend on their brief files existing yet. For Accordion and Collapsible, only `docs/src/app/(docs)/react/components/{accordion,collapsible}/page.mdx` were read (Subtitle, Anatomy, Examples headings) — no source-code dive, per this session's budget. Tabs' side of the comparison draws on this session's full source/test/docs research (see [`../tabs/brief.md`](../tabs/brief.md)).

**Taxonomy placement, stated up front**: `taxonomy.md` places `accordion` and `collapsible` under category 4, "Disclosure & structure," but places `tabs` under category 3, "Navigation & app chrome" — with an explicit flag: "`tabs` is disclosure-adjacent (shows/hides panels — category 4) but its APG pattern and usage archetype are navigational ('switch views'); placed here, cross-linked from Disclosure" [E] (taxonomy.md, category 3 table note). This file *is* that cross-link: Tabs sits formally in Navigation, but its show/hide-panel mechanics are close enough to Accordion/Collapsible that a reader choosing between them needs a single place to compare all three, which is exactly what taxonomy.md anticipated when it flagged the ambiguity rather than silently filing Tabs away from its two nearest behavioral neighbors.

- **Accordion** — "A set of collapsible panels with headings." [E] `docs/src/app/(docs)/react/components/accordion/page.mdx`

## The three-way distinction

1. **One at a time (peer, mutually exclusive) vs independent vs coordinated multi.**
   - **Tabs**: exactly one panel is the active selection among peers at (almost) all times — `Tabs.Root` holds a single `value`, and every `Tabs.Panel` is matched against that one value; there is no "all panels open" or "all panels closed" steady state built into the model (the closest exception is the disabled/missing-selection fallback producing `value: null`, which is an edge case, not a designed multi-open mode). [E] `../tabs/brief.md` §6 (Root's single-`value` model, verified against `TabsRoot.tsx`/`TabsPanel.tsx` directly).
   - **Collapsible**: a single region, on its own, with no siblings to coordinate against — its own two-part anatomy (`Root > (Trigger, Panel)`) has no `List`/composite-of-siblings part at all. [E] Collapsible Anatomy, `page.mdx`: `<Collapsible.Root><Collapsible.Trigger /><Collapsible.Panel /></Collapsible.Root>` — nothing analogous to Tabs' `List` or Accordion's `Item` exists in this tree.
   - **Accordion**: multiple Collapsible-shaped regions (`Accordion.Item`, each wrapping its own `Header > Trigger` and `Panel`) coordinated by a shared `Accordion.Root`, with an explicit `multiple` prop controlling whether more than one item can be open simultaneously — "You can set up the accordion to allow multiple panels to be open at the same time using the `multiple` prop" [E] Accordion Examples "Open multiple panels", `page.mdx`. `research/b-library-principles/principles.md` (B-P40) additionally notes this is the library-wide "one `multiple` prop across multi-active components" convention, and that "accordion's default flipped false in #3141" [E] — i.e. Accordion defaults to single-open-at-a-time (closer to Tabs' one-selected model) unless `multiple` is explicitly turned on, rather than defaulting to "anything can be open."
2. **Content relationship: peer views of one subject vs freestanding regions vs a titled list of regions.**
   - Tabs' panels are framed by the library's own words as views onto the *same page* ("toggling between related panels on the same page") — they're read as alternates of one another, like views of a single object (an Overview/Projects/Account trio for one workspace, in Tabs' own hero demo). [E]/[I] inferred from subtitle wording plus the hero demo's own content (`docs/src/app/(docs)/react/components/tabs/demos/hero`).
   - Collapsible's one panel has no "sibling views" framing at all — it's a single freestanding expandable region (an FAQ answer, a "show more" block). [I] inferred from the two-part anatomy having nothing to be a peer of.
   - Accordion's panels are explicitly headed ("a set of collapsible panels **with headings**" [E] subtitle) — each Item is framed as its own titled unit within a list, closer to "a list of FAQ questions" than to "views of one workspace."
3. **Visibility model: exactly one visible vs one region's own state vs N independently toggled.**
   - Tabs: exactly one Panel is the "open" one at a time by construction (single `value` on Root) — you cannot land in the "everything visible" or (outside the disabled-fallback edge case) "nothing visible" state by ordinary use. [E] `../tabs/brief.md` §6.
   - Collapsible: one boolean per instance — open or closed, nothing to compare it against.
   - Accordion (default, `multiple=false`): behaves like Tabs' one-active-among-peers model but for expand/collapse rather than tab selection — opening one Item can close the previously open one. Accordion (`multiple=true`): any subset of Items, including all or none, can be open simultaneously — genuinely independent per-Item state, unlike Tabs which never allows zero-or-many. [E]/[I] — the single/multiple behavioral split itself is [E] (Accordion's own docs example + B-P40); that it converges with Tabs' model specifically when `multiple=false` is this file's own [I] synthesis, not a maintainer statement comparing the two directly.
4. **Navigational framing vs pure disclosure.** Tabs' own APG pattern (Tabs, https://www.w3.org/WAI/ARIA/apg/patterns/tabs/) and ARIA roles (`tablist`/`tab`/`tabpanel`) frame the interaction as *navigating between views*, not merely *disclosing* content — this is precisely why taxonomy.md filed it under Navigation despite the shared show/hide mechanics. [E] `../tabs/brief.md` §7 (role wiring verified against source). Collapsible and Accordion use disclosure-shaped semantics instead (a trigger button whose pressed/expanded state reveals or hides a panel, with no `tablist`-style composite roving-focus contract layered on top) — [I] inferred from Collapsible/Accordion's Anatomy trees showing `Trigger`/`Header` rather than a `List`+`Tab` composite, without a source-level a11y read of either component in this session.
5. **Keyboard model.** Tabs is a roving-tabindex composite: one Tab stop for the whole `tablist`, arrow keys move focus among tabs (B-P29, "single tab stop, roving highlight" — Tabs is one of the composites this principle names). [E] `../tabs/brief.md` §7. [G] This file did not independently verify whether Accordion's Triggers form a similar single-tab-stop composite or are each their own Tab stop (would require Accordion source, out of this session's budget for that component) — a future Accordion brief should confirm this and this file should be updated to link it directly rather than leave it open.

## Decision table

| | Collapsible | Accordion (default) | Accordion (`multiple`) | Tabs |
|---|---|---|---|---|
| Number of regions | 1 | N (Items) | N (Items) | N (peer panels) |
| Coordination | none (standalone) | Root coordinates Items | Root coordinates Items | Root coordinates panels via single `value` |
| Open states possible | open / closed | exactly one Item open (or none) | any subset (0..N) open | exactly one panel selected (edge case: none, on disabled/missing fallback) |
| Content framing | one freestanding region | a headed list of regions | a headed list of regions | peer views of one subject |

## Per-pair boundaries

**Collapsible vs Accordion** — cardinality. A single expandable region with nothing to coordinate against is Collapsible; the moment there are multiple such regions that should behave as a set (only one open at a time by default, or an explicit toggle for "several at once"), reach for Accordion instead of hand-rolling multiple independent Collapsibles — [I] inferred directly from Accordion's `Item`-wrapping-`Collapsible`-shaped anatomy plus its subtitle explicitly describing itself as "a set of collapsible panels," i.e. Accordion's own docs frame it as the multi-instance answer to Collapsible's single-instance shape. [G] No maintainer statement directly telling users "use Accordion instead of N Collapsibles" was searched for in this session (out of budget for Accordion/Collapsible issue-mining); this boundary is inferred from API shape alone.

**Accordion vs Tabs** — the two "one active among several" models. With `multiple=false` (Accordion's default), Accordion and Tabs both resolve to "exactly one region visible among peers" as their steady state, which is the closest these three components get to converging. The distinguishing question is the interaction/ARIA contract, not the visibility math: Tabs is a `tablist` — a roving-focus composite whose whole point is fast switching between views via a persistent row/column of triggers, and the previously-active panel disappears from the accessibility tree entirely when not `keepMounted`. Accordion's Items keep their trigger+header framing even when collapsed (each Item's `Header`/`Trigger` stays visible and in normal tab order — [I], inferred from the Anatomy tree showing `Header > Trigger` per `Item`, not a single shared composite List), so an Accordion reads as a stack of independently-labeled sections you're paging through top-to-bottom, while Tabs reads as a fixed set of alternate views selected from a compact control. [I] Use Accordion when each region has meaningfully separate content that benefits from its own persistent heading in a vertical stack (FAQ-style); use Tabs when the regions are true peers of one subject switched via a compact horizontal/vertical control that itself disappears from flow. [G] No maintainer statement comparing Accordion and Tabs directly was found — searched issues/PRs mentally against terms already covered in this session's Tabs mining ("accordion vs tabs", "tabs vs accordion") with no hits recalled from the Tabs-side issue trail; this boundary is [I] only.

**Collapsible vs Tabs** — the least likely confusion of the three, but the clearest cardinality boundary. A single disclosed region has no `value`/selection concept and no peer to compare itself against; the instant a second, mutually-exclusive alternative view enters the picture (even just two), the model becomes Tabs' territory (or Accordion's, per the pair above, if the "peer views of one subject" framing doesn't fit and a headed vertical list does). [I] inferred directly from Collapsible's anatomy having no analog to Tabs' `List`/multi-panel `value` matching at all — there is no API surface on Collapsible that could express "which one of several is currently shown," which is the entire reason Tabs' Root exists.
