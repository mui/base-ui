# Cluster: binary controls — toggle (+ toggle group) vs switch vs checkbox

Decision guidance shared by the three components that all render a two-state on/off control. Per-component briefs link here instead of repeating it. Evidence: `Toggle.tsx`, `ToggleGroup.tsx`, `packages/react/src/switch/root/SwitchRoot.tsx`, `packages/react/src/checkbox/root/CheckboxRoot.tsx`, `research/b-library-principles/_mining/discussions.md` #3525, `research/b-library-principles/principles.md` B-P30/B-P31.

## Purpose per component

- **Toggle** (`@base-ui/react/toggle`) — "A two-state button that can be on or off" [E docs]. Implements the ARIA **toggle button** pattern: `aria-pressed` on a native-semantics `<button>`. No form participation by design — `Toggle.tsx` destructures and discards `form`/`type` with inline comments ("never participates in form validation," "cannot change button type") [E].
- **Toggle Group** (`@base-ui/react/toggle-group`) — "Provides a shared state to a series of toggle buttons" [E docs]. A context/composite-roving-focus coordinator over plain `Toggle` children, not a distinct ARIA widget of its own; single- or multi-select via one `multiple` prop, array-valued in both modes [E `ToggleGroup.tsx`].
- **Switch** (`@base-ui/react/switch`) — `role="switch"` on its root [E `SwitchRoot.tsx` L128]; ships a hidden `<input type="checkbox">` for native form participation, with an optional `uncheckedValue` for the unchecked-state submission value [E `SwitchRoot.tsx` L246-247].
- **Checkbox** (`@base-ui/react/checkbox`) — `role="checkbox"` on its root [E `CheckboxRoot.tsx` L313]; also ships a hidden `<input type="checkbox">` plus `uncheckedValue` [E `CheckboxRoot.tsx` L415-416] — nearly identical form-participation shape to Switch, differentiated primarily by ARIA role and (per `research/b-library-principles/principles.md` B-P33) the fact that a group of Checkboxes is a *multi-select* pattern where a Switch is always singular.

## Decision dimensions

1. **ARIA semantics** — toggle-button `aria-pressed` (Toggle/ToggleGroup) vs `role="switch"` (Switch, an on/off physical-switch metaphor, always announced as boolean state) vs `role="checkbox"` (Checkbox, a selection/agreement metaphor, supports `aria-checked="mixed"` for indeterminate — Toggle/Switch have no indeterminate state in source).
2. **Form participation** — Toggle/ToggleGroup: **none** — no hidden input anywhere in either component's source [E, confirmed by direct source inspection, no `<input type="hidden">` in `Toggle.tsx`/`ToggleGroup.tsx`]. Switch and Checkbox: **both** ship a hidden `<input type="checkbox">` that participates in native form submission and constraint validation, with an identical `uncheckedValue` escape hatch for what gets submitted when unchecked [E, source line citations above]. This is the single sharpest, most source-verifiable line in this cluster: **if the value must submit with a `<form>`, Toggle/ToggleGroup are the wrong choice by construction, not by convention** — the capability simply doesn't exist in their source.
3. **Immediate-effect vs submit-time semantics** — [I] Toggle/ToggleGroup read as UI-state controls whose effect is immediate and local (bold/italic in an editor, a view-mode switcher) since they have no submission story at all; Switch/Checkbox read as form-field values whose effect is typically deferred to form submission (though both also support `onCheckedChange` for immediate-effect use, e.g. a settings toggle that saves instantly) — the distinguishing signal is less "when does it take effect" (both families support immediate `onChange`-style callbacks) and more "does this value need to travel with a `<form>` POST/submit," which only Switch/Checkbox can do natively.
4. **Cardinality** — Toggle stands alone or as one item in a ToggleGroup (array-valued group state, single or multiple selection via one `multiple` prop, per `research/b-library-principles/principles.md` B-P40's "one `multiple` prop" convention); Switch is inherently singular (one on/off value, no "SwitchGroup"); Checkbox has a sibling **CheckboxGroup** for multi-select sets that need to submit as a native form value (not covered in source detail in this pass — [G] CheckboxGroup source not read for this cluster note, flagged rather than assumed).
5. **Deselectability** — Toggle/ToggleGroup (single mode) can always return to "nothing pressed," which is the exact property mj12albert's maintainer answer (#3525, below) uses to distinguish the family from RadioGroup's exclusive-and-non-clearable contract; a single Switch/Checkbox is binary by construction (there is no "half-pressed" third state to return to, though Checkbox alone supports a distinct `indeterminate` visual/ARIA state that Toggle/Switch do not [I, inferred from Checkbox's `role="checkbox"` supporting `aria-checked="mixed"` per ARIA spec — not independently verified against Checkbox source in this pass, flagged [G]]).

## Decision table

| You need | Use | Why |
|---|---|---|
| A UI-state toggle with no form submission (bold/italic, view mode, favorite/star) | Toggle | `aria-pressed`, deselectable, zero form machinery — the leanest option when nothing needs to submit |
| A row of related toggle buttons sharing pressed-state (segmented control, alignment picker) | Toggle Group | Coordinates an array of Toggle values; single (`multiple=false`, still clearable) or multiple selection from one prop |
| A single on/off setting that should announce as a physical switch and can submit with a form | Switch | `role="switch"`, hidden `<input type="checkbox">` + `uncheckedValue` for native submission |
| A single agree/select checkbox, or one item in a multi-select set that submits with a form | Checkbox (+ CheckboxGroup for sets) | `role="checkbox"`, same hidden-input form participation as Switch, selection/agreement metaphor rather than physical-switch metaphor |
| An exclusive choice that must always have exactly one selection and can never be cleared | **Not this cluster** — RadioGroup | See boundary below; Toggle Group's single mode is deliberately still clearable, which is the wrong contract for true radio semantics |

## Per-pair boundaries

- **Toggle Group ↔ RadioGroup** [E, the cluster's most directly cited maintainer boundary]: "ToggleGroup implements the toggle-button pattern (`aria-pressed`, deselectable); a single-select-you-can't-clear widget *is* a RadioGroup; making a toggle announce as a radio is semantically wrong" (mj12albert, mui/base-ui#3525, open/unresolved as of the mining date). The practical test: if clicking the currently-selected item again should be a no-op or impossible (rather than deselecting it), the widget's correct ARIA identity is RadioGroup, not single-mode Toggle Group — no amount of styling makes the two interchangeable, because the underlying accessibility tree announcement differs. This remains a live community friction point, not a settled non-issue — some users continue to want `role="radio"` semantics on single-select Toggle Group and maintainers have held the line against it. [E]
- **Toggle ↔ Switch/Checkbox** [E from source, [I] for the framing]: the deciding, source-verifiable question is "does this value need to travel with a `<form>` submission or participate in native constraint validation?" Toggle/ToggleGroup structurally cannot (no hidden input in source); Switch/Checkbox structurally can (both ship the identical hidden-`<input type="checkbox">` + `uncheckedValue` mechanism). [G] No maintainer issue was found in the mined corpus explicitly contrasting "Toggle vs Switch" or "Toggle vs Checkbox" — this boundary is derived from direct source inspection (Toggle's discarded `form`/`type` props, Switch/Checkbox's shared hidden-input code), not from a quoted maintainer decision, and is tagged accordingly throughout this note.
- **Switch ↔ Checkbox** [I, since neither this note nor the two components' briefs were the subject of a dedicated Tier-1/2 research pass]: both share nearly identical form-participation source shape (`role` differs; hidden-input + `uncheckedValue` mechanism is close to identical between `SwitchRoot.tsx` and `CheckboxRoot.tsx`). The conventional distinction — Switch for "on/off" settings with immediate real-world effect (airplane mode, dark mode), Checkbox for "select/agree" choices frequently submitted in bulk as part of a form (terms acceptance, multi-select filters) — reflects the *metaphor* each ARIA role carries (physical switch vs a checkable box) rather than a difference enforced by Base UI's code. [G] No Base UI maintainer statement drawing this exact line was found in the mined corpus for this pass; flagged as inference from ARIA convention, not library-specific evidence.
