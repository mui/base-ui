## 1. Identity

- **Name / slug**: Checkbox — `@base-ui/react/checkbox`. Multi-part compound component, namespace export `Checkbox.*`.
- **Parts** (from `index.parts.ts`, one-liners from JSDoc):
  | Part | Renders | One-liner |
  |---|---|---|
  | `Checkbox.Root` | `<span>` + hidden `<input type="checkbox">` | "Represents the checkbox itself. Renders a `<span>` element and a hidden `<input>` beside." |
  | `Checkbox.Indicator` | `<span>` (conditionally, transition-aware) | "Indicates whether the checkbox is ticked. Renders a `<span>` element." |
- **Status**: stable, no `[New]`/`[Preview]` tag. Introducing commit: `6f74390e8` "[Checkbox] Component and Hook (#159)" — found via `git log --diff-filter=A -- "**/checkbox/**"`, offline, not yet corroborated by reading PR #159's body directly [E for the commit; I/G for full PR-body intention, since this specific lookup was not spent from the 5-lookup `gh` budget in this pass — prioritized instead toward radio-group and checkbox-group origin questions].
- **Taxonomy** (Phase A): **selection & input** → binary/choice controls sub-cluster, alongside radio/radio-group/checkbox-group/switch/toggle/toggle-group (`research/c-components/_clusters/binary-controls.md`). Purpose/IA: a single independently-toggleable boolean or tri-state (checked/unchecked/indeterminate) value — the base primitive that Checkbox Group coordinates into a multi-select array (`research/c-components/checkbox-group/brief.md`).
- **Cross-component role**: Checkbox is one of the components directly consuming `useFieldRootContext` (alongside radio, radio-group, checkbox-group, select, switch, combobox, number-field, otp-field — `research/c-components/field/brief.md` §1); it is also the sole component with built-in **parent-checkbox tri-state coordination** logic shared with Checkbox Group via the `parent` prop + `useCheckboxGroupParent` hook (see `research/c-components/checkbox-group/brief.md` §6, since the coordination *logic* lives in checkbox-group's `useCheckboxGroupParent.ts` even though the `parent` *prop* is declared on `Checkbox.Root`).

## 2. Intention

- [E] **Valid-HTML labeling forced the same `<span role="checkbox">` shape as Radio and Switch.** Checkbox was swept into the identical library-wide root-shape change (PR #3205, "[checkbox][radio][switch] Render root as `span` instead of `button`," merged 2025-11, breaking, 15 comments): "the reason these components (Checkbox, Radio, Switch) render a `<span>` is because otherwise an enclosing `<label>` creates invalid HTML since it wraps two form controls" — `<label><Checkbox.Root /></label>` would otherwise emit `<label><button/><input/></label>`, invalid HTML (atomiks, issue #4122 comment). This is the load-bearing design constraint behind the entire component shape, exactly as for Radio (`research/c-components/radio/brief.md` §2) — not an aesthetic choice.
- [E] **Native `<input type="checkbox">` off-state behavior is the explicit contract, matched deliberately over time.** The clearest single arc: "[checkbox][switch] Match native off state in form submission" (#3406, atomiks, merged 2025-12) changed Checkbox/Switch so that **an unchecked checkbox submits nothing by default**, exactly like a native `<input type="checkbox">` (which never appears in `FormData` when unchecked) — before this fix, Base UI's checkbox likely submitted some placeholder value while unchecked [I, inferred from the PR's framing as a "match native" fix rather than a net-new feature; the PR diff itself was inspected (`0c6d6c435`, 10-line change to `CheckboxRoot.tsx` + 235-line test rewrite) confirming the change touched exactly the unchecked-submission path]. The escape hatch, `uncheckedValue`, is opt-in and off by default — see §8.
- [E] **The component's identity spans three states, not two — `indeterminate` is a first-class, independently-settable prop**, not merely a derived display trick: `Checkbox.Root indeterminate` sets `aria-checked="mixed"` and the native input's `.indeterminate` DOM property regardless of the `checked` prop's value, and is explicitly **not overridden by `checked`** (`CheckboxRoot.test.tsx` "should not be overridden by `checked` prop" — `<Checkbox.Root indeterminate checked />` still renders `aria-checked="mixed"`). This tri-state design is the structural feature that differentiates Checkbox from Switch/Toggle in the binary-controls cluster note (`research/c-components/_clusters/binary-controls.md` §"Decision dimensions" point 1, which had flagged this as `[I]`/`[G]` pending a dedicated Checkbox pass — **now confirmed [E] from source and tests**, closing that gap).
- [E] **A checkbox can be a group coordinator, not just a group member.** The `parent` boolean prop turns a `Checkbox.Root` into the "select all" control for a sibling `<CheckboxGroup>` — an unusual design choice for a "single value" component to also carry group-orchestration behavior, but the source shows this is deliberately kept inside Checkbox.Root itself (`PARENT_CHECKBOX` constant + `groupContext?.parent` wiring in `CheckboxRoot.tsx`) rather than a separate `CheckboxGroup.Parent` part. See `research/c-components/checkbox-group/brief.md` §6 for the full tri-state cycling algorithm.
- [E] **Enter submits the owning form; it does not toggle the checkbox.** This is a deliberate, tested divergence from `<button>` semantics and was hardened in two stages: first, Enter is suppressed from toggling ("does not activate with Enter key" test); then a dedicated fix made Enter **submit the checkbox's owning form** instead of doing nothing, matching native `<input type="checkbox">` inside a `<form>` (`[checkbox] Submit the associated form on Enter key press`, #4713) — the source shows an elaborate `preventBaseUIHandler()`/microtask-deferred `getDefaultFormSubmitter(formToSubmit)?.click()` dance in `CheckboxRoot.tsx`'s `onKeyDown` specifically to reproduce this native quirk correctly across event-propagation edge cases.

## 3. When to use

- [E] A single independently-toggleable option — "on its own" (agree-to-terms, single settings flag) or as one member of a set where **each item's state is independent of the others** (the defining cardinality difference from Radio, `research/c-components/radio/brief.md` §4) — inferred from the boolean/tri-state `checked`/`indeterminate` model.
- [E] Inside forms where the checked state should participate in native submission and validation: the hidden-input + `uncheckedValue` + native-off-state-matching design exists specifically for this (§2, §8).
- [E] When a "select all" / parent-child tri-state relationship is needed over a set of related checkboxes: the `parent` prop + Checkbox Group's `allValues` exists as a built-in, first-class pattern, not a userland workaround (docs "Parent checkbox" + "Nested parent checkbox" sections, `checkbox-group/page.mdx`).
- [I] When the labeling pattern needs to support an enclosing `<label>` (the default `<span>` shape) *or* a sibling `htmlFor`/`id` label with a real native `<button>` (`nativeButton` + `render`) — both patterns are first-class and documented, so "when to use Checkbox" doesn't force a particular labeling shape; the choice is orthogonal.

## 4. When not to use + alternatives

Full binary/choice-control comparative guidance (Toggle/ToggleGroup/Switch/Checkbox dimensions, decision table, and the ToggleGroup-vs-RadioGroup boundary): [`research/c-components/_clusters/binary-controls.md`](../_clusters/binary-controls.md) — read first; this section adds Checkbox-specific refinements the cluster note flagged as pending.

- **A single on/off setting that should announce as a physical switch, with immediate real-world effect → Switch, not Checkbox**: [E, cluster note] — same hidden-input form-participation shape as Checkbox (`role` differs: `switch` vs `checkbox`; Switch has no `indeterminate` state). The cluster note's own inference (tagged [I]/[G] there) can now be partially sharpened from this pass: Checkbox alone supports `aria-checked="mixed"`/`indeterminate` — **confirmed [E]** in this brief (§2) — which is a genuine structural capability Switch lacks, not just a metaphor difference. Use Switch when there is no third "mixed" state ever needed; use Checkbox when there might be (parent/child groups) or when the selection/agreement metaphor (rather than physical on/off) fits the content.
- **An exclusive, always-exactly-one-of-N choice → Radio / RadioGroup, not Checkbox**: cardinality boundary — see `research/c-components/radio/brief.md` §4. A checkbox's checked state never affects any sibling checkbox's state (each is independent); a radio's checked state always affects every sibling (mutually exclusive).
- **Several independent checkboxes that should coordinate as one array-valued form field → Checkbox Group**, not several standalone `Checkbox.Root`s wired up by hand: see `research/c-components/checkbox-group/brief.md` §4 for the full boundary; a standalone Checkbox is the right choice when the value truly stands alone (a single "I agree" checkbox), while a set that should submit as one named array belongs under `<CheckboxGroup>`.
- **A UI-state toggle with no need for form submission at all → Toggle**: [E, cluster note] — Toggle/ToggleGroup have zero hidden-input machinery in source, by construction; if the value never needs to travel with a `<form>` POST, Toggle is the leaner primitive.

## 5. Anatomy & composition

- [E] Canonical tree (docs Anatomy):
  ```jsx
  <Checkbox.Root>
    <Checkbox.Indicator />
  </Checkbox.Root>
  ```
- [E] **Every `Checkbox.Root` renders both a `<span role="checkbox">` and a hidden sibling `<input type="checkbox">`** — never a single element, identical in shape to Radio (`radio/brief.md` §5). The hidden input's visibility style switches between `visuallyHidden`/`visuallyHiddenInput` based on whether `name` is set (`CheckboxRoot.tsx`).
- [E] **A third, separate hidden input appears conditionally**: when `uncheckedValue` is set, `name` is present, the checkbox is unchecked, is **not** part of a group, and is **not** a `parent` checkbox, an *additional* `<input type="hidden">` renders carrying the `uncheckedValue` (`CheckboxRoot.tsx:415-417` — `{!checked && !groupContext && name && !parent && uncheckedValue !== undefined && <input type="hidden" .../>}`). This is a third DOM node beyond the visible span + the main hidden checkbox input, and only exists in that specific state combination — important for anyone inspecting the rendered DOM and wondering why there are sometimes two hidden inputs.
- [E] **`Checkbox.Indicator` renders conditionally on `checked || indeterminate`** (not `checked` alone) — `rendered = rootState.checked || rootState.indeterminate` in `CheckboxIndicator.tsx` — and is transition-aware via the same `useTransitionStatus`/`useOpenChangeComplete` pattern as Radio's Indicator and Select's ItemIndicator (`select/brief.md` §6 Animation). Default `keepMounted={false}`.
- [E] **Parent-checkbox anatomy is structurally identical to a regular checkbox** — a `Checkbox.Root parent` renders the same `<span role="checkbox">` + hidden input shape, distinguished only by the `PARENT_CHECKBOX` (`data-parent`) attribute and its `name` being force-unset (`name: parent ? undefined : name` in `inputProps`) so it's excluded from form submission — see `research/c-components/checkbox-group/brief.md` §5 for the full group-level anatomy (`aria-controls` wiring, `id` scheme).
- [E] **Text spec for a future anatomy diagram**: (1) Root `<span role="checkbox">` (shows three visual variants: unchecked / checked / indeterminate) → (2) Indicator `<span>` (conditionally rendered, shown for checked and indeterminate states with distinct icon content, e.g. checkmark vs. dash — the docs "Parent checkbox" demo literally swaps icon by `state.indeterminate`) → (3) hidden `<input type="checkbox">` (visually hidden, real form participant, carries `.indeterminate` DOM property) → (4, conditional) a second hidden `<input type="hidden">` carrying `uncheckedValue`, present only when unchecked + `uncheckedValue` set + not grouped/parent.

## 6. Behavior ("How it works")

- [E] **Controlled/uncontrolled**: `checked`/`onCheckedChange`/`defaultChecked` triplet (`CheckboxRoot.tsx` props); `eventDetails.cancel()` vetoes the change — pinned by "does not update its state when onCheckedChange cancels the event."
- [E] **Space toggles; Enter does not toggle but submits the form instead.** Confirmed directly: "can be activated with Space key" and "does not activate with Enter key" (`CheckboxRoot.test.tsx`). Enter's form-submission behavior (rather than a no-op) was added later and deliberately reproduces native quirks including propagation-stopping edge cases (§2, #4713).
- [E] **Clicking the label, the visible span, or the hidden input all toggle identically** — separately tested for wrapping labels ("should toggle the checkbox when a wrapping `<label>` is clicked"), explicitly linked labels ("…explicitly linked `<label>`"), and direct input clicks ("should update its state if the underlying input is toggled").
- [E] **`readOnly` blocks all toggle paths, including via the label** — a dedicated test confirms clicking the wrapping `<label>` of a `readOnly` checkbox produces no state change ("should not change its state when its label is clicked"), a subtlety not obviously implied by `readOnly` blocking direct clicks alone.
- [E] **`indeterminate` is a prop the app must set and clear manually** — there is no automatic timer or interaction that clears it; the checkbox stays `aria-checked="mixed"` and visually indeterminate through clicks unless the app changes the prop (`CheckboxRoot.test.tsx` "should not change its state when clicked" under the indeterminate describe block — clicking an indeterminate (non-grouped) checkbox leaves it `mixed`, it does not "resolve" to checked/unchecked on its own). This matters for the parent-checkbox pattern, where Checkbox Group's `useCheckboxGroupParent` hook computes and passes `indeterminate` down automatically (`research/c-components/checkbox-group/brief.md` §6) — but a **manually-set** `indeterminate` on a *grouped* parent checkbox is independently supported and asserted too ("sets grouped parent aria when manually indeterminate").
- [E] **Native form-submission parity was directly verified against a real `<input type="checkbox">` in the same test** ("matches native checkbox form submission behavior," Chromium-only): both an unchecked native checkbox and an unchecked Base UI checkbox submit `null`/`[]` for their name; both submit `"on"` once checked with no explicit `value` — i.e., Base UI's default (no `uncheckedValue`, no `value`) submission behavior is **verified byte-for-byte identical** to the native element, not merely "close to."
- [E] **`value` defaults to falling back to `name`** when neither is explicitly distinguished (`CheckboxRoot.tsx`: `const value = valueProp ?? name`) — relevant when a checkbox is inside a `CheckboxGroup` and needs a `value` to be individually addressable in the group's array.
- [E] **Disabled semantics use `aria-disabled`, not the native `disabled` attribute, on the visible span** — "uses aria-disabled instead of HTML disabled" — while the hidden `<input>` itself does carry the real `disabled` attribute so native form-submission exclusion still works correctly.
- [E] **SSR**: hidden inputs carry `suppressHydrationWarning` (library-wide convention, confirmed present in source `<input {...inputProps} suppressHydrationWarning />`).

## 7. Accessibility contract

- [E] **Pattern**: W3C ARIA APG **Checkbox** pattern — https://www.w3.org/WAI/ARIA/apg/patterns/checkbox/ (dual/mixed variant applicable given `indeterminate`/`aria-checked="mixed"` support) [I for the explicit APG citation — not directly quoted in Base UI's own docs, but the `role="checkbox"` + `aria-checked` (including `"mixed"`) implementation is a textbook match, same inference style as Radio's APG pattern citation].
- [E] **ARIA managed by the component** (`CheckboxRoot.tsx`): `role="checkbox"`, `aria-checked` (`"mixed"` when `indeterminate`, else boolean), `aria-readonly`, `aria-required`, `aria-labelledby` (auto-linked, see below); hidden input: `aria-hidden="true"`, `tabIndex={-1}`.
- [E] **The `aria-labelledby` auto-link fix (#4122 → #4142) applies to Checkbox identically to Radio and Switch** — same WCAG-failure root cause (enclosing `<label>` doesn't associate with a `<span role="checkbox">`), same external verification across NVDA/Windows Narrator/JAWS (grace-snow, issue #4122), same fix (auto-generated `aria-labelledby` from `Field.Label` context, no API change required). See `research/c-components/radio/brief.md` §7 for the full quoted thread — identical mechanism, not repeated here.
- [E] **Keyboard interaction table** (evidence: `CheckboxRoot.tsx` handlers + tests cited in §6):

  | Key | Context | Action |
  |---|---|---|
  | Tab | anywhere before the control | Moves focus to the checkbox (standalone control — one Tab stop, no composite/roving-tabindex behavior since Checkbox is not a group widget on its own) |
  | Space | a focused checkbox | Toggles checked ↔ unchecked (does not affect `indeterminate` state directly — indeterminate is an app-controlled prop, not a third click-cycle state, §6) |
  | Enter | a focused checkbox | **Does not toggle**; submits the checkbox's owning `<form>` instead, matching native `<input type="checkbox">` inside a form (#4713) |
  | Click (label, span, or hidden input) | anywhere in the labeling chain | Toggles checked ↔ unchecked (unless `disabled`/`readOnly`) |
- [E] **Screen reader notes**: `aria-checked="mixed"` is the standard AT-facing announcement for `indeterminate` (per ARIA spec — checkbox is one of the few widget roles supporting the mixed value); label via enclosing `<label>` (now correctly linked post-#4142), `Field.Label`, sibling `<label htmlFor>` + `nativeButton`, or `aria-label` when no visible label exists.
- **Open a11y-adjacent issues (honesty list)**: [E] #3263 (closed, expected-behavior) explicitly states its `:hover`-stops-working-via-label report "also applies to Checkbox" — see `research/c-components/radio/brief.md` §7/Pitfall 2 for the full citation; not repeated here since it's the identical CSS-authoring surprise for the identical `<span>`-root cause. [E] #4329 (open) — `useCheckboxRootContext()` export request, same as Radio (§7 cross-reference).

## 8. Prop-level guidance (decision-relevant props)

- [E] **`uncheckedValue` (Root, default `undefined`)** — "The value submitted with the form when the checkbox is unchecked. By default, unchecked checkboxes do not submit any value, matching native checkbox behavior" (prop JSDoc). This is the `#3406` escape hatch: use it when a backend needs an explicit "off"/`"false"` sentinel rather than the field being absent from the payload entirely (common with some server frameworks that otherwise can't distinguish "checkbox was on this page but unchecked" from "checkbox wasn't rendered at all"). Verified to correctly toggle between `uncheckedValue` and the checked `value` across repeated check/uncheck cycles in the same form submission (`CheckboxRoot.test.tsx` "should submit uncheckedValue when checkbox is unchecked and uncheckedValue is specified" — 3-cycle test) and to respect `disabled` (a disabled+unchecked checkbox with `uncheckedValue` set still submits nothing, "does not submit uncheckedValue when disabled").
- [E] **`indeterminate` (Root, default `false`)** — sets tri-state `aria-checked="mixed"` + native `.indeterminate`; independent of and not overridden by `checked` (§2, §6). Use for parent/summary checkboxes over a partially-selected child set (the primary evidenced use case — the entire Checkbox Group "parent checkbox" pattern exists to compute and drive this prop automatically, `checkbox-group/brief.md` §6) or any other "some but not all" UI state.
- [E] **`parent` (Root, default `false`)** — "Whether the checkbox controls a group of child checkboxes. Must be used in a Checkbox Group" (prop JSDoc). Declaring `parent` unsets the checkbox's `name` (excluded from submission) and switches its `checked`/`indeterminate`/`onCheckedChange` to be driven by the group's computed tri-state via `useCheckboxGroupParent` rather than local state — see the group brief for the full algorithm.
- [E] **`nativeButton` (Root, default `false`)** — identical convention and rationale to Radio's (`radio/brief.md` §8): declare `true` + `render={<button/>}` for the sibling-label pattern; the default `<span>` supports enclosing labels without the invalid-HTML problem.
- [E] **`value` (Root, optional, string)** — "Identifies it within a Checkbox Group, falling back to `name` when omitted. When submitting a form, a checked box submits `value`; with no `value`, it submits the native `'on'`" (prop JSDoc) — the same native-parity behavior verified in §6's native-comparison test.
- [E] **`disabled`/`readOnly`/`required`/`name`/`form`/`inputRef`** — same shape and cascade rules as Radio's identical props (`radio/brief.md` §8); Field.Root-level `disabled`/`name` cascade and take precedence over Root-level props, confirmed by `CheckboxRoot.test.tsx`'s Field describe block.
- [E] **Styling contract highlights** (from `*DataAttributes.ts`): `data-checked`/`data-unchecked` (mutually exclusive), **`data-indeterminate`** (Checkbox's distinguishing attribute versus Radio/Switch — neither of those has an indeterminate equivalent), `data-disabled`, `data-readonly`, `data-required`, plus the Field tri-state set (`data-valid`/`data-invalid`/`data-touched`/`data-dirty`/`data-filled`/`data-focused`); Indicator additionally carries `data-starting-style`/`data-ending-style` for exit-transition animation, and — unusually — mixes in `fieldValidityMapping` directly at the Indicator level too (`CheckboxIndicator.tsx`, not just Root), a subtlety not present in Radio's otherwise-identical Indicator wiring [E, direct source comparison].

## 10. Pitfalls & FAQ

1. **"My enclosing `<label>` doesn't announce the checkbox's name to screen readers"** → identical WCAG failure and fix to Radio's Pitfall 1 (`radio/brief.md` §10) — see that citation; correction is the same (`nativeButton` + `render`, or upgrade past #4142, or explicit `aria-label`).
2. **"`.checkbox:hover` stopped firing when I hover the label"** → identical CSS-authoring surprise to Radio's Pitfall 2, explicitly confirmed to also affect Checkbox in the same issue thread (#3263: "This also applies to Checkbox"). [E]
3. **"My unchecked checkbox doesn't appear in `FormData` at all"** → this is the *intended*, native-matching default behavior since #3406, not a bug — set `uncheckedValue` if your backend needs an explicit off-value. [E]
4. **"`indeterminate` won't clear itself after the user clicks"** → correct; indeterminate is app-controlled, not a self-resolving click-cycle state — clear it explicitly in your `onCheckedChange` handler, or let the Checkbox Group `parent` pattern compute it for you automatically. [E]
5. **"Enter key does nothing / submits my form unexpectedly"** → Enter never toggles a checkbox (matches `<input type=checkbox>`, which also doesn't toggle on Enter in real browsers); if focus is on a checkbox inside a `<form>`, Enter will submit that form via its default submitter, same as pressing Enter in any other form field. [E, #4713]
6. **"I want to read the checkbox's internal state to animate it externally"** → same open request as Radio's Pitfall 5 (#4329, open, shared ask across Checkbox/Radio/Switch). [E]
