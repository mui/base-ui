## 1. Identity

- **Name / slug**: Radio — `@base-ui/react/radio` (2 parts: `Radio.Root`, `Radio.Indicator`) — plus **Radio Group** — `@base-ui/react/radio-group` (1 part, single default export `RadioGroup`, no namespace). Treated as **one research unit** because Radio is never meaningfully used outside a Radio Group [E]: "Radio is always placed within Radio Group" (`radio/page.mdx` Anatomy section) — a standalone `Radio.Root` outside `<RadioGroup>` falls back to a degenerate `checked = value === ''` comparison (`RadioRoot.tsx:85`) rather than any real group-selection semantics, which is not a documented or intended usage pattern [I].
- **⚠️ Packaging asymmetry (verify prominently)**: `radio-group` is a **fully public, independently exported subpath** (`@base-ui/react/radio-group`, its own `index.ts`, its own test file, its own git history under `[radio group]`/`[RadioGroup]` commit scopes) — but it has **no standalone docs page**. `docs/src/app/(docs)/react/components/radio-group/` contains only `types.md`/`types.ts` (generated API-reference data), no `page.mdx`. All of Radio Group's user-facing documentation — Anatomy, Examples, Usage guidelines, and its API reference section — lives *inside* `radio/page.mdx` under a `### RadioGroup` heading (docs `page.mdx:120-126`). Radio Group is also **absent from the docs nav** (confirmed against the flat A–Z component list, `research/PROMPT.md` §3.2) — a user browsing the component list would never discover it exists as its own thing; they'd only find it by opening the Radio page. This is a real packaging inconsistency worth flagging as a documentation gap, not a design decision with a cited rationale [E, inferred from file layout — no maintainer statement found explaining *why* the split docs/no-nav-entry state exists, tagged [G]].
- **Parts**:
  | Part | Renders | One-liner (from JSDoc) |
  |---|---|---|
  | `Radio.Root` | `<span>` + hidden `<input type="radio">` | "Represents the radio button itself. Renders a `<span>` element and a hidden `<input>` beside." |
  | `Radio.Indicator` | `<span>` (conditionally, transition-aware) | "Indicates whether the radio button is selected. Renders a `<span>` element." |
  | `RadioGroup` | `<div>` | "Provides a shared state to a series of radio buttons. Renders a `<div>` element." |
- **Status**: stable, no `[New]`/`[Preview]` tag. Radio Group's introducing GitHub issue is **#26** "[RadioGroup] Implement RadioGroup" (michaldudak, 2024) [E]; its implementing PR is **#487** "[RadioGroup] Create new `RadioGroup` component" (atomiks, merged 2024-07-24, commit `0c87b6ee5`) [E]. Radio itself has no separately identifiable introducing commit in the file-tracked history before the library-wide kebab-case import-path rename (`#867`, 2024-11-26) — it was very likely folded into the same early implementation wave as RadioGroup or an even earlier pre-rename commit that `git log --follow` could not trace through the directory restructure [G].
- **Taxonomy** (per PROMPT.md §3.2/§8.2 category conventions): **selection & input** → **binary/choice controls** sub-cluster, shared with checkbox, checkbox-group, switch, toggle, toggle-group (see `research/c-components/_clusters/binary-controls.md`). Purpose/IA: a mutually-exclusive, always-visible option picker — the user sees every choice at once and picks exactly one, contrasted with Select's collapsed-dropdown IA for the same "pick one" task (see §4).

## 2. Intention

- [E] **Origin**: issue #26 frames Radio Group as solving a named, cross-referenced pain point: "This is the place to gather ideas, measure interest and discuss the API and implementation details of the radio button and radio group components and hooks," linking the Base UI umbrella tracking issue (#10) and, critically, **mui/material-ui#6218** — "the problem this project solve[s]" (michaldudak). This mirrors Select's origin story (a ground-up answer to accumulated MUI-legacy pain, per `research/c-components/select/brief.md` §2) but for the radio-selection primitive specifically.
- [E] **Design pole: exclusive, always-one-selected, form-native.** RadioGroup wraps a `CompositeRoot` (roving-tabindex composite widget, `RadioGroup.tsx`) rather than a popup — the entire point is that all options are visible simultaneously and reachable by a single Tab stop with arrow-key movement between them, matching the library's general "single tab stop, roving highlight" composite-widget convention (`research/b-library-principles/principles.md` B-P29).
- [E] **Native `<input type="radio">` semantics are the contract, not just an implementation detail.** Every radio ships a hidden native input (`RadioRoot.tsx` `inputProps`, `type: 'radio'`) that participates in real browser radio-group behavior — form submission, `FormData`, native validation, and (crucially) the native "no radio selected submits `null`" behavior is explicitly pinned by tests: "should return null when no radio is selected (matching native behavior)" (`RadioGroup.test.tsx`) and the introducing fix "return null in form data when no option selected" (`#2473`, `[radio group]`). This is unlike Select, which always has *some* serialized value (even `""`); RadioGroup's unselected state is a first-class `null`, matching how a native `<form>` treats an untouched radio group.
- [E] **Valid-HTML labeling forced the `<span role="radio">` shape.** Radio was swept into the library-wide "roots render `<span>` not `<button>`" change: "the reason these components (Checkbox, Radio, Switch) render a `<span>` is because otherwise an enclosing `<label>` creates invalid HTML since it wraps two form controls" — `<label><Checkbox.Root /></label>` would otherwise produce `<label><button/><input/></label>`, which is invalid (atomiks, issue #4122 comment, referencing the decision PR `#3205`). This is a load-bearing design constraint that shapes the entire component, not a stylistic choice — see §7 and Pitfall 1.
- [E] **Arrow-key navigation both moves focus *and* selects — by design, matching native `<input type=radio>` groups.** This is the single most important behavioral fact to verify (per the assignment's explicit instruction) and it is unambiguously confirmed by tests, not inferred: `RadioGroup.test.tsx` "should automatically select radio upon navigation" asserts that after `ArrowDown` moves focus from radio A to radio B, **B is immediately `aria-checked="true"` and `data-touched` fires on the group** — the user did not press Space or click; arrow navigation alone committed the selection. The composite navigation wiring lives in `RadioRoot.tsx`'s `onFocus` handler: `if (... !touched) { inputRef.current?.click(); setTouched(false); }` — focusing an untouched radio via keyboard synthetically clicks its own hidden input. This mirrors the real native HTML radio-group behavior (arrowing through a native radio group changes the checked radio, unlike Tabs/Menu/Listbox roving-focus patterns where arrow keys only move a highlight).
- [I] **Explicit non-goal: no visual anatomy beyond Root+Indicator.** Radio has the leanest part count in the entire choice-control family (2 parts vs. Checkbox's 2, but far below Select's 19) — inferred from the deliberately small `index.parts.ts` surface; no maintainer statement rejecting additional parts (e.g. a `Radio.Trigger`) was found for Radio specifically, though the identical `Checkbox.Trigger` idea was floated and left unbuilt in #4122 (see §10 Pitfall 1) [I, extending by analogy].

## 3. When to use

- [E] A small, always-visible set of mutually exclusive options where the user must always end up choosing exactly one (or, if not required, may leave the group at its unselected `null` state) — inferred directly from the exclusive-selection mechanics and the native-`null`-on-unselected behavior (§2).
- [E] Inside forms, where the selected value should submit natively: RadioGroup's `name`/`form` props and hidden-input wiring exist specifically for this (`RadioGroup.tsx` props; forms handbook `Field.Item` group-labeling guidance quoted in `research/c-components/field/brief.md` §3).
- [E] When every option should be simultaneously visible and comparable without an extra open/close interaction — this is RadioGroup's structural difference from Select (a popup-based picker for the same "choose one" task); see §4.
- [I] When the exclusive-choice semantics must be unambiguous to assistive technology as a `role="radiogroup"`/`role="radio"` pair, not merely styled to *look* like a segmented control — this is the deciding factor raised in the Toggle-Group-vs-RadioGroup boundary (#3525, §4) [I, extending the maintainer's stated reasoning to the "when to use" direction].

## 4. When not to use + alternatives

Full binary/choice-control comparative guidance: [`research/c-components/_clusters/binary-controls.md`](../_clusters/binary-controls.md) (covers Toggle/ToggleGroup/Switch/Checkbox; extend rather than duplicate). This section adds Radio/RadioGroup-specific boundaries not yet in that cluster note.

- **Many options, or options that should stay collapsed until interacted with → Select**: [I] derived from API affordances, not a found maintainer statement (searched issues-prs.md/discussions.md for "radio vs select" and "select vs radio" — none found, tagged [G]). The dividing signal by analogy to Select's own documented boundary (`research/c-components/select/brief.md` §4, "Choosing a predefined value… especially inside forms," "when the item count is small enough that keyboard typeahead suffices instead of filtering") is inverted here: Radio Group is the right choice specifically when the option count is small enough that showing all of them at once, with no popup, is the better UX — once the list grows past a handful of items, the always-visible IA becomes a liability (vertical space, scanning cost) and Select's collapsed-dropdown IA becomes preferable. No Base UI maintainer has stated this threshold explicitly.
- **A single-select-you-can't-clear-back-to-nothing widget disguised as a toggle group → use RadioGroup, not ToggleGroup**: [E] direct maintainer boundary, the sharpest cited line in this space: "ToggleGroup implements the toggle-button pattern (`aria-pressed`, deselectable); a single-select-you-can't-clear widget *is* a RadioGroup; making a toggle announce as a radio is semantically wrong" (mj12albert, mui/base-ui#3525, **open as of mining date** — a live, unresolved community friction point, not a settled non-issue). The practical test from the maintainer's framing: if re-clicking the currently-selected item should be a no-op (never returning the group to "nothing selected"), the correct component is RadioGroup; ToggleGroup's single mode remains deliberately clearable and is the wrong contract for true exclusive-choice semantics no matter how it's styled to look like a segmented control.
- **A single on/off value, or a submit-time bulk boolean set → Checkbox / CheckboxGroup, not Radio**: cardinality differs — Radio always resolves to "exactly one of N, or none" where Checkbox/CheckboxGroup resolve to "any subset of N, independently toggleable." See `research/c-components/checkbox/brief.md` §4 and the cluster note.
- **A list of actions/commands rather than a persisted value → Menu**: [I], same reasoning as Select's boundary against Menu (`select/brief.md` §4) — Radio Group persists and (optionally) submits a value; Menu performs commands. No maintainer statement specific to Radio-vs-Menu was found [G].

## 5. Anatomy & composition

- [E] Canonical tree (docs Anatomy, `radio/page.mdx`):
  ```jsx
  <RadioGroup>
    <Radio.Root>
      <Radio.Indicator />
    </Radio.Root>
    {/* …repeated per option */}
  </RadioGroup>
  ```
- [E] **RadioGroup renders via `CompositeRoot`** (`RadioGroup.tsx`), the same roving-tabindex composite-widget primitive used by other composite controls in the library; each `Radio.Root` renders via `CompositeItem` when inside a group context (`RadioRoot.tsx`: `isRadioGroup ? <CompositeItem .../> : element` — the same component renders differently depending on whether a `RadioGroupContext` is present).
- [E] **`Radio.Indicator` is conditionally rendered and transition-aware**, not a fixed empty box: it computes `rendered = rootState.checked` and only mounts (absent `keepMounted`) while checked or during exit transition (`RadioIndicator.tsx`, `useTransitionStatus`/`useOpenChangeComplete` — the identical pattern Checkbox's Indicator uses, and the same pattern Select uses for its ItemIndicator per `select/brief.md` §6 Animation).
- [E] **Each `Radio.Root` renders both a `<span role="radio">` and a hidden sibling `<input type="radio">`** — never a single element. The hidden input's `style` switches between `visuallyHidden` and `visuallyHiddenInput` depending on whether `name` is set (`RadioRoot.tsx` `inputProps.style: name ? visuallyHiddenInput : visuallyHidden`), matching the same convention documented for Checkbox (`research/b-library-principles/principles.md` B-P31).
- [E] **RadioGroup itself also renders a hidden input** — not per-radio-only: `groupInputRef`/`registerInputRef` machinery in `RadioGroup.tsx` maintains a single pointer to *the currently checked radio's* hidden input (re-pointed via `setInputRef` whenever selection changes, and falling back to the first enabled radio's input when the value is cleared to `null` — `useValueChanged` handler, `RadioGroup.tsx:161-174`). This is why `RadioGroup`'s own `inputRef` prop returns the checked radio's native input, not a group-level input of its own.
- [E] **Text spec for a future anatomy diagram**: (1) RadioGroup container (invisible `<div role="radiogroup">`) → (2) Radio.Root #1 (`<span role="radio">`, roving-tabindex composite item) → (3) Radio.Indicator #1 (conditionally rendered `<span>`) → (4) hidden `<input type="radio">` #1 (visually hidden, real form participant) → repeat (2)-(4) per option.
- [I] **Composition rule**: Field.Item wraps each individual radio option for per-option labeling/description inside a group, while the group-level label comes from Fieldset.Legend (`Field.Root` → `Fieldset.Root render={<RadioGroup/>}` → `Fieldset.Legend` + N × `Field.Item` → `Radio.Root`) — this is the exact canonical pattern shown in both the docs Form-integration example and pinned by `RadioGroup.test.tsx`'s `Fieldset`/`Field.Item` describe blocks [E, cited from both docs and tests].

## 6. Behavior ("How it works")

- [E] **Controlled/uncontrolled**: `value`/`onValueChange`/`defaultValue` triplet on RadioGroup (`RadioGroup.tsx` props JSDoc); `eventDetails.cancel()` vetoes the change while staying uncontrolled — pinned by three dedicated cancellation tests covering click, hidden-input click, and arrow-key navigation, all confirming **no state change, no `data-touched`/`data-dirty`/`data-filled`** when canceled (`RadioGroup.test.tsx` "does not change state when canceled via…").
- [E] **Arrow-key navigation selects, not just focuses** — see §2 for the mechanism; confirmed for both vertical (`ArrowDown`/`ArrowUp`) and horizontal (direction-aware `ArrowLeft`/`ArrowRight`, swapped under `DirectionProvider` RTL) axes in the same composite (`RadioGroup.test.tsx` "should manage arrow key navigation" describe block, parametrized over `ltr`/`rtl`). **Wraps around** (last→first, first→last) but does **not** support Home/End: `RadioGroup.tsx` passes `enableHomeAndEndKeys={false}` to `CompositeRoot` explicitly — Home/End do nothing in a RadioGroup, unlike Select's listbox navigation which does support them (`select/brief.md` §7 keyboard table). Holding **Shift** while pressing arrow keys suppresses the auto-select-on-focus behavior and moves focus only, matching the underlying `MODIFIER_KEYS = [SHIFT]` guard in `RadioGroup.tsx` (test: "when Shift is pressed arrow keys move focus normally").
- [E] **Space (not Enter) selects a focused-but-unchecked radio.** `RadioRoot.tsx`'s `onKeyDown` explicitly `preventDefault()`s on `Enter` (radios never activate on Enter — this is a deliberate difference from native `<button>` semantics, matching Checkbox's identical Enter-suppression, §"checkbox" brief §6). Space selection resolves **on keyup**, not keydown: `RadioGroup.test.tsx` "should select an item with Space on keyup" explicitly asserts `handleChange` has **not** fired while Space is still held down (`[Space>]`) and only fires after release (`[/Space]`) — this was a fixed regression, `[radio] Fix selection on space key press (#4930)`.
- [E] **Clicking the hidden hidden hidden hidden input directly also works and is separately tested** — clicking either the visible `<span role="radio">` or its paired native `<input>` produces identical selection behavior and identical cancellation semantics (`RadioGroup.test.tsx`, separate click-on-item vs click-on-input test blocks).
- [E] **Disabled radios are structurally excluded from form data, matching native behavior**, including edge cases: a disabled *and selected* radio is excluded from `onFormSubmit` (test: "excludes a disabled selected radio from onFormSubmit to match native form data"); re-enabling before submit restores it (test: "includes a selected radio again when it is re-enabled before form submission"); and when *all* radios start disabled, RadioGroup still correctly focuses the first radio once any become enabled (test: "focuses the first enabled radio when all radios start disabled"). This required a dedicated fix, `[radio] Fix disabled selected form submission (#4926)`.
- [E] **Focus/validation lifecycle**: RadioGroup's own `onBlur` (fires only when focus truly leaves the group, via `contains()` check — the shadow-DOM-safe utility per `AGENTS.md`) sets Field `touched`+`focused=false` and commits `onBlur`-mode validation (`RadioGroup.tsx` `defaultProps.onBlur`); `onKeyDownCapture` marks `touched`+`focused=true` the moment any arrow key is pressed, ahead of the actual selection (same handler block).
- [E] **`RadioGroup.name` and `Field.Root name` interplay**: Field-level `name` takes precedence over a RadioGroup-level `nameProp` (`RadioGroup.tsx`: `const name = fieldName ?? nameProp` — same precedence rule documented for Field generally, `research/c-components/field/brief.md` §8).
- [E] **SSR**: hidden inputs carry `suppressHydrationWarning` (library-wide convention, `research/b-library-principles/principles.md` B-P35, confirmed present in `RadioRoot.tsx`'s rendered `<input {...inputProps} suppressHydrationWarning />`).

## 7. Accessibility contract

- [E] **Pattern**: W3C ARIA APG **Radio Group** pattern — https://www.w3.org/WAI/ARIA/apg/patterns/radio/. `role="radiogroup"` on the container, `role="radio"` + `aria-checked` on each option, single Tab stop with roving arrow-key focus/selection inside — this is a textbook implementation of the pattern, not a variant [I, pattern-name inference from the `role`/keyboard-model match; no Base UI doc explicitly cites the APG URL for Radio the way Select's brief cites the Combobox pattern URL — tagged [G] for the explicit citation, though the pattern identity itself is unambiguous from source].
- [E] **ARIA managed by the component** (`RadioRoot.tsx:138-182`, `RadioGroup.tsx:218-244`):
  - RadioGroup: `role="radiogroup"`, `aria-required`, `aria-disabled`, `aria-readonly`, `aria-labelledby` (auto-resolved from `Field.Label`/`Fieldset.Legend`/explicit `aria-labelledby`, in that fallback order: `elementProps['aria-labelledby'] ?? labelId ?? fieldsetContext?.legendId`).
  - Radio.Root: `role="radio"`, `aria-checked`, `aria-required`, `aria-readonly`, `aria-labelledby` (auto-linked via `useAriaLabelledBy` — the same mechanism fixed for Checkbox/Switch in #4142, see below).
  - Hidden input: `tabIndex={-1}`, `aria-hidden="true"` — never independently reachable or announced; exists purely for native form participation.
- [E] **The `aria-labelledby` auto-link fix (#4122 → #4142) applies identically to Radio.** The root problem: because Radio.Root renders `<span role="radio">` (not a native `<input>`/`<button>`), an enclosing `<label>` does not create an implicit accessible-name association — confirmed as a **WCAG compliance failure with no workaround**, verified independently across NVDA, Windows Narrator, and JAWS by an external accessibility reviewer (grace-snow, issue #4122): "checkboxes, radios and switches are all announced as un-named controls by NVDA, Windows Narrator, Android Talkback… While JAWS and VoiceOver will use nearby text to take a 'best guess' at the intended label, other screen readers will not do this kind of auto-correction." The fix (#4142, shipped) auto-generates and links `aria-labelledby` from the enclosing `Field.Label` context so the accessible name resolves correctly without any API change required from consumers [E, full thread read from cache — atomiks confirmed the root cause is the same `<label>`-wraps-two-controls HTML-validity constraint that produced the `<span>` shape in the first place, PR #3205].
- [E] **Keyboard interaction table** (evidence: `RadioGroup.tsx` `CompositeRoot` wiring + `RadioRoot.tsx` handlers + tests cited in §6):

  | Key | Context | Action |
  |---|---|---|
  | Tab | anywhere before/in the group | Moves focus to (or through) the group as **one stop** — lands on the checked radio if one exists, otherwise the first enabled radio |
  | ArrowDown / ArrowRight (LTR) | focus in the group | Moves focus to next radio **and selects it** (fires `onValueChange`), wraps from last to first |
  | ArrowUp / ArrowLeft (LTR) | focus in the group | Moves focus to previous radio **and selects it**, wraps from first to last |
  | ArrowLeft / ArrowRight | RTL (`DirectionProvider`) | Horizontal keys swap meaning; vertical keys (`ArrowUp`/`ArrowDown`) unaffected |
  | Shift + Arrow keys | focus in the group | Moves focus **only** — does not select (suppresses the auto-select-on-focus behavior) |
  | Home / End | focus in the group | **No effect** — `enableHomeAndEndKeys={false}`, a deliberate difference from listbox-style composites like Select |
  | Space | a focused, unchecked radio | Selects on **keyup** (not keydown) |
  | Enter | a focused radio | **No effect** — explicitly `preventDefault()`ed; radios never activate on Enter |
  | Click | the visible radio or its hidden input | Selects (unless `disabled`/`readOnly`) |
- [E] **Screen reader notes**: label the group via `Fieldset.Legend` (preferred, form-integrated) or a sibling element + `aria-labelledby` (docs "Labeling a radio group" example); label each option via an enclosing `<label>` (simplest, and now correctly linked per #4142) or `Field.Label`/`Field.Item` composition. Disabled radios inside the group remain keyboard-discoverable per the library's composite-widget policy (roving-tabindex composites keep disabled items focusable — `research/b-library-principles/principles.md` B-P28, citing issue #656) [I, extending the general composite-widget policy to Radio by architectural analogy; no Radio-specific test asserting disabled-radio focusability was found in this pass, tagged [G]].
- **Open a11y-adjacent issues (honesty list)**: [E] #3263 (closed, expected-behavior, **not reopened as a bug**) — a user reported that `.radio:hover` used to fire when hovering the *enclosing label* in beta 4 and earlier, and stopped doing so in beta 5/6; the reporter explicitly notes "this also applies to Checkbox, and other CSS states like `:active`" and links it to the `<span>`-root PR (#3205) as the likely cause — i.e., a **CSS-authoring surprise**, not an accessibility regression, but one that trips up stylers who relied on native `<label>` hover propagation to a wrapped `<button>`. [E] #4329 (open, feature request) — request to publicly export `useRadioRootContext()`/`useCheckboxRootContext()` etc. for building custom subcomponents/animations; unresolved as of mining date.

## 8. Prop-level guidance (decision-relevant props)

- [E] **`value` (Radio.Root, required)** — the unique identifying value for this option within the group; `RadioGroup`'s `checkedValue === value` comparison is how selection is determined (`RadioRoot.tsx:85`). Generic `Value` typing was added later (`e937bccff`, "[radio] Add generic `Value` typing to `Radio`", #4033) so non-string values type-check; `RadioRoot.spec.tsx` pins the generic contract (`<Radio.Root<string | null> value={null} />` compiles; a mismatched explicit generic is a `@ts-expect-error`).
- [E] **`nativeButton` (Radio.Root, default `false`)** — declare `true` (with `render={<button/>}`) when you want each radio to render as an actual `<button>` instead of the default `<span>`; required specifically for the sibling-label pattern (`htmlFor`/`id`) so keyboard/ARIA adapt correctly (docs "Rendering as a native button" section; library-wide `nativeButton` convention, #1909). The default `<span>` shape exists to support **enclosing** labels without the invalid-HTML problem (§2, §7).
- [E] **`disabled`/`readOnly`/`required` — cascade from RadioGroup to each Radio.Root, and further from `Field.Root`/`Fieldset`.** `RadioRoot.tsx` merges `disabledGroup || disabledProp` etc.; a `Field.Root disabled` cascades all the way down (`RadioGroup.test.tsx` "should receive disabled prop from Field.Root"). Use group-level props for "the whole choice is off-limits," per-radio props for "this one option is unavailable but others remain choosable."
- [E] **`name`/`form` (RadioGroup)** — `name` on RadioGroup (or the enclosing `Field.Root`, which wins) identifies the submitted value; `form` targets an owning `<form>` when the group is rendered outside it (mirrors Select's identical `form` escape hatch, `select/brief.md` §8).
- [E] **`inputRef` (RadioGroup)** — unusually, this points to *the checked radio's* native input, re-pointed dynamically as selection changes, and falls back to the first enabled radio's input when the value is `null` (§5, §6) — do not expect a stable single DOM node if selection changes at runtime.
- [E] **`value`/`defaultValue`/`onValueChange` (RadioGroup)** — the standard controlled/uncontrolled triplet; `onValueChange(value, eventDetails)` receives full modifier-key details (Shift/Ctrl/Alt/Meta preserved through both click and keyboard paths, #3935) and supports `eventDetails.cancel()` (§6).
- [E] **Styling contract highlights** (from `*DataAttributes.ts`): Radio.Root/Indicator share an identical set — `data-checked`/`data-unchecked` (mutually exclusive, always one present), `data-disabled`, `data-readonly`, `data-required`, plus the Field tri-state set `data-valid`/`data-invalid`/`data-touched`/`data-dirty`/`data-filled`/`data-focused`; Indicator additionally carries `data-starting-style`/`data-ending-style` for exit-transition animation. RadioGroup itself carries only `data-disabled` as its own attribute (`RadioGroupDataAttributes.ts`) plus the full Field validity set via `fieldValidityMapping`.

## 10. Pitfalls & FAQ

1. **"My enclosing `<label>` doesn't announce the radio's name to screen readers"** → confirmed WCAG failure prior to the fix; NVDA/Narrator/Talkback announce the control as unnamed (JAWS/VoiceOver "best-guess" from nearby text masked the bug for some testers). Correction: upgrade to the version containing #4142 (auto-`aria-labelledby`), or in the meantime render a native `<button>` via `nativeButton` + `render`, or set `aria-label`/`aria-labelledby` explicitly. [E, issue #4122]
