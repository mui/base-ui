# Number Field — component research brief

Tier 2 (leaner depth). Mined 2026-07-07 from source, docs, tests, experiments, git history, and 2 `gh` PR lookups (read-only). Evidence tags: [E] direct evidence, [I] inference, [G] acknowledged gap.

## 1. Identity

- **Name / slug**: Number Field — `@base-ui/react/number-field`. Multi-part compound component, namespace export `NumberField.*`.
- **7 parts** (from `packages/react/src/number-field/index.parts.ts`; one-liners from part JSDoc):
  | Part | Renders | One-liner |
  |---|---|---|
  | Root | `<div>` | Groups all parts of the number field and manages its state [E] |
  | Group | `<div>`, `role="group"` | Groups the input with the increment and decrement buttons [E] |
  | Input | `<input type="text">` | The native input control in the number field [E] |
  | Increment | `<button>` | A stepper button that increases the field value when clicked [E] |
  | Decrement | `<button>` | A stepper button that decreases the field value when clicked [E] |
  | ScrubArea | `<span>` | An interactive area where the user can click and drag to change the field value [E] |
  | ScrubAreaCursor | `<span>` (portaled to `body`) | A custom element to display instead of the native cursor while using the scrub area [E] |
- **Status**: stable, no `[Preview]`/`[New]` tag on the docs page. One of the **alpha.0 founding trio** — "This is an initial release of Base UI as the @base_ui/react package. It features the Checkbox, Number Field, and Switch as the first components to be rewritten with a fresh new API" [E] (`research/b-library-principles/_mining/history.md`; founding commit `f68970d0f`, PR #186, 2024-04-11).
- **Taxonomy**: falls in the **selection & input** category alongside Slider — both are numeric-value editors with a non-typing gesture (drag/scrub vs drag-thumb), but Number Field's primary input mode is typed text, unlike Slider's primary gesture-driven mode. No dedicated cluster note exists yet for number-field↔slider (that boundary belongs primarily to slider's brief per this task's scope — see §4).
- **Purpose / IA statement**: [E] "A numeric input element with increment and decrement buttons, and a scrub area" (`docs/src/app/(docs)/react/components/number-field/page.mdx` Subtitle) — a typed, formatted, steppable numeric form control, distinct from a generic `Input`.

## 2. Intention

- [E] **Introducing PR**: mui/base-ui#186, "[number field] Component and Hook" (atomiks, merged 2024-04-11, commit `f68970d0f`), closing #185 and #224. The PR body's checked TODO list is a direct window into founding design intent and is worth quoting at length: "Add Home and End key support if min/max are specified" (implemented, confirmed in current source); "Dynamically change the `inputMode=\"numeric\"` prop for iOS... it must be 'text'" (implemented, `setDynamicInputModeForIOS` in `NumberFieldRoot.tsx`); "**Research if spinbutton ARIA props become necessary for iOS when `inputMode` is 'text'**" — this line is the closest the project comes to an explicit spinbutton-ARIA decision point, and the shipped component has never added `role="spinbutton"` or `aria-value*` attributes (verified directly against source — see §7); "Automatically remove floating point errors" (implemented as `removeFloatingPointErrors` in `utils/validate.ts`, still actively maintained as of 2026); "Move `ScrubArea` logic into the `useNumberField` hook, or possibly a separate hook. Depends how granular the tree-shaking could possibly be" (the scrub logic did move into the dedicated `NumberFieldScrubArea` component, per `0f3ef0ee3 [NumberField] Move scrubbing logic to ScrubArea component (#1859)`, alpha-era).
- [E] **PR #590, "[number field] Modernize implementation"** (atomiks, merged 2024-09-11, commit `f8906ef52`): body is terse ("Namespace types", "`useComponentRenderer`"), and the diff confirms it as an **internal API-grammar migration**, not a design-intent PR — it deleted the standalone `NumberFieldRoot.types.ts` files and folded types into namespaces, matching the library-wide Root/part-namespace convention (B-P8) that was being rolled out across components in the same era. Treat it as infrastructure alignment, not a rethink of the component's behavior.
- [I] **Design pole: a formatted, typed numeric input first, with stepping as an enhancement layer.** Inferred from: the Input renders `type="text"` (not `type="number"`) so `Intl.NumberFormat`-formatted text (currency symbols, units, grouping separators) can be displayed and edited (`NumberFieldRoot.tsx` `formatNumber`/`parseNumber` pipeline; confirmed by extensive locale/currency/unit tests in `NumberFieldInput.test.tsx`); a parallel hidden `<input type="number">` exists purely to carry native constraint validation (`min`/`max`/`step`/`required`) and form submission, decoupled from the visible formatted text.
- [E] **Explicit non-goals, from discussions mining** (`research/b-library-principles/_mining/discussions.md`):
  - **No exported hooks** — #1814: an ex-`@mui/base` user who had built a calculated text-input on `useNumberInput` asked for `useNumberFieldButton`/similar hooks back; mj12albert answered "no hooks in the new library — being refactored away; compose instead (controlled NumberField with your own inner input)." This mirrors the library-wide render-prop/no-context-hooks doctrine (B-P11); `useNumberFieldButton`/`useNumberFieldStepperButton` exist in source today but are internal, unexported implementation details shared between Increment/Decrement, not public API.
  - **No math-expression evaluation** — #2152: a user wanted a Unity-style field that evaluates "5+20" via math.js; mj12albert and colmtuite noted `preventBaseUIHandler` could skip internal logic but would disable too much of the component's own behavior to be a clean fit; the asker ended up forking, and maintainers mused a wholly separate component would be the right answer rather than bolting expression evaluation onto Number Field. **This draws a firm boundary**: Number Field validates and formats typed numbers; it does not interpret arbitrary typed expressions.
- [E] **Alignment with Slider**: `e693fd9df [number field] Align value changes with Slider (#2726)` and `c2420fbc0 [slider][number field] Granular change reasons (#3132)` show the two components' event-details/reason vocabularies were deliberately kept in sync as the library-wide `eventDetails` pattern (B-P16) rolled out — not a claim that the two components are interchangeable, just that their change-callback *shape* is shared infrastructure.
- [E] Performance/precision is treated as an ongoing identity concern, not a one-time fix: the git log shows a dense, still-active 2026 cluster of parsing/formatting correctness fixes (`4e97595d7 Preserve numeric precision while keeping default display formatting`, `f937ba784 Fix formatter locale cache`, `c6e781fad Respect Intl rounding options on blur`, `0a5c4f661 Fix keyboard editing with multi-character format symbols` — all within the last few months of the mining date), signaling that correct Unicode/locale/rounding behavior is treated as core surface area worth continual investment, not an edge case.

## 3. When to use

- [E] Any numeric value that benefits from formatted display (currency, percent, units) alongside typed entry and click-to-step controls — the component exists specifically to bridge `Intl.NumberFormat`-quality display with editable numeric text (source: the entire `parse.ts`/`validate.ts`/`formatNumber` pipeline; tests cover currency (`USD`, `EUR`, `BRL`), percent, permille, unit (`km/h`), and scientific notation).
- [E] Forms needing native constraint validation (`min`/`max`/`step`/`required`) on a numeric field, with the formatted display never polluting the submitted value — confirmed by the test "should not include formatting in the submitted value" (`NumberFieldRoot.test.tsx`) and the hidden `<input type="number">` carrying the raw numeric value.
- [E] Values with a bounded, steppable range where users benefit from increment/decrement affordances and/or a drag-to-scrub gesture, not just free-text entry — the docs subtitle foregrounds "increment and decrement buttons, and a scrub area" as the component's identity, not just "a numeric input."
- [I] Quantity/amount pickers where precision matters more than glanceable range context (inferred contrast with Slider — see §4); RHF/shadcn-style forms are a confirmed real use case (`research/b-library-principles/_mining/discussions.md` #3552, "[number field] with shadcn UI forms").
- [I] Locale-sensitive numeric entry (RTL Arabic/Persian digit input, CJK Han numerals, space-grouped European formats) — heavily tested (`isNumeralChar` covers Arabic/Persian/Han/fullwidth numerals; `NumberFieldInput.test.tsx` has dedicated de-DE, fr-FR, pt-BR, pl-PL cases), suggesting real international-forms usage is an explicit target, though no maintainer statement was found naming this as a driving use case directly [G].

## 4. When not to use + alternatives

- [I] **Continuous/glanceable range selection → Slider.** From the Number Field side only (the from-Slider-side framing of this same boundary is in [`research/c-components/slider/brief.md`](../slider/brief.md) §4): Number Field's primary interaction is typed text with formatting; its scrub gesture is a secondary, pointer-only enhancement gated behind an explicit opt-in part (`NumberField.ScrubArea` must be composed in — it is not part of the minimal Root+Input+Group anatomy). Slider's primary interaction is the drag gesture itself — there is no typed-entry path on Slider at all, only drag/click-track/keyboard-step. Where a range's current position relative to its bounds matters more than an exact number, or where formatting/units/typed entry aren't needed, Slider is the fit; Number Field's format pipeline is the deciding evidence that it exists for value precision + display quality, not glance.
- [E] **Plain unformatted text/number entry → `Input` (or bare `<input type="number">`).** Number Field's entire surface (7 parts, format/parse/validate pipeline, scrub area, dual hidden+visible inputs) is overhead when no formatting, locale-awareness, or stepping affordance is needed; a plain `Input` or native `<input type="number">` suffices. No direct maintainer quote comparing the two was found in the mined corpus [G], but this follows directly from Number Field's stated purpose (formatted, steppable numeric entry) versus `Input`'s (a generic text-entry primitive).
- [E] **Expression/formula evaluation → not a documented use case for Number Field.** Per #2152 (§2), the maintainers explicitly declined to extend Number Field into math-expression territory and suggested a separate component instead; users needing that must compose their own layer on top or fork.
- [E] **Form-library interplay caveat**: #3552 ("[number field] with shadcn UI forms") shows Number Field's native constraint validation (`step` defaulting to `1`, producing a `stepMismatch`) can silently block submission in shadcn/RHF-style forms that don't call `noValidate`; the maintainer fix was `noValidate` on the form plus `format={{ maximumFractionDigits: 0 }}` for integer-only fields — this is a form-library integration pitfall to document (§10), not a reason to avoid the component, but worth surfacing prominently since it's the exact kind of "silent form submission failure" documented as a common misuse pattern library-wide (B-M8/B-P31 territory).

## 5. Anatomy & composition

- [E] Canonical tree (docs Anatomy, `page.mdx`):
  `Root > (ScrubArea > ScrubAreaCursor, Group > (Decrement, Input, Increment))`.
- [E] **Minimal composition** is just `Root > Input` — every other part is an optional enhancement layer. Confirmed directly: `NumberFieldGroup`, `NumberFieldIncrement`/`Decrement`, and `NumberFieldScrubArea`/`ScrubAreaCursor` all read from `NumberFieldRootContext` but nothing in `NumberFieldRoot.tsx` requires their presence; `NumberFieldRoot.test.tsx`'s form-submission tests render `Root > Input` alone.
- [E] **Group is a semantic wrapper, not a required layout part**: renders `role="group"` on a `<div>` (`NumberFieldGroup.tsx`) — purely an ARIA/visual grouping convenience for Decrement+Input+Increment, distinct from popup-style "explicit anatomy" mandatory parts elsewhere in the library (B-P18 does not apply here; Group's absence changes nothing structurally, only ARIA grouping semantics).
- [E] **ScrubArea/ScrubAreaCursor are a matched, fully optional pair**: `ScrubAreaCursor` throws if rendered outside a `ScrubArea` ("Base UI: NumberFieldScrubAreaContext is missing. NumberFieldScrubArea parts must be placed within `<NumberField.ScrubArea>`.", `NumberFieldScrubAreaContext.ts`) but `ScrubArea` itself has no required children — it works standalone (with pointer lock but no visible custom cursor) or paired with `ScrubAreaCursor` for the virtual-cursor visual (docs anatomy always pairs them, but this is a convention, not an enforced requirement).
- [E] **ScrubAreaCursor renders via `ReactDOM.createPortal(..., ownerDocument(domElement).body)`** — it always portals to `<body>` regardless of where `ScrubArea` sits in the tree (`NumberFieldScrubAreaCursor.tsx`), the one Portal-like behavior in this component, though it is automatic and has no explicit `Portal` part or `container` prop (unlike popup components' explicit Portal grammar, B-P18).
- [I] **Increment/Decrement share one internal implementation**: both call `useNumberFieldStepperButton(componentProps, forwardedRef, isIncrement)` (`useNumberFieldStepperButton.ts`), differing only in the boolean direction flag and which boundary (`max` vs `min`) disables them — confirmed directly from source; this explains why their DataAttributes files, ARIA labels, and disabled-boundary logic are structurally identical mirror images (`git log` commit `9945e7fc2 [number field] Deduplicate steppers and simplify internals (#5045)` names this consolidation explicitly).
- Text spec for a future anatomy diagram: (1) ScrubArea [invisible interactive layer, optional], (2) ScrubAreaCursor [portaled virtual cursor, visible only while scrubbing, optional], (3) Group [semantic wrapper, optional], (4) Decrement [button], (5) Input [text box, the only structurally required part beyond Root], (6) Increment [button]. A dotted/annotated line should mark the hidden native `<input type="number">` that Root renders as a sibling to its children — invisible, `aria-hidden`, `tabIndex={-1}` — carrying native form semantics only.

## 6. Behavior ("How it works")

- [E] **Controlled/uncontrolled**: uncontrolled by default via `defaultValue`; controlled via `value`/`onValueChange` (`useControlled` in `NumberFieldRoot.tsx`). Matches the library-wide triplet convention (B-P15).
- [E] **Two change callbacks, not one** — this is a distinctive wrinkle worth calling out: `onValueChange(value, eventDetails)` fires on every parseable intermediate change (each keystroke that parses, each step), while a *separate* `onValueCommitted(value, eventDetails)` fires only when a value is finalized — on blur after typing, on pointer release after scrubbing/button-press, and simultaneously with `onValueChange` for keyboard/wheel interactions (JSDoc on `NumberFieldRootProps.onValueCommitted`; extensively tested in `NumberFieldRoot.test.tsx`'s `describe('prop: onValueCommitted')` block, e.g. "fires on blur with committed numeric value", "fires on keyboard interactions (ArrowUp/Down/Home/End)", "fires once per press-release even after mouseleave/mouseenter during a hold"). `onValueCommitted`'s docs explicitly warn: "**Warning**: This is a generic event not a change event" — it cannot cancel the change (only `onValueChange`'s `eventDetails.cancel()` can veto).
- [E] **Reason vocabulary** (`NumberFieldRootChangeEventReason`, `NumberFieldRoot.tsx`): `input-change`, `input-clear`, `input-blur`, `input-paste`, `keyboard`, `increment-press`, `decrement-press`, `wheel`, `scrub`, `none`. Commit reasons are a subset (no `input-change`/`input-paste`, since those never directly commit). Direct text-entry reasons (`input-*`, `none`) behave differently from step-based reasons: only direct-entry reasons can produce out-of-range values when `allowOutOfRange` is set (see §8); step-based interactions always clamp (`setValue` in `NumberFieldRoot.tsx`, `isInputReason` branch).
- [E] **Format handling is `Intl.NumberFormat`-based**, confirmed directly from source (not inferred): `formatNumber`/`getFormatter` (`packages/react/src/utils/formatNumber.ts`, shared with Slider) wrap a cached `Intl.NumberFormat` instance; `parseNumber` (`utils/parse.ts`) reverses locale-specific grouping/decimal/currency/unit/percent/permille/exponent symbols and multiple numeral systems (ASCII, Arabic-Indic, Persian, fullwidth, Han) back to a JS `number`. The `format` prop is a raw `Intl.NumberFormatOptions` object passed straight through — no Base UI-specific format DSL.
- [E] **min/max/step/largeStep/smallStep semantics**: `step` (default `1`) governs plain Arrow-key/click/wheel/scrub increments; holding Shift multiplies to `largeStep` (default `10`); holding Alt uses `smallStep` (default `0.1`) — confirmed by `getStepAmount()` in `NumberFieldRoot.tsx` and dedicated `describe('prop: largeStep')`/`describe('prop: smallStep')` test blocks (Chromium-only, `describe.skipIf(isJSDOM)`, since they rely on real modifier-key event timing). `min`/`max` default to `Number.MIN_SAFE_INTEGER`/`Number.MAX_SAFE_INTEGER` internally when unset. `step="any"` disables native step validation entirely while keeping the base step amount at `1` for interactive stepping (prop JSDoc).
- [E] **`snapOnStep` behavior** (default `false`): when enabled, incrementing/decrementing snaps to the nearest multiple of `step` relative to a base (`min` if finite, else `0`), using directional snapping (floor toward the step boundary in the direction of travel) for regular steps and nearest-multiple snapping for Alt/`smallStep` moves (`snapToStep` in `utils/validate.ts`; tests "should snap when incrementing to the nearest multiple of the `step` prop").
- [E] **`allowOutOfRange` (default `false`)**: when `true`, *direct text entry only* (typing, pasting, clearing, autofill — the `input-*`/`none` reasons) may produce values outside `min`/`max` so native browser range-overflow/underflow validation can fire and surface to the user; step-based interactions (arrows, buttons, wheel, scrub) still clamp regardless (prop JSDoc; `setValue`'s `shouldClampValue` logic; extensive `describe('prop: allowOutOfRange')` tests confirming both the unclamped-typing path and the still-clamped-stepping path, plus that the minus key isn't blocked at the keyboard level when out-of-range entry is allowed even with `min={0}`).
- [E] **Floating-point cleanup**: `removeFloatingPointErrors` (`utils/validate.ts`) cleans binary floating-point noise introduced by step arithmetic (e.g., `0.1 + 0.2`) using a `toPrecision(15)` pass bounded by an epsilon-scaled delta cap, while explicitly preserving legitimate high-precision values typed or pasted directly (not touched unless the format has rounding options) — a deliberate, carefully-commented trade-off (`MAX_FLOATING_POINT_CLEANUP_DELTA` comments explain the ~5.2e5-magnitude boundary above which genuine stepping noise is left uncleaned to avoid corrupting real precision at scale).
- [E] **The scrub interaction, in mechanical detail** — the component's signature behavior:
  - On `pointerdown` (mouse only; touch is tracked separately via `isTouchInput`), the input is focused, `isScrubbing` flips true, and — outside WebKit — `document.body.requestPointerLock()` is requested (`NumberFieldScrubArea.tsx`). WebKit/Safari is explicitly excluded from pointer lock: "WebKit causes significant layout shift with the native message, so we can't use it" (source comment); `ScrubAreaCursor`'s own JSDoc independently confirms: "disabled in Safari to avoid a layout shift that this notification causes there."
  - While scrubbing, `pointermove` deltas (`movementX`/`movementY`) accumulate against a `pixelSensitivity` threshold (default `2`) before triggering a step in the configured `direction` (`'horizontal'` default, or `'vertical'`) — confirmed by `describe('prop: pixelSensitivity')` and `describe('prop: direction')` tests dispatching real `PointerEvent`s in Chromium/Firefox.
  - A virtual cursor (`ScrubAreaCursor`, a `position: fixed` `<span>` portaled to `<body>`) tracks accumulated movement and visually teleports to the opposite edge of the viewport (or a custom `teleportDistance` box) when it would otherwise leave the screen (`getViewportRect`, `onScrub` in `NumberFieldScrubArea.tsx`) — this replaces the native OS cursor, which is invisible during pointer lock.
  - On `pointerup`, pointer lock is released (`exitPointerLock()`), the scrubbing state clears, and `onValueCommitted` fires with reason `scrub`. **Firefox-specific fix**: "Firefox needs a small delay here when soft-clicking as the pointer lock will not release otherwise" — a 20ms `useTimeout` delay gates the release on Gecko (`33692bda1 [number field] Fix pointer lock release when soft clicking in Firefox (#3378)`).
  - **Quick-tap/click compatibility**: because `preventDefault()` on `pointerdown` suppresses the browser's native `click` event, the component manually re-dispatches a synthetic `click` event on `pointerup` if no scrub movement occurred, so `onClick` handlers on the ScrubArea (or children, e.g. a `<label>`) still fire for a simple tap (`e198c8357 [NumberField] Correctly handle quick touches (#1294)`; tests "should fire onClick when clicked without scrubbing" and "should fire onClick on child elements").
  - **Touch is excluded from pointer lock and the virtual cursor entirely**: `ScrubAreaCursor` explicitly does not render for touch input (test "does not render when using touch input"); touch scrubbing still works via the same `pointermove` accumulation, with `touchstart` default-prevented to stop page scroll during the gesture.
  - **Pinch-zoom is respected**: the virtual cursor inversely scales against `visualViewport.scale` so it doesn't visually grow/shrink with pinch-zoom (`subscribeToVisualViewportResize.ts`), and wheel scrubbing explicitly ignores `ctrlKey` (pinch-zoom gesture) events so as not to hijack browser zoom (tested: "does not scrub on wheel while pinch-zooming (ctrlKey)").
- [E] **`allowWheelScrub` (default `false`)**: when enabled, mouse-wheel input over a *focused* input increments/decrements by one step per discrete wheel tick, committing immediately per tick like a keyboard step (not accumulated like scrub) — implemented via a native (non-passive) `wheel` listener attached directly to the input DOM node, since React's synthetic `onWheel` is passive and can't `preventDefault()` to stop page scroll (source comment in `NumberFieldRoot.tsx`); wheel scrubbing is a no-op when the input isn't focused (tested).
- [I] **SSR**: the visible input's initial value is formatted server-side, and a `suppressHydrationWarning` is applied because "the server's locale may differ from the client's locale... causing a hydration mismatch" (source comment, `NumberFieldRoot.tsx`) — the maintainers chose to accept a possible hydration-formatting mismatch rather than flash an empty field, "preferable to rendering an empty input field and then updating it" (same comment). This matches the library's general SSR stance of internal correctness without a dedicated consumer-facing SSR doc (B-P35).

## 7. Accessibility contract

- [E] **The Input does NOT use the ARIA spinbutton pattern.** Verified directly by reading `NumberFieldInput.tsx` and grepping the entire `number-field/` source tree: there is no `role="spinbutton"`, no `aria-valuenow`, `aria-valuemin`, `aria-valuemax`, or `aria-valuetext` anywhere in source. The Input instead renders `type="text"` with `role` left as the browser default for a text `<input>` (confirmed by tests: "has textbox role" in `NumberFieldInput.test.tsx`; `screen.getByRole('textbox')` is used pervasively across the entire `NumberFieldRoot.test.tsx` suite). It additionally sets `aria-roledescription="Number field"` (a role *description*, not a role) — the JSDoc for this prop explicitly clarifies: "A user-friendly description of the input's role for assistive tech. This is a role description, not an accessible name — use `Field.Label` or `aria-label` to name the control."
  - [E] **This was a considered decision, not an oversight**: the founding PR #186's own checklist included "Research if spinbutton ARIA props become necessary for iOS when `inputMode` is 'text'" as an explicit open TODO — the shipped component never added them, and no later PR reversed that. [I] The likely rationale, inferred from the textbox-based ARIA and the format/typing-first design (§2): a spinbutton pattern's `aria-valuenow`/`aria-valuetext` model fits a value that's *set* (like a native `<input type="number">` or a slider), whereas this component's primary interaction is free-form formatted text editing where the visible text and the announced value should simply be the field's text content, read like any other text input. This is an inference from the shipped design, not a maintainer quote — no issue or PR comment explaining the final ARIA choice was found in the mined corpus [G].
  - [I] The WAI-ARIA APG **spinbutton pattern** (https://www.w3.org/WAI/ARIA/apg/patterns/spinbutton/) is the closest matching *named* ARIA design pattern for a stepper-and-value control, but Number Field does not implement it — this is a documentation-relevant gap between "what the component visually resembles" (a spinner/stepper) and "what ARIA role it actually exposes" (a plain textbox with a role description), and is worth calling out explicitly on the component's docs page so authors don't assume spinbutton semantics are present.
- [E] **ARIA actually rendered** (verified directly, `NumberFieldInput.tsx`/`useNumberFieldButton.ts`):
  - Input: implicit textbox role (native `<input type="text">`), `aria-roledescription="Number field"`, `aria-invalid` (Field integration), `aria-labelledby` (auto-linked from `Field.Label` via `useLabelableContext`), `required`/`disabled`/`readOnly` as native attributes.
  - Increment/Decrement buttons: `aria-label` ("Increase"/"Decrease" — hardcoded, not currently localizable via a prop [I, inferred from source; no i18n prop found for these labels]), `aria-controls` (pointing at the Input's `id`), `tabIndex={-1}` — deliberately excluded from the Tab order, with the source comment explaining why: "Keyboard users shouldn't have access to the buttons, since they can use the input element to change the value. On the other hand, `aria-hidden` is not applied because touch screen readers should be able to use the buttons." `aria-disabled` (not native `disabled`, so the buttons stay in the a11y tree while unavailable) appears at range boundaries; when `readOnly`, the buttons are exposed as unavailable through disabled semantics rather than `aria-readonly` (which is invalid on the `button` role) — confirmed by test "does not render aria-readonly on stepper buttons".
  - Group: `role="group"` (plain semantic grouping, `NumberFieldGroup.tsx`).
  - ScrubArea/ScrubAreaCursor: `role="presentation"` on both (confirmed by tests "has presentation role" in each) — the scrub gesture is explicitly **decorative/non-essential** from an ARIA standpoint, not a primary interaction path a screen-reader or keyboard-only user is expected to discover through the accessibility tree.
- [E] **Keyboard interaction table** (evidence: `NumberFieldInput.tsx` `onKeyDown`, `NumberFieldInput.test.tsx`, `NumberFieldRoot.test.tsx`):
  | Key | Context | Action |
  |---|---|---|
  | Any numeral (any supported numeral system), decimal/group/sign/currency/unit/percent symbols the current format allows | Input focused | Typed directly into the text field; parseable intermediate values fire `onValueChange` with reason `input-change` |
  | ArrowUp | Input focused | Increments by `step` (or `largeStep` with Shift, `smallStep` with Alt); reason `keyboard`; commits immediately (`onValueCommitted` fires) |
  | ArrowDown | Input focused | Decrements symmetrically; same reason/commit behavior |
  | Home | Input focused, `min` is set | Jumps directly to `min`; reason `keyboard`, commits immediately. **No-op if `min` is unset** — the browser's native text-caret Home behavior takes over instead (source comment: "Let the browser handle ... Home/End without min/max") |
  | End | Input focused, `max` is set | Jumps directly to `max`, symmetric to Home |
  | PageUp / PageDown | Input focused | **Not implemented as stepping keys** — explicitly falls through to native browser handling (source comment lists PageUp alongside Insert/F-keys as keys the component deliberately does not act on). This diverges from the ARIA spinbutton pattern's *optional* PageUp/PageDown-as-large-step convention, consistent with Number Field not implementing that pattern at all (see above). |
  | Backspace / Delete / ArrowLeft / ArrowRight / Tab / Enter / Escape | Input focused | Native text-editing/navigation behavior, explicitly passed through (`NAVIGATE_KEYS` set in `NumberFieldInput.tsx`) |
  | Mouse wheel | Input focused, `allowWheelScrub` enabled | One step per discrete tick, direction by scroll sign, commits immediately per tick |
  | Click-drag on ScrubArea | Pointer down inside `ScrubArea` | Continuous stepping proportional to pointer movement; pointer-only, `role="presentation"` (decorative) — see the a11y caveat immediately below |
  | Click-and-hold on Increment/Decrement | Pointer down on stepper button | Repeats at a fixed tick delay after an initial start delay (`usePressAndHold`); commits once on release |
- [E] **The scrub interaction has no keyboard equivalent, and is marked accordingly**: both `ScrubArea` and `ScrubAreaCursor` render `role="presentation"`, and the stepper buttons are deliberately removed from the Tab order (`tabIndex={-1}`) with the explicit rationale that keyboard users have the Input's own Arrow/Home/End stepping as their equivalent path. This means the scrub gesture is architecturally treated as a *pointer-only convenience layered on top of an already-complete keyboard/typing interaction*, not a required or primary path — consistent with the library's two-sided a11y-contract framing (B-P27) where the library guarantees a complete keyboard path and the scrub gesture is additive. No ARIA APG pattern directly matches drag-to-scrub; the closest named pattern remains spinbutton, which (per above) this component does not use.
- [I] The absence of `aria-label`-equivalent localization props on Increment/Decrement ("Increase"/"Decrease" hardcoded English strings) is a documentation-relevant limitation for non-English-first apps — inferred from source, no open issue found discussing it in the mined corpus [G].

## 8. Prop-level guidance (decision-relevant props)

- [E] **`min` / `max`** — bound the value; unset defaults to `Number.MIN_SAFE_INTEGER`/`Number.MAX_SAFE_INTEGER` internally, but the *native* hidden input only receives the literal `min`/`max` you pass (unset stays unset there too), which matters for native `stepMismatch`/`rangeOverflow`/`rangeUnderflow` validity — confirmed via the hidden-input JSX in `NumberFieldRoot.tsx` and the source comment "stepMismatch validation is broken unless an explicit `min` is added. See https://github.com/react/react/issues/12334."
- [E] **`step`** (default `1`, or `'any'`) — governs stepping amount and (unless `'any'`) native step validation; `'any'` disables step validation entirely while keeping the interactive step amount at `1`. The prop JSDoc explicitly ties this to a common footgun: "To always enable step validation on form submission, specify the `min` prop explicitly in conjunction with this prop" — directly explains the #3552 shadcn-forms pitfall (§4, §10).
- [E] **`largeStep`** (default `10`, Shift-modified) / **`smallStep`** (default `0.1`, Alt-modified) — secondary step amounts; `smallStep` is also the snapping base for `snapOnStep` when Alt is held.
- [E] **`snapOnStep`** (default `false`) — snap to the nearest step multiple on increment/decrement; see §6 for directional-vs-nearest snapping detail.
- [E] **`allowOutOfRange`** (default `false`) — lets direct text entry (typing/paste/clear/autofill) exceed `min`/`max` so native range-overflow/underflow validation fires; step interactions always clamp regardless. Added relatively recently: `c8fa08756 [number field] Add allowOutOfRange prop (#3919)`.
- [E] **`format`** (`Intl.NumberFormatOptions`, no default) — the entire formatting/locale surface is delegated to the standard `Intl.NumberFormat` options object; no Base UI-specific format mini-language. Combine with **`locale`** (`Intl.LocalesArgument`, defaults to runtime locale) to control both together.
- [E] **`allowWheelScrub`** (default `false`) — enables wheel-based single-step-per-tick scrubbing while the input is focused; distinct from the drag-based `ScrubArea` gesture (§6).
- [E] **`disabled` / `readOnly`** — `disabled` also merges with Field's own disabled state (`fieldDisabled || disabledProp`, `NumberFieldRoot.tsx`); `readOnly` on the Root disables the stepper buttons via computed disabled state but is deliberately *not* exposed as `aria-readonly` on those buttons (invalid ARIA for `role="button"`) — only the Input itself carries native `readonly`.
- [E] **`onValueChange` vs `onValueCommitted`** — the two-callback split is the single most decision-relevant behavioral prop pair in this component (§6); document them together, never independently, since their relationship (fires-every-parseable-change vs fires-once-on-finalize) is easy to misread as redundant.
- [E] **`inputRef`** — ref to the *hidden* native `<input type="number">`, not the visible text Input (JSDoc: "A ref to access the hidden input element"); the visible Input's own DOM ref is obtained the normal way (`ref` on `NumberField.Input` itself).
- [E] **`name` / `form`** — standard hidden-input form-participation props; `form` lets the hidden input target an external `<form>` by id when the component isn't rendered inside one (mirrors the library-wide external-form pattern, tested: "submits to an external form when `form` is provided").
- [E] **Data attributes that matter for styling** (identical set across every part's `*DataAttributes.ts` file — Root, Group, Input, Increment, Decrement, ScrubArea, ScrubAreaCursor all export the same ten): `data-scrubbing` (present on **every part**, not just ScrubArea, while a scrub gesture is active — useful for e.g. dimming the Increment/Decrement buttons while scrubbing), `data-disabled`, `data-readonly`, `data-required`, plus the Field-integration set `data-valid`/`data-invalid`/`data-touched`/`data-dirty`/`data-filled`/`data-focused` (only populated when wrapped in `Field.Root`). There is no per-part-unique data attribute (e.g. no Increment-only or Input-only attribute) — the entire state surface is Root-level state duplicated onto every part, per the library-wide doctrine (B-P13).
- [E] **No CSS-variable file exists for this component** — confirmed directly by directory listing (`find packages/react/src/number-field -iname "*cssvars*"` returns nothing) and by the absence of any `--`-prefixed custom property in `NumberFieldRoot.tsx`/`NumberFieldScrubArea.tsx`/etc. This is consistent with the component having no popup/positioning surface that would need measurement variables (contrast Select/Popover's `--available-width`, B-P14) — Number Field's only dynamic visual element (the scrub cursor) is positioned via a directly-set inline `transform` style (`updateCursorTransform` in `NumberFieldScrubArea.tsx`), not a CSS variable. Tag this as [E] absence, not [G]: the search was conclusive, not merely unperformed.

## 10. Pitfalls & FAQ

- **Form silently fails to submit / unexpected native validation popup** → almost always a `step`/`stepMismatch` issue: the default `step={1}` triggers native step validation once an explicit `min` is present (see the `NumberFieldRoot.Props.step` JSDoc's own footnote about this), and shadcn/RHF-style forms that don't call `noValidate` will block submission with a native browser popup the developer didn't expect. Fix: add `noValidate` to the `<form>` if you want Base UI's own `Field`/`Form` validation to be authoritative, and/or set `format={{ maximumFractionDigits: 0 }}` for integer-only fields, and/or set `step="any"` to opt out of step validation entirely (mui/base-ui#3552). [E]
- **"Why doesn't Number Field export a hook like `useNumberInput`?"** → by design; no hooks are exported from any current-generation Base UI component, this one included — compose a controlled `NumberField.Root` with your own inner elements instead of reaching for internal hooks (mui/base-ui#1814). The internal `useNumberFieldButton`/`useNumberFieldStepperButton` exist but are not public API. [E]
- **"Can I make it evaluate math expressions like '5+20'?"** → no, and this is an acknowledged boundary, not a bug: maintainers considered and declined it, citing that the override mechanism (`preventBaseUIHandler`) would disable too much of the component's own logic to be a clean fit, and mused a separate component would be the right vehicle instead (mui/base-ui#2152). [E]
- **Expecting `role="spinbutton"` / `aria-valuenow` to be present** → they are not, and never have been, despite the component's spinner-like visual affordances; the Input is a plain textbox with `aria-roledescription="Number field"`. Screen-reader testing/automation that specifically queries for a spinbutton role will not find this component. Verified directly from source (§7), not inferred. [E]
- **Expecting PageUp/PageDown to do a "large step"** → they don't; only Home/End (when `min`/`max` are set) and Shift+Arrow (`largeStep`) provide bigger jumps. PageUp/PageDown fall through to native browser text-field behavior untouched. [E]
- **Expecting `onValueCommitted` to be cancelable** → it is not; only `onValueChange`'s `eventDetails.cancel()` can veto a change. The commit callback's own JSDoc calls this out explicitly ("This is a generic event not a change event"). [E]
- **Styling `ScrubAreaCursor` and having it not appear in Safari, or on touch** → this is intentional, not a bug: pointer lock (and therefore the virtual cursor) is disabled in WebKit to avoid a native layout-shifting notification, and is never used for touch input at all. [E]
- **`data-scrubbing` unexpectedly present on the Increment/Decrement buttons, not just ScrubArea** → intentional; every part shares the same Root-level `data-scrubbing` state (per the library-wide duplicated-data-attribute doctrine, B-P13), so styling e.g. a dimmed stepper during a scrub gesture is possible directly. [E]
- **Increment/Decrement buttons unreachable by Tab** → intentional (`tabIndex={-1}`); the keyboard path is the Input's own Arrow/Home/End handling, not the buttons — see §7's rationale quote. [E]
