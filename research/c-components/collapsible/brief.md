## 1. Identity

- **Name / subpath**: Collapsible — `@base-ui/react/collapsible`. Multi-part compound component, namespace export `Collapsible.*`.
- **Parts** (`index.parts.ts`, one-liners from part JSDoc):
  - `Collapsible.Root` — "Groups all parts of the collapsible." Renders a `<div>`. Owns open/closed state, disabled, and the shared transition status.
  - `Collapsible.Trigger` — "A button that opens and closes the collapsible panel." Renders a `<button>`.
  - `Collapsible.Panel` — "A panel with the collapsible contents." Renders a `<div>`. Owns height/width measurement, the CSS-transition-vs-CSS-animation auto-detection engine, and `hiddenUntilFound`/`keepMounted`.
- **Status**: stable, no `[New]`/`[Preview]` tag. Founding-era, but with an unusually long and still-active revision history for a 3-part component: introduced `8224de7d2 [Collapsible] Add Collapsible components and hooks (#481)` (2024-08-28), and still receiving substantive behavior fixes as recently as `e18d78832 [collapsible] Fix trigger and panel state bugs (#4848)` and `3d0be4e37 [collapsible] Refactor panel logic (#4565)` — both within the last few months of this writing [E] (git log). This is one of the most actively-revised Tier-3 components in the corpus; the churn is concentrated entirely in the Panel's animation-measurement engine (§6), not in the API surface, which has been stable since early on.
- **Taxonomy** (Phase A): **disclosure & structure** cluster (alongside accordion, tabs, separator, toolbar, scroll-area). Purpose: a single, independent show/hide panel controlled by one button. IA: the smallest possible disclosure primitive — one Trigger, one Panel, no coordination with any sibling.
- **Disclosure-cluster relationship to Accordion — the defining fact of this component's identity**: **Accordion does not reimplement Collapsible's logic; it literally imports and reuses it.** `packages/react/src/accordion/item/AccordionItem.tsx` calls `useCollapsibleRoot` directly (imported from `../../collapsible/root/useCollapsibleRoot`); `packages/react/src/accordion/trigger/AccordionTrigger.tsx` and `.../accordion/panel/AccordionPanel.tsx` both call `useCollapsibleRootContext` from `../../collapsible/root/CollapsibleRootContext`, and `AccordionPanel.tsx` calls `useCollapsiblePanel` directly from `../../collapsible/panel/useCollapsiblePanel` [E] (verified via `grep -rn "useCollapsibleRoot\|useCollapsiblePanel\|useCollapsibleRootContext" packages/react/src/accordion/`). Concretely: **Collapsible is the primitive; Accordion is the coordinated-set wrapper built on top of it** — each Accordion item is, internally, one Collapsible whose open/close is additionally coordinated by the Accordion Root's single/multiple-open bookkeeping. The docs subtitles make the same distinction in plain language: Collapsible is "**A** collapsible panel controlled by a button" (singular); Accordion is "**A set of** collapsible panels with headings" (plural, coordinated) [E] (`collapsible/page.mdx:3`, `accordion/page.mdx:3`). This is Collapsible's side of the disclosure cluster comparison (§4) — the parallel agent's `_clusters/disclosure.md` should state the inverse (Accordion's side) and can link back here for the implementation-sharing evidence.

## 2. Intention

- [E] **Origin**: `8224de7d2 [Collapsible] Add Collapsible components and hooks (#481)` (2024-08-28) — no rich RFC-style body was located for this introducing PR beyond the commit itself [G] (searched `gh search issues`/`git log --grep` for a standalone "[collapsible] Implement" design discussion; found none — consistent with the pattern seen in Field/Fieldset, where the earliest components shipped without an extensive public design write-up).
- [E] **Stated purpose, verbatim**: "A collapsible panel controlled by a button" (docs Subtitle, `page.mdx:3`) — the plainest possible statement: one button, one panel, no further structure implied.
- [E] **Explicit non-goal, stated by contrast with its own sibling**: Collapsible does not coordinate with other Collapsibles — that coordination is exactly what Accordion adds. The shared-implementation fact (§1) is itself the strongest evidence of intention: the maintainers built the open/close/measure/animate primitive exactly once and reused it, rather than letting Accordion duplicate the logic — matching the library's stated anti-duplication stance ("If two components can share logic..., define the logic/handlers in the parent and share it through a context to the child," AGENTS.md) applied at the cross-component level rather than within a single component.
- [E] **The component's real complexity center is measurement-driven animation, not open/close state** — the open/close state machine itself (`useCollapsibleRoot.ts`) is ~40 lines of ordinary controlled/uncontrolled boilerplate (principles B-P15's `x`/`onXChange`/`defaultX` triplet, applied to `open`); nearly all of Collapsible's git history (§9) and its 1190-line Panel test file are about correctly measuring and animating height/width across CSS transitions, CSS keyframe animations, SSR, `React.Activity`, and the `beforematch` find-in-page event. [I] This asymmetry — trivial state, sophisticated measurement — is the clearest signal of where the maintainers judged the hard problem to actually live: not "is it open," but "how do you smoothly animate a `height: auto` panel," a CSS-native problem with no built-in browser solution.
- [I] **The heart of the intention**: give a single button-controlled panel the two things `<details>`/`<summary>` can't provide out of the box — animatable open/close transitions (native `<details>` can't transition `height: auto`) and full styling control over the trigger — while still supporting the one thing `<details>` uniquely offers that most custom disclosure widgets drop: participation in the browser's native find-in-page search via `hiddenUntilFound` (§6). Inferred from the `hiddenUntilFound`/`beforematch` implementation directly mirroring `<details>`'s own `hidden="until-found"` semantics (the prop's JSDoc literally says "uses `hidden='until-found'`" — the same attribute value `<details>` uses internally).

## 3. When to use

- [E] A single, independently-toggleable panel — settings sections, "show more" content, expandable detail panels — controlled by exactly one trigger button (docs Subtitle; the component's entire API surface assumes a 1:1 Trigger:Panel relationship, with no `value`/identity prop of any kind, unlike Accordion's `AccordionItem` which requires a `value`).
- [E] When the panel's content needs to participate in the browser's built-in Ctrl/Cmd+F find-in-page search even while collapsed — `hiddenUntilFound` exists specifically for this (§6, §8).
- [E] When you need the panel to stay in the DOM while closed (for animation libraries, for preserving scroll position/focus/form state inside the panel, or for SEO/crawlability reasons) — `keepMounted` (§6, §8).
- [I] As the building block for a custom accordion-like UI where you want more control over cross-item coordination than `Accordion` provides — since Accordion is built from the same primitive (§1), a consumer could in principle wire several `Collapsible.Root`s together with their own coordination logic instead of using `Accordion`, though no maintainer statement recommends this over just using Accordion, and doing so forfeits Accordion's already-built single/multiple-open bookkeeping. [G] no issue or docs passage was found recommending or discouraging this pattern directly.

## 4. When not to use + alternatives — Collapsible's side of the disclosure cluster

- **A coordinated set of same-purpose panels (where opening one may need to close another, or where "how many are open" is itself meaningful) → Accordion, not multiple independent Collapsibles.** Accordion adds exactly the single/multiple-open coordination Collapsible deliberately omits, while reusing Collapsible's own open/close/measurement engine underneath (§1). [E] the plural-vs-singular subtitle distinction (§1, §2) plus the direct code-sharing are the two strongest pieces of evidence; [G] no maintainer issue/PR comment was found stating "use Accordion instead of N Collapsibles" in so many words — the boundary is drawn by API shape (Accordion requires per-item `value`; Collapsible has none) rather than by an explicit written rule.
- **A tabbed, single-visible-panel-at-a-time UI where all panel labels are always visible together → Tabs, not Collapsible.** [I] inferred from taxonomy adjacency (both are "disclosure & structure") and the general shape difference (Tabs shows all triggers with one panel visible; Collapsible/Accordion show/hide their own panel(s) inline); no direct citation comparing them was found in this pass — [G] this exact comparison likely belongs in the shared `_clusters/disclosure.md` rather than being re-derived per component; flagging rather than fabricating a maintainer quote.
- **Content that should always be visible needs no Collapsible at all** — trivial but worth stating: Collapsible (and its `hiddenUntilFound` variant) is for content the user may reasonably want hidden by default; don't reach for it as a generic "put a button above a div" pattern when the content should simply always render.
- **You need the platform's own zero-JS disclosure semantics and don't need custom trigger styling or animation timing control** → plain `<details>`/`<summary>` may suffice; Collapsible's whole value-add over `<details>` is styleable triggers plus animatable, JS-observable open/close transitions (§2) — if neither matters for a given use case, the native element is simpler. [I] — a direct consequence of the "why does this exist over the native element" scope test (B-P4), not a maintainer quote.

## 5. Anatomy & composition

```jsx
import { Collapsible } from '@base-ui/react/collapsible';

<Collapsible.Root>
  <Collapsible.Trigger />
  <Collapsible.Panel />
</Collapsible.Root>;
```

(docs Anatomy, `page.mdx:15-24` — flat, 3-part tree; no Portal/Positioner layering, since Collapsible is inline content, not a popup.)

- **Hero demo** (`docs/.../collapsible/demos/hero/css-modules/index.tsx`) — the canonical shape, reproduced in full since it is also the exact animation recipe (§6, §8):
  ```jsx
  <Collapsible.Root className={styles.Collapsible}>
    <Collapsible.Trigger className={styles.Trigger}>
      Recovery keys
      <CaretRightIcon className={styles.Icon} />
    </Collapsible.Trigger>
    <Collapsible.Panel className={styles.Panel}>
      <div className={styles.Content}>
        <div>alien-bean-pasta</div>
        <div>wild-irish-burrito</div>
        <div>horse-battery-staple</div>
      </div>
    </Collapsible.Panel>
  </Collapsible.Root>
  ```
  [E] Note the extra inner `<div className={styles.Content}>` wrapping the actual content, distinct from `Collapsible.Panel` itself — Panel carries the `overflow: hidden` + animated `height` (§6), while `Content` carries the padding/gap; this separation exists so padding doesn't distort the height measurement Panel performs (`scrollHeight`/`scrollWidth` on the Panel node, §6) — an unstated-but-inferable reason for the two-node pattern. [I]
- **Nesting**: Collapsibles can be nested inside one another's Panel (exercised directly in the "nested" experiment, `transitions.tsx:109-123` — a `Collapsible.Root` containing, inside its Panel's `Content`, a second independent `Collapsible.Root`) [E] — no special API is needed; each level manages its own state independently, consistent with Collapsible having no `value`/identity prop to collide.
- **No mandatory Portal**: unlike every popup component (Select, Popover, Dialog, Menu, …), Collapsible renders inline in normal document flow — it is not gated by the library's Portal-required doctrine (principles B-P18) since it isn't a popup; its content is always part of the same layout box the Trigger lives in.
- **Trigger↔Panel wiring**: `Collapsible.Trigger` sets `aria-controls={open ? panelId : undefined}` and `aria-expanded={open}` (§7); `panelId` is either an auto-generated id (`useBaseUiId` fallback) or the `id` prop explicitly set on `Collapsible.Panel`, registered back into Root's context via `setPanelIdState` [E] (`useCollapsibleRoot.ts:25-27`; `CollapsiblePanel.tsx:61-69`) — mirroring the same registration-by-mount-effect pattern seen in Field (Label→Control) and Fieldset (Legend→Root).
- **Visual anatomy diagram spec**: (1) Trigger button (with an optional rotating disclosure-caret icon, per hero demo) → (2) Panel box directly beneath, height driven by `--collapsible-panel-height` → (3) an inner Content wrapper carrying visual padding, offset from Panel itself → (4) a side callout showing the `aria-expanded`/`aria-controls` arrow from Trigger to Panel.

## 6. Behavior ("How it works")

- [E] **Controlled/uncontrolled**: `open`/`onOpenChange`/`defaultOpen` triplet on Root, default `defaultOpen: false` (`CollapsibleRoot.tsx:85-103`). `onOpenChange(open, eventDetails)` follows the standard `eventDetails` grammar; the only reasons ever produced are `REASONS.triggerPress` (a real trigger click/keypress) and `REASONS.none` (the `beforematch`-driven open, see below) — `CollapsibleRootChangeEventReason = typeof REASONS.triggerPress | typeof REASONS.none` (`CollapsibleRoot.tsx:106`). `eventDetails.cancel()` vetoes the state change while staying uncontrolled — pinned by `CollapsibleRoot.test.tsx` "eventDetails.cancel() prevents opening/closing while uncontrolled".
- [E] **`disabled`**: suppresses the trigger's click handling entirely — `CollapsibleRoot.test.tsx` "does not toggle or call onOpenChange when clicked while disabled" and the keyboard-interaction describe block repeats the same assertion per key. `Collapsible.Trigger`'s own `disabled` prop defaults to the context's `disabled` (`contextDisabled`) but can be overridden per-Trigger — though since there is exactly one Trigger per Collapsible, this override has little practical use beyond matching the general per-part `disabled`-prop convention other components share.
- [E] **`mounted` vs `open` — the two-state model that drives everything else**: `open` is the logical/controlled state; `mounted` (from `useTransitionStatus`) additionally stays `true` for the duration of a close animation so the Panel isn't yanked out of the DOM before its exit transition finishes — "Whether the collapsible panel is mounted for transition and hidden-state purposes. This can be `false` while the element remains in the DOM when `keepMounted` or `hiddenUntilFound` is enabled" (`useCollapsibleRoot.ts:99-103` JSDoc). `Panel`'s own `shouldRender = keepMounted || hiddenUntilFound || mounted || open` (`useCollapsiblePanel.ts:382`) is the single line that determines whether Panel returns `null` at all.
- [E] **Animation-type auto-detection**: on every open/close, `getAnimationType()` inspects the Panel's own computed styles (`getComputedStyle(element).animationName`/`.animationDuration`/`.transitionDuration`) to classify the situation as `'css-transition' | 'css-animation' | 'none'` — and **warns in development if both a non-zero-duration transition and a non-zero-duration animation are detected simultaneously**: "CSS transitions and CSS animations both detected on Collapsible or Accordion panel. Only one of either animation type should be used." (`useCollapsiblePanel.ts:408-430`) — an explicit, enforced constraint on how consumers may combine the two animation mechanisms.
- [E] **Height/width measurement**: `getDimensions()` reads `element.scrollHeight`/`scrollWidth` (`useCollapsiblePanel.ts:401-406`) and writes the results into two CSS custom properties on the Panel's inline `style` — `--collapsible-panel-height` and `--collapsible-panel-width` — falling back to the literal string `'auto'` when no measurement is active (`CollapsiblePanel.tsx:111-116`). This is the mechanism a consumer's CSS hooks into for the transition recipe (§8).
- [E] **CSS-transition path**: on open, `resetLayoutStyles()` temporarily neutralizes `justify-content`/`align-items`/`align-content`/`justify-items` (flex/grid alignment properties that would otherwise distort a `scrollHeight` measurement) for one animation frame, measures, then restores them; on close, dimensions are captured *before* the deferred "ending" phase applies closed styles, so the transition always starts from a real measured pixel value rather than jumping from `auto` (`useCollapsiblePanel.ts:195-207`, `230-245`).
- [E] **CSS-keyframe-animation path is handled differently and more defensively** than the transition path, because keyframes replay from scratch on every mount rather than smoothly interpolating from a live value: initially-open panels **skip their first-paint mount animation** entirely (to avoid a layout-shift flash on first render, `shouldPreventMountAnimationRef`) while still caching the expanded size so the *first close* can animate from a real pixel value instead of `auto` (`useCollapsiblePanel.ts:171-179`); `React.Activity` teardown/resume is separately guarded (`shouldPreventActivityResumeAnimationRef`) so an already-open keyframe panel that gets Activity-hidden-then-shown again doesn't replay its open animation a second time — "React.Activity unmounts Effects while preserving component state. If that teardown happens while an already-open keyframe panel is visible, remember to suppress the replayed open animation on the next committed reveal." (`useCollapsiblePanel.ts:124-131`).
- [E] **`beforematch` / find-in-page interaction** (only relevant when `hiddenUntilFound` is set): the browser's native find-in-page fires a `beforematch` event on a `hidden="until-found"` element right before revealing it; Collapsible listens for this and calls `onOpenChange(true, eventDetails)` + `setOpen(true)`, additionally flagging `shouldSkipNextOpenRef` so **that specific reveal skips the author's authored transition/animation duration** (forced to `0s` temporarily) — "`beforematch` opens should reveal the panel immediately so find-in-page does not wait for the author-defined transition or animation to finish" (`useCollapsiblePanel.ts:184-187`). The temporary zero-duration override is then carefully restored before the *next* real close, so a later user-triggered close still animates normally (`useCollapsiblePanel.ts:203-206`, and the dedicated test block `describe('prop: hiddenUntilFound')`/the `keeps the temporary zero animation duration until the panel closes` and `restores the transition duration before the first close after a beforematch open` tests).
- [E] **`keepMounted={false}` + `hiddenUntilFound` conflict is explicitly guarded and warned**: "The `keepMounted={false}` prop on `Collapsible.Panel` is ignored when `hiddenUntilFound` is enabled, since the panel must remain mounted while closed." (`CollapsiblePanel.tsx:35-44`, a dev-mode-only `useIsoLayoutEffect` + `warn()` call) — `hiddenUntilFound` structurally requires the element to stay in the DOM (browser find-in-page can't search removed nodes), so it silently overrides an explicit `keepMounted={false}` while telling the consumer it did so.
- [E] **`data-ending-style` frame timing fix, with a filed root cause**: closing-transition detection for `CollapsiblePanel` waits one animation frame before watching for animation completion specifically because "Chrome can still register the exit transition one frame later when an Accordion closes one item while opening another" — citing [mui/base-ui#3099](https://github.com/mui/base-ui/issues/3099) directly in a source comment (`useCollapsiblePanel.ts:299-301`).
- [E] **SSR**: initially-open panels suppress their mount-animation both via computed-style detection (`shouldPreventMountAnimationRef.current = open` at first render, `useCollapsiblePanel.ts:58`) and via a check against inline styles specifically — the test suite has a dedicated `describe('server-side rendering')` block with both "suppresses the initial keyframe animation when rendered open" and "...from inline styles when rendered open," confirming this was hardened against more than one code path producing the same visible bug (a flash of the open animation on first paint).

## 7. Accessibility contract

Collapsible does not map to a single named WAI-ARIA APG pattern the way, say, Select maps to Combobox — but its shape (one button toggling the visibility of associated content) is exactly the interaction the APG catalogs as the **Disclosure (Show/Hide) Pattern**: https://www.w3.org/WAI/ARIA/apg/patterns/disclosure/. [I] connecting Collapsible's implementation to this named pattern; no direct maintainer citation of the APG disclosure page was found in source comments or docs — [G] searched `page.mdx` and source files for "disclosure" or a W3C APG URL; none present. This is itself a documentation gap worth flagging: unlike Select (which does cite its APG combobox pattern URL, per the select brief §7), Collapsible's docs make no APG reference at all today.

**What Collapsible manages automatically:**

| Mechanism | How |
|---|---|
| Trigger↔Panel relationship | `aria-controls` on Trigger points at Panel's id, present only while `open` (`CollapsibleTrigger.tsx:55`) |
| Expanded/collapsed state | `aria-expanded={open}` on Trigger, always present regardless of open state (`CollapsibleTrigger.tsx:56`) — pinned by `CollapsibleRoot.test.tsx`'s "sets ARIA attributes" test |
| Native button semantics | `Trigger` uses the shared `useButton` hook with `nativeButton = true` default and `focusableWhenDisabled: true` — disabled triggers stay focusable/discoverable (the same composite-vs-standalone convention documented library-wide, principles B-P29), though as a single, standalone (non-composite) trigger this mainly matters for consistency rather than roving-tabindex needs |
| Hidden-panel semantics | Panel sets the native `hidden` boolean attribute when `!open && !mounted`, or the string value `"until-found"` (forced via `setAttribute` due to a React limitation — "React only supports a boolean for the `hidden` attribute," `useCollapsiblePanel.ts:352-354`, citing a live React issue) when `hiddenUntilFound` is active |

**Keyboard interaction table** (evidence: `useButton`'s native-button delegation + `CollapsibleRoot.test.tsx`'s dedicated `describe.skipIf(isJSDOM)('keyboard interactions')` block, which parametrizes over the relevant keys):

| Key | Context | Action |
|---|---|---|
| Enter / Space | Trigger focused | Toggles the panel (native `<button>` activation via `useButton`; blocked entirely when `disabled`, test-pinned per key) |
| Tab | anywhere | Ordinary single-tab-stop button focus — Collapsible has no composite/roving-tabindex behavior since there is only one interactive element |

**Screen reader notes**: a Trigger with visible text content (the hero demo's "Recovery keys" + caret icon) gets its accessible name from that text automatically — no `aria-label`/`Field.Label` wiring is needed or provided, since Trigger is a plain button with children, not a Field-integrated control. [I] — consequence of Trigger rendering a bare `<button>`; no dedicated labeling mechanism exists in source because none is needed for ordinary button content.

**Known issues / honesty**: [E] **#4915** (open, `type: expected behavior`) — a user expected `data-starting-style` to persist through the *entire* opening animation (to gate `overflow-hidden` for its duration); maintainers (mj12albert, atomiks) clarified it exists for exactly one frame — the transition *source* state, not a duration — and atomiks conceded a docs sentence describing it "is definitely imprecise and should be updated" [E] (cited via `research/b-library-principles/_mining/discussions.md:317-321`; this brief did not re-read the issue directly, relying on the already-mined summary). [E] **#1608** (open, `has workaround`) — Tailwind v4's `[hidden]{display:none!important}` reset defeats JS animation libraries' (e.g. Motion) inline styles, breaking exit animations entirely; workaround is `hidden={undefined}` on the Panel [E] (mined summary, same file). [E] **#4553** (closed, `type: bug`) — "[collapsible] Panel does not close when controlled by external trigger and keepMounted," title/labels only retrieved via cached search results (`research/b-library-principles/_mining/_cache/search-term-animation-issues.json`); body unread — flagged as a real, since-fixed bug rather than an open gap, relevant to the "controlled" story plan item. [E] **#2984** (closed, `type: bug`) — "[collapsible] Starting styles do not apply when using Tailwind," title/labels only, same cache file — another instance of the Tailwind-interaction theme, since fixed. [G] no dedicated screen-reader-specific (NVDA/JAWS/VoiceOver) bug report was found for Collapsible in the mined corpus, unlike several other components (e.g. Select's #4754/#4122-adjacent threads) — either genuinely absent or simply not yet surfaced by the searches run so far.

## 8. Prop-level guidance

**`Collapsible.Root`**
- `open` / `onOpenChange` / `defaultOpen` (default `false`) — the standard controlled triplet; use `onOpenChange`'s `eventDetails.reason` to distinguish a real trigger press (`'trigger-press'`) from a `beforematch`-driven reveal (`'none'`) if your app needs to react differently to the two (e.g. skip an analytics event for find-in-page-driven opens).
- `disabled` (default `false`) — suppresses trigger interaction entirely; does not hide the panel or change its mounted state on its own.

**`Collapsible.Trigger`**
- `nativeButton` (default `true`) — the library-wide convention (principles B-P10): set to `false` only when `render`-swapping Trigger to a non-`<button>` element, so ARIA/keyboard adapt correctly.

**`Collapsible.Panel`** — the two props that matter most for this component, both decision-relevant:
- `hiddenUntilFound` (default `false`) — "Allows the browser's built-in page search to find and expand the panel contents. Overrides the `keepMounted` prop and uses `hidden='until-found'` to hide the element without removing it from the DOM." Use when panel content should be discoverable via Ctrl/Cmd+F even while collapsed (FAQ-style pages, documentation sections) — this is the single most distinctive prop Collapsible offers over a hand-rolled show/hide `div`. It silently overrides `keepMounted={false}` (with a dev warning, §6) because the DOM-presence requirement is structural, not optional, once this prop is set.
- `keepMounted` (default `false`) — "Whether to keep the element in the DOM while the panel is hidden. This prop is ignored when `hiddenUntilFound` is used." Use when: driving the panel with a JS animation library that needs the exit-animation node present (`AnimatePresence`-style patterns, see the `animate-presence.tsx` experiment, §9 "Real-world/manual-testing evidence"); preserving scroll position, focus, or uncontrolled form state inside the panel across closes; or when the panel's content should be crawlable/present in the DOM for reasons unrelated to animation.
- **Styling contract (data attributes, identical shape across Root/Trigger/Panel where applicable)**: `data-open`/`data-closed` (Root, Panel), `data-panel-open` (Trigger only — deliberately named differently from Root/Panel's `data-open`, since it describes "my associated panel's state," not "my own state," letting a Trigger be styled based on the panel it controls) [E] (`CollapsibleTriggerDataAttributes.ts`), `data-starting-style`/`data-ending-style` (Root, Panel — the standard one-frame/transition-duration semantics, principles B-P25, subject to the imprecision the maintainers themselves conceded in #4915).
- **CSS variables (Panel only)**: `--collapsible-panel-height`, `--collapsible-panel-width` — both typed `{number}` in JSDoc despite being consumed as pixel-suffixed strings in the actual recipe (`CollapsiblePanelCssVars.ts`; consumed via `height: var(--collapsible-panel-height)` in the hero demo CSS, §5) — these are the exact "measurements travel as CSS variables" pattern the principles doc names as a library-wide convention (B-P14), applied here to solve the one problem CSS alone can't: animating to/from `height: auto`.
- **The canonical animation recipe, verbatim from the hero demo CSS** (`docs/.../collapsible/demos/hero/css-modules/index.module.css`):
  ```css
  .Panel {
    height: var(--collapsible-panel-height);
    overflow: hidden;
    transition: height 150ms ease-out;
  }
  .Panel[hidden]:not([hidden='until-found']) {
    display: none;
  }
  .Panel[data-starting-style],
  .Panel[data-ending-style] {
    height: 0;
  }
  ```
  [E] This is the complete, minimal working transition recipe: animate `height` from `0` (the starting/ending-style override) to the measured `--collapsible-panel-height` value, with `overflow: hidden` masking the height change, and `display: none` only when truly `hidden` (excluding the `"until-found"` string value, which must **not** get `display: none` or the browser can no longer find-in-page it).

## 10. Pitfalls & FAQ

1. **"My exit animation never plays / the panel just disappears instantly"** → almost certainly a `keepMounted`/animation-type mismatch: without `keepMounted`, a component using a JS animation library (Motion, etc.) may need the node to persist through the exit — see the `animate-presence.tsx` experiment for the working `AnimatePresence` + `keepMounted` + `render={<motion.div/>}` pattern (§6, §8); if using pure CSS, check that a `transition`/`animation` with non-zero duration is actually defined on the Panel and that both aren't defined simultaneously (the dev warning from `getAnimationType()`, §6).
2. **"`data-starting-style` disappeared after one frame — I wanted to gate `overflow: hidden` for the whole opening animation"** → expected: it exists for exactly one frame (the transition *source* state), not the animation's duration; the maintainers acknowledge this is easy to misread and that a docs sentence on the general animation page is imprecise on this point [E] (#4915).
3. **"Tailwind v4 broke my JS-driven exit animation"** → Tailwind v4's `[hidden]{display:none!important}` reset defeats inline styles set by JS animation libraries; workaround is `hidden={undefined}` on the Panel [E] (#1608).
4. **"`keepMounted={false}` isn't actually removing my panel from the DOM"** → check whether `hiddenUntilFound` is also set — it silently overrides `keepMounted={false}` with a dev-mode warning, since find-in-page requires DOM presence (§6, `CollapsiblePanel.tsx:35-44`).
5. **"Controlling `open` externally + `keepMounted` used to have a bug where the panel wouldn't close"** → fixed; see the closed issue **#4553** (title/labels only retrieved this pass — body unread) for the historical symptom, and #4555/#4565's later refactors for the fix lineage (§9).
6. **"My initially-open panel flashes its mount animation on first paint"** → guarded against directly (`shouldPreventMountAnimationRef`, §6) via two separate code paths (computed-style detection and inline-style detection, both covered by dedicated SSR tests) — if it's still happening, it may be a regression worth reporting rather than expected behavior.
7. **"Nesting one Collapsible inside another's Panel — is that supported?"** → yes, exercised directly in the maintainers' own manual-testing experiment (`transitions.tsx`'s "nested" section, §5); no special API needed since each level tracks its own state independently.
