## 2. One story per use case

Columns: name — renders/setup — controlled? — play? — doc section served.

1. **Basic** — hero recreation: Root(defaultValue)+List+Tab×3+Indicator, Panel×3 in a `PanelViewport` — uncontrolled — no — Hero / Anatomy.
2. **ClickActivatesFocusDoesNot** *(required interaction story — pins the #3176 policy)* — Basic-style 3-tab set with default `activateOnFocus={false}` + an `onValueChange` log — uncontrolled — **play**: (a) `userEvent.tab()` into the tablist / programmatic `.focus()` on an unactive tab → assert `onValueChange` NOT called and the previously-active panel is still visible; (b) `userEvent.click()` on that same unactive tab → assert `onValueChange` IS called and its panel becomes visible; (c) ArrowRight to move roving focus to the next tab → assert focus moved but panel did NOT change → Behavior §6 / A11y contract §7 (this is the single most important test-bearing story for the component, per the #3176 decision log entry).
3. **ActivateOnFocusContrast** *(pairs with story 2 to demonstrate the resolved policy fight)* — same 3-tab set with `<Tabs.List activateOnFocus>` — uncontrolled — **play**: ArrowRight moves focus AND assert the panel changes immediately; also assert a main-button `pointer({ keys: '[MouseLeft>]' })` press-focus on an unactive tab activates it, while a `button: 2` (right-click) focus does not → Behavior §6 / Prop guidance §8 (`activateOnFocus`).
4. **KeyboardHomeEndNavigation** — 5+ tab set, keyboard-only interaction — uncontrolled — **play**: focus first tab, ArrowRight repeatedly to traverse, End to jump to last tab (assert focus, not activation, since default `activateOnFocus=false`), Home to jump back to first tab, assert `loopFocus` wrap at both ends — A11y contract §7 keyboard table.
5. **DisabledTabs** — mixed enabled/disabled tabs including a disabled first tab (auto-fallback to next enabled) — uncontrolled — **play**: assert initial selection skips the disabled tab; arrow-navigate onto the disabled tab and assert focus lands there but it never activates in either `activateOnFocus` mode; attempt to click it and assert no change — A11y contract §7 / Behavior §6 (disabled-focusable convention) / Pitfalls §10.
6. **VerticalOrientation** — `orientation="vertical"` sidebar-style tab list — uncontrolled — **play**: ArrowDown/ArrowUp move focus (not ArrowLeft/Right); assert `aria-orientation="vertical"` on the tablist and `data-orientation="vertical"` on parts — Behavior §6 / Prop guidance §8 (`orientation`).
7. **RTLHorizontal** — `DirectionProvider direction="rtl"` wrapping a horizontal tab set — uncontrolled — play (ArrowRight moves to the *previous* tab, ArrowLeft to the *next* — confirms the swapped semantics asserted in `TabsRoot.test.tsx`'s parametrized RTL matrix) — A11y contract §7 (RTL note).
8. **AnimatedIndicatorUnderline** — CSS `transition` on `left`/`width` (or `transform`) driven by `--active-tab-left`/`--active-tab-width`; click between tabs to show the sliding underline — uncontrolled — play (click a distant tab; assert the indicator's inline custom properties update to the new tab's measurements) — Behavior §6 (Indicator mechanics) / Prop guidance §8 (Indicator CSS vars).
9. **IndicatorPrehydration** — `<Tabs.Indicator renderBeforeHydration />` with an SSR-rendered story harness (or a note-only demo explaining the mechanism if the Storybook harness doesn't do real SSR) — uncontrolled — no — Behavior §6 (prehydration script) / SSR notes.
10. **ActivationDirectionSlide** — recreates `experiments/tabs/tabs-animations.tsx`: panels slide left/right (or up/down in vertical mode) based on `data-activation-direction`, styled via `[data-activation-direction="right"]` etc. selectors on Panel — uncontrolled — play (click a tab to the right, then one to the left; assert the direction attribute flips accordingly) — Behavior §6 (activation direction) / Decision log §9 (#245 review thread that shaped this contract).
11. **PanelKeepMountedVsUnmount** — two Panels, one `keepMounted`, one not; a counter or input inside each panel demonstrates state loss on the non-keepMounted panel after switching away and back — uncontrolled — play (type into an input on panel A, switch to B, switch back to A; assert the keepMounted panel retained its input value while the default panel reset) — Prop guidance §8 (`keepMounted`) / Pitfalls §10 (JS-animation `keepMounted` caveat, contextualized).
12. **PanelCssTransition** — CSS transition on `data-starting-style`/`data-ending-style` for a fade/slide enter-exit on the Panel itself (not the Indicator) — uncontrolled — play (switch tabs; assert the outgoing panel briefly carries `data-ending-style` before removal, mirroring `TabsPanel.test.tsx`'s own animation tests) — Behavior §6 (Panel transition-status contract) / Pitfalls §10 (Motion/Tailwind v4 callout, framed as "here's the supported CSS-only path instead").
13. **TabsAsLinks** — recreates the docs "Links" example: `Tabs.Tab nativeButton={false} render={<a href="#…" />}` — uncontrolled — no — Examples "Links" / Anatomy composition rule (`nativeButton`).
14. **ControlledValue** — external `value`/`onValueChange` state with buttons that programmatically set the active tab from outside the tablist — controlled — play (click an external button; assert the corresponding tab becomes active and receives the roving-tabindex highlight even though focus never entered the tablist) — Behavior §6 (controlled mode, highlight-follows-external-change) / Decision log.
15. **DynamicTabsAddRemove** — tabs addable/removable at runtime (recreates the spirit of `experiments/tabs/tabs-onValueChange.tsx`), with an event log showing `onValueChange` reasons (`initial`/`disabled`/`missing`/`none`) as tabs are added, disabled, and removed — uncontrolled — play (remove the active tab; assert `onValueChange` fires with `reason: 'missing'` and the next tab becomes active) — Behavior §6 (automatic-fallback reasons) / Decision log §9 (#3372/#3373 keepMounted/omitted-value fixes, contextualized).
