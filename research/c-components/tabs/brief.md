# Tabs — component research brief

Tier 2 (full brief, capped mining). Mined 2026-07-07 from source, tests, docs, experiments, git history, and one targeted mui/base-ui PR lookup (read-only). Evidence tags: [E] direct evidence, [I] inference, [G] gap.

## 1. Identity

- **Name / slug**: Tabs — `@base-ui/react/tabs`. Multi-part compound component, namespace export `Tabs.*`.
- **Status**: stable, no `[New]`/`[Preview]` tag [E] (`docs/src/app/(docs)/react/components/tabs/page.mdx`). One of the library's oldest lineages: a Material-UI-era `TabsUnstyled` (mui/material-ui#29597, 2022) → `[Tabs] Overhaul Tabs API` (mui/base-ui#245, 2024-04-29, merged michaldudak) → `[tabs] Modernize implementation` (mui/base-ui#751, 2024-11-25, mj12albert — rewired onto the shared Composite internals) [E] (`git log --follow`).
- **Taxonomy**: `research/a-documentation-standard/taxonomy.md` places `tabs` formally under **category 3, "Navigation & app chrome"** (alongside `navigation-menu`, `menubar`), signature "switch between peer views in place (role=tablist)" [E] (taxonomy.md line 56). The same file flags the placement as contested: "`tabs` is disclosure-adjacent (shows/hides panels — category 4) but its APG pattern and usage archetype are navigational ('switch views'); placed here, cross-linked from Disclosure" [E] (taxonomy.md line 58). This brief follows that formal placement and defers the disclosure-side comparison to [`../_clusters/disclosure.md`](../_clusters/disclosure.md).
- **Purpose / IA statement**: "A component for toggling between related panels on the same page" [E] (page.mdx Subtitle) — switches between mutually exclusive peer views without navigating away or opening an overlay; no portal, no `value`-as-form-data, no popup machinery.
- **5 parts** (from `packages/react/src/tabs/index.parts.ts`; one-liners from part JSDoc):

  | Part | Renders | One-liner |
  |---|---|---|
  | Root | `<div>` | Groups the tabs and the corresponding panels |
  | List | `<div>` | Groups the individual tab buttons |
  | Tab | `<button>` | An individual interactive tab button that toggles the corresponding panel |
  | Indicator | `<span>` | A visual indicator that can be styled to match the position of the currently active tab |
  | Panel | `<div>` | A panel displayed when the corresponding tab is active |

  No Portal/Positioner/Popup/Backdrop — confirmed by reading every part file directly [E]; anatomy is flat (Root → List → (Tab, Indicator), Panel as a Root sibling).

## 2. Intention

- [E] **Introducing PR (current architecture)**: mui/base-ui#245 "[tabs] Overhaul Tabs API" (michaldudak, merged 2024-04-29, closing #212 and #81). PR body states the concrete rewrite scope: "Replaced the slots API with the render prop. Changed the `selectionFollowsFocus` prop to `activateOnFocus` and moved it to TabsList. Added the `loop` prop to the Tabs. Added the `keepMounted` prop to the TabPanel." A to-do list included "Lava-lamp animation support" and "Rendering the indicator on the server" — i.e. the Indicator's SSR/prehydration story was scoped in from this PR, not bolted on later.
- [E] The `selectionFollowsFocus` → `activateOnFocus` rename in #245 is the earliest evidence that click-vs-focus activation was already a named, deliberate axis of the API before the later #3176 policy flip (§9) — the PR predates the render-prop-era library convention of false-by-default booleans (B-P40) but the boolean itself, and its opt-in framing, is unchanged today.
- [E] `#245` review thread also fixed the CSS variable's underlying data attribute during review: colmtuite requested renaming `data-movement-direction` → `data-activation-direction` and normalizing its values to `up|down|right|left` "to accommodate the RTL issue we discussed" — the activation-direction contract (§6, §8) was argued into its current shape in this same PR, not added incidentally.
- [E] `#751` "[tabs] Modernize implementation" (mj12albert, 2024-11-25) rewired Tabs onto the shared `Composite` internals (`CompositeRoot`/`CompositeItem`/`CompositeList`) that Menu, Toolbar, RadioGroup, and other roving-tabindex composites use — this is when Tabs stopped having bespoke keyboard-navigation code and started sharing the library's single-tab-stop composite-widget machinery (B-P29).
- [I] **Design pole: a roving-tabindex composite widget with a native tablist ARIA contract, not a popup.** Inferred from: zero Portal/Positioner/Popup parts (§1); the Composite-widget wiring shared with Menu/Toolbar/RadioGroup (#751); the fact that `TabsPanel` uses `useTransitionStatus`/`useOpenChangeComplete` (the same primitives popups use for exit animation, §6) while carrying none of the popup-specific data attributes (`data-open`/`data-closed`, no backdrop, no dismiss-on-outside-press) — Tabs borrows the animation *primitive*, not the popup *architecture*.
- [E] **Explicit non-goal, stated by users and answered by maintainers**: users expect Radix-style Motion (framer-motion) exit animations "out of the box" the way they work for Popover/Dialog; maintainers clarified the animation docs "target popup components" and Tabs' conditionally-rendered panels "unmount instantly despite `keepMounted`" under a naive Motion integration (mui/base-ui#1494, closed not-planned) — Tabs is deliberately not given popup-style JS-animation wiring; CSS transitions on `data-starting-style`/`data-ending-style` are the supported path (§6, §10).
- [E] Performance and correctness under dynamic tab sets is treated as core identity, not an edge case: a long, still-active bugfix trail exists for adding/removing/disabling tabs mid-session and keeping `onValueChange`, the Indicator, and `aria-controls`/`aria-labelledby` linkage consistent (#2093, #2107, #2164, #3024, #3372, #3373, #4165, #4347, #4532, #4704, #4903, #4935 — all `[tabs]`-scoped commits, `git log --grep='^\[tabs\]'`).

## 3. When to use

- [E] Toggling between related panels that live on the same page/screen without navigating away (page.mdx Subtitle).
- [E] Content-switching UIs where exactly one of several peer views should be visible at a time and the choice is ephemeral/session-scoped rather than a persisted form value — inferred from the absence of any `name`/form-participation prop on `Tabs.Root` (no hidden input, unlike Select/RadioGroup) [I], contrasted directly against Select/RadioGroup which do carry form semantics.
- [E] Rendering tabs as links (routed views) is a first-class supported pattern: "Use the `render` prop and set `nativeButton={false}` on `<Tabs.Tab>` to render tabs as anchor elements" (page.mdx Examples "Links") — supports Next.js `<Link>` directly in the shipped example.
- [E] Cases needing an animated active-indicator (underline/pill) that tracks the selected tab's position/size, including through SSR — the dedicated `Tabs.Indicator` part with a prehydration script exists specifically for this (§6, §8).
- [I] Cases needing vertical or vertical+RTL tab layouts with correct arrow-key semantics — inferred from `orientation` prop plus the `DirectionProvider`-aware test matrix (`TabsRoot.test.tsx` runs every keyboard-navigation assertion across horizontal/ltr, horizontal/rtl, and vertical).

## 4. When not to use + alternatives

Full three-way comparative guidance (accordion vs collapsible vs tabs): [`../_clusters/disclosure.md`](../_clusters/disclosure.md).

- [E] **A single, independent show/hide region → Collapsible.** Tabs coordinates *peer* views with mutual exclusivity built into the Root's single `value`; a lone expandable region with no siblings to coordinate needs none of that Root-level activation-direction/roving-tabindex machinery. [I] inferred from API shape: Collapsible has no `List`/roving composite at all (per taxonomy.md category 4 placement).
- [E] **A coordinated set of independently-openable regions (zero, one, or many open) → Accordion.** Tabs enforces exactly one active peer at all times (no `null`-everything steady state beyond the disabled-fallback edge case, §6); Accordion's whole purpose is to coordinate open/closed state across siblings without that single-selection constraint. [I] inferred from taxonomy.md's category-4 description of accordion as "coordinated set of collapsibles (single/multiple open)" vs Tabs' single mandatory active value.
- [E] **A list of actions to perform, not views to persist and reveal → Menu.** [I] The dividing line (by analogy to the Select-vs-Menu boundary already documented for Select, `../select/brief.md` §4): Tabs' `value` identifies which panel is currently visible; Menu items execute a command and the menu closes. No maintainer statement comparing Tabs vs Menu directly was found — [G] searched issues for "tabs vs menu", "tabs menu difference", found nothing on-topic.
- [E] **Routed, multi-page navigation with URL changes → a router `<Link>`/`nav`, optionally wrapped by Tabs' own link-rendering support.** Tabs supports rendering its own Tab as a link (§3) for exactly this hybrid case, but a page-level primary navigation structure with many top-level destinations is `navigation-menu`'s job, not Tabs' — [I] inferred from taxonomy.md's own placement of `navigation-menu` as "site/app navigation with rich flyouts" vs Tabs "switch between peer views in place," and from Tabs having no submenu/flyout parts at all.

## 5. Anatomy & composition

- [E] Canonical tree (page.mdx Anatomy):
  `Root > (List > (Tab, Tab, …, Indicator), Panel, Panel, …)`.
- [E] **List is required for keyboard/ARIA wiring but is a separate part from Root** — `role="tablist"` and `aria-orientation` live on `List` (`TabsList.tsx` `defaultProps`), while the selection state (`value`) and cross-part linkage (`aria-controls`/`aria-labelledby` id maps) live on `Root`'s context (`TabsRootContext.ts`). Panels are **not** children of List; they are siblings of List under Root, matched to tabs purely by `value` (verified directly in every demo/test/experiment file read: `Root > (List > Tab*, Panel*)`, never `List > (Tab*, Panel*)`).
- [E] **Indicator is optional and lives inside List**, as a sibling of the Tab buttons (`<Tabs.List><Tabs.Tab/>…<Tabs.Indicator/></Tabs.List>` — every demo/experiment file). It renders `null` entirely when no tab is selected (`if (value == null) { return null; }`, `TabsIndicator.tsx`) and sets `hidden` on itself until its size is non-zero, so it never flashes at `(0,0)` before layout settles [E] (`displayIndicator = isTabSelected && width > 0 && height > 0`, same file; the underlying fix predates this brief's window, commit `490011ee0` "[Tabs] Do not display the indicator before layout is settled (#1027)").
- [E] **Panel is matched to its Tab purely by `value` equality**, not DOM position or index — `TabsPanel` reads `getTabIdByPanelValue`/`registerMountedTabPanel` from `TabsRootContext`, and `TabsRoot.test.tsx` explicitly asserts Panels can be declared in any order relative to their Tabs and the `aria-labelledby`/`aria-controls` linkage still resolves correctly ("sets the aria-labelledby attribute on tab panels to the corresponding tab id", lines 65-88, using an intentionally shuffled Tab/Panel order).
- [E] **No Portal, Positioner, Popup, or Backdrop parts exist.** Confirmed by reading `index.parts.ts` directly (5 exports only: Root, List, Tab, Indicator, Panel) — Tabs is not layered like Select/Menu/Popover; everything renders in normal document flow.
- [E] Composition rule for link-style tabs: `render={<Link href="…" />}` + `nativeButton={false}` on `Tab` (page.mdx Examples "Links") — the `nativeButton` convention (library-wide, B-P10) tells the Tab to stop assuming a native `<button>` is being rendered.
- Text spec for a future anatomy diagram: (1) Root [invisible grouping div], (2) List [role=tablist], (3) Tab ×N [role=tab, one per peer view], (4) Indicator [role=presentation, floats over List tracking the active Tab's box], (5) Panel ×N [role=tabpanel, sibling of List under Root, only the one matching the active Tab's `value` is visible/mounted by default].

## 6. Behavior ("How it works")

- [E] **Controlled/uncontrolled**: `value`/`onValueChange`/`defaultValue` triplet on Root (B-P15 pattern); `defaultValue` defaults to `0` when omitted entirely (`TabsRootProps.defaultValue`, JSDoc `@default 0`) — this differs from most other components' `undefined` default, but the omitted-`defaultValue` path is still routed through the same "implicit initial value" fallback logic as an explicit one (see next bullet), so `0` functions as a sentinel starting point, not a hard requirement that a tab with value `0` exists.
- [E] **Automatic-selection fallback is Root's own responsibility, not just a `defaultValue` convenience.** `TabsRoot.tsx`'s mount effect distinguishes an *implicit* initial value (no `defaultValue`/`value` prop given at all) from an *explicit* one, and in uncontrolled mode automatically falls back to the first enabled tab whenever the current selection is disabled or missing — with four distinct `onValueChange` reasons surfacing exactly why a change happened: `'initial'` (first automatic selection, including implicit defaults landing on a later tab because the first is disabled), `'disabled'` (previously-valid selection became disabled), `'missing'` (selected tab was removed or an explicit `defaultValue` never matched a mounted tab), and `'none'` (a real user click/keyboard activation). Controlled roots (`value` prop supplied) opt out of all automatic fallback entirely and keep exactly the value the parent gave them, even if it points at a disabled or absent tab (`TabsRoot.test.tsx`, "should still honor explicit value prop even if it points to a disabled tab"). Automatic changes cannot be canceled via `eventDetails.cancel()` (Root JSDoc, and test "does not cancel automatic value changes").
- [E] **Activation semantics — the #3176 story, verified directly in source and tests.** `TabsTab.tsx`'s `onClick` handler activates the tab whenever it is not already active and not disabled (lines 125-136) — **clicking always activates**, independent of `activateOnFocus`. The `onFocus` handler only activates when `activateOnFocus` is `true` **and** either the focus did not arrive via a pointer press (`!isPressingRef.current`, i.e. keyboard Tab or touch) **or** it arrived via a pointer press on the tab's *main* mouse button (`isPressingRef.current && isMainButtonRef.current`, tracked by `onPointerDown`). With the default `activateOnFocus={false}` (set on `TabsList`, not `TabsRoot`), focus alone — via `Tab`-keying into the list, `element.focus()`, or clicking without the click handler running — never activates a panel change; only an explicit click (or, once focused, Enter/Space acting as native button activation) does. This is exactly the outcome the mining note describes for mui/base-ui#3176 ("[tabs] Change `activateOnFocus` to false", part of the arc #1323 + #2006 → #3176, WCAG 2.5.2 pointer-cancellation policy beating "feels more responsive" per B-P28) — confirmed against the *current* source, not just the historical PR.
  - `activateOnFocus={true}` **does restore something like pointer-down-style activation for mouse users specifically**, because of the `onPointerDown`/`isMainButtonRef` gate: a mouse press that focuses an unactive tab activates it on `onFocus` (which fires essentially at pointerdown time for an unfocused element), matching the mining note's claim "opting into `activateOnFocus` restores pointer-down activation." This is pinned down directly by the test "when `activateOnFocus = true` should call onValueChange on pointerdown" (`TabsRoot.test.tsx` line 438, using `user.pointer({ keys: '[MouseLeft>]', … })` — a press-and-hold, not a full click). For keyboard/touch focus, `activateOnFocus={true}` also activates (test "when `activateOnFocus = true` should call onValueChange if an unactive tab gets focused", line 505, using `element.focus()` directly plus an arrow-key move) — so the true nuance is: `activateOnFocus={true}` activates on *any* route to focus, and additionally special-cases mouse interaction to require the main button (guarding against activating on a right-click-to-focus, tested by "should not call onValueChange on non-main button clicks", line 473, using `fireEvent.click(…, { button: 2 })`).
  - **Keyboard arrow navigation always moves roving focus; it only also activates when `activateOnFocus={true}`.** Every arrow-key/Home/End test in `TabsRoot.test.tsx`'s "keyboard navigation when focus is on a tab" and "when focus is on a tab regardless of orientation" describe blocks is written in matched `activateOnFocus=false` (moves focus, `handleChange` not called) / `activateOnFocus=true` (moves focus *and* calls `onValueChange`) pairs — including for disabled tabs, which always receive focus during arrow traversal but never activate regardless of `activateOnFocus` (composite disabled-focusable rule, B-P29).
- [E] **`activateOnFocus` and `loopFocus` live on `Tabs.List`, not `Tabs.Root`** (`TabsListProps`) — both default `false`/`true` respectively; this differs from a component like Select where interaction-config props sometimes live at Root.
- [E] **Orientation**: `orientation` prop on Root (`'horizontal'` default), read by List for `aria-orientation` and arrow-key axis (ArrowLeft/Right for horizontal, ArrowUp/Down for vertical — `TabsRoot.test.tsx` keyboard-navigation matrix), and mirrored as `data-orientation` on every part via the shared `tabsStateAttributesMapping`.
- [E] **Activation direction** (`tabActivationDirection`, computed in `TabsRoot.tsx`'s `computeActivationDirection`): compares the previous and new active tab's bounding-rect positions (falling back to a value-based left/right or up/down guess when a tab element isn't registered yet, e.g. a newly added and immediately selected tab) to produce `'left' | 'right' | 'up' | 'down' | 'none'`. Exposed as `data-activation-direction` on Root/List/Tab/Panel/Indicator and via the `state.tabActivationDirection` render-prop argument on Tab — purpose-built for direction-aware slide transitions between panels (demoed in `docs/src/app/(private)/experiments/tabs/tabs-animations.tsx`: "Panels animate based on data-starting-style, data-ending-style, and activation direction").
- [E] **Indicator animation mechanics**: `Tabs.Indicator` measures the active Tab's `getBoundingClientRect()` relative to `List`, accounting for scroll offset and any CSS `transform: scale(...)` on ancestors (tested directly: "should account for scroll and border when the tab list is transformed", `TabsIndicator.test.tsx`), and exposes six `px`-valued CSS custom properties (`--active-tab-{left,right,top,bottom,width,height}`) as inline `style` on itself. A `ResizeObserver` on the List and on every registered Tab keeps these variables live when tabs resize or the list itself resizes (`registerTabResizeObserverElement`/`registerIndicatorUpdateListener` in `TabsList.tsx`), with a documented perf guard: "perf: single tab resize does not fan out excessive indicator rerenders" (`TabsIndicator.test.tsx`) asserts a single resize does not cascade into unbounded re-renders even with 100 tabs.
- [E] **Prehydration script (`renderBeforeHydration`, default `false`)**: when set, `Tabs.Indicator` additionally renders an inline `<script>` (`prehydrationScript.template.js`) as a `React.Fragment` sibling of the indicator span. The script runs synchronously during initial HTML parse, walks up from itself to the closest `[role="tablist"]`, finds the tab carrying `[data-active]`, replays the *exact same* position/size math as the React component (byte-for-byte parallel logic, verified by reading both files), and writes the six CSS variables directly via `element.style.setProperty`, then removes the `hidden` attribute the SSR-rendered indicator started with — closing the gap between "HTML painted, indicator invisible" and "React hydrated, indicator positioned" that would otherwise show a flash of an unstyled/mispositioned indicator. JSDoc: "Whether to render itself before React hydrates. This minimizes the time that the indicator isn't visible after server-side rendering." Confirmed server-rendered ("renders the inline pre-hydration script during server-side rendering") and confirmed to unmount cleanly post-hydration without a hydration-mismatch error ("keeps the script during hydration and removes it afterwards", `TabsIndicator.test.tsx`); respects `CSPProvider`'s `nonce` ("applies the CSP nonce to the pre-hydration script").
- [E] **SSR without `renderBeforeHydration`**: the Indicator still server-renders with `suppressHydrationWarning` and starts `hidden` until client-side measurement completes on mount — so the non-prehydration path is "briefly invisible after hydrate," and `renderBeforeHydration` is the opt-in fix for that gap, not a requirement for SSR correctness in general [I], inferred from the `hidden`/`suppressHydrationWarning` props on the element itself plus the default `false`.
- [E] **Panel mount lifecycle and transition-status contract.** `Panel`'s `hidden` attribute and actual DOM presence follow `useTransitionStatus`/`useOpenChangeComplete` — the same primitives popup components use for exit-animation timing — giving Panel `data-starting-style`/`data-ending-style` data attributes (`TabsPanelDataAttributes.ts`, added by commit `52dd939ab` "[tabs] Add transition attributes to `<Tabs.Panel>` part (#3880)") alongside `data-hidden`. **This refines, rather than confirms verbatim, the mining note's claim that "Tabs/Collapsible panels lack the popup animation machinery": Panel does carry the CSS starting/ending-style contract since #3880** and this is directly exercised by tests ("triggers enter animation via data-starting-style when mounting", "applies data-ending-style before unmount", `TabsPanel.test.tsx`). What Panel genuinely lacks, verified by the absence of the corresponding code, is the *popup*-specific machinery: no Portal/`keepMounted`-on-Portal, no `data-open`/`data-closed` (popups use an open/closed pair; Panel only exposes `data-hidden` plus the transition-status pair), no backdrop, no dismiss-on-outside-press. So "Tabs is not a popup" (§1, §4) is accurate; "Tabs has zero animation hooks" is not — it has the transition-status pair without the popup envelope around it.
- [E] Without `keepMounted` (default `false`), an inactive Panel unmounts from the DOM entirely once its exit transition completes (`useOpenChangeComplete`'s `onComplete` calls `setMounted(false)`); with `keepMounted={true}`, the Panel element stays in the DOM at all times with `hidden`/`inert` toggling instead of unmounting — this is the prop the #1494 Motion-compatibility issue (§10) explicitly names as insufficient on its own for third-party JS animation libraries, because Motion needs the *element*, not just non-unmounting, to drive its own exit sequencing.
- [E] **Suspense compatibility**: a Panel whose content suspends (React 19 `use()`) while being newly activated is exercised directly by a dedicated test ("renders a panel that suspends when opened with the boundary outside the root", `TabsPanel.test.tsx`) and by the `[tabs] Fix suspended panel activation (#4903)` commit — `onValueChange` still fires with the correct value/reason even while the panel's own content is suspended above a `<Suspense>` boundary outside Root.
- [E] **Disabled tabs stay keyboard-focusable but are never selected or highlighted** — `useButton({ disabled, native: nativeButton, focusableWhenDisabled: true })` in `TabsTab.tsx` is the same composite-widget disabled-focusable convention as Menu/Select/RadioGroup (B-P29); if the first tab is disabled, the Root automatically selects the next enabled tab on mount, but this "does not work like this during server-side rendering, as it is not known during pre-rendering which Tabs are disabled" — the component's own JSDoc explicitly tells integrators to set `defaultValue`/`value` to an already-enabled tab's value to avoid an SSR/client mismatch (`TabsTabProps.disabled` JSDoc, `TabsTab.tsx`).
- [E] **Works inside popup components** without special wiring — `TabsRoot.test.tsx`'s "popups" describe block directly tests Tabs nested inside both `Popover` and `Dialog` ("works inside Popover", "works inside Dialog").

## 7. Accessibility contract

- [E] **Pattern**: W3C WAI-ARIA APG **Tabs** pattern — https://www.w3.org/WAI/ARIA/apg/patterns/tabs/. Confirmed by role wiring read directly from source: `List` sets `role="tablist"` (`TabsList.tsx` `defaultProps`); `Tab` sets `role="tab"` + `aria-selected` + `aria-controls` (`TabsTab.tsx`); `Panel` sets `role="tabpanel"` + `aria-labelledby` (`TabsPanel.tsx`).
- [E] **ARIA managed by the component**:
  - List: `role="tablist"`, `aria-orientation` (only emitted when `orientation="vertical"`; omitted for the horizontal default — test "does not add aria-orientation by default").
  - Tab: `role="tab"`, `aria-selected` (`true`/`false` per active state), `aria-controls` (the currently-mounted matching Panel's `id` — re-synced live as panels mount/unmount when `keepMounted` is `false`, test "syncs aria-controls to the mounted tab panel when keepMounted is false"), roving `tabIndex` (`0` on exactly the highlighted tab, `-1` on the rest — test "puts the selected child in tab order").
  - Panel: `role="tabpanel"`, `aria-labelledby` (the owning Tab's `id`), `tabIndex={0}` when open / `-1` when not, `inert` when not the active panel (`inertValue(!open)`, `TabsPanel.tsx`) — so an inactive-but-`keepMounted` panel is excluded from the accessibility tree and from tab order even before it fully unmounts.
- [E] **Full keyboard interaction table** (verified directly from `TabsRoot.test.tsx`'s exhaustive matrix — every row below is asserted for horizontal/ltr, horizontal/rtl via `DirectionProvider`, and vertical, with disabled-tab variants):

  | Key | Context | `activateOnFocus=false` (default) | `activateOnFocus=true` |
  |---|---|---|---|
  | Click | any tab | Activates the clicked tab (unless already active or `disabled`) | Same — click always activates regardless of this prop |
  | Enter / Space | focused tab (native `<button>`) | Activates via native button semantics | Same |
  | ArrowRight/Left (horizontal) or ArrowDown/Up (vertical) | focus inside List | Moves roving focus to the next/previous tab (skips nothing — disabled tabs remain stops); does **not** change the active panel | Moves roving focus **and** activates the newly focused tab, unless it is `disabled` |
  | Home | focus inside List | Moves focus to the first tab without activating | Moves focus to the first tab and activates it (unless `disabled`) |
  | End | focus inside List | Moves focus to the last tab without activating | Moves focus to the last tab and activates it (unless `disabled`) |
  | Arrow keys at list boundary | `loopFocus=true` (default) | Wraps focus to the opposite end | Same wrap behavior, plus activation per the `activateOnFocus` column |
  | Arrow keys at list boundary | `loopFocus=false` | Focus stays at the boundary tab (no wrap) | Same, plus activation |
  | Any navigation onto a `disabled` tab | either mode | Focus lands on the disabled tab (composite-widget focusable-when-disabled convention); it is **never** activated, in either `activateOnFocus` mode | Never activated |
  | Tab / Shift+Tab | — | Leaves the tablist as a single stop (roving tabindex — only the active/highlighted tab is in the page tab order) | Same |

  RTL note: with `DirectionProvider direction="rtl"`, ArrowLeft/ArrowRight swap meaning (ArrowRight moves to the *previous* tab) — asserted directly in the parametrized test matrix, not inferred.
- [E] **Screen reader / focus-highlight sync**: when the active tab changes externally (controlled `value` prop) while focus is *outside* the tablist, the roving-tabindex highlight (and hence which tab is `tabIndex=0`) follows the new active tab; when focus is *inside* the tablist at the time of an external change, the highlight deliberately stays on the currently-focused tab so arrow-key navigation continues relative to where the user's focus already is, rather than jumping (`TabsRoot.test.tsx` "highlight synchronization on external value change relative to focus" describe block, both cases tested explicitly).
- **Known open issues (honesty list)**: [G] searched `mui/base-ui` issues for "tabs accessibility", "tabs screen reader", "tabs voiceover/nvda/jaws", and the `label:"component: tabs"` combined with `label:accessibility` — no open a11y-specific bug reports surfaced distinct from the already-resolved #3176 activation-timing story and the already-covered disabled-tab/highlight edge cases. ## 8. Prop-level guidance (decision-relevant props)

- [E] **`activateOnFocus` (List, default `false`)** — the star prop for this component. Leave at the default for WCAG 2.5.2-conformant "select-then-activate" behavior (arrow keys move focus, Enter/Space/click commits) — this is the outcome of the #1323/#2006 → #3176 policy decision (§9). Set to `true` for classic "focus follows selection" tab behavior (arrow keys immediately switch panels) — useful for tab sets where switching panels is cheap/instant and users expect immediate feedback, but be aware it also activates on a plain `element.focus()` call and on keyboard/touch focus generally, not only on click (§6).
- [E] **`orientation` (Root, default `'horizontal'`)** — set to `'vertical'` for sidebar-style stacked tabs; flips the arrow-key axis (ArrowUp/Down instead of ArrowLeft/Right) and emits `aria-orientation="vertical"` on List; also flips which axis `tabActivationDirection` measures (`up`/`down` instead of `left`/`right`).
- [E] **`value` / `defaultValue` / `onValueChange` (Root)** — standard controlled/uncontrolled triplet (B-P15). `defaultValue` defaults to `0`, but omitting it entirely still routes through the "implicit initial value" fallback path (reason `'initial'`) rather than hard-requiring a tab with value `0` to exist — if no tab matches, Root falls back to the first enabled tab and still reports `reason: 'initial'`. Pass `null` explicitly for "no tab selected" as a real state (Root JSDoc: "When the value is `null`, no Tab will be active" — confirmed by test "does not call onValueChange on initial render when defaultValue is null").
- [E] **`loopFocus` (List, default `true`)** — set `false` to stop arrow-key wrap-around at the first/last tab; independent of `activateOnFocus`.
- [E] **`disabled` (Tab)** — disabled tabs stay keyboard-focusable (never removed from the roving tab-stop sequence) but can never become active, whether via click, focus, or programmatic `value`. If the first tab in DOM order is `disabled`, Root auto-selects the next enabled tab on mount **in client-rendered uncontrolled mode only** — the JSDoc explicitly warns this does not apply during SSR pre-rendering, so set `defaultValue`/`value` to an already-enabled tab when the first tab may be disabled.
- [E] **`keepMounted` (Panel, default `false`)** — set `true` to keep an inactive panel's DOM subtree alive (hidden + `inert`) instead of unmounting it after its exit transition completes. Needed for: preserving panel-local state/scroll position across tab switches, and as one half of third-party JS-animation compatibility (the other half being driving `open` externally — see #1494 in §10, since `keepMounted` alone does not make Motion/framer-motion exit animations work).
- [E] **`renderBeforeHydration` (Indicator, default `false`)** — set `true` to eliminate the "indicator invisible for one paint after SSR, before hydration measures it" gap via the inline prehydration script (§6). Trade-off: adds an inline `<script>` tag to the response (respects `CSPProvider` nonce; excluded from client bundles per commit `99018b2c7` "[tabs][slider] Exclude the prehydration script from client bundles (#5003)").
- [E] **`nativeButton` (Tab, default `true`)** — set `false` together with `render={<a .../>}` (or any non-button element) to render tabs as links; part of the library-wide `nativeButton` convention (B-P10) that tells the component ARIA/keyboard wiring not to assume a native `<button>`.
- [E] **Indicator CSS variables** (`TabsIndicatorCssVars.ts`, all `number` pixel values, present only while a tab is selected):

  | Variable | Represents |
  |---|---|
  | `--active-tab-left` | Distance from the List's left edge to the active tab's left edge |
  | `--active-tab-right` | Distance from the active tab's right edge to the List's right edge |
  | `--active-tab-top` | Distance from the List's top edge to the active tab's top edge |
  | `--active-tab-bottom` | Distance from the active tab's bottom edge to the List's bottom edge |
  | `--active-tab-width` | The active tab's width |
  | `--active-tab-height` | The active tab's height |

  Typical usage: an underline indicator uses `left`/`width` (horizontal) or `top`/`height` (vertical) with a CSS `transition` for the sliding animation; a full-bleed "pill" background can use all four inset variables with `position: absolute; inset: var(--active-tab-top) var(--active-tab-right) var(--active-tab-bottom) var(--active-tab-left)`.
- [E] **Data attributes that matter for styling** (from `*DataAttributes.ts`, shared `data-orientation`/`data-activation-direction` present on every part via `tabsStateAttributesMapping`): `data-active` (Tab, present on the selected tab), `data-disabled` (Tab), `data-hidden` (Panel, present while not the active panel), `data-starting-style`/`data-ending-style` (Panel, transition-status pair, §6), `data-index` (Panel, its position among Panels — useful for index-based stagger animations).

## 10. Pitfalls & FAQ

- **"Framer Motion / Radix-style exit animations don't work on my tab panels"** → the animation docs' `keepMounted` + JS-library recipes target *popup* components (Popover/Dialog/Menu); Tabs panels have no popup envelope to hook into, and a conditionally-rendered panel unmounts instantly even with `keepMounted` if the animation library isn't also driving `open` externally. Correction: use CSS transitions on `data-starting-style`/`data-ending-style` (supported directly, §6, since #3880) instead of a JS animation library; if a JS library is required, keep the panel's element alive yourself (`keepMounted={true}`) and drive its exit sequencing manually rather than relying on Base UI's popup-oriented `actionsRef`/`onOpenChangeComplete` recipes, which Tabs does not expose. (mui/base-ui#1494, closed not-planned) [E]
- **"Tailwind v4's `[hidden]{display:none!important}` reset defeats my animation library on a hidden tab panel"** → this is the general Tailwind v4 + JS-animation conflict documented library-wide (B-P26), not Tabs-specific, but it compounds with the #1494 issue above for Tabs/Collapsible since neither has the popup-style workaround already built in. Workaround: unset `hidden` explicitly (`hidden={undefined}`) when driving visibility via a JS library instead of the `hidden` attribute (mj12albert, #1608, cited in principles.md B-P26). [E]
- **"Both the old and new tab panel are visible for a moment" (Preact only)** → a Preact `useTransitionStatus`/`useLayoutEffect` incompatibility caused a double-render flash; root cause is fixed upstream in Preact 10.29.0. Workaround while on an older Preact: style `[data-ending-style] { display: none }` explicitly rather than relying on timing. (mui/base-ui#4127, closed external, 9 comments) [E]
- **"My tabs auto-select the wrong tab (or none) right after the page loads"** → if the first tab in DOM order is `disabled`, Root's automatic-fallback-to-first-enabled-tab logic only runs client-side; during SSR pre-rendering it's not known which tabs will end up disabled. Correction: always set `defaultValue`/`value` explicitly to an enabled tab's value when the first tab might be disabled (`TabsTabProps.disabled` JSDoc, directly in source). [E]
- **"I expected `activateOnFocus={true}` to activate on every mouse interaction that focuses a tab, including via keyboard-simulated events or right-click"** → it only activates on *main-button* pointer presses (`event.button === 0`/unset) that result in focus, plus any non-pointer focus (keyboard, touch); a right-click-to-focus does not activate (tested directly, `TabsRoot.test.tsx` "should not call onValueChange on non-main button clicks"). Correction: this is intentional main-button gating, not a bug — reach for `onFocus`/`onPointerDown` overrides on `Tab` only if a different arbitration is genuinely needed. [E]
- **"Why does my Panel still show up in the tab order / to screen readers while hidden?"** → without `keepMounted`, this shouldn't happen (Panel unmounts on close); with `keepMounted={true}`, the Panel correctly receives `inert` and `tabIndex={-1}` while inactive — if a custom `render` function drops the spread props (including `inert`), the a11y contract breaks. Correction: always spread all received props onto the underlying node in a custom `render` function (library-wide composition rule, B-P9). [I] inferred from the general render-prop contract; no Tabs-specific issue found stating this exact symptom, [G] searched "tabs inert screen reader" with no distinct hits.
