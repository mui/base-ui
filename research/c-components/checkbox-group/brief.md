## 1. Identity

- **Name / slug**: Checkbox Group — `@base-ui/react/checkbox-group`. **Single-part** component (default export `CheckboxGroup`, no namespace, no sub-parts of its own) — grouped with the library's other single-part exports (Button, Input, Form, Menubar, Toggle, ToggleGroup, Separator, per `research/PROMPT.md` §3.2). Composes with the separately-exported `Checkbox` namespace rather than defining its own item part.
- **Status**: stable, no `[New]`/`[Preview]` tag. Introducing commit: `9c04f2c3e` "[CheckboxGroup] Implement components and hooks (#458)" — found via local git log (`git log --oneline --diff-filter=A -- "**/checkbox-group/**"`), offline, no `gh` lookup spent on this one [E for the commit locator; PR #458's body itself not read in this pass — deprioritized given the shared budget, tagged [G] for the full origin narrative beyond the commit subject].
- **Parts**: none of its own; composes `Checkbox.Root`/`Checkbox.Indicator` as children. The `index.parts.ts` file that once existed for it was explicitly removed: `[internal] Remove index.parts.ts from Checkbox Group (#4660)` [E, git log] — a signal that maintainers deliberately treat it as a single-part component, not a compound one, consistent with its single default export.
- **Taxonomy** (Phase A): **selection & input** → binary/choice controls sub-cluster (`research/c-components/_clusters/binary-controls.md`), specifically the multi-select coordinator for a set of independent `Checkbox.Root`s. Purpose/IA: turn N independently-toggleable checkboxes into one array-valued, form-participating field — the direct multi-select counterpart to Radio Group's single-value coordination of N radios (`research/c-components/radio/brief.md`).
- **Cross-component role**: consumes `useFieldRootContext` (same integration bus as every other form control, `research/c-components/field/brief.md` §1); is the sole home of the `useCheckboxGroupParent` hook that computes the "select all" parent-checkbox tri-state — even though the `parent` boolean *prop* that opts a checkbox into this behavior is declared on `Checkbox.Root` (`research/c-components/checkbox/brief.md` §1 cross-component role). This split (coordination logic here, opt-in prop there) is worth naming explicitly since it means understanding "parent checkboxes" requires reading both components' source.

## 2. Intention

- [E] **Enable submitting a checkbox group's value as one named form field — the state-of-the-question was explicitly "how do I submit an array"**: `[form][checkbox group] Enable submitting checkbox group value as one field (#1948)` is the PR that wired `useRegisterFieldControl` into `CheckboxGroup.tsx`, and its accompanying commit also shipped a dedicated docs demo (`checkbox-group/demos/form/`) — i.e., "submit as one array-valued field" was significant enough to warrant its own demo at the time it landed, not a side effect of some other change. Before this, presumably each checkbox in a group submitted independently under its own `name`, requiring the consuming app to reconstruct the array from individual `FormData` entries [I, inferred from the PR title's "as one field" framing implying a prior "as many fields" state — not independently confirmed by reading the pre-#1948 diff].
- [E] **A checkbox group's `required` semantics on individual members means "all must be checked," which is a genuinely non-obvious and easy-to-misread contract.** This is directly confirmed by a dedicated test: "keeps a required error while another required checkbox in the group is unchecked" — checking only one of two `required` checkboxes in a group **does not** clear the group's validation error; the error only clears once **every** `required` checkbox in the group is checked (`CheckboxGroup.test.tsx`). This is the opposite of what "required" might intuitively suggest for a multi-select set (e.g., "pick at least one") — see Pitfall 1.
- [E] **The parent-checkbox "select all" pattern is a first-class, deliberately-engineered feature, not a userland recipe** — evidenced by (a) two dedicated docs demos (`Parent checkbox`, `Nested parent checkbox`), (b) a dedicated hook module (`useCheckboxGroupParent.ts`, ~150 lines of tri-state cycling logic), (c) a dedicated test file (`useCheckboxGroupParent.test.tsx`, 14 test cases), and (d) a whole cluster of maintenance PRs hardening it against edge cases: `#4941` "Fix parent group cancellation and indeterminate state," `#4912` "Fix parent checkbox with custom `validate` fn," `#2155` "Fix `onCheckedChange` not running when parent checkbox is present." This is a mature, actively-maintained feature, contrasting with the still-open, community-requested "tree of grouped checkboxes with Select All" pattern for **Combobox** specifically, which maintainers explicitly declined to build ("trees are out of combobox scope… can't marry CheckboxGroup with Combobox," mj12albert, issue #2695) — i.e., the parent-checkbox pattern is scoped deliberately to Checkbox Group's flat list, not generalized into a tree widget or extended to other components.
- [E] **The tri-state parent-click cycle is a specific, tested algorithm, not "toggle everything."** Reading `useCheckboxGroupParent.ts` directly: clicking the parent when **all-or-nothing** is checked (`status` tracking is irrelevant in this branch) simply flips between "select all" and "select none." But when the group is genuinely **mixed** (some but not all children checked, arrived at via individual child clicks), clicking the parent follows an explicit `mixed → on → off → mixed → …` cycle (`nextStatus` state machine in `getParentProps.onCheckedChange`) that **preserves which specific children were checked before the mixed state**, restoring exactly that prior subset when cycling back to "off" rather than simply clearing everything — confirmed by the test "preserves initial state if mixed when parent is clicked."
- [I] **Disabled children are treated as immovable within the parent's "select all"/"select none" computation** — `useCheckboxGroupParent.ts`'s `getParentProps` explicitly partitions `allValues` into "none" (disabled-and-checked, can't be changed) vs. "all valid" (not-disabled, or disabled-and-already-checked) before deciding what "select all" even means, so a disabled-and-checked child is never un-checked by a parent "select none" click, and a disabled-and-unchecked child is never checked by a parent "select all" click. This mirrors Radio Group's similar disabled-radio exclusion from its own aggregate operations (`radio/brief.md` §6), extended here to a genuinely more complex multi-child computation [E for the source logic; I for the "mirrors Radio Group's design philosophy" framing, since no single maintainer statement ties the two together explicitly].

## 3. When to use

- [E] Several related checkboxes whose checked states should coordinate as **one named, array-valued form field** — the explicit motivating use case for `#1948` (§2); docs' own framing: "Provides shared state to a series of checkboxes" (Subtitle).
- [E] Any set of checkboxes needing a "select all" / parent-child summary control — the built-in `parent`/`allValues` pattern exists specifically for this (§2), including **nested** groups (a parent-of-a-parent scenario — the docs "Nested parent checkbox" demo shows a top-level `CheckboxGroup` whose own `parent` checkbox additionally coordinates with a *second*, nested `CheckboxGroup`'s own parent/children, propagating "manage-users" membership bidirectionally between the two groups via app-level `onValueChange` handlers rather than any built-in nested-group API — i.e., "nesting" here is a composition pattern the app wires up with two independent `CheckboxGroup`s and shared handler logic, not a single component that natively understands hierarchy).
- [E] Grouping labeling: `Field.Item` wraps each option, `Fieldset.Legend` labels the whole group — the identical composition pattern documented for Radio Group (`radio/brief.md` §5) and confirmed by the docs "Form integration" example and `CheckboxGroup.test.tsx`'s Fieldset/Field.Item usage throughout.
- [I] When the individual checkboxes must remain independently addressable and stylable as full `Checkbox.Root`/`Checkbox.Indicator` compositions (as opposed to some flattened multi-select list-item abstraction) — inferred from the fact that Checkbox Group composes real `Checkbox.Root` children rather than rendering its own item part.

## 4. When not to use + alternatives

Full binary/choice-control comparative guidance: [`research/c-components/_clusters/binary-controls.md`](../_clusters/binary-controls.md) (does not yet cover Checkbox Group in comparative depth — flagged there as `[G]`; this section is the first dedicated pass).

- **Large or filterable option sets that need to submit as an array → Combobox with `multiple`, not Checkbox Group**: [I, no maintainer statement found contrasting these two directly — searched discussions.md/issues-prs.md for "checkbox group vs combobox"/"checkbox group vs multiple select," none found, tagged [G]]. The closest adjacent evidence is issue **#2695**, where a user explicitly asked for "a Tree of Groups of Checkboxes with a 'Select All' root Item" *inside Combobox*, and mj12albert declined ("trees are out of combobox scope… can't marry CheckboxGroup with Combobox") while providing a **modified multiple-select Combobox demo with its own select-all item** as the alternative — i.e., maintainers' own recommended pattern for "many options, filterable, with select-all" is a **Combobox in `multiple` mode**, not Checkbox Group, once the option count grows past what's comfortable to render as a flat always-visible list of checkboxes. The dividing signal, by the same reasoning Select's brief uses against Radio Group (`select/brief.md` §4): Checkbox Group is right when every option should stay simultaneously visible with no popup/filtering; once that stops being true, the collapsed, filterable listbox pattern (Combobox `multiple`) is the documented escape hatch — evidenced directly by mj12albert's own workaround demo, even though no prose decision doc states this boundary outright.
- **A tree/hierarchical multi-select with nested select-all at arbitrary depth → not built-in anywhere**: the only "nesting" Checkbox Group supports is the app-composed two-`CheckboxGroup`-instances pattern shown in the "Nested parent checkbox" demo (§3) — there is no native multi-level tree widget, and the Combobox-tree feature request was explicitly declined (#2695, above). If true arbitrary-depth tree selection is the requirement, Checkbox Group (nested-demo pattern) is the closest first-party building block, but it requires hand-wired parent/child value propagation between independent group instances, not a single hierarchical API.
- **An exclusive, always-exactly-one choice → Radio Group, not Checkbox Group**: cardinality boundary — see `research/c-components/radio/brief.md` §4; Checkbox Group's value is any subset of N (including all or none), Radio Group's is always exactly one (or `null`).
- **A single independent checkbox that never needs to coordinate with siblings → standalone `Checkbox.Root`**, no group wrapper needed — see `research/c-components/checkbox/brief.md` §4.
- **Why is there no equivalent "ButtonGroup"-style non-form container?** — tangential but relevant framing evidence: issue **#4008** asked "why is there no ButtonGroup when you have a Checkbox Group?" — the maintainer answer (cited in `research/b-library-principles/_mining/issues-prs.md` #4008) was that "'button group' (non-toggles) is just `role='group'` around some normal buttons, so there isn't anything that a headless lib could add," pointing to ToggleGroup/Toolbar instead. This confirms Checkbox Group's *raison d'être* is specifically the array-valued, native-form-submitting **state coordination** it provides (§2) — a capability plain buttons don't need and CSS/`role="group"` alone can't supply — not merely "a div that visually groups checkboxes," which would indeed need no dedicated component. [E]

## 5. Anatomy & composition

- [E] Canonical tree (docs Anatomy):
  ```jsx
  <CheckboxGroup>
    <Checkbox.Root />
    {/* …repeated per option, each with its own value */}
  </CheckboxGroup>
  ```
- [E] **CheckboxGroup renders a plain `<div role="group">`** (`CheckboxGroup.tsx` — `role: 'group'`, not a composite/roving-tabindex widget like Radio Group's `role="radiogroup"` + `CompositeRoot`). This is a structural difference from Radio Group worth naming: **Checkbox Group does not manage focus or arrow-key navigation between its children at all** — each `Checkbox.Root` is an independent Tab stop, since each is independently toggleable (Space on each, no shared composite navigation model). Confirmed by absence of any `CompositeRoot`/`CompositeItem` usage in `CheckboxGroup.tsx`/`CheckboxRoot.tsx`'s non-radio-group code paths, and by the complete absence of arrow-key tests in `CheckboxGroup.test.tsx` (contrast with Radio Group's extensive arrow-key describe block, `radio/brief.md` §6).
- [E] **Parent-checkbox anatomy**: a `Checkbox.Root parent` gets its `id` derived from the group (`parentId` via `useBaseUiId()`), and each child's `id` is deterministically `${parentContext.id}-${value}` (`CheckboxRoot.tsx:103`) — this scheme feeds the parent's `aria-controls` attribute, which is a **space-separated list of every child's id**: `'aria-controls': allValues.map((v) => \`${id}-${v}\`).join(' ')` (`useCheckboxGroupParent.ts`), confirmed by test "should apply space-separated aria-controls attribute with child names." A source comment flags a known sharp edge here: "TODO: custom `id` on child checkboxes breaks this" (referencing open issue **mui/base-ui#2691**) — i.e., if a consumer sets an explicit `id` prop on a child `Checkbox.Root` inside a parent-coordinated group, the parent's `aria-controls` computation (which assumes the derived `${parentId}-${value}` scheme) silently breaks. [E, source comment + linked issue number]
- [E] **A parent checkbox's `name` is force-unset**, excluding it from native form submission entirely (`CheckboxRoot.tsx`: `name: parent ? undefined : name`) — confirmed by the dedicated test "excludes parent checkboxes from form submission," which submits a group of 3 real values + 1 parent and asserts `FormData.getAll('apple')` contains exactly the 3 child values, never a parent entry.
- [E] **Text spec for a future anatomy diagram**: (1) CheckboxGroup container (`<div role="group">`, invisible layout wrapper) → (2, optional) parent `Checkbox.Root` (`data-parent` attribute, `aria-controls` pointing at every child) → (3+) child `Checkbox.Root`s, each independently focusable/toggleable, each contributing its `value` to the group's array when checked.
- [I] **Composition rule**: identical Field.Item/Fieldset.Legend labeling composition to Radio Group (§3; `research/c-components/field/brief.md` §5's group anatomy diagram already documents this pattern generically for "checkbox group or radio group" — not repeated here).

## 6. Behavior ("How it works")

- [E] **Controlled/uncontrolled**: `value`/`onValueChange`/`defaultValue` triplet (all `string[]`), with `defaultValue` defaulting to `[]` when omitted entirely (`CheckboxGroup.tsx`: `defaultValueProp ?? []`, confirmed by test "should treat an omitted defaultValue as an empty array"). `eventDetails.cancel()` vetoes the change (test "does not update the group when onValueChange cancels the event").
- [E] **Each child checkbox's click independently pushes/removes its `value` from the group array** — `CheckboxRoot.tsx`'s `onChange` handler: `nextChecked ? [...groupValue, value] : groupValue.filter((item) => item !== value)`. No ordering guarantee beyond append-on-check (confirmed by test sequences: clicking red→green→blue produces `['red']`→`['red','green']`→`['red','green','blue']`; unchecking green from that state produces `['red','blue']`, i.e., filter-based removal, not an ordered re-sort).
- [E] **Group-level `disabled` cascades to every child and takes precedence over any individual child's own `disabled={false}`** — confirmed by "takes precedence over individual checkboxes" test. Individual children can still be independently disabled while the group itself is enabled.
- [E] **`required` on individual group members means "every required checkbox in the group must be checked," not "at least one must be checked."** This is the single most counter-intuitive behavioral fact in this brief (§2) — validated by three distinct tests covering the happy path, the disabled-exemption path ("ignores a disabled required checkbox when validating the group" — a disabled required checkbox does not block the group's validity), and an unmount-safety path ("keeps validating the remaining required checkbox after a checked sibling unmounts" — even if the DOM node backing the shared validation-input-ref is removed, the field registry still correctly tracks the remaining required-but-unchecked sibling).
- [E] **Clicking a parent checkbox does not fire the individual children's own `onCheckedChange` callbacks** — only the parent's own `onCheckedChange` fires; per-child callbacks are silent during a parent-driven bulk change (`useCheckboxGroupParent.test.tsx` "should control child checkboxes": `parentCheckedChange` fires once per parent click, `childCheckedChange` (registered on one specific child) never fires across two parent clicks that check then uncheck every child). This matters for anyone attaching side effects to individual child `onCheckedChange` handlers expecting them to fire during "select all."
- [E] **The parent's `indeterminate`/`checked` are automatically computed from the live child value array whenever the parent is *not* manually overridden** — `checked = value.length === allValues.length`, `indeterminate = value.length !== allValues.length && value.length > 0` (`useCheckboxGroupParent.ts`) — but a **manually-set** `indeterminate` prop on the parent (independent of the computed tri-state) is separately supported and does not get silently overwritten (`CheckboxRoot.test.tsx` "sets grouped parent aria when manually indeterminate").
- [E] **Validation-mode matrix (`onSubmit`/`onChange`/`onBlur`) all apply at the group level**, each with its own dedicated test ("prop: validationMode=onSubmit," "…onChange," "…onBlur") — the group's aggregate array value (not any single child) is what gets validated, consistent with Field's general validation-mode contract (`research/c-components/field/brief.md` §6).
- [E] **SSR / uncontrolled initialization was itself a bugfix target**: `[checkbox group] Fix uncontrolled default initialization (#4535)` — flagged here as evidence that the controlled/uncontrolled boundary for an array-valued group (as opposed to a simple boolean or string) had real edge cases worth a dedicated fix.

## 7. Accessibility contract

- [E] **Pattern**: no single W3C ARIA APG pattern name applies to Checkbox Group as a composite — it is `role="group"` around N independent `role="checkbox"` widgets, each following the plain APG **Checkbox** pattern individually (see `research/c-components/checkbox/brief.md` §7) [I, inference from the plain `role="group"` + no composite-navigation source, §5 — no APG "checkbox group" pattern exists as a distinct named pattern the way "Radio Group" does, since unlike radios, checkboxes in HTML/ARIA have never needed a dedicated composite-widget pattern: each is independently focusable and operable].
- [E] **ARIA managed by the component** (`CheckboxGroup.tsx`): `role="group"`, `aria-labelledby` (from `Field.Label`/`Fieldset.Legend`, via `useLabelableContext`'s `labelId`). No `aria-required`/`aria-disabled` of its own beyond what cascades individually to children — contrast with Radio Group, which sets `aria-required`/`aria-disabled`/`aria-readonly` on the group container itself (`radio/brief.md` §7); Checkbox Group's `disabled` cascades to children (§6) but is not independently exposed as an ARIA attribute on the group `<div>` itself [E, direct source comparison — `CheckboxGroup.tsx`'s `defaultProps` object contains only `id`/`role`/`aria-labelledby`, no `aria-disabled`/`aria-required` keys, unlike `RadioGroup.tsx`'s equivalent object].
- [E] **The `aria-labelledby` auto-link fix (#4122 → #4142) applies to Checkbox Group's individual children identically to standalone Checkbox** — same mechanism, not repeated (`research/c-components/radio/brief.md` §7 has the full quoted thread; the affected-components list in the original report explicitly names "Checkbox Group" as one of the five affected surfaces).
- [E] **Keyboard interaction table** (evidence: `CheckboxGroup.tsx`/`CheckboxRoot.tsx` — no composite wiring at the group level, §5):

  | Key | Context | Action |
  |---|---|---|
  | Tab | anywhere in the group | Moves focus to the **next individual checkbox** — each checkbox is its own Tab stop; the group itself is not a single Tab stop the way Radio Group is |
  | Space | a focused child checkbox | Toggles that checkbox, pushing/removing its `value` from the group array (§6) |
  | Space | a focused **parent** checkbox | Cycles the tri-state select-all/select-none/restore-mixed algorithm (§2) |
  | Enter | any focused checkbox | Submits the owning form, does not toggle — identical to standalone Checkbox (`checkbox/brief.md` §6, #4713) |
- [E] **Screen reader notes**: label the group via `Fieldset.Legend` + `aria-labelledby` (auto-wired) or a sibling element (docs "Labeling a checkbox group" example, identical pattern to Radio Group); label each option the same way individual Checkboxes are labeled (`checkbox/brief.md` §7). The parent checkbox's `aria-controls` (§5) gives AT users a programmatic link from "select all" to the specific controls it affects, though `aria-controls` support varies significantly across screen readers/browsers in practice [I, general ARIA knowledge, not Base-UI-specific evidence].
- **Open a11y-adjacent issues (honesty list)**: [E] the `id`-scheme fragility noted in §5 (custom child `id` silently breaking `aria-controls`, mui/base-ui#2691, referenced directly in a source `TODO` comment) is the clearest open, source-acknowledged gap specific to this component. [G] No dedicated screen-reader bug report specific to the parent-checkbox `aria-controls` wiring itself (as opposed to the general labeling issue) was found in the mined cache for this pass.

## 8. Prop-level guidance (decision-relevant props)

- [E] **`allValues` (Root, `string[]`)** — "Names of all checkboxes in the group. Use this when creating a parent checkbox" (prop JSDoc). Required for the parent-checkbox pattern to compute its tri-state correctly; without it, `useCheckboxGroupParent` defaults to empty arrays and the parent's computed `checked`/`indeterminate` will not reflect reality. Must be paired with a **controlled** `value`+`onValueChange` (docs "Parent checkbox" step 1: "Make `<CheckboxGroup>` a controlled component") — the parent pattern is not supported in fully-uncontrolled mode [E, inferred from the docs' explicit numbered-steps requirement; not independently tested against an uncontrolled-with-`allValues` combination in this pass, tagged [I] for the "not supported" framing specifically].
- [E] **`value`/`defaultValue`/`onValueChange` (Root)** — standard controlled/uncontrolled array triplet (§6); `defaultValue` safely defaults to `[]`.
- [E] **`disabled` (Root, default `false`)** — cascades to every child, takes precedence over any individual child's `disabled={false}` (§6). Use for "the whole set is off-limits right now" (e.g., a permissions section gated behind another control) rather than disabling each child individually.
- [E] **`parent` (on `Checkbox.Root`, not on `CheckboxGroup` itself)** — opts a specific child into the "select all" coordinator role; see `research/c-components/checkbox/brief.md` §8 for the prop's own documentation (declared there since it lives on `Checkbox.Root`), and §2/§6 above for the coordination algorithm it triggers when a `CheckboxGroup` ancestor with `allValues` is present.
- [E] **`required` on individual children (not a group-level prop)** — see §6's counter-intuitive "all must be checked" contract; there is no group-level `required` prop that means "at least one selected" — if that's the desired validation (a common real-world requirement, e.g. "choose at least one interest"), it must be implemented via a custom `Field.Root validate` function reading the group's array length, not via the individual-checkbox `required` prop [I, inferred from the absence of any such prop in `CheckboxGroupProps`/`CheckboxRootProps` and the confirmed "all must be checked" semantics of the existing `required`].
- [E] **Styling contract highlights** (from `CheckboxGroupDataAttributes.ts`): the group itself carries only `data-disabled` as its own attribute, plus the full Field validity set via `fieldValidityMapping` (`data-valid`/`data-invalid`/`data-touched`/`data-dirty`/`data-filled`/`data-focused`) — no `data-checked`/`data-unchecked`/`data-indeterminate` of its own, since "checked" isn't a meaningful concept for the group container itself (only for the optional parent checkbox and each individual child, both of which use `CheckboxRootDataAttributes`/`CheckboxIndicatorDataAttributes`, documented in `research/c-components/checkbox/brief.md` §8).

## 10. Pitfalls & FAQ

1. **"I set `required` on some checkboxes expecting 'pick at least one,' but the error won't clear until I check all of them"** → this is the intended, tested contract: `required` on individual group members means "every required one must be checked," not "at least one." Correction: implement "at least one" via a custom `Field.Root validate` function checking the group's array length, since no built-in prop expresses that semantics. [E, §6/§8]
2. **"My parent checkbox's `aria-controls` doesn't point at the right children"** → likely cause: a custom `id` was set on one of the child `Checkbox.Root`s, which breaks the parent's derived `${parentId}-${value}` id scheme it relies on for `aria-controls`. This is a known, source-acknowledged limitation (mui/base-ui#2691, referenced in a `TODO` comment) — avoid setting explicit `id`s on children of a parent-coordinated group until it's fixed. [E]
3. **"Clicking 'select all' didn't trigger my per-checkbox `onCheckedChange` side effects"** → intended behavior: parent-driven bulk changes only fire the parent's own `onCheckedChange`, not each child's. Attach side effects to the group's `onValueChange` (which fires once with the full new array) instead of individual child callbacks if they need to react to "select all." [E]
4. **"I want a tree of grouped checkboxes with select-all at each level"** → not a single built-in feature; the closest first-party pattern is composing two independent `CheckboxGroup` instances with hand-wired `onValueChange` propagation between them (the "Nested parent checkbox" demo), and a Combobox-hosted tree was explicitly declined by maintainers (#2695, "trees are out of combobox scope"). [E]
5. **"Why is there no ButtonGroup if there's a CheckboxGroup?"** → not a Checkbox Group pitfall per se, but a frequently-asked adjacent question with a citable maintainer answer worth reusing in docs: plain buttons don't need a coordinating component the way checkboxes (array-valued, form-submitting) do; use `role="group"`/ToggleGroup/Toolbar instead (#4008). [E]
