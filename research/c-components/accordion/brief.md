# Accordion — component research brief

Tier 2 (full brief, capped mining). Mined 2026-07-07 from source, docs, tests, and mui/base-ui git history (2 `gh` PR lookups: #4965, #4961 — the APG roving-focus removal pair). Evidence tags: [E] direct evidence, [I] inference, [G] gap.

## 1. Identity

- **Name / slug**: Accordion — `@base-ui/react/accordion`. Multi-part compound component, namespace export `Accordion.*`.
- **Status**: stable, no `[New]`/`[Preview]` tag. Introduced 2024-10-30 (commit `8054a8502`, mui/base-ui#577). [E]
- **Purpose / IA statement**: "A set of collapsible panels with headings" [E] (`docs/src/app/(docs)/react/components/accordion/page.mdx` Subtitle). Taxonomy: **Disclosure & structure** category — "coordinated set of collapsibles (single/multiple open)" [E] (`research/a-documentation-standard/taxonomy.md`).
- **5 parts** (from `packages/react/src/accordion/index.parts.ts`; one-liners from part JSDoc):
  | Part | Renders | One-liner |
  |---|---|---|
  | Root | `<div>` | Groups all parts of the accordion |
  | Item | `<div>` | Groups an accordion header with the corresponding panel |
  | Header | `<h3>` | A heading that labels the corresponding panel |
  | Trigger | `<button>` | A button that opens and closes the corresponding panel |
  | Panel | `<div>` | A collapsible panel with the accordion item contents |
- No internal-only/`@internal` parts. [E]
- **Built directly on Collapsible's primitives, not a parallel reimplementation**: `AccordionItem.tsx` calls `useCollapsibleRoot` and provides `CollapsibleRootContext`; `AccordionPanel.tsx` calls `useCollapsiblePanel` (the same hook Collapsible's own Panel uses) for its `shouldRender`/height/width/transition-status logic; `AccordionTrigger.tsx` reads `panelId`/`open`/`handleTrigger` straight from `useCollapsibleRootContext()`. [E] Each `Accordion.Item` is, structurally, one Collapsible whose open/close callback is intercepted and redirected into the shared `Accordion.Root` value array (`AccordionItem.tsx` `onOpenChange` calls `handleValueChange(value, nextOpen, eventDetails)` after the item-level `onOpenChange` prop runs and isn't canceled).

## 2. Intention

- [E] **Introducing PR**: mui/base-ui#577 "[accordion] New Accordion components and hooks" (merged 2024-10-30, commit `8054a8502`), landing in the same lineage sweep as Collapsible ("[Collapsible][Accordion] Modernize implementation", #889; "[Collapsible][Accordion] Rework animations and transitions", #1601).
- [E] **Design pole: Accordion is a coordination layer over Collapsible, not a separate accessibility widget.** Every architectural decision traced in source confirms this — no independent open/close, animation, or height-measurement logic exists in the accordion package; it all lives in `collapsible/`. The component's entire distinct value-add is: (a) an array-valued `value`/`onValueChange` that coordinates *which* items are open across siblings, (b) `multiple` semantics for whether more than one can be open at once, and (c) an `<h3>` Header wrapper plus the ARIA labeling wiring that gives the group its accordion-specific accessible shape. [I] inferred directly from the shared-hook architecture; no explicit maintainer quote frames it this way, but the source leaves no other reading available.
- [E] **Accessibility contract was actively revised toward W3C APG guidance mid-lifecycle, not fixed at launch.** Two related 2026-06-08 PRs removed behavior that had shipped since #577:
  - **#4965 "[accordion] Align Accordion keyboard navigation with APG"**: "The Accordion APG pattern no longer recommends roving focus, as documented in https://github.com/w3c/aria-practices/pull/3434. This PR proposes removing roving focus from Accordion to align with the latest APG guidance." [E, PR body verbatim]
  - **#4961 "[accordion] Remove region role from `Accordion.Root`"**: "Based on the WAI-ARIA accordion pattern, region is typically applied to accordion panels rather than the root container. Since `Accordion.Root` serves as a structural wrapper and does not have an accessible label of its own, removing the region role seems more aligned with the APG guidance." Same PR body also flags the roving-focus removal as a companion change. [E, PR body verbatim]
  - Net effect: `orientation` and `loopFocus` props are now formally `@deprecated` in source ("This prop no longer affects keyboard focus behavior" — `AccordionRoot.tsx` JSDoc, citing the same w3c/aria-practices#3434 link three times), and `role="region"` moved off Root entirely (it was already correctly placed on `Panel` — `AccordionPanel.tsx` line 118 — so no functional gap resulted, just removal of a redundant/misplaced role on Root).
- [E] **Explicit non-goal, confirmed by absence**: Accordion does not implement composite roving-focus keyboard navigation between headers (no `Composite`/`useCompositeRoot` usage anywhere in `accordion/`, contrast Toolbar/Menubar/RadioGroup/ToggleGroup which all do) — this is now a *documented alignment with the current APG spec*, not a missing feature. `AccordionRoot.test.tsx`'s entire "keyboard interactions" describe block (`describe.skipIf(isJSDOM)('keyboard interactions', ...)`) tests only `Enter`/`Space` toggling a focused trigger via native `Tab` to reach it — there is no test anywhere in the accordion suite exercising ArrowUp/ArrowDown/Home/End moving focus between headers, because no such behavior exists to test. [E]
- [E] Field/Form integration was never in scope — unlike Select, Accordion has no `name`, no hidden input, and does not appear in the Field/Form handbook pages; it is a pure disclosure/structure component, not a form value. [I] inferred from the complete absence of form-related props/wiring in `AccordionRootProps`/`AccordionItemProps`.

## 3. When to use

- [E] FAQ-style content, settings panels, or any "set of collapsible panels with headings" where readers scan headings and expand only what they need (docs Subtitle). [E]
- [E] When exactly one section should typically be open at a time, keeping the page compact — the default (`multiple: false`). [E]
- [E] When multiple sections may reasonably be open simultaneously (e.g., comparing two FAQ answers side by side) — via `multiple` (docs "Open multiple panels" example). [E]
- [E] When content must be discoverable by the browser's native in-page "find" (Ctrl/Cmd+F) even while visually collapsed — via `hiddenUntilFound`, which uses `hidden="until-found"` instead of unmounting/`display:none`, so browser find-in-page can locate and auto-expand the right panel (`AccordionRootProps` JSDoc: "Allows the browser's built-in page search to find and expand the panel contents"). [E]
- [I] Nested/hierarchical content (an accordion item whose panel contains further structured content, including another accordion) — no explicit maintainer guidance found either endorsing or discouraging this; [G] no test or docs example of nested accordions was found in this pass, though nothing in source structurally prevents it (each `Accordion.Item` provides its own independent `CollapsibleRootContext`/`AccordionItemContext`, so nesting should compose the same way any two independently-scoped Collapsible trees would).

## 4. When not to use + alternatives

Related disclosure-shaped components (accordion/collapsible/tabs) are being compared in a separate, in-progress cluster note: [`research/c-components/_clusters/disclosure.md`](../_clusters/disclosure.md) *(a parallel Tier-2 pass is authoring this cluster note; it may not exist as a file yet at the time this brief was written — link is forward-looking and will resolve once that pass lands)*. This brief states Accordion's own side of that comparison directly, since the cluster note is not yet available to read-and-link against:

- [E] **A single show/hide region with no siblings to coordinate → Collapsible.** Accordion is, structurally, nothing more than several `Collapsible`s wrapped in a value-coordinating `Accordion.Root`/`Accordion.Item` pair (§1, §2) — if there is only ever one region and nothing to keep in sync with siblings, reach for `Collapsible` directly rather than a one-item Accordion; the extra Root/value/multiple machinery buys nothing in that case. [I] inferred from the shared-hook architecture; no maintainer statement explicitly tells users to prefer Collapsible for the single-item case, but the API overhead is directly visible in source (Accordion.Root requires a `value` array even for one item; Collapsible.Root does not).
- [E] **Switching between mutually exclusive, persistent views (not stacked disclosure) → Tabs.** The taxonomy explicitly notes Tabs is "disclosure-adjacent (shows/hides panels) but its APG pattern and usage archetype are navigational (‘switch views’)" — the dividing line is that an accordion's panels are all part of one continuous scrollable document (any subset can be open, panels appear in their document position), while tabs present one panel at a time, replacing the previous one in a fixed viewport region, typically as `role="tablist"` navigation rather than a document-flow disclosure list. [E, `research/a-documentation-standard/taxonomy.md`]
- [I] **A single boolean setting with an immediate visual/functional effect (not "content to reveal") → Switch/Checkbox**, not Accordion/Collapsible — derived from API affordances (Accordion's `value` is a set of *identities*, not a boolean), no direct maintainer statement found contrasting these. [G]
- [G] No maintainer-stated boundary between Accordion and a plain `<details>`/`<summary>` native disclosure element was found in the mined corpus for this pass — the general "headless primitive where the native element can't be styled enough" scope test (`research/b-library-principles/principles.md` B-P4) plausibly applies here too (native `<details>` markers/animations are notoriously hard to style consistently cross-browser), but this specific application of the test was not directly evidenced.

## 5. Anatomy & composition

- [E] Canonical tree (docs Anatomy):
  `Root > Item > (Header > Trigger, Panel)`, repeated per item — no Portal/Positioner layering (this is an in-document disclosure component, not a popup).
  ```jsx
  <Accordion.Root>
    <Accordion.Item>
      <Accordion.Header>
        <Accordion.Trigger />
      </Accordion.Header>
      <Accordion.Panel />
    </Accordion.Item>
  </Accordion.Root>
  ```
- [E] **Header renders `<h3>` unconditionally** — there is no prop to change the heading level (contrast some other libraries' `as`/level props). If an app needs a different heading rank in context, the library-wide `render` prop is the escape hatch (swap the rendered element while Base UI still supplies the ARIA/behavior wiring) — [I] inferred from the fixed `'h3'` literal in `useRenderElement('h3', ...)`; no accordion-specific docs guidance on heading-level selection was found. [G]
- [E] **Trigger renders a native `<button>` by default** (`nativeButton = true`), goes through the shared `useButton` hook with `focusableWhenDisabled: true` — disabled triggers remain focusable (consistent with the library-wide composite-widget policy of keeping disabled options discoverable by assistive tech, cited elsewhere in `research/b-library-principles/principles.md`). Swapping the rendered element (e.g. to a `<span>`/`<div>`) requires `nativeButton={false}` for keyboard/ARIA behavior to adapt correctly — pinned by the "rendering interactive/non-interactive triggers" test matrix that runs the entire keyboard-interactions suite twice, once per `nativeButton` value. [E]
- [E] **Panel exposes both `hiddenUntilFound` and `keepMounted`, settable at Root (applies to all items) or overridden per-Panel** (`AccordionPanelProps` picks both from `AccordionRoot.Props`; `hiddenUntilFoundProp ?? contextHiddenUntilFound`, `keepMountedProp ?? contextKeepMounted`). `keepMounted={false}` combined with `hiddenUntilFound` triggers a dev-only warning (both at Root and at Panel level) explaining the override is ignored, "since the panel must remain mounted while closed" — `hidden="until-found"` structurally requires the DOM node to exist. [E]
- [E] **Panel animates via two CSS custom properties, not layout observation alone**: `--accordion-panel-height` and `--accordion-panel-width` are set inline on the Panel element every render, sourced from `useCollapsiblePanel`'s measured `height`/`width` (falling back to `'auto'` when undefined, i.e. before first measurement or once fully open with nothing to animate toward). Consumer CSS keys a `height: var(--accordion-panel-height)` transition off this, with `[data-starting-style]`/`[data-ending-style]` collapsing `height: 0` at the animation boundaries (docs hero demo CSS, `_index.module.css`: `.Panel { height: var(--accordion-panel-height); transition: height 150ms ease-out; } .Panel[data-starting-style], .Panel[data-ending-style] { height: 0; }`). [E]
- [E] **Panel gets `role="region"` and `aria-labelledby={triggerId}` directly and unconditionally** (`AccordionPanel.tsx`: `props: [props, { 'aria-labelledby': triggerId, role: 'region', style: {...} }, ...]`) — this is the correct, current placement per APG guidance and per #4961's stated rationale (Root itself has no accessible label of its own and should not have carried the region role). [E]
- [E] **Trigger id auto-generation and manual override both work, bidirectionally**: `AccordionTrigger` calls `setTriggerId(idProp)` in an effect when an explicit `id` is supplied, else `AccordionItemContext` supplies a `defaultTriggerId` (via `useBaseUiId`); Panel's `aria-labelledby` always tracks whichever id is current, live, including when a manual id is added/changed/removed after mount (pinned by 4 dedicated tests: "references manual trigger id in panel aria-labelledby", "updates panel labeling when a manual trigger id is added or changed", "restores panel labeling when a manual trigger id is removed"). Same bidirectional pattern exists for the Panel's own id feeding `aria-controls` on Trigger. [E]
- Text spec for a future anatomy diagram: (1) Item [repeated per entry], (2) Header [`<h3>`, invisible structurally], (3) Trigger [`<button>`, `aria-expanded`/`aria-controls`], (4) Panel [`role="region"`, `aria-labelledby`, animated height/width].

## 6. Behavior ("How it works")

- [E] **Controlled/uncontrolled**: uncontrolled by default via `defaultValue` (an array, default `[]`); controlled via `value`/`onValueChange` (`AccordionRootProps` JSDoc, `useControlled` under the hood). `onValueChange(value, eventDetails)` fires with the *would-be* next array; `eventDetails.isCanceled` (settable via the shared `BaseUIChangeEventDetails` cancel mechanism) blocks the internal `setValue` call while staying uncontrolled — the same escape hatch pattern used library-wide (Select, Menu, etc.), pinned by 6 dedicated cancel tests spanning uncontrolled/controlled/multiple permutations. [E]
- [E] **Single vs multiple toggling logic** (`AccordionRoot.tsx` `handleValueChange`): when `multiple` is false, opening item X always replaces the whole value array with `[X]` (or `[]` if X was already the sole open item — a toggle-closed, not "always stays open" — behavior); when `multiple` is true, opening pushes X onto the existing array, closing filters it out. There is no way to have `multiple=false` behave as "exactly one must always be open and cannot be collapsed" — closing the currently-open item is always possible. [E]
- [E] **`multiple` defaults to `false`** — this was a breaking, deliberate change (#3141, "Change `multiple` prop to be false by default"); prior to that, the prop was named `openMultiple` (renamed in #2764, "[accordion][toggle group] Unify multiple prop name to `multiple`", part of a library-wide convergence on one `multiple` prop name across Accordion/Select/ToggleGroup, `research/b-library-principles/_mining/issues-prs.md` #1075). A docs bug shipped briefly stating the old default incorrectly (#3448, "Fix incorrect Accordion multiple prop default") before being corrected.
- [E] **Item value fallback**: if `Accordion.Item` isn't given an explicit `value` prop, a React `useId()`-derived fallback is used (`useBaseUiId()` in `AccordionItem.tsx`) — this replaced an earlier composite-index-based fallback (#2664, "Use `useId` instead of composite index as fallback value") specifically so item identity survives reordering/conditional rendering rather than drifting with position. An explicit `value` is required whenever the accordion is controlled or an item needs a stable initial open/closed state (`AccordionItemProps` JSDoc example shows `<Accordion.Item value="a" />` for this exact reason).
- [E] **Space/Enter activation timing was hardened, not naive**: `#94accb823 "[accordion] Activate trigger on Space keydown"` and the dedicated "keyboard activation timing" test describe block confirm Space triggers on **keyup** (matching native button semantics — pressing and holding Space, then moving focus away before releasing, should not fire) — this timing correctness was itself a bugfix generation (#9069ba886 "[accordion] Fix trigger behavior bugs", #3684 "Fix keyboard navigation with non-interactive trigger elements", #3219792 "Fix keyboard interactions with elements in the panel"), i.e. getting native-button-equivalent keyboard timing right on a custom composite took several iterations even without any roving-focus complexity. [E]
- [E] **Panel exit-animation race, fixed**: `#3099` ("[accordion] Panel exit animation sometimes skipped") — `display:none` was applied prematurely mid-exit-animation when switching panels in single (`multiple={false}`) mode, causing layout jumps; genuinely fixed across #2985/#3101 (`research/b-library-principles/_mining/discussions.md` #3099 entry). This is corroborated by a still-present, dedicated regression test: "keeps the closing panel visible until its exit transition completes when switching items" (`AccordionPanel.test.tsx`). [E]
- [E] **`hidden="until-found"` interacts with `keepMounted` structurally, not just by convention** — enabling `hiddenUntilFound` (Root- or Panel-level) forces the panel to remain mounted regardless of the `keepMounted` value, with a dev warning if a consumer explicitly (and contradictorily) sets `keepMounted={false}` alongside it. Pinned directly: `expect(screen.getByText(PANEL_CONTENT).getAttribute('hidden')).toBe('until-found')`. [E]
- [E] **The `dir` attribute is no longer set implicitly on Root** — `#5117 "[accordion] Remove implicit root dir attribute"` removed a `useDirection()` call and its resulting `dir={direction}` prop from `AccordionRoot.tsx`, aligning Accordion with the library-wide `DirectionProvider` convention that the provider coordinates *behavior* only and does not itself apply a `dir` HTML attribute anywhere (`research/a-documentation-standard/taxonomy.md`'s DirectionProvider entry: "behavior only — does not set `dir`"). [E]
- [E] **`data-orientation` is still emitted despite the prop being deprecated** — `orientation` (default `'vertical'`) still flows into `AccordionRootDataAttributes.orientation`/`AccordionPanelDataAttributes.orientation` as a pure styling hook (e.g. flipping a chevron icon's rotation axis, or laying items out in a row); it simply no longer changes any keyboard-focus behavior, per the JSDoc's explicit deprecation notice. This is a meaningful, easy-to-miss nuance: the prop's *name* still exists and *does something visually*, but its original behavioral purpose (roving-focus direction) is gone. [E]
- [G] No SSR-specific fix commits were found for Accordion beyond the shared Collapsible/library-wide `useIsoLayoutEffect` conventions; server-rendered-open-then-animate suppression is tested (`AccordionPanel.test.tsx` "suppresses the initial keyframe animation from inline styles when rendered open"), which is Collapsible-shared machinery rather than Accordion-specific.

## 7. Accessibility contract

- [E] **Pattern**: WAI-ARIA APG **Accordion** pattern (https://www.w3.org/WAI/ARIA/apg/patterns/accordion/, https://www.w3.org/WAI/ARIA/apg/patterns/accordion/examples/accordion/ — both cited verbatim in PR #4961's body), specifically the **current** revision of that pattern, which as of w3c/aria-practices#3434 no longer recommends roving-tabindex arrow-key navigation between headers. Base UI tracks the current spec, not the older, more elaborate version some other libraries still implement. [E]
- [E] **ARIA managed by the component** (confirmed by exhaustive source read of Root/Item/Header/Trigger/Panel):
  - Root: no ARIA role or state at all — a plain structural `<div>` (contrast: it *did* carry `role="region"` before #4961 removed it).
  - Item: no ARIA role or state at all — a plain structural `<div>` grouping one Header+Panel pair.
  - Header: no ARIA role or state beyond being a real `<h3>` (native heading semantics are the entire contribution here — headings give screen-reader users a "jump between headings" navigation mode for free, which is *why* APG accordions require a heading wrapper around the trigger in the first place).
  - Trigger: `aria-expanded` (always present, boolean), `aria-controls` (present **only while open** — `open ? panelId : undefined`, meaning a closed trigger has no `aria-controls` at all, not an `aria-controls` pointing at a hidden/absent element), `id` (bidirectionally synced with Panel's `aria-labelledby`, §5).
  - Panel: `role="region"`, `aria-labelledby={triggerId}` (always present, bidirectionally synced, §5).
  - No `aria-multiselectable`-equivalent state exists anywhere — `multiple` is a pure behavioral prop with no corresponding ARIA reflection, [I] because the APG accordion pattern itself has no such state to expose (unlike, say, a listbox's `aria-multiselectable`).
- [E] **Keyboard interaction table** (evidence: exhaustive read of `AccordionTrigger.tsx`'s `useButton` wiring + the "keyboard interactions"/"keyboard activation timing" test blocks in `AccordionRoot.test.tsx`; absence of any Composite/roving-focus usage confirmed by source read):

  | Key | Context | Action |
  |---|---|---|
  | Tab / Shift+Tab | anywhere | Moves focus to the next/previous focusable element in **native DOM tab order** — including between accordion headers, since each Trigger is an ordinary tab-stop, not a roving-tabindex composite member (this is the direct, tested consequence of #4965) |
  | Enter | Trigger focused | Toggles the item's open state (native `<button>` activation on `nativeButton`; explicit `onClick: handleTrigger` wiring covers non-native-button `render` swaps too) |
  | Space | Trigger focused | Toggles the item's open state, firing on **keyup** to match native button semantics, not keydown-eager (hardened by #4284 "Activate trigger on Space keydown" — the PR title is about the keydown *not* mistakenly double-firing; the dedicated "keyboard activation timing" tests pin the keyup-fire contract) |
  | ArrowUp / ArrowDown / Home / End | anywhere | **No accordion-specific behavior** — these keys do nothing special to move focus between headers; this is the deliberate, current-APG-aligned design (§2, §6), not an omission |
- [E] **Disabled items remain focusable** (`focusableWhenDisabled: true` passed to `useButton` in `AccordionTrigger.tsx`) — consistent with the library-wide policy of keeping disabled composite/interactive members discoverable by keyboard/AT users rather than silently skipping them in tab order.
- [E] **Screen reader notes**: the Header→Trigger→Panel labeling chain means a screen-reader user navigating by headings level lands on "What is Base UI? button, collapsed" (heading + button role + expanded state all announced together, since Trigger *is* the sole child content of Header in the canonical anatomy) then can activate it, then can navigate into the newly-revealed `role="region"` labeled by that same trigger's text. This chain is fully automatic as long as the canonical `Header > Trigger` nesting is followed; nothing about it requires additional `aria-label`/`aria-labelledby` authored by the consumer. [I] inferred from the ARIA wiring traced above; not independently verified with an actual screen reader in this pass. [G]
- **Open a11y-adjacent issues (honesty list)**: [E] the mined `discussions.md` per-component index lists accordion issues **3099** (exit-animation race, closed/fixed — see §6), **1678**, **4114**, **1148** (all three appear only as bare "index entries" in `sources.md` without full digest text captured in this pass's mining artifacts — **[G] their content was not independently re-verified**; a deeper Phase-C pass should re-open these three directly rather than rely on the index alone). No open a11y-specific bug beyond the now-resolved APG-alignment PRs (#4965/#4961, both merged 2026-06-08) was found for Accordion specifically. [G] A direct `label:accessibility component:accordion` search was not run in this Tier-2-capped pass (budget went to the git-log/PR-body evidence instead, which proved higher-yield); a Tier-1-depth pass should run it.

## 8. Prop-level guidance (decision-relevant props)

- [E] **`value` / `defaultValue` (Root)** — array of item identities that are open; omit both for a fully uncontrolled, all-closed-by-default accordion (`defaultValue` defaults to `[]` internally). Generic-typed: `Accordion.Root<Value>` lets `onValueChange` and item `value`s be strongly typed beyond the default permissive `any[]` (`AccordionRoot.spec.tsx` pins string/number/nullable/union generic scenarios; generic typing added #4138 "Add generic `Value` typing").
- [E] **`multiple` (Root, default `false`)** — set `true` to allow more than one panel open simultaneously; renamed from `openMultiple` (#2764) as part of a library-wide `multiple`-prop-name convergence (also affecting ToggleGroup); default flipped to `false` as a breaking change (#3141) after initially defaulting `true`.
- [E] **`disabled` (Root and/or Item)** — Root-level `disabled` cascades to every item (`contextDisabled` OR'd with each item's own `disabled` prop in `AccordionItem.tsx`); set at Item level alone to disable just one panel while leaving siblings interactive. Disabled triggers stay focusable (§7).
- [E] **`hiddenUntilFound` (Root and/or Panel, default `false`)** — enables browser find-in-page discovery of collapsed content via `hidden="until-found"`; forces the panel to stay mounted regardless of `keepMounted`, with a dev warning on the contradictory explicit-`false` combination. Panel-level setting overrides Root-level (verified: a Root-level `hiddenUntilFound` with one Panel explicitly set `hiddenUntilFound={false} keepMounted={false}` correctly un-mounts only that one overridden panel).
- [E] **`keepMounted` (Root and/or Panel, default `false`)** — keep closed panels in the DOM (e.g. for SEO, or to avoid remount cost/flash on reopen); ignored whenever `hiddenUntilFound` is active (which structurally requires mounting).
- [E] **`orientation` / `loopFocus` (Root)** — both `@deprecated`; no longer affect keyboard focus behavior following the APG update (w3c/aria-practices#3434, applied in mui/base-ui#4965). `orientation` still drives the `data-orientation` styling hook (purely visual now, e.g. chevron rotation or row-vs-column layout) — do not rely on either prop for any behavioral guarantee going forward; they are retained for backward compatibility and pure-CSS use only.
- [E] **`nativeButton` (Trigger, default `true`)** — set `false` when `render` swaps Trigger to a non-`<button>` element, so ARIA/keyboard behavior adapts correctly (library-wide `NativeButtonProps` convention, same as Select's Trigger). Both `true`/`false` paths are run through the entire keyboard-interaction test matrix.
- [E] **`onOpenChange` (Item) vs `onValueChange` (Root)** — `Item.onOpenChange(open, eventDetails)` fires first, per-item, and can cancel via `eventDetails.isCanceled` to stop the value from ever propagating to Root's `handleValueChange`; `Root.onValueChange(value, eventDetails)` fires with the whole next array and has its own independent cancel path. Both can be used together (item-level for local side effects, root-level for the coordinated array), or either alone.
- [E] **Styling contract highlights** (from `*DataAttributes.ts`/`*CssVars.ts`): Root — `data-disabled`, `data-orientation` (deprecated-but-live styling hook); Item/Header — `data-index`, `data-disabled`, `data-open`; Trigger — `data-panel-open` (note: **not** `data-open` — Trigger uses a distinctly-named attribute so a consumer can't accidentally confuse "is the trigger open" with "is the trigger disabled/pressed" semantics that `data-open` might otherwise suggest for a button), `data-disabled`; Panel — `data-index`, `data-open`, `data-orientation`, `data-disabled`, `data-starting-style`, `data-ending-style`; Panel CSS vars — `--accordion-panel-height`, `--accordion-panel-width` (the animation-driving pair, §5/§6).

## 10. Pitfalls & FAQ

- **Arrow keys don't move focus between accordion headers** → this is not a bug; current W3C APG guidance removed roving-focus arrow-key navigation from the Accordion pattern (w3c/aria-practices#3434), and Base UI deliberately aligned with it in #4965 (2026-06-08). Use Tab/Shift+Tab; if an app specifically wants the older roving-focus behavior back, it must be built in userland — Base UI does not offer an opt-in flag for it. [E]
- **`orientation`/`loopFocus` props seem to do nothing** → correct, as of the same 2026-06-08 change; both are `@deprecated` and no longer affect keyboard behavior. `orientation` still flips `data-orientation` for pure CSS purposes (e.g. chevron rotation), but carries no behavioral guarantee. [E]
- **`role="region"` missing from `Accordion.Root` after upgrading** → intentional removal (#4961); the region role was arguably misplaced on Root all along (Root has no accessible label of its own) and already existed correctly on `Accordion.Panel`. No action needed unless custom code was specifically depending on Root's now-removed role.
- **Panel snaps to the wrong height / jumps during exit** on rapid open/close of different items in single (non-`multiple`) mode → historically a real race condition (#3099), fixed via #2985/#3101; if still seeing this, confirm the library version is current, since a dedicated regression test now guards it.
- **Panel won't unmount even with `keepMounted={false}`** → check whether `hiddenUntilFound` is active anywhere in the chain (Root or Panel level) — it structurally forces mounting and silently wins over an explicit `keepMounted={false}`, with only a dev-mode console warning (not a thrown error) to flag the contradiction.
- **`aria-controls` is missing from a closed trigger** → intentional, not a bug: `aria-controls` is only rendered while the panel is open (`open ? panelId : undefined`); this avoids a closed trigger pointing at DOM that may not exist (e.g. under `keepMounted={false}`).
- **`multiple` defaulted differently than expected** → confirm which version is in use; the default flipped from `true` to `false` as a breaking change in beta.5 (#3141), and a docs bug briefly stated the old default incorrectly (#3448, now fixed).
- **No community-sourced FAQ beyond the exit-animation race could be independently re-verified in this pass** → [G] `discussions.md`'s per-component index lists three additional bare issue numbers for accordion (1678, 4114, 1148) that were not individually re-opened and read during this Tier-2-capped pass (mining budget went to the higher-yield git-log/PR-body evidence trail instead — see §7). A future pass should read those three directly before treating this pitfalls list as complete.
