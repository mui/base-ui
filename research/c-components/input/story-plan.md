# Input — story plan

7 stories. Input is a single-part thin wrapper over `Field.Control` (brief.md §1–2) — the story set exists to prove both halves of its duality: "works standalone" and "works automatically once nested in Field," plus the state-attribute styling contract it inherits verbatim from Field.

1. **Basic (Field-labeled)** — `Field.Root > Field.Label + Input`, styled per the fieldset/field hero demo pattern (`docs/.../fieldset/demos/hero/css-modules/index.tsx` shows the canonical `Field.Root`+`Field.Label`+`Field.Control` triple; substitute `Input` for `Field.Control` to prove they're interchangeable). Serves DoD anatomy + hero-equivalent sections. Not a play story — static rendering + snapshot-style assertion that the label is programmatically associated (`getByLabelText`).
2. **Standalone, no Field** — the actual hero-demo pattern: plain `<label>` wrapping `<Input placeholder=… />`, no `Field.Root` anywhere (brief §5, §6). Proves Input never throws or misbehaves outside a Field tree and that `data-disabled` is the only attribute that still means anything (brief §1, §6). Play: assert no `data-touched`/`data-valid` attributes are present even after interacting.
3. **Validation states styling** — `Field.Root` with `required` + `Field.Error`, Input as the control. Play: `render` → assert no invalid styling before interaction → blur empty required field → `waitFor` `data-invalid`/`data-touched` to appear on the `Input` element itself (not just Field.Root) → type a value → `waitFor` `data-dirty` appears, `data-invalid` clears. One assertion per `waitFor`, per AGENTS.md test guidelines. Serves the a11y contract section (aria-invalid) and the styling-contract section (DoD §10) simultaneously.
4. **Disabled, cascading from Fieldset** — `Fieldset.Root disabled` wrapping a `Field.Root > Field.Label + Input`. Play/assertion: `Input` receives `disabled` and `data-disabled` despite never receiving the prop directly — demonstrates the native `disabled` cascade fieldset brief §6 documents and that Fieldset's test suite (`FieldsetRoot.test.tsx` "keeps nested fieldsets disabled...") already pins for `Field.Control`; Input inherits it unchanged. Cross-links fieldset's disabled-cascade story.
5. **Standalone with `aria-label`** — no wrapping label, no Field — `<Input aria-label="Search" />`. Proves the "consumer owns the accessible name" rule (brief §7) has more than one valid resolution; useful negative-space story showing what happens when neither a `<label>` nor Field.Label is present (a11y-tree assertion: accessible name equals the `aria-label` value; contrast against a bare unlabeled `<Input />` which the story explicitly does NOT render, to avoid demonstrating an anti-pattern without commentary).
6. **Controlled value with external clear button** — addresses the open pitfall (#4643, brief §10 item 2): a controlled `value`/`onValueChange` pair plus an external "Clear" button that resets `value` to `''`. Since the upstream issue's resolution was never read (gh budget reserved for higher-value fieldset lookups), this story is written from first principles (the standard controlled-triplet pattern) and tagged `needs-work` until cross-checked against the issue's actual resolution — flag this explicitly in the story's docstring/comment.
7. **Form submit** — `Input` with `name` inside `<Form>`, submitting and asserting the value is present in the submitted `FormData`/`onSubmit` payload — exercises the hidden-input/native-submission contract Input inherits wholesale from `Field.Control` (brief §6, cross-referencing field brief's Form-error-interplay section). Reuses the Form component's existing test-proven submit pattern rather than inventing a new one.

**Omitted per brief §12**: format-on-blur (no such capability exists on Input — it belongs to NumberField only; fabricating a story here would misrepresent the API).

**MDX docs page** notes: since Input's own docs page today has almost no prose (one Usage-guidelines bullet, no Examples section), the MDX docs page should lean heavily on cross-links to Field's page/brief for anything beyond "this is a native input" — do not duplicate Field's validation-mode/eventDetails explanations verbatim; link out per the library's own composition-over-duplication convention (AGENTS.md: "Avoid duplicating logic where necessary").
