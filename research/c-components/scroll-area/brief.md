# Scroll Area ‚Äî component research brief

Tier 2 (full brief, capped mining). Mined 2026-07-07 from source, docs, tests, experiments, and mui/base-ui git history (1 `gh` PR lookup: the introducing PR #665). Evidence tags: [E] direct evidence, [I] inference, [G] gap.

## 1. Identity

- **Name / slug**: Scroll Area ‚Äî `@base-ui/react/scroll-area`. Multi-part compound component, namespace export `ScrollArea.*`.
- **Status**: stable, no `[New]`/`[Preview]` tag. Introduced 2024-11-05 (commit `2d1cd9e0c`, mui/base-ui#665). [E]
- **Purpose / IA statement**: "A native scroll container with custom scrollbars" [E] (`docs/src/app/(docs)/react/components/scroll-area/page.mdx` Subtitle). It is explicitly **not** a virtualized-scroll or custom-scroll-physics component ‚Äî it wraps the browser's own `overflow: scroll` behavior and layers styleable decoration on top. Taxonomy: **Disclosure & structure** category ‚Äî "custom scrollbars over native scrolling" [E] (`research/a-documentation-standard/taxonomy.md`).
- **6 parts** (from `packages/react/src/scroll-area/index.parts.ts`; one-liners from part JSDoc):
  | Part | Renders | One-liner |
  |---|---|---|
  | Root | `<div>` | Groups all parts of the scroll area |
  | Viewport | `<div>` | The actual scrollable container of the scroll area |
  | Content | `<div>` | A container for the content of the scroll area |
  | Scrollbar | `<div>` | A vertical or horizontal scrollbar for the scroll area |
  | Thumb | `<div>` | The draggable part of the scrollbar that indicates the current scroll position |
  | Corner | `<div>` | A small rectangular area that appears at the intersection of horizontal and vertical scrollbars |
- No internal-only/`@internal` parts. [E]

## 2. Intention

- [E] **Introducing PR**: mui/base-ui#665 "[ScrollArea] Create new `ScrollArea` component" (atomiks, merged 2024-11-05, commit `2d1cd9e0c`), closing mui/mui-x#649.
- [E] **Explicit rationale from the PR thread**, not the docs: oliviertassinari initially floated scoping it out of core into a hypothetical "Base UI X" premium tier ("most applications don't have a custom scrollbar‚Ä¶ it shows the potential opportunity cost of having it in the core"). colmtuite (maintainer) pushed back directly: "It's just a basic building block of high-quality web dev‚Ä¶ Very many web apps (Slack, Linear, Radix docs etc.) have custom scrollbars. It's not possible to build a high-quality web UI without them‚Ä¶ We need it for our new docs (for the sidebar and code blocks)." The component shipped in core, settling the debate in practice. [E]
- [E] **Design pole: decorate native scrolling, never replace it.** The component does not reimplement scroll physics ‚Äî it reads real `scrollTop`/`scrollLeft`/`scrollHeight` etc. from the underlying `<div>` and drives thumb position/visibility from those measurements (`ScrollAreaViewport.tsx` `computeThumbPosition`). Dragging the thumb or clicking the track writes back to the native `scrollTop`/`scrollLeft` properties directly (`ScrollAreaRoot.tsx` `handlePointerMove`; `ScrollAreaScrollbar.tsx` `onPointerDown`) rather than maintaining a parallel virtual scroll state. [E]
- [E] **API naming settled through review, not shipped as first proposed**: the PR review (vladmoroz) pushed back on several names that did not survive to release ‚Äî a `type="inlay"` scrollbar-style prop (named after VS Code's "inlay hints"; atomiks explained the etymology in review) and a `gutter` prop were both removed before/shortly after merge (`gutter` removed in #1023, "Remove `gutter` prop"); a `hidden` force-visibility prop was reconsidered because it read as inconsistent with `keepMounted` conventions elsewhere in the library. What shipped is leaner than the first draft. [E]
- [E] **Safari forced the injected-`<style>` design** (see ¬ß5, ¬ß8): "I think we can remove them by default, but Safari requires a `<style>` tag, right?" (atomiks, PR #665 review) ‚Äî WebKit does not support hiding scrollbars via inline `style` attributes alone for `scrollbar-width`/`::-webkit-scrollbar`, so a real `<style>` element is unavoidable. This is the origin of the CSP exception documented in `research/b-library-principles/principles.md` B-P22.
- [E] Performance was treated as an ongoing identity trait post-launch, not an afterthought: dedicated perf commits `[scroll area] Perf improvements` (#3536), `[scroll area] Improve CSS vars performance` (#3156, disables CSS custom-property inheritance via `CSS.registerProperty` ‚Äî see ¬ß6), and `[scroll area] Fix thumb-drag divide-by-zero and per-scroll re-render` (#5099, most recent). [E]

## 3. When to use

- [E] Any custom-styled scrollable region where the browser's native scrollbar chrome should be replaced with themeable markup, while keeping native scroll behavior (momentum, keyboard, wheel, touch) intact ‚Äî the canonical "Slack/Linear-style app shell" case cited in the introducing PR discussion. [E]
- [E] Long lists or overflowing panels where a **gradient scroll-fade** affordance is wanted at the edges, driven by the viewport overflow CSS variables (docs "Gradient scroll fade" example, `--scroll-area-overflow-y-start`/`-end`). [E]
- [E] Popups/menus that need a scrollable region **inside** their popup content while keeping the popup's own layering/positioning untouched ‚Äî verified by two dedicated experiments, `experiments/scroll-area/inside-select.tsx` (ScrollArea wrapping 300 `Select.Item`s inside `Select.Popup`, with `alignItemWithTrigger={false}` and `tabIndex={-1}` on the Viewport) and `experiments/scroll-area/inside-menu.tsx` (ScrollArea wrapping 100 `Menu.Item`s plus a live submenu inside `Menu.Popup`). [E] Both experiments explicitly set `tabIndex={-1}` on `ScrollArea.Viewport`, foreshadowing the later `#4220` focus-trap fix (¬ß6).
- [E] Composing with `<Tabs.List>` when the tab list itself needs the viewport overflow values for a mask fade ‚Äî the docs "Combining with Tabs" example renders `<Tabs.List render={<ScrollArea.Viewport />}>` directly so the same element receives both the tablist semantics and the scroll state. [E]
- [I] Sidebars, code blocks, and dashboard panels with fixed dimensions and unpredictable content length ‚Äî inferred from colmtuite's PR-review justification ("We need it for our new docs (for the sidebar and code blocks)") and the "both scrollbars" grid demo.

## 4. When not to use + alternatives

- [E] **Large virtualized lists** ‚Äî Scroll Area does not virtualize; it only decorates a real overflowing `<div>`. [G] No maintainer statement explicitly contrasting Scroll Area with a virtualization library was found in the mined corpus (`discussions.md`'s per-component index has no entries at all for scroll-area ‚Äî the searched issue corpus is essentially silent on this component). Treat virtualization as an orthogonal, compatible concern the app must add itself if content is huge ‚Äî [I] inferred from the component's own measurement code operating on real `scrollHeight`/`scrollWidth`, which would need to reflect actual (not virtual) content size.
- [E] **Average, non-decorated overflow** ‚Äî oliviertassinari's PR-review counterpoint stands as a documented (if not adopted) position: "For the average UIs, I don't think it's a big need‚Ä¶ I personally never implemented a custom scrollbar." Plain CSS `overflow: auto` with the browser's native scrollbar remains the simpler default when no custom styling of the scrollbar itself is required. [E]
- [E] **Select's own popup scrolling** ‚Äî `Select.Popup`/`Select.List` has its own built-in scroll handling plus dedicated `Select.ScrollUpArrow`/`Select.ScrollDownArrow` parts (hover-to-scroll arrows, not rendered on touch) that are architecturally independent of `ScrollArea.*` ‚Äî they do not share a context or composition point. [E] (`packages/react/src/select/scroll-up-arrow/SelectScrollUpArrow.tsx`; select brief ¬ß5). [I] The `inside-select.tsx` experiment demonstrates that `ScrollArea` *can* be nested inside `Select.Popup` as an alternative scrolling mechanism (with `alignItemWithTrigger={false}`, since item-aligned positioning and a custom scroll viewport don't combine cleanly), but this is a documented experiment, not the default select scrolling story ‚Äî the two mechanisms are not designed to interoperate structurally, only to coexist when deliberately composed.
- [I] Full cluster context is thin: this is one of the two named "unstyled" exceptions library-wide (B-P22), and the taxonomy places it in Disclosure & structure alongside Accordion/Collapsible/Separator, but no cluster note commits ScrollArea to a direct comparison the way pickers/overlays are compared elsewhere ‚Äî [G] no dedicated cluster covers scroll-area; if one is warranted it would sit with virtualization-adjacent concerns, which are out of Base UI's scope entirely.

## 5. Anatomy & composition

- [E] Canonical tree (docs Anatomy):
  `Root > (Viewport > Content, Scrollbar > Thumb, Scrollbar[orientation=horizontal] > Thumb, Corner)`.
- [E] **Root is a non-scrolling coordination layer** (`role="presentation"`) that owns all shared refs/state via `ScrollAreaRootContext` ‚Äî hover state, scrolling-x/y state, corner/thumb sizing, overflow-edge state, drag handlers ‚Äî and injects the scrollbar-hiding `<style>` tag as a sibling of its rendered element (`ScrollAreaRoot.tsx`: `{!disableStyleElements && styleDisableScrollbar.getElement(nonce)}` immediately before `{element}`). [E]
- [E] **Viewport is the only element that actually scrolls**: it renders with inline `style={{ overflow: 'scroll' }}` and the injected `base-ui-disable-scrollbar` class (`className: styleDisableScrollbar.className`), which sets `scrollbar-width: none` and `::-webkit-scrollbar{display:none}` to hide the native scrollbar while the custom `Scrollbar`/`Thumb` parts render the visible chrome. `role="presentation"`. [E] Its `tabIndex` is computed, not fixed: `hiddenState.x && hiddenState.y ? -1 : 0` ‚Äî it only enters the tab sequence when there is real overflow to scroll (accessibility-insights guidance cited directly in a source comment: "Keep non-scrollable viewports out of tab order," linking `https://accessibilityinsights.io/info-examples/web/scrollable-region-focusable/`). [E]
- [E] **Content is optional but recommended**: it renders `role="presentation"` with `style={{ minWidth: 'fit-content' }}` and is the element a `ResizeObserver` watches to recompute thumb geometry when content size changes independently of viewport size. Its own overflow-state data attributes mirror the Root/Viewport (useful for edge-fade masks applied directly to Content instead of Viewport).
- [E] **Scrollbar is conditionally rendered**: `shouldRender = keepMounted || !isHidden` ‚Äî by default (`keepMounted={false}`) a scrollbar for an axis with no overflow does not render at all, not merely hidden via CSS; `keepMounted` forces it to stay in the DOM (useful for measuring/animating a scrollbar that appears later). Before the very first thumb-position measurement completes, the track is set `visibility: hidden` via `hideTrackUntilMeasured` to avoid a flash of a wrongly-sized/positioned track. [E]
- [E] **Thumb has no independent geometry logic**: its size comes entirely from Root-level CSS custom properties set by the Scrollbar (`--scroll-area-thumb-height`/`--scroll-area-thumb-width`, defined in `ScrollAreaScrollbarCssVars`), and its position from a directly-mutated `transform: translate3d(...)` set imperatively in `computeThumbPosition` (bypassing React state, similar in spirit to Select's `data-highlighted` fast path). Its own `visibility` stays `hidden` until `hasMeasuredScrollbar` is true. [E]
- [E] **Corner only renders when both axes overflow simultaneously** (`hiddenState.corner`); its size is derived from the scrollbar thickness of whichever axis is present, propagated back to Root via `--scroll-area-corner-height`/`--scroll-area-corner-width` CSS vars so `Scrollbar` tracks can reserve space for it (`bottom: var(--scroll-area-corner-height)` on the vertical track, etc.) without a layout measurement round-trip. [E] The docs explicitly recommend rendering `<ScrollArea.Corner>` whenever both scrollbars are used, "to prevent the scrollbars from intersecting."
- [E] **Inline-style / injected-`<style>` caveat (CSP-relevant)**: `<ScrollArea.Viewport>` is one of exactly two components library-wide that inject a `<style>` tag out of necessity (the other is `Select.Popup`/`Select.List` under `alignItemWithTrigger`) ‚Äî "inject a style tag to disable native scrollbars," governed by `CSPProvider`'s `nonce`/`disableStyleElements` (`docs/src/app/(docs)/react/utils/csp-provider/page.mdx`; `research/b-library-principles/principles.md` B-P22). The tag itself: `<style nonce={nonce} href="base-ui-disable-scrollbar" precedence="base-ui:low">.base-ui-disable-scrollbar{scrollbar-width:none}.base-ui-disable-scrollbar::-webkit-scrollbar{display:none}</style>` (`packages/react/src/utils/styles.tsx`). Safari's WebKit engine is the forcing constraint (¬ß2). [E]
- Text spec for a future anatomy diagram: (1) Root [invisible coordination layer], (2) Viewport [scrollable, `overflow:scroll`], (3) Content [measured, `min-width:fit-content`], (4) Scrollbar (vertical), (5) Thumb (vertical), (6) Scrollbar (horizontal), (7) Thumb (horizontal), (8) Corner [only when both 4 and 6 present].

## 6. Behavior ("How it works")

- [E] **No controlled/uncontrolled value at all** ‚Äî unlike most Base UI compound components, Scroll Area has no `value`/`onValueChange` or `open`/`onOpenChange` pair. Its only "state" is derived measurement (hover, scrolling, overflow edges), not something the consumer sets. [E] (confirmed by exhaustive read of `ScrollAreaRootProps` ‚Äî the only public prop is `overflowEdgeThreshold`).
- [E] **Measurement pipeline**: `computeThumbPosition` (in `ScrollAreaViewport.tsx`) is the single source of truth, invoked from the native `onScroll` handler, a `ResizeObserver` on the viewport, a `ResizeObserver` on the Content wrapper, and after any subtree animation finishes (`viewport.getAnimations({ subtree: true })`, awaited via `Promise.allSettled`). It computes: hidden-state per axis (`clientHeight >= scrollHeight`, i.e. no overflow), thumb size/position (clamped to `MIN_THUMB_SIZE = 16`px), corner size, and the four overflow-edge CSS variables/data-attributes, then writes them via `setState` calls that all bail out on referential-equal no-op updates (explicit comments call out avoiding "a fresh object literal on every scroll frame" from rebuilding the whole root context and re-rendering every scroll-area part). [E]
- [E] **Scrolling state and its timeout**: `data-scrolling` is set on Root/Viewport/Scrollbar/Thumb/Content while a scroll is in progress and cleared after `SCROLL_TIMEOUT = 500`ms of inactivity (`constants.ts`), independently tracked per axis (`scrollingX`/`scrollingY`) so a horizontal-only style rule doesn't fire during vertical-only scrolling. [E] Programmatic scrolls (assigning `scrollTop` directly, e.g. from thumb-drag) are explicitly excluded from marking `data-scrolling` via wheel/touch/pointer/keydown listeners that flip a `programmaticScrollRef` flag, debounced by a 100 ms "scroll end" heuristic (`scrollend` event threshold cited via MDN link in source) so momentum scrolling still counts as user-driven even though no further interaction events fire during it. [E]
- [E] **Hover state**: tracked via `onPointerEnter`/`onPointerMove` on Root testing `contains(rootRef.current, event.target)`, explicitly excluded for `pointerType === 'touch'` (`handlePointerEnterOrMove`); a mount-time check (`viewportRef.current?.matches(':hover')`) compensates for `onMouseEnter` not firing on load if the pointer is already over the region. [E] Fixed bug: pointer events from sibling portals no longer spuriously trigger hover (#2542, "Prevent pointer events from sibling portals triggering hover"). [E]
- [E] **Thumb dragging**: `setPointerCapture`/`releasePointerCapture` on the Thumb element itself; drag delta is translated into a scroll-ratio against the track's available travel distance (`scrollbarEl.offsetHeight - thumbHeight - padding - margin`), guarded against divide-by-zero/negative ranges when a short or heavily padded track drives the denominator to zero (explicit source comments on this guard in both `ScrollAreaRoot.tsx` and `ScrollAreaScrollbar.tsx`; most recently hardened in #5099 "Fix thumb-drag divide-by-zero and per-scroll re-render"). `pointercancel` is handled defensively since it implicitly releases capture ‚Äî code checks `hasPointerCapture` before calling `releasePointerCapture` to avoid throwing. [E]
- [E] **Track click**: clicking the Scrollbar track (not the Thumb ‚Äî clicks routed to the thumb, including through a shadow-DOM retarget, are ignored via `contains(thumb, target)`) jumps the thumb so its center lands under the pointer, then converts that offset into an absolute `scrollTop`/`scrollLeft`. [E]
- [E] **Wheel handling on the Scrollbar track itself** (not just the Viewport): a `wheel` listener (`{ passive: false }`) on the Scrollbar redirects wheel deltas to the Viewport's scroll properties directly, with edge-chaining ‚Äî "At an edge (or with no overflow), let the wheel event chain to the parent/page instead of swallowing it via `preventDefault`" ‚Äî so scrolling over the scrollbar chrome itself still respects page-level scroll chaining at the boundaries. Ctrl+wheel (pinch-zoom gesture) is explicitly excluded (`event.ctrlKey` check). [E]
- [E] **RTL support**: horizontal scroll math is direction-aware throughout (`useDirection()` from the internal `DirectionContext`) ‚Äî RTL uses a negative `scrollLeft` range from `0` to `-maxScroll` in some browsers, and the code branches for both conventions; a Safari-specific RTL horizontal-scrollbar fix shipped separately (#3776, "Fix RTL horizontal scrollbar on Safari"). Logical CSS properties (`insetInlineStart`/`insetInlineEnd`) are used throughout instead of physical `left`/`right` so the whole component flips correctly under `DirectionProvider` (originally #1194, "Read `DirectionProvider` and use logical positioning CSS props"). [E]
- [E] **CSS variable inheritance is deliberately disabled** for the four overflow CSS vars, via `CSS.registerProperty({ inherits: false, ... })`, registered once module-wide ‚Äî "removes inheritance‚Ä¶ which improves rendering performance in complex scroll areas with deep subtrees. Instead, each child must manually opt-in‚Ä¶ by specifying `inherit`" (source JSDoc, citing a Motion.dev blog post on CSS variable performance). This optimization is **skipped entirely on WebKit** (`platform.engine.webkit` check) because Safari does not support child `inherit` opt-in when `inherits: false` is set ‚Äî a documented, source-level browser carve-out. [E] (#3156 "Improve CSS vars performance"; #3208 "Fix CSS vars inheritance on Safari").
- [E] **Focus trap interaction (Dialog-adjacent fix)**: `#4220` fixed a focus-trap bug specific to a *non-scrollable* Viewport ‚Äî the fix and its accompanying test live in `DialogRoot.test.tsx`, confirming ScrollArea's Viewport `tabIndex` logic (¬ß5) directly affects focus-trap cycling when a ScrollArea sits inside a modal Dialog. [E]
- [G] No SSR-specific fix commits were found in the scroll-area git log (contrast with Select's `suppressHydrationWarning` fixes) ‚Äî the component appears not to need special hydration handling beyond the general library-wide `useIsoLayoutEffect` convention already in use throughout its source.

## 7. Accessibility contract

- [E] **No dedicated WAI-ARIA APG pattern exists for "custom scrollbar decoration over native scroll"** ‚Äî there is no ARIA "scrollbar" widget role in play here by design. Root and Viewport both render `role="presentation"`; Scrollbar/Thumb/Corner carry no ARIA roles or states at all (confirmed by exhaustive read ‚Äî no `role=`, `aria-*` prop, anywhere in `ScrollAreaScrollbar.tsx`, `ScrollAreaThumb.tsx`, or `ScrollAreaCorner.tsx`). [E] This is a deliberate, honest design choice, not an oversight: the actual scrolling affordance and its accessible semantics belong entirely to the native, real `overflow: scroll` `<div>` (the Viewport) ‚Äî the decorative Scrollbar/Thumb/Corner parts are purely visual/pointer-driven mirrors of what the browser already exposes correctly.
- [E] **Keyboard scrolling stays fully native and is not reimplemented**: no keyboard event handling exists anywhere in `ScrollAreaScrollbar.tsx` or `ScrollAreaThumb.tsx` for arrow keys, Page Up/Down, Home/End, or Space ‚Äî all keyboard scrolling works because the Viewport `<div>` is a real scrollable element that the browser already handles when it is focused (which is precisely why the `tabIndex` computation in ¬ß5/¬ß6 matters: a non-overflowing Viewport is deliberately taken **out** of the tab order per the cited accessibility-insights guidance, and an overflowing one is deliberately kept **in** it at `tabIndex={0}`). There is no custom Scroll Area keyboard interaction table to document beyond "the Viewport is (or isn't) tabbable, and once focused, all native scroll keys work exactly as they would on any real scrollable region." [E]
- [E] **The Thumb and Scrollbar are pointer-only affordances, honestly**: dragging the Thumb or clicking the Scrollbar track are convenience shortcuts for sighted mouse/touch users; nothing in source makes the Thumb keyboard-focusable or keyboard-operable (no `tabIndex`, no `onKeyDown` anywhere in `ScrollAreaThumb.tsx`). A keyboard or screen-reader user scrolls via the native Viewport instead, which remains fully capable. [I] This matches the honesty framing the task calls for: decorative scrollbar chrome vs. native scroll semantics ‚Äî the chrome is genuinely decorative, and the semantics are genuinely native, with no gap papered over by ARIA roles that would misrepresent non-interactive `<div>`s as interactive scrollbar widgets.
- [G] Searched `discussions.md`'s per-component index and full-text for "scroll area"/"scrollarea"/"scrollbar" ‚Äî **the mined issue/discussion corpus has no entries at all for this component** (it is entirely absent from the per-component index table, unlike every other Tier 1/2 component). No open accessibility bug reports, no screen-reader-specific complaints, and no APG-alignment discussion were found for Scroll Area in the corpus mined for Deliverable B. This is itself worth stating plainly rather than implying non-existent research depth: either the component genuinely generates very little support friction (plausible, given how thin its public API surface is ‚Äî a single `overflowEdgeThreshold` prop plus 6 parts with almost no configuration), or it has simply not been targeted by a dedicated issue search in prior mining passes. Both are live possibilities and neither can be ruled out from this pass alone.
- [I] **Touch**: `handleTouchModalityChange`/`touchModality` state exists in the Root context but nothing in the read source branches Scrollbar/Thumb rendering on it directly the way Select's scroll arrows hide on touch ‚Äî [G] the touch-modality state's downstream consumer(s) were not traced beyond its presence in context; flagged rather than asserted.

## 8. Prop-level guidance (decision-relevant props)

- [E] **`overflowEdgeThreshold` (Root, default `0`)** ‚Äî the only meaningfully configurable prop on Root. Accepts a single number (applied to all four edges) or a partial `{ xStart, xEnd, yStart, yEnd }` object; values are clamped to `>= 0`. Controls how many pixels of scroll must occur past an edge before that edge's `data-overflow-*-start`/`-end` attribute (and matching CSS var) activates ‚Äî useful for suppressing edge-fade flicker on the very first pixel of scroll. Changing it recomputes overflow edges reactively (pinned by test "recomputes overflow edges when overflowEdgeThreshold changes"). [E]
- [E] **`orientation` (Scrollbar, default `'vertical'`)** ‚Äî the only way to get a horizontal scrollbar; there is no `orientation="both"` ‚Äî render two `<ScrollArea.Scrollbar>` elements (one per orientation) as the docs' "Both scrollbars" example does, plus a `<ScrollArea.Corner>` to prevent them intersecting.
- [E] **`keepMounted` (Scrollbar, default `false`)** ‚Äî set `true` to keep a currently-non-overflowing axis's Scrollbar/Thumb DOM present (still visually hidden until measured) rather than unmounting it entirely; needed for exit-transition styling on a scrollbar that becomes hidden, or to avoid layout thrash when overflow toggles frequently. `describeConformance` for Scrollbar is run with `keepMounted` explicitly set, signaling this is the conformance-relevant default to test against. [E]
- [E] **Overflow data attributes/CSS vars are the primary styling surface** (from `*DataAttributes.ts`/`*CssVars.ts`, mirrored across Root/Viewport/Content): `data-scrolling` (per-axis-aware but a single boolean per element ‚Äî true if either axis is scrolling), `data-has-overflow-x`/`data-has-overflow-y` (static: is there any overflow at all on this axis), `data-overflow-x-start`/`-x-end`/`-y-start`/`-y-end` (dynamic: has the user scrolled far enough from that specific edge, gated by `overflowEdgeThreshold`); the four numeric CSS vars `--scroll-area-overflow-{x,y}-{start,end}` (pixel distance from each edge, only set on Viewport) are the mechanism behind the docs' gradient-scroll-fade recipe. `Scrollbar` additionally exposes `data-orientation`, `data-hovering`, `data-visible` is **not** an attribute here (contrast Select's ScrollArrows) ‚Äî visibility is controlled by conditional rendering (`shouldRender`), not a data attribute. [E]
- [E] **`--scroll-area-thumb-height`/`-width` and `--scroll-area-corner-height`/`-width`** are internal plumbing CSS vars (Scrollbar/Root respectively) that consumers do not need to set themselves ‚Äî they document the sizing contract for anyone inspecting computed styles, not a public configuration surface. [E]
- [E] **`data-scrolling` excludes programmatic scroll** ‚Äî a documented, deliberate behavior (#1908, "Ignore `data-scrolling` during programmatic scroll"), meaning a style keyed to `[data-scrolling]` will not fire when, e.g., a consumer calls `scrollIntoView()` or sets `scrollTop` directly from application code, only for user-driven wheel/touch/keyboard/thumb-drag scrolling.
- [G] No prop exists to force scrollbars permanently visible without `keepMounted` doing double duty as both "stay mounted" and (via CSS keyed on the absence of `data-hovering`/`data-scrolling`) "stay visually visible" ‚Äî an "always visible" style is achieved purely in the consumer's own CSS by *not* gating opacity/visibility on `[data-hovering]`/`[data-scrolling]` at all, not via a Base UI prop. This is worth calling out precisely in a docs Examples section (no such example currently exists in `page.mdx` ‚Äî only the hover/scroll-conditional hero style is shown).

## 10. Pitfalls & FAQ

- **Scrollbar doesn't show at all even though content overflows** ‚Üí check `keepMounted` isn't accidentally masking a real measurement problem, and confirm the Scrollbar isn't relying on `[data-hovering]`/`[data-scrolling]` CSS that never matches your interaction pattern (e.g., styling only `:hover` on a touch device, where `hovering` state is deliberately excluded ‚Äî `handlePointerEnterOrMove` ignores `pointerType === 'touch'`). [E, from source]
- **Custom scrollbar styling looks fine except on Safari** ‚Üí the scrollbar-hiding mechanism on Viewport requires an injected `<style>` tag specifically because Safari doesn't respect `scrollbar-width`/`::-webkit-scrollbar` via inline `style` attributes; if a strict CSP blocks the tag without a nonce, wrap the tree in `<CSPProvider nonce={...}>` (or `disableStyleElements` + manual CSS if you'd rather not use a nonce at all). [E, docs `csp-provider/page.mdx`]
- **Wrong scroll direction / broken clicks in RTL** ‚Üí confirm the tree is under a `DirectionProvider` with `dir="rtl"` (ScrollArea reads it, but does not set `dir` itself, per the library-wide DirectionProvider convention ‚Äî see `research/b-library-principles/principles.md`), and note Safari needed a dedicated RTL horizontal-scrollbar fix historically (#3776-era). [E]
- **A style keyed on `[data-scrolling]` never fires for an app-triggered scroll** (e.g., "scroll to top" button) ‚Üí intentional; `data-scrolling` is defined to track *user*-driven scrolling only (#1908). Drive your own state for programmatic-scroll styling if needed. [E]
- **Nesting inside `Select.Popup` and expecting `Select.ScrollUpArrow`/`ScrollDownArrow` to combine with it** ‚Üí they don't share any composition point; the two experiments that verify ScrollArea-inside-a-popup (`inside-select.tsx`, `inside-menu.tsx`) use `ScrollArea.*` as the *entire* scroll mechanism for that popup, not alongside Select's own scroll-arrow parts. Pick one mechanism per popup, not both. [E, from experiments]
- **Focus gets trapped strangely when a ScrollArea with no overflow sits inside a modal Dialog** ‚Üí fixed upstream (#4220); if seeing this on an older version, upgrade ‚Äî the fix is specifically about the Viewport's `tabIndex` computation interacting with Dialog's focus trap.
