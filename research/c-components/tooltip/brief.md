# Tooltip — component research brief

Tier 2 (full brief, capped mining — top ~8 PRs/issues). Mined 2026-07-07 from source, tests, docs, git history, and mui/base-ui issues/PRs (read-only). Evidence tags: [E] direct evidence, [I] inference, [G] gap.

## 1. Identity

- **Name / slug**: Tooltip — `@base-ui/react/tooltip`. Multi-part compound component, namespace export `Tooltip.*`.
- **Status**: stable, no `[New]`/`[Preview]` tag. One of the oldest lineages in the repo — introduced 2024-05-29 (mui/base-ui#264, first commit `ace9e1703`), predating most of today's popup infrastructure. [E]
- **Purpose / IA statement**: "A popup that appears when an element is hovered or focused, showing a hint for sighted users" [E] (docs Subtitle). It is a **visual-only, non-essential** supplementary label — never a container for content a user must be able to access via touch or screen reader [E] (docs Usage guidelines, §3–4 below). Taxonomy: **overlays/popups** cluster [E] (`research/a-documentation-standard/taxonomy.md` line 36: "tooltip | anchored, hover/focus, informational, never focusable content"); full comparative treatment in [`../_clusters/overlays.md`](../_clusters/overlays.md).
- **8 parts** (from `packages/react/src/tooltip/index.parts.ts`; one-liners from part JSDoc):

  | Part | Renders | One-liner |
  |---|---|---|
  | Provider | no element | "Provides a shared delay for multiple tooltips. The grouping logic ensures that once a tooltip becomes visible, the adjacent tooltips will be shown instantly." |
  | Root | no element | Groups all parts of the tooltip; doesn't render its own HTML element |
  | Trigger | `<button>` | "An element to attach the tooltip to" |
  | Portal | `<div>` | Moves the popup to a different part of the DOM (default: `<body>`) |
  | Positioner | `<div>` | Positions the tooltip against the trigger |
  | Popup | `<div>` | A container for the tooltip contents |
  | Arrow | `<div>` | Displays an element positioned against the tooltip anchor |
  | Viewport | `<div>` | A viewport for displaying content transitions (optional — only for multi-trigger, animated content) |
- Non-element API: `Tooltip.createHandle()` / `Tooltip.Handle` — imperative control + detached-trigger association (see §6, §9). [E] `index.parts.ts`
- **Preview Card differentiator** (for the sibling brief): Preview Card shares this exact Trigger/Portal/Positioner/Popup/Arrow/Viewport skeleton almost part-for-part [I] (structural comparison, not independently verified against preview-card source per this task's scope) but **has no `Provider`**. Tooltip's Provider is the one part in this family with no Preview Card counterpart — it exists specifically because tooltips are expected to appear in dense clusters (icon toolbars, data tables) where sequential hovers should feel instant after the first "warm-up," a affordance Preview Card's link-hover use case does not need to the same degree. [I] inferred from Provider's stated purpose plus the absence of any `PreviewCardProvider` file reference anywhere in `_clusters/overlays.md` or the tooltip docs' own "Detached triggers"/"Multiple triggers" sections, which describe Provider-like grouping only in Tooltip terms.

## 2. Intention

- [E] **Introducing PR**: mui/base-ui#264 "[Tooltip] Component and Hook" (atomiks, merged 2024-05-29, commit `ace9e1703`), closing #32. PR body: "a lot of the implementation details are tested by Floating UI already, so we only need to test the wrapping APIs" — signaling from day one that Tooltip is conceived as a thin, well-tested wrapper around Floating UI's hover/focus/dismiss primitives rather than a novel interaction model.
- [E] **Provider shipped in the same introducing PR** — `TooltipProvider.tsx` has no separate `git log --diff-filter=A` commit distinct from #264; the delay-grouping mechanism (`FloatingDelayGroup`) was part of the component's initial design, not a later addition.
- [I] **Design pole: a tooltip is a visual annotation on something whose real purpose lies elsewhere.** This is the maintainers' own explicit mental model, stated most fully in mui/base-ui#3530 (atomiks): "A popover's trigger's purpose is to open the popover itself (and every platform is supported), while a tooltip's trigger's purpose is unrelated to the tooltip (and touch platforms are excluded)." The entire component — hover/focus-only opening, no ARIA role, no `aria-describedby` wiring (see §7) — follows from this one sentence. [E] cited fully in `../_clusters/overlays.md`.
- [E] **Explicit non-goals** (docs "Alternatives to tooltips" section, `page.mdx`):
  - Not a way to deliver **essential/critical information** — "A user should not miss critical information if they never see a tooltip."
  - Not for **touch input** — "tooltips are disabled on touch devices" by design, because iOS has no system-standard touch tooltip affordance and Android's long-press convention conflicts with the browser's own contextual-menu gesture.
  - Not a substitute for **labeling the trigger** — "The tooltip's trigger must have an `aria-label` attribute that closely matches the tooltip's content" (Usage guidelines bullet 2); the tooltip is additive polish, not the accessible name.
  - Not for **"Copied!"-style contextual feedback** — docs explicitly redirect this use case to an anchored Toast "to ensure the message is announced to screen readers" (`page.mdx` "Contextual feedback messages"; cross-referenced in `../_clusters/overlays.md` Toast section, #3096).
  - Not for **info-icon popups whose entire purpose is to be opened** — that's Popover with `openOnHover` (docs "Infotips"; #3530).
- [E] Performance and correctness under **dense clusters of tooltips** is treated as a first-class design concern, evidenced by a long, still-active bugfix lineage specifically about groups of adjacent tooltips: shared-tooltip-across-trigger-gaps fix (#3452), contained-triggers perf (#3385), detached-trigger perf run across the whole test suite (#3479), provider delay-group lifecycle fix (#4887), nested-tooltip-inside-trigger flicker fix (#4501). [E] `git log --grep='^\[tooltip\]'`

## 3. When to use

- [E] Labeling an icon-only button or other control whose accessible name needs a visible, sighted-user-facing echo (docs Usage guidelines bullet 2: pair the tooltip content with the trigger's `aria-label`).
- [E] Supplying "extra clarity for sighted mouse or keyboard users" when the underlying description is **not critical** to understanding the element (docs "Description text").
- [E] Any UI where the trigger's own purpose (a save button, a delete icon, a toolbar action) is unrelated to "opening a popup" — the tooltip is a bonus annotation on top of an already-functional control (#3530 litmus test, canonized in docs as: "If the trigger's purpose is unrelated to opening the popup, it's a tooltip").
- [E] Dense clusters of related controls (icon toolbars, data-grid header cells) where `Tooltip.Provider` lets a user sweep across several triggers and get near-instant tooltips after the first one's full delay (see §6) — this is the scenario the Provider's own JSDoc and its dedicated tests exist for.
- [E] Content that can also be conveyed by animating/transitioning between multiple trigger-specific payloads without a full remount — the `Viewport` + `payload`/`Handle` combination (docs "Animating the Tooltip" + "Detached triggers with payload").

## 4. When not to use + alternatives

Full comparative guidance, decision table, and the complete #3530 mental model: [`../_clusters/overlays.md`](../_clusters/overlays.md) — read that note first; only the tooltip-specific nuance not already stated there is repeated here.

- [E] **Info icon whose whole job is to reveal more content → Popover with `openOnHover`.** "This way, touch users and screen reader users can access the content" (docs "Infotips"). This is the single most load-bearing alternative and is the crux of #3530.
- [E] **Description text that matters to understanding the element → inline text, or Popover if space-constrained.** "If the description is important to understanding the element, don't hide it behind a tooltip" (docs "Description text").
- [E] **Transient contextual feedback ("Copied", "Saved") → an anchored Toast, not a Tooltip.** Stated on the tooltip docs page itself ("Contextual feedback messages") and reinforced in `../_clusters/overlays.md`'s Toast section (#3096) — this is the one alternative most likely to be missed by someone reaching for Tooltip out of habit, since a tooltip-styled feedback bubble looks superficially similar to a toast.
- [E] **Rich, hoverable link previews → Preview Card**, not Tooltip — both are hover-only and sighted-only, but Preview Card's trigger is specifically an `<a>` whose destination must independently contain the previewed content (`../_clusters/overlays.md` "Tooltip vs Preview Card").
- [I] **Any content a touch-only user must reach** cannot go in a Tooltip full stop — there is no touch fallback by design (§6), so if touch support is a hard requirement, Tooltip is disqualified outright regardless of content weight.

## 5. Anatomy & composition

- [E] Canonical tree (docs Anatomy):
  `Provider > Root > (Trigger, Portal > (Positioner > (Popup > (Arrow, Viewport))))`.
- [E] **Provider is optional and wraps outside Root** — it provides a shared `delay`/`closeDelay`/`timeout` context plus a Floating UI `FloatingDelayGroup` that any number of sibling `Tooltip.Root` instances can participate in; a lone `Tooltip.Root` with no ancestor Provider works fine and simply uses its own per-Trigger `delay`/`closeDelay` defaults (§6, §8).
- [E] **Portal is mandatory** for the popup subtree, matching the library-wide "explicit anatomy" doctrine (library-wide arc, `principles.md` B-P18) — `TooltipPortal.tsx` renders `null` unless `mounted || keepMounted`, and `TooltipPositioner`/`TooltipPopup`/`TooltipArrow` all throw via their context hooks (`useTooltipPortalContext`, `useTooltipPositionerContext`) if used outside a `<Tooltip.Portal>`/`<Tooltip.Positioner>`.
- [E] **Arrow and Viewport are both optional.** Arrow only matters if you want a visual pointer to the anchor; Viewport "is only required if one popup can be opened by multiple triggers, its content changes based on the trigger, and switching between them is animated" (Viewport JSDoc, confirmed verbatim by docs API reference note).
- [E] **Trigger can live inside or outside Root** ("contained" vs "detached" trigger). A contained Trigger is the default, simplest case. A **detached** Trigger is placed anywhere in the tree and associated with a Root via a shared `handle` created by `Tooltip.createHandle()` — this is the mechanism behind the `TooltipRoot.detached-triggers.test.tsx` suite (§6, §9). Multiple Triggers — contained or detached, sharing one Root/handle — can all open the same tooltip; which one is "active" is tracked per-trigger (`isOpenedByThisTrigger`/`isTriggerActive` state, `TooltipTrigger.tsx`).
- [E] **Payload / function-as-child.** `Tooltip.Root`'s `children` may be a function `({ payload }) => ReactNode`; each `Tooltip.Trigger` can carry a `payload` prop (typed via `Tooltip.createHandle<PayloadType>()`), so one Popup can render different content depending on which of several Triggers opened it — the basis for the docs' "Multiple triggers" and "Animating the Tooltip" examples.
- Text spec for a future anatomy diagram: (1) Provider [invisible, optional, wraps everything], (2) Root [invisible], (3) Trigger, (4) Portal [invisible], (5) Positioner [invisible box], (6) Popup, (7) Arrow, (8) Viewport [optional, wraps Popup's children when present].

## 6. Behavior ("How it works")

- [E] **Opening**: hover (`restMs`-style "rest" delay, mouse-only — see touch note below) or focus. `useHoverReferenceInteraction(floatingRootContext, { mouseOnly: true, ... })` in `TooltipTrigger.tsx` is the exact hook wiring; `useFocus(...)` is wired separately alongside it. No click-to-open path exists.
- [E] **Delay defaults**: `OPEN_DELAY = 600` ms (`packages/react/src/tooltip/utils/constants.ts`, the only exported constant in that file) is the default hover "rest" delay before opening, applied per-`Tooltip.Trigger` via its `delay` prop (`delay ?? OPEN_DELAY`, `TooltipTrigger.tsx`). Default `closeDelay` is `0` (Trigger prop JSDoc `@default 0`) — closing on unhover is instant unless a `closeDelay` is set on the Trigger or inherited from a Provider.
- [E] **Focus opens with no delay** — the `useFocus` hook is independent of the hover rest-timer path (`TooltipTrigger.test.tsx`/`.detached-triggers.test.tsx`: `trigger.focus()` followed only by `flushMicrotasks()`, no `clock.tick(OPEN_DELAY)`, before asserting the popup is visible).
- [E] **`Tooltip.Provider` delay-grouping mechanism — the precise timing model** (`provider/TooltipProvider.tsx`, `provider/TooltipProvider.test.tsx`, `trigger/TooltipTrigger.tsx`):
  1. `Tooltip.Provider` accepts `delay`, `closeDelay`, and `timeout` (**default `400`** ms) props. It wraps its children in Floating UI's `FloatingDelayGroup` with `delay={{ open: delay, close: closeDelay }}` and `timeoutMs={timeout}`.
  2. Every `Tooltip.Trigger` inside the Provider calls `useDelayGroup(floatingRootContext, { open: isOpenedByThisTrigger })`, which returns `{ delayRef, isInstantPhase, hasProvider }`. `isInstantPhase` becomes `true` for the group's members for a window after any member closes.
  3. `TooltipTrigger.getOpenDelay()` computes the *effective* open delay for that trigger: while the group is in its post-close instant window (tracked internally by Floating UI's delay group and surfaced as `delayRef.current.open === 0`), the effective delay collapses to `0`; otherwise it falls back to the Trigger's own `delay` prop, then the Provider's `delay` prop, then the hardcoded `600` ms default.
  4. Concretely: hover Trigger A inside a `<Tooltip.Provider timeout={400}>` → it waits the full open delay (Provider's `delay`, or `600` ms if unset) → tooltip shown. Move the mouse to Trigger B (also inside the Provider) **within `timeout` (400 ms by default) of A's tooltip closing** → B's effective open delay collapses to `0` and its tooltip appears instantly. Wait longer than `timeout` after A's tooltip has fully closed, then hover B → B pays the full delay again, because the group's "warm" window has expired.
  5. This is pinned by `TooltipRoot.test.tsx`'s "`keeps the tooltip open when moving across spaced triggers without a closeDelay`" test (uses `<Tooltip.Provider timeout={400}>` around three handle-backed detached triggers and asserts the popup stays visible while sweeping across all three with no `closeDelay` set) and by `TooltipProvider.test.tsx`'s dedicated `delay`/`closeDelay` prop suites (asserting the Provider's own `delay`/`closeDelay` values are honored, that per-Trigger `delay` overrides the Provider's, and that `closeDelay` updates take effect on the *next* render even mid-open).
  6. Animation is kept **instant** during this warm-up handoff specifically so consecutive tooltips in a group don't visibly fade in/out on every hop — `TooltipRoot.tsx`'s `instantType` state is forced to `'delay'` whenever `isInstantPhase` is true or a close is due to another tooltip opening (`reason === 'none'`), and `TooltipRoot.test.tsx`'s `'toggles instant animations for adjacent tooltips only while opening'` test asserts `data-instant="delay"` and zero running CSS animations on the second popup in exactly this scenario.
  - [I] This means the Provider does not change *what* opens a tooltip (still hover/focus) — it only changes the *effective open delay* for the group's members during a short post-interaction window, and forces the transition to render instantly during that window.
- [E] **Touch is excluded by design, verified at the source-code level (not just via docs/#3530).** `TooltipTrigger.tsx` passes `mouseOnly: true` to `useHoverReferenceInteraction`; that hook (`floating-ui-react/hooks/useHoverReferenceInteraction.ts` lines 222 & 436) contains the literal guard `if (mouseOnly && !isMouseLikePointerType(instance.pointerType)) { return; }` before opening on a hover-type interaction — a touch pointer's `pointerType` is `'touch'`, which fails `isMouseLikePointerType`, so the open path never runs. `TooltipRoot.test.tsx`'s own conformance suite is set up with `triggerMouseAction: 'hover'` (never touch), and every open-on-hover unit test in the file fires `pointerDown`/`pointerEnter` with explicit `pointerType: 'mouse'`. [G] No dedicated `it('does not open on touch pointerType')` unit test was found scoped specifically inside the tooltip package (the guard lives in the shared `floating-ui-react` hook, tested generically there rather than per-consuming-component) — the behavioral claim rests on (a) this source-level guard being unconditionally wired with `mouseOnly: true`, (b) docs prose, and (c) the fully-cited #3530 maintainer explanation, not on a tooltip-specific touch assertion in the test suite.
- [E] **Nested tooltips**: a tooltip trigger can itself contain another `Tooltip.Root`/`Trigger` (e.g. a toolbar button that is itself annotated, containing a nested icon that has its own tooltip). Hovering the nested (inner) trigger opens only the inner tooltip and suppresses the outer one, at any nesting depth — pinned by three dedicated tests in `TooltipRoot.test.tsx`'s `'nested tooltips'` describe block (`'should not open the outer tooltip when hovering over a nested tooltip trigger'`, `'...when moving between sibling nested triggers'`, `'...ancestor tooltips when hovering over a third-level nested trigger'`). Moving from the nested trigger back out to the parent's own hover area at zero delay reopens the parent (same block, final test). A dedicated bugfix (#4501, "Fix flickering when nesting tooltip inside another tooltip's trigger") hardened this.
- [E] **Handle / detached-trigger pattern** (`store/TooltipHandle.ts`, `store/TooltipStore.ts`, introduced together in #3040 "Keep state in a store" and #3071 "Support detached triggers"): `Tooltip.createHandle<Payload>()` returns a `TooltipHandle` with `open(triggerId)`, `close()`, and a read-only `isOpen` getter. Passing the same `handle` to a `<Tooltip.Root handle={h}>` and to one or more `<Tooltip.Trigger handle={h}>` elements anywhere in the tree lets those Triggers control the Root's tooltip **without being physically nested inside it** — e.g. rendering the Trigger in a table row and the Root/Popup once at the page root. Before any Root using the handle is mounted (or after it unmounts), imperative calls on the handle are silently ignored (with a `console.warn`, pinned by `TooltipRoot.detached-triggers.test.tsx`'s `'handle-backed root ownership'` suite) — there's an inert `NullStore` fallback (`createNullTooltipStore`) so a detached Trigger's hover/click handlers are safe to fire even with no Root attached yet. A handle can be shared by many detached Triggers (each carrying its own typed `payload`) and only one Root at a time; the test suite explicitly warns if a handle stays attached to more than one mounted Root simultaneously.
- [E] **Controlled/uncontrolled**: uncontrolled by default via `defaultOpen`/`defaultTriggerId`; fully controlled via `open`/`onOpenChange` + `triggerId` (needed only when multiple triggers share one Root, to track *which* trigger is "active"). `onOpenChange(open, eventDetails)` reasons include `triggerHover | triggerFocus | triggerPress | outsidePress | escapeKey | disabled | imperativeAction | none` (`TooltipRootChangeEventReason`, `TooltipRoot.tsx`) — `'none'` specifically marks a close caused by a *different* tooltip opening (e.g. moving between detached triggers), used to keep animations instant (see above). `eventDetails.cancel()` vetoes the state change; a tooltip-specific extra escape hatch, `eventDetails.preventUnmountOnClose()`, keeps the Portal/Positioner/Popup subtree mounted through a close so exit-animation state (or expensive-to-remount content) survives — pinned by `TooltipTrigger.test.tsx`'s `'removes data-popup-open as soon as open becomes false'` and multiple `preventUnmountOnClose()` suites in `TooltipRoot.test.tsx`.
- [E] **Dismissal**: `useDismiss` closes on outside press (`referencePress`, gated by the Trigger's `closeOnClick` prop, default `true`) and on Escape (`escapeKey` reason) — see §7 for the pinned Escape test. `eventDetails.allowPropagation()` lets an Escape keep bubbling to an ancestor popup while still closing the tooltip itself (`TooltipRoot.test.tsx` `'allowPropagation() prevents stopPropagation on Escape while still closing'`).
- [E] **`closeOnClick` (Trigger, default `true`)**: clicking a trigger whose tooltip is about to open (still within its open delay) cancels the pending open via `store.cancelPendingOpen(event)` rather than reporting a public open-state change — this is a *silent* cancellation distinct from a normal close, evidenced by `TooltipStore.ts`'s comment: "Used by trigger clicks to clear a delayed hover open without reporting a public open-state change." If already open, a click closes it (unless `closeOnClick={false}`), added in #4140 "Add `closeOnClick` prop."
- [E] **`disableHoverablePopup` (Root, default `false`)**: when `false` (default), the Popup itself participates in the hover interaction via `safePolygon()` (a triangular "safe zone" so the cursor can travel from trigger to popup without closing it) and `useHoverFloatingInteraction` on `TooltipPopup.tsx`; when `true`, `pointer-events: none` is applied to the Positioner (#1917) so the popup is purely decorative and can never be hovered into.
- [E] **`trackCursorAxis` (Root, default `'none'`)**: `'x' | 'y' | 'both'` makes the Positioner follow the pointer along the given axis instead of anchoring rigidly to the Trigger's box (`useClientPoint`); `'both'` implicitly disables the safe-polygon hoverable-popup behavior (Positioner is `inert` while `trackCursorAxis === 'both'`).
- [E] **Animation**: standard popup hooks — `data-open/closed/starting-style/ending-style` on Popup, `data-instant` (values `'delay' | 'dismiss' | 'focus'`) marks transitions that should skip CSS animation entirely, `--transform-origin`/`--positioner-width`/`--positioner-height` CSS vars on Positioner. `Viewport`'s content-transition system adds its own `data-current`/`data-previous`/`data-activation-direction`/`data-transitioning` attributes and `--popup-width`/`--popup-height` CSS vars for freezing dimensions mid-transition (docs "Animating the Tooltip" explicitly instructs setting `width: var(--positioner-width)` / `height: var(--positioner-height)` on the Positioner when Viewport is used, "otherwise content-driven resizing can make the popup thrash or flip to another side").
- [G] **SSR**: no tooltip-specific SSR test or docs note was found in this pass (searched `TooltipPortal.test.tsx`, `TooltipRoot.test.tsx` for `hydration`/`suppressHydrationWarning` — none present, unlike Select's hidden-input SSR handling). [I] Likely simply inherits the library-wide default-unmounted-when-closed popup behavior with no special-case SSR code needed, since Tooltip has no hidden form input or other server-rendered-then-hydrated artifact — but this is inferred from absence, not confirmed by a maintainer statement.

## 7. Accessibility contract

- [E] **`Tooltip.Popup` renders with NO ARIA role and no `aria-describedby` wiring to the Trigger — verified directly in source, not assumed.** `TooltipPopup.tsx`'s `useRenderElement('div', componentProps, { state, ref: [...], props: [popupProps, getDisabledMountTransitionStyles(transitionStatus), elementProps], stateAttributesMapping })` call contains no `role` key anywhere; `grep -rn "role" packages/react/src/tooltip/` returns zero matches across the entire part family, and `grep -rn "aria-describedby|describedby"` likewise returns zero matches. `TooltipPopup.test.tsx` correspondingly never asserts `getByRole('tooltip')` — its only assertions are conformance + "renders the children." This is a **deliberate absence, not an oversight**: it is the direct mechanical consequence of the tooltip being defined as a purely visual annotation (§2) — giving it `role="tooltip"` plus a live `aria-describedby` link would make its content part of the accessible name/description tree, which is exactly what the maintainers' mental model in #3530 says a tooltip must *not* be (that's Popover's job). [I] The corollary in `../_clusters/overlays.md`'s decision table ("Popup role: none (visual only)" for Tooltip) is consistent with this, though that note's own footnote citation for the "no role/aria attrs" claim was written about a different row's sourcing; this brief independently re-confirms it directly against `TooltipPopup.tsx`.
- [G] **Because of the above, the W3C ARIA APG "Tooltip" pattern (https://www.w3.org/WAI/ARIA/apg/patterns/tooltip/) — which specifies `role="tooltip"` plus `aria-describedby` from the trigger — is NOT the pattern Base UI's Tooltip actually implements.** [I] Base UI's approach instead matches the docs' own prescribed technique of pairing a purely-visual Tooltip with a separate `aria-label` set directly on the Trigger by the consumer (Usage guidelines bullet 2) — i.e., the accessible name comes from `aria-label`, not from an ARIA relationship to the tooltip's DOM content. No maintainer PR/issue was found in this pass explicitly discussing *why* the APG pattern's `role`/`aria-describedby` prescription was not adopted (searched `TooltipPopup` + `role` + `aria-describedby` in git log subject lines and in the `TooltipPopup.tsx` blame history — nothing found); this is inferred from the #3530 design philosophy plus the absence of the attribute in source, not from an explicit rejection statement. A future session with `gh` budget could search issues for "tooltip role aria-describedby" to see if a user ever requested it and how it was answered.
- [E] **Keyboard interaction table** (evidence: `TooltipTrigger.tsx`'s `useHoverReferenceInteraction`/`useFocus`/`useDismiss` wiring; tests as noted):

  | Key / interaction | Context | Action |
  |---|---|---|
  | Tab (focus lands on trigger) | closed | Opens immediately, no delay (`useFocus`; `TooltipRoot.test.tsx` `'should open when the trigger is focused'` — asserted with only `flushMicrotasks()`, no delay tick) |
  | Shift+Tab / blur away | open | Closes immediately (`'should close when the trigger is blurred'`) |
  | Escape | open | Closes (`escapeKey` reason); `allowPropagation()` available to let Escape continue bubbling to an ancestor popup while still closing this one (`TooltipRoot.test.tsx` line ~818–845) |
  | Mouse hover-rest on trigger | closed | Opens after `delay` (Trigger prop, default `600` ms; collapses to near-`0` inside a "warm" `Tooltip.Provider` group, §6) |
  | Mouse leaves trigger (and popup, if hoverable) | open | Closes after `closeDelay` (Trigger prop, default `0`) |
  | Touch tap on trigger | any | **Never opens** — `mouseOnly: true` guard (§6); the trigger's own tap/click behavior (e.g. a button's `onClick`) still fires normally, only the tooltip-hover path is suppressed |
  | Click on trigger | closed, delay pending | Cancels the pending open silently (`closeOnClick` default `true`) rather than reporting a state change |
  | Click on trigger | open | Closes (unless `closeOnClick={false}`) |
- [E] **Screen reader notes**: because the Popup carries no role/description linkage (see above), screen reader users get **no tooltip announcement at all** by design — the docs' own guidance is that the Trigger's `aria-label` must independently and redundantly state what the tooltip would have shown ("must closely match the tooltip's content"), so a screen reader user loses nothing essential, only the sighted-user visual hint. This is the accessibility contract in full: *the tooltip itself is invisible to assistive technology; the burden is placed on the author to label the trigger directly.*
- [E] **Focus-visible / disabled-trigger handling**: a `disabled` Trigger (or a Root-level `disabled`) still forwards `data-trigger-disabled`, and a real HTML-`disabled` button on a *nested* trigger is specifically handled — `TooltipTrigger.test.tsx`'s `'opens on delayed hover when rendered as a disabled toolbar button'` confirms a `Toolbar.Button disabled` trigger renders `aria-disabled="true"` (not the native `disabled` attribute, so it stays hoverable/focusable) and its tooltip still opens on hover after its delay — consistent with the library-wide "disabled-but-focusable" composite-widget convention noted for Select in the sibling brief.
- **Known open issues (honesty list)**: [G] a targeted search for tooltip-specific accessibility bug reports was not run in this pass beyond what `../_clusters/overlays.md` and `discussions.md` already surfaced (no dedicated tooltip a11y bug was flagged in either source). This pass's `gh` budget (2 lookups) was spent on the introducing PR #264; #1420/#1447/#1539 (indexed in the per-component discussion index but not detailed in `discussions.md`'s top-35 writeups) were **not independently verified in this pass** — flagged here as [G] for a future session.

## 8. Prop-level guidance (decision-relevant props)

- [E] **`Tooltip.Provider.delay` / `.closeDelay` / `.timeout`** — wrap several `Tooltip.Root`s to give them a shared open/close delay and, more importantly, a shared "warm" grouping window (`timeout`, default `400` ms) so hopping between adjacent tooltips feels instant after the first pays the full delay. Use it for icon toolbars, dense data-grid headers, or any cluster of related hover targets. A per-`Trigger.delay`/`.closeDelay` always overrides the Provider's value for that one trigger (`TooltipProvider.test.tsx` `'respects trigger delay prop over provider delay prop'`).
- [E] **`Trigger.delay` (default `600`, i.e. `OPEN_DELAY`)** — the per-trigger open-hover-rest delay; lower it (even to `0`) for triggers where instant feedback matters more than avoiding accidental hover-triggering, e.g. inside a Provider group where you want the *first* hover in a cluster to also be fast.
- [E] **`Trigger.closeDelay` (default `0`)** — raise this when users need a moment to move the cursor from the trigger onto the popup content (only matters if `disableHoverablePopup` is `false`, since the safe-polygon already helps the direct trigger→popup traversal).
- [E] **`Root.disableHoverablePopup` (default `false`)** — set `true` when the popup content must never be reachable by hovering into it (e.g. a truly decorative label with pointer-events that should pass through to whatever is underneath); this also nulls out the `safePolygon()` grace-zone, so use it only when hoverable content genuinely isn't needed.
- [E] **`Root.trackCursorAxis` (default `'none'`)** — set to `'x'`/`'y'`/`'both'` to have the tooltip follow the pointer along an axis instead of staying anchored to the trigger's box; useful for tooltips on large draggable/continuous controls (e.g. a slider thumb or a canvas) where the "point of interest" moves within the trigger. `'both'` makes the Positioner `inert` and disables hoverable-popup behavior.
- [E] **`Root.disabled` (default `false`)** — suppresses opening entirely and force-closes if already open (`TooltipRoot.tsx`'s `useIsoLayoutEffect` watching `openState && disabled`); use for a trigger control that is temporarily non-interactive and should not surface a hint while disabled.
- [E] **`Trigger.disabled` (default `false`)** — disables *only the tooltip-open behavior* for that trigger, explicitly **not** the native `disabled` attribute on the underlying element (JSDoc: "this doesn't apply the `disabled` attribute to the trigger element... pass the `disabled` prop to the trigger element via the `render` prop" if you also want the element itself disabled).
- [E] **`Trigger.closeOnClick` (default `true`)** — set `false` for triggers whose click action is unrelated to the tooltip and shouldn't dismiss it (added #4140); default behavior treats a click as a dismissal signal, matching most buttons' expectation that clicking "does the thing" and the hint is no longer needed.
- [E] **`handle` (Root + Trigger, via `Tooltip.createHandle<Payload>()`)** — reach for this only when a Trigger cannot be practically nested inside its Root (e.g. Trigger rendered once per table row, Root/Popup rendered once at the page level) or when several physically distant Triggers must share one tooltip. For same-subtree cases, plain contained Triggers inside one Root are simpler and should be preferred (library-wide "handles are the sanctioned imperative surface, but uncontrolled parts come first" guidance, `principles.md` B-P17).
- [E] **`Root.actionsRef` (`{ unmount, close }`)** — `unmount()` for JS animation libraries that need to force-remove the popup after their own exit animation; `close()` as a plain imperative close without needing controlled `open` state.
- [E] **`onOpenChange` `eventDetails.preventUnmountOnClose()`** — tooltip-specific escape hatch (not shared with every popup component) to keep the Portal/Positioner/Popup subtree mounted through a close, e.g. to let a library-driven exit animation read from a still-mounted DOM node, or to avoid re-mount cost for expensive tooltip content that gets reopened frequently.
- [E] **Styling contract highlights** (from `*DataAttributes.ts` / `*CssVars.ts`):
  - Trigger — `data-popup-open`, `data-trigger-disabled`.
  - Popup/Positioner/Arrow — `data-open`/`data-closed`, `data-starting-style`/`data-ending-style`, `data-side`, `data-align`, `data-instant` (`'delay' | 'dismiss' | 'focus'`); Positioner adds `data-anchor-hidden`; Arrow adds `data-uncentered`.
  - Positioner CSS vars — `--available-width`/`--available-height`, `--anchor-width`/`--anchor-height`, `--transform-origin`, `--positioner-width`/`--positioner-height` (the last two specifically needed when `Viewport` is used, per docs API note).
  - Viewport — `data-current`/`data-previous` (per content-transition container), `data-activation-direction` (space-separated tokens, matchable with `~=`), `data-transitioning`, `data-instant`; CSS vars `--popup-width`/`--popup-height` for freezing dimensions on the outgoing container.

## 10. Pitfalls & FAQ

- **Tooltip stays open after the trigger is clicked, but I want it to show "Copied!" instead** → check `eventDetails.reason` in `onOpenChange` and call `eventDetails.cancel()` to veto the click-driven close (`reason === 'trigger-press'`-equivalent); noted as a near-invisible escape hatch (`principles.md` B-M4) that most users don't discover unassisted (mui/base-ui#1311, has-workaround, closed). Consider instead: is this really contextual feedback that belongs in an anchored Toast (§4)? [E]
- **Tooltip flashes/doesn't animate smoothly when sweeping across several icons in a toolbar** → wrap them in a shared `<Tooltip.Provider>`; without it, every trigger pays its own full `delay` and animates independently. With a Provider, hops within the `timeout` window (default `400` ms) are instant and animation-suppressed by design (§6). [E]
- **"Copied to clipboard" / transient status message rendered as a Tooltip** → wrong component; docs explicitly redirect this to an anchored Toast so the message is announced to screen readers (docs "Contextual feedback messages"; #3096). [E]
- **Tooltip never shows up when testing on a phone/tablet or in a touch-emulated browser** → this is intentional, not a bug — touch is excluded by the `mouseOnly: true` guard (§6); design a touch-reachable alternative (visible text, or Popover with `openOnHover`) if the content matters on touch. [E]
- **Expecting `role="tooltip"` or an `aria-describedby` link to show up in the rendered DOM** → neither exists; Base UI's Tooltip is deliberately visual-only. Set `aria-label` directly on the Trigger with content matching the tooltip (§7). [E]
- **Detached-trigger `handle.open('id')` silently does nothing** → a `Tooltip.Root` using that same handle must already be mounted; calls made before mount (or after unmount) are dropped with a `console.warn`, and no state is replayed once a Root does mount (`TooltipRoot.detached-triggers.test.tsx`). [E]
- **Two mounted `Tooltip.Root`s sharing one handle** → supported only transiently (e.g. during a remount/handoff); a lingering overlap triggers a `console.warn` ("more than one mounted root") — don't rely on more than one Root being attached to a handle at steady state. [E]
- **Popup or its animation state resets unexpectedly on close** → if you need the exit-animation subtree (or expensive content) to survive a close, call `eventDetails.preventUnmountOnClose()` in `onOpenChange`; otherwise the Portal subtree unmounts as soon as `open` becomes `false` and any transient DOM state is lost. [E]
- **Docs feedback on record**: none of the "top ~35 by activity" discussions catalog entries beyond #3530/#1311 were surfaced as tooltip-specific open feedback in this Tier-2 pass; #1420/#1447/#1539 are indexed per-component but **left unverified** — flagged for a future deeper pass. [G]
