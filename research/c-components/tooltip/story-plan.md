# Tooltip — story plan

Feeds Phase E. Source of truth: [`brief.md`](./brief.md). Styling: follow the docs hero demo CSS (CSS Modules, raw oklch values, `docs/src/app/(docs)/react/components/tooltip/demos/hero/css-modules/`).

## Notes on interaction reliability (read before implementing)

- **Prefer keyboard focus-open as the primary, reliable interaction assertion.** Base UI's own test suite treats hover and focus as two independent paths (`useHoverReferenceInteraction` vs `useFocus`), and focus-open has no delay to race against (§6 of brief.md) — `trigger.focus()` + a microtask flush is enough, with no fake-timer choreography needed. This makes it the most reliable play-function interaction for Storybook, which runs against real timers and a real (if headless) browser rather than jsdom's fake-timer harness.
- **Hover-open is attempted as a secondary, best-effort interaction only.** Testing Library's/Storybook's simulated `hover`/`pointerMove` events on Chromium via play functions can be flaky for rest-delay-based hover detection (movement thresholds, `pointerType` synthesis, and the `600` ms default delay all compound the flake risk) — this is called out explicitly wherever a story attempts it below. If a hover assertion proves unreliable in practice, keep the story but drop the play-function assertion down to a documentation-only demonstration (render + manual QA note) rather than fighting a flaky automated interaction.
- **Touch modality cannot be reliably simulated in a Storybook play function.** Per brief.md §6, touch exclusion is enforced via a `pointerType` guard deep in a shared Floating UI hook; synthesizing a genuinely touch-typed pointer event in a browser automation context is unreliable and not worth the flake risk. The touch story below is documentation-only by design, not a testing gap to "fix" later.

## 2. One story per use case

Columns: name — renders/setup — controlled? — play? — doc section served.

1. **Basic** — hero recreation: `Provider`(none, standalone `Root`) — `Root > Trigger("Info") + Portal > Positioner > Popup("Helpful hint text") + Arrow` — uncontrolled — no — Hero / Anatomy (§5).
2. **KeyboardFocusOpen** *(primary required interaction story)* — Basic layout with an extra focusable element before the trigger in tab order — play: `Tab` to focus the trigger → assert popup content becomes visible with no delay wait needed → `Tab` away (or blur) → assert popup is gone — uncontrolled — **play (reliable)** — Behavior (§6 focus-open path) / A11y keyboard table (§7).
3. **HoverOpenBestEffort** — Basic layout — play (best-effort): simulate pointer hover-rest on the trigger, wait past the default `600` ms open delay, assert popup visible; unhover, assert popup closes — uncontrolled — **play (best-effort, may be flaky — see notes above; if unreliable in CI, drop the assertion and keep as a manual-QA-only render)** — Behavior (§6 hover-open path).
4. **ProviderDelayGrouping** *(must-have, distinctive to this component)* — two sibling `Tooltip.Root`s (each with its own `Trigger` + `Popup`, e.g. "Bold" and "Italic" icon buttons side by side) wrapped in one shared `<Tooltip.Provider timeout={400}>` — play: focus/hover Trigger A → wait the full default delay (`600` ms) → assert Popup A visible → move focus/hover directly to Trigger B within the `400` ms grouping window → assert Popup B becomes visible **near-instantly** (no full-delay wait) and with `data-instant="delay"` present during the handoff → wait past the `timeout` window after unfocusing B, then focus a third trigger C → assert C pays the full delay again (group has gone "cold") — uncontrolled — **play** — Behavior §6 (Provider delay-grouping mechanism, the mechanism most needing pinned proof), Anatomy §5 (Provider), Prop guidance §8.
5. **TouchBehaviorNote** *(documentation-only by design, not play-tested)* — renders `Basic` unchanged, but its docs/description block explicitly states: "Base UI's Tooltip never opens on touch input by design — a `pointerType` guard in the shared hover hook suppresses the open path for non-mouse-like pointers (brief.md §6). This is not a bug and is not simulated here; touch-modality emulation in a browser-automation play function is unreliable enough that this story intentionally renders only the mouse/keyboard-accessible tooltip and documents the touch exclusion in prose, citing mui/base-ui#3530." — uncontrolled — no (documentation note only, per the reliability constraints above) — Intention §2 / When-not-to-use §4 / Behavior §6 (touch exclusion) / A11y §7.
6. **PositioningAndArrow** — `side`/`align`/`sideOffset` controls (Storybook args) on the Positioner + visible `Arrow` styled against `data-side`/`data-align`/`data-uncentered` — uncontrolled — no — Anatomy §5 (Arrow) / Prop guidance §8 (styling contract data attributes).
7. **ControlledOpen** — external `open`/`onOpenChange` state on `Root` with a log of `eventDetails.reason` values (hover/focus/escape/outside-press/imperative) rendered next to the demo — controlled — play (toggle open via an external button; assert reason logged) — Behavior §6 (controlled/uncontrolled, `eventDetails` reasons).
8. **DetachedTriggerHandle** — `Tooltip.createHandle()` shared between a `Trigger` rendered in one part of the story and the `Root`/`Popup` rendered elsewhere in the DOM tree, plus an external "Open programmatically" / "Close" button pair calling `handle.open(id)`/`handle.close()` — uncontrolled (handle-driven) — play (click external buttons; assert popup appears/disappears; assert focus/hover on the physically-separate trigger also still works) — Anatomy §5 (detached trigger) / Behavior §6 (Handle pattern) / Decision log §9 (#3040, #3071).
9. **MultipleTriggersWithPayload** — one `Root` with 2–3 contained `Trigger`s each carrying a distinct typed `payload`, `Root`'s function-as-child rendering the active payload's text inside `Popup` (recreates docs "Detached triggers with payload" using contained triggers) — uncontrolled — play (hover/focus Trigger 1, assert payload text; switch to Trigger 2, assert text updates and popup does not flicker/remount — cites `TooltipRoot.test.tsx` "should reuse the popup and positioner DOM nodes when switching triggers") — Anatomy §5 (payload / function-as-child) / Behavior §6.
10. **DisabledTrigger** — one enabled and one `disabled` (Root-level `disabled` prop) tooltip side by side — uncontrolled — play (assert the enabled one opens on focus; assert the disabled one never opens on focus or hover) — Prop guidance §8 (`Root.disabled` vs `Trigger.disabled`).
11. **InToolbarContext** — recreates the pattern from `TooltipTrigger.test.tsx`'s disabled-toolbar-button coverage: a `Toolbar.Root` containing a `Toolbar.Button disabled` wrapped by `Tooltip.Trigger`, demonstrating that a toolbar-disabled (aria-disabled, still focusable) button's tooltip still opens on hover/focus — uncontrolled — play (hover/focus the aria-disabled toolbar button; assert tooltip opens) — Pitfalls/FAQ §10 (composite-widget "disabled but focusable" convention) / real-world toolbar integration.
