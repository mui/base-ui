# Preview Card — story plan

Feeds Phase E. Source of truth: [`brief.md`](./brief.md). Styling: follow the docs hero demo CSS (CSS Modules, raw oklch values) — see `docs/src/app/(docs)/react/components/preview-card/demos/hero/css-modules/index.tsx` for the reference look (underlined link trigger, image + bold term + summary paragraph inside the popup, `sideOffset={8}`).

Tier 2 target: 6–10 stories. This plan lists 8.

## Chosen approach for the Tooltip-vs-Preview-Card distinction story (§4 of brief.md)

**Chosen form: a single story rendering both a Tooltip and a Preview Card side by side in one sentence of prose**, rather than two separate story files cross-referenced by name. Rationale: the boundary is about *trigger element and purpose* (an `<a href>` whose point is navigation vs. a non-link control whose point is something else), which is easiest to see when both triggers sit in the same paragraph and the reader can hover each in turn and feel the difference (one is a plain text hint on a button-like element; the other opens on a real link and previews its destination). A split into two independent stories would let a reader view them without ever comparing, defeating the point. The story's docs-block prose states the litmus test verbatim from brief.md §4 (colmtuite, mui/base-ui#4778): "If you're not rendering a link, you should not use PreviewCard." This story lives in the Preview Card story file (not Tooltip's) since Preview Card is the component whose scope is narrower and needs the sharper caveat; Tooltip's own sibling story-plan may cross-link back to it.

## One story per use case

Columns: name — renders — controlled? — play? — doc section served.

1. **HoverOpenLinkPreview** *(required primary interaction story)* — Root + Trigger(`href`, underlined link inline in a paragraph) + Portal + Positioner(`sideOffset={8}`) + Popup + Arrow, plain-text popup content — uncontrolled — **play**: hover the trigger, advance fake timers/wait past the 600ms default `delay`, assert the popup becomes visible with its content, assert `role`/`aria-*` are absent from both Trigger and Popup (documents the "no ARIA wiring" fact from brief.md §7), then unhover and wait past the 300ms `closeDelay` and assert it closes — Hero / Anatomy / Behavior (§6) / A11y contract (§7).
2. **RichLinkPreviewContent** — recreation of the shipped hero demo: Trigger wrapping the word "typography" inline in a sentence, Popup containing an `<img>` + `<strong>` term + summary `<p>` (the exact archetype from `docs/.../hero/css-modules/index.tsx`) — uncontrolled — no — Anatomy / "When to use" (§3) — demonstrates that Preview Card content is deliberately richer/more composed than a Tooltip's plain-text hint, per the parent brief's callout that this needs its own demonstration.
3. **TooltipVsPreviewCardDistinction** — one paragraph rendering both: a `Tooltip.Root` wrapping a plain icon `<button>` with a one-line hint ("Copy to clipboard"), and a `PreviewCard.Root` wrapping an `<a href>` inline in the same sentence with a richer popup (reuses story 2's content) — uncontrolled — **play**: hover the Tooltip trigger, assert its popup has no image/rich content (short text only); hover the Preview Card trigger, assert its popup shows the image+summary; assert the Tooltip trigger is a `<button>` while the Preview Card trigger is an `<a>` via role queries (`getByRole('button')` vs `getByRole('link')`) — When not to use + alternatives (§4), the dedicated distinction subsection.
4. **PositioningWithArrow** — Trigger + Positioner with explicit `side="top"`/`align="start"`/`sideOffset` props exposed via story controls, visible `Arrow` styled against `data-side`/`data-align`/`data-uncentered` — uncontrolled — no — Anatomy / Prop-level guidance (Positioner props, styling contract).
5. **KeyboardFocusOpen** — same basic Trigger+Popup shape as story 1, but the play function drives it via keyboard only (Tab to the trigger, no mouse events) — uncontrolled — **play**: focus the trigger via `Tab`, wait past `delay`, assert the popup opens; blur (Tab away or Shift+Tab), assert it closes; separately assert Escape closes an open card and that re-focusing reopens it afterward — Behavior (§6, the "focus opens it too, same delay" finding) / A11y keyboard table (§7). This is the story most directly answering the parent task's "precisely note focus behavior" instruction.
6. **DetachedTriggerWithHandle** — a `PreviewCard.createHandle()` connecting a `<PreviewCard.Trigger handle={...}>` rendered far from a `<PreviewCard.Root handle={...}>` (e.g., trigger in a "body text" column, Root/Popup markup declared once elsewhere in the story) — uncontrolled — **play**: hover the detached trigger, assert the popup (declared elsewhere in the tree) opens and shows the correct trigger's `payload` — Anatomy (Handle/detached-trigger pattern) / Behavior (§6).
7. **MultipleTriggersSharedPopup** — one `PreviewCard.Root` (or one handle) with three `PreviewCard.Trigger`s each carrying a distinct `payload`, function-as-children `Root` rendering `payload`-driven content, optionally wrapped in `PreviewCard.Viewport` for an animated content swap — uncontrolled — **play**: hover trigger 1, assert content A; move to trigger 2, assert content B appears (and, if `Viewport` is included, assert a `data-activation-direction` attribute appears during the swap) — Anatomy (Viewport, payload pattern) / Behavior (§6, multi-trigger reuse).
8. **DelayTuning** — two Triggers side by side with different explicit `delay`/`closeDelay` values (e.g., `delay={0}` vs the 600ms default) plus a live on-screen readout of elapsed hover time — uncontrolled — **play**: hover the `delay={0}` trigger and assert near-instant open; hover the default-delay trigger and assert it does *not* open before ~600ms then does after — Prop-level guidance (§8, delay/closeDelay defaults) / Pitfalls (§10, "my hover event opens after longer than I expected").
