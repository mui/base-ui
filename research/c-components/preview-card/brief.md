# Preview Card — component research brief

Tier 2 (full brief, capped mining — capped at top ~8 issues/PRs). Mined 2026-07-06/07 from source, tests, docs, git history, and 2 targeted mui/base-ui issue lookups (read-only). Evidence tags: [E] direct evidence, [I] inference, [G] gap. Shares the popup/positioner/popup/arrow/backdrop part family with Tooltip almost exactly — see [`../_clusters/overlays.md`](../_clusters/overlays.md) for the full comparative cluster analysis; this brief focuses on what is specific to Preview Card and precisely on what differs from Tooltip.

## 1. Identity

- **Name / slug**: Preview Card — `@base-ui/react/preview-card`. Multi-part compound component, namespace export `PreviewCard.*`. [E] `packages/react/src/preview-card/index.ts`
- **Status**: stable, no `[New]`/`[Preview]` tag in the docs nav. [E] docs nav (§3.2 of `research/PROMPT.md`)
- **Purpose / IA statement**: "A popup that appears when a link is hovered, showing a preview for sighted users." [E] docs Subtitle (`docs/src/app/(docs)/react/components/preview-card/page.mdx`). Taxonomy category: **overlays/popups** (per `research/a-documentation-standard/taxonomy.md`), the same cluster as Popover/Tooltip/Dialog/Alert Dialog/Drawer. Purpose: a visual, hover-triggered enrichment of an ordinary hyperlink — not an independent piece of UI with its own affordance; information architecture: it always hangs off exactly one navigational element (a link) and shows content that "reflects the link destination" [E] docs Usage guidelines.
- **8 parts** (from `packages/react/src/preview-card/index.parts.ts`; one-liners from part JSDoc):

  | Part | Renders | One-liner |
  |---|---|---|
  | Root | no element | "Groups all parts of the preview card. Doesn't render its own HTML element." |
  | Trigger | `<a>` | "A link that opens the preview card." |
  | Portal | `<div>` | "A portal element that moves the popup to a different part of the DOM. By default, the portal element is appended to `<body>`." |
  | Backdrop | `<div>` | "A presentational overlay displayed beneath the popup." |
  | Positioner | `<div>` | "Positions the popup against the trigger." |
  | Popup | `<div>` | "A container for the preview card contents." |
  | Arrow | `<div>` | "Displays an element positioned against the preview card anchor." |
  | Viewport | `<div>` | "A viewport for displaying content transitions. This component is only required if one popup can be opened by multiple triggers, its content changes based on the trigger, and switching between them is animated." |

  Plus a non-element API: `PreviewCard.createHandle()` / `PreviewCard.Handle` (imperative handle + detached-trigger association, same family as `Dialog.createHandle`/`Handle` and Tooltip's equivalent). [E] `index.parts.ts`

## 2. Intention

- [E] **Introducing commit**: `00aebe315 [PreviewCard] Create new component (#469)` — the earliest commit touching `packages/react/src/preview-card`. Confirmed via `git log --oneline --all --follow -- packages/react/src/preview-card` (125 total commits found).
- [E] **Design goal, in the maintainers' own words** (mui/base-ui#4170, atomiks, closing an accessibility complaint): "It doesn't work for touch or screen reader users as it's purely an enhancement for sighted keyboard/mouse pointer use, but this isn't an issue provided the Preview Card doesn't contain any content not accessible via the link itself (as the name implies and **common use cases on the web show, e.g. Wikipedia's preview cards**)." This is the clearest maintainer statement of the archetype the component models itself on — a real, named, precedented UI pattern (Wikipedia link-hover previews), not a hypothetical.
- [E] Further corroboration (mui/base-ui#4778, oliviertassinari, closing a touch-support request): "For reference, it's not the case with **GitHub preview card or Wikipedia preview card**. … This component is about taking advantage of the unique **hover** feature of the *mouse* … it was designed so that with *touch* or *keyboard* modalities, it's as if the component doesn't exist." Note: this quote and the surrounding test suite (`PreviewCardRoot.test.tsx` "should open when the trigger is focused") together show that **focus is the one input modality the component does still support** — see §6; the "as if it doesn't exist" framing in #4778 is specifically about touch, not about the sighted-keyboard-focus path.
- [E] Sibling relationship to Tooltip is explicit in maintainer language: mui/base-ui#4170 opens "This is expected for Preview Card (**similar to Tooltip**)"; colmtuite (mui/base-ui#4778) draws the boundary directly: "The pattern is a visual-only enhancement for sighted, desktop users only. It should only ever be used on links, where all of the data and actions inside the popup are also accessible by following the link. If you're not rendering a link, you should not use PreviewCard. **Popover with `openOnHover` might be a good fit**, depending on your use case."
- [E] **Explicit non-goal, stated directly by a maintainer, twice**: it is by design that Preview Card cannot be opened by touch, screen reader, or (without a mouse/keyboard-focus path) speech input — this was raised as a bug/removal request (#4170: "Preferably, remove this component. It isn't necessary, and creates an inequitable experience between users of different modalities") and closed as expected behavior both times (#4170, #4778). The library's answer is not "we'll fix this" but "document it, and only use this component where the popup content already exists at the link destination."
- [E] **Docs-visible remediation from #4170**: the issue also flagged that hero-demo links lacked an underline (WCAG 1.4.1 Use of Color concern) and that there was no usage-guidelines warning; both were fixed — atomiks confirms an underline was added to the demo link and a Usage guidelines section now exists on the docs page (the same section quoted in §3/§4 below). [E] docs page `## Usage guidelines` section + issue comment with before/after screenshot.
- **Non-goal, structural**: [I] no `PreviewCard.Provider` part exists anywhere in the source tree (verified: `ls packages/react/src/preview-card/` — no `provider/` directory, no `Provider` export in `index.parts.ts`). Tooltip's sibling brief documents a `Tooltip.Provider` for cross-tooltip delay-grouping; Preview Card has no equivalent. Each `PreviewCard.Root` (or handle) owns its own delay state independently — there is no library-level mechanism to make multiple preview cards on a page share a "already showing one, so open the next instantly" grouping behavior the way grouped tooltips can. This is inferred from the total absence of the part/prop rather than a maintainer statement that it was deliberately omitted — [G] searched git log and issue titles for "preview card provider" / "preview card group" and found nothing; the omission may simply reflect that link-hover-preview UIs (unlike toolbar tooltips) are not typically clustered densely enough on one page to need grouping, but no maintainer text confirms this reasoning.

## 3. When to use

- [E] Showing a preview of what a **link** points to before the user commits to navigating — the canonical evidenced examples are Wikipedia-style article previews and GitHub-style profile/PR hover previews (both named directly by maintainers in #4170 and #4778 as the real-world precedent this component follows).
- [E] The shipped hero demo is itself a "when to use" instance: hovering the word "typography" (a `<PreviewCard.Trigger href="https://en.wikipedia.org/wiki/Typography">`) shows an image + bolded term + one-paragraph summary of the linked Wikipedia article. [E] `docs/.../demos/hero/css-modules/index.tsx`
- [E] Multiple links on a page that should each preview differently, optionally sharing one popup instance via a `handle` for performance/DOM-reuse — the "Detached triggers" / "Multiple triggers" docs examples exist specifically for this ("A single preview card can be opened by multiple trigger elements… by using the same `handle` for several detached triggers, or by placing multiple `<PreviewCard.Trigger>` components inside a single `<PreviewCard.Root>`"). [E] docs `## Examples` → "Multiple triggers"
- [E] When the destination page is itself the source of truth for all information in the preview — i.e., the popup is a convenience/no-navigation-cost peek, never a place to put facts the user can only get by hovering (docs Usage guidelines, quoted in full in §4).
- [I] By extension of the "must exist at the destination" rule: content that is inherently link-attached (article summaries, user/profile cards, product cards linked from a catalog) rather than content unrelated to a link's target.

## 4. When not to use + alternatives

Full comparative guidance (decision table, all six overlay-cluster members): [`../_clusters/overlays.md`](../_clusters/overlays.md). This section restates and sharpens the one boundary that matters most for Preview Card so a reader does not have to cross-reference: **Tooltip vs. Preview Card.**

### The Tooltip-vs-Preview-Card boundary, stated precisely

Both are hover-only (never touch, per §6), sighted-only, non-modal, and never move focus into themselves. The dividing line is the **trigger's element and purpose**, not the visual weight of the popup:

- **Preview Card's trigger is a link whose purpose is navigation.** [E] `PreviewCardTrigger.tsx` renders an `<a>` element unconditionally — confirmed by the part JSDoc ("Renders an `<a>` element"), by `PreviewCardTrigger.spec.tsx` ("`PreviewCard.Trigger` exposes the native `<a>` props in its `render` callback" — the type test asserts `props.href: string | undefined` flows through), and by `describeConformance(<PreviewCard.Trigger />, () => ({ refInstanceof: window.HTMLAnchorElement, ...}))` in `PreviewCardTrigger.test.tsx`. The popup exists to preview *where that link goes* before the user commits to leaving the page.
- **Tooltip's trigger is typically a non-link control** (a button/icon) whose purpose is unrelated to the tooltip content — the tooltip is a label for a tool, not the point of interacting with the trigger. [E] cluster note, citing mui/base-ui#3530.
- **Content-locality rule differs slightly in wording but is the same idea**: Preview Card content "must also exist at the link destination" [E] preview-card docs Usage guidelines — because the destination is a concrete page one hyperlink away. Tooltip content should never be interactive "unless … there are other means of accessing that content (see: description for Preview Card)" [E] mui/base-ui#3530 — i.e., Tooltip's own docs point to Preview Card as the pattern to use *instead* when the hover content needs to be richer/more interactive than a plain text hint, provided a real link is available.
- **A maintainer has stated the litmus test directly** (mui/base-ui#4778, colmtuite): "The pattern is intended for links only, in both Base UI and Radix. … If you're not rendering a link, you should not use PreviewCard. Popover with `openOnHover` might be a good fit, depending on your use case." In other words: no `<a href>` and no navigable destination → don't reach for Preview Card at all (not even as "Tooltip but bigger") — reach for Popover in hover mode, or Tooltip if the content is a short hint.
- **Practical decision rule to hand a component author**: "Am I annotating a link to a page whose content I'm previewing?" → Preview Card. "Am I labeling a control (usually not a link) with a short hint?" → Tooltip. "Am I attaching rich/interactive content to something that isn't a link and doesn't have `openOnHover`-eligible click semantics?" → Popover.

### Other overlaps (brief pointers; full detail in the cluster note)

- [E] **vs Popover** — Popover supports every input modality (including touch) and moves focus in; Preview Card by design never does either. If touch/keyboard/screen-reader access to the content is required, that alone disqualifies Preview Card (both #4170 and #4778 close as "use something else," not "we'll fix accessibility").
- [I] **vs plain `<a title="...">` or native browser preview** — Preview Card exists because the browser's native title-tooltip is unstyleable and slow to appear; Preview Card is the "primitive where the native element can't be styled" case the library's general scope test applies elsewhere (Select's rationale in `research/c-components/select/brief.md` §2 makes the same argument for a different component) — no direct Preview Card citation for this specific framing, marked [I].

## 5. Anatomy & composition

- [E] Canonical tree, exactly as shown in docs Anatomy (`docs/.../preview-card/page.mdx` `## Anatomy`):

  `Root > (Trigger, Portal > (Backdrop, Positioner > Popup > (Arrow, Viewport)))`

  Note the nesting precisely: **Backdrop is a sibling of Positioner inside Portal, not nested under it** — same layering convention as the rest of the popup family (Select, Menu, Popover). Viewport is optional and lives inside Popup, wrapping the actual content.
- [E] **No Provider part** — confirmed by directory listing (`arrow/ backdrop/ index.parts.ts index.ts popup/ portal/ positioner/ root/ store/ trigger/ utils/ viewport/`, no `provider/`) and by `index.parts.ts`'s export list (Root, Portal, Trigger, Positioner, Popup, Arrow, Backdrop, Viewport, createHandle, Handle — no Provider). This is the single biggest structural absence relative to Tooltip, whose sibling brief documents a `Tooltip.Provider` for cross-component delay grouping. Concretely: if a page has ten Preview Card triggers, hovering the second one within the first one's close-delay window does **not** get a shortened/instant open the way grouped tooltips do — each `PreviewCard.Root` (or each handle-backed cluster of triggers sharing one Root) tracks its own delay state independently. [I] inferred from the absence of any shared-state export; no maintainer text found explaining the omission [G].
- [E] **Handle / detached-trigger pattern exists and mirrors Tooltip's.** `store/PreviewCardHandle.ts` (`PreviewCardHandle` class extending `BasePopupHandle`) + `store/PreviewCardStore.ts` (`createNullPreviewCardStore` inert fallback) let a `<PreviewCard.Trigger>` live anywhere in the tree — even outside `<PreviewCard.Root>` — as long as both share a `handle` created via `PreviewCard.createHandle()`. `root/PreviewCardRoot.detached-triggers.test.tsx` is a dedicated 640-line test file exercising: multiple detached triggers sharing one popup, imperative `handle.open(triggerId)`/`handle.close()`, payload passing per-trigger, warnings when a handle is attached to more than one mounted Root simultaneously, and DOM-node reuse (`should reuse the popup and positioner DOM nodes when switching triggers`) — verified present and exercising exactly the same shape of pattern the Tooltip sibling brief documents for Tooltip.
- [E] **Payload / function-as-children pattern**: `PreviewCard.Root`'s `children` can be a render function `({ payload }) => ReactNode`, letting one popup render different content depending on which trigger (identified by an optional per-trigger `payload` prop) opened it — this is unique among the overlay-cluster popups documented so far and is central to the "one preview card, many links" use case. [E] `PreviewCardRoot.tsx` (`typeof children === 'function' ? children({ payload }) : children`), docs "Detached triggers with payload" example.
- [E] Composition rule: `Portal` is required for the popup subtree to render at all — `PreviewCardPortal.tsx` returns `null` unless `mounted || keepMounted`. `Viewport` is explicitly optional and only needed for the multi-trigger animated-content-transition case (see docs API reference note: "The `Viewport` is optional — reach for it only when a single popup is opened by multiple triggers, its content differs per trigger, and the switch between them is animated. When used, set `width: var(--positioner-width)` and `height: var(--positioner-height)` on the `Positioner`…").
- Text spec for a future anatomy diagram: (1) Trigger (the `<a>` link, inline in prose), (2) Portal boundary [invisible], (3) Backdrop [invisible, pointer-events: none], (4) Positioner [invisible box, anchored to Trigger], (5) Popup, (6) Arrow, (7) Viewport [optional, wraps content], (8) the actual preview content (image/heading/description in the reference hero demo).

## 6. Behavior ("How it works")

- [E] **Hover-open model with an explicit default delay, precisely stated.** `packages/react/src/preview-card/utils/constants.ts`:
  ```ts
  export const OPEN_DELAY = 600;
  export const CLOSE_DELAY = 300;
  ```
  These are re-exported as the JSDoc `@default` values on `PreviewCardTrigger.Props.delay` (600ms) and `.closeDelay` (300ms). **Both props live on `Trigger`, not `Root`** — confirmed by the decision-log commit `be2bf142c [preview card] Move delay props to trigger (#3182)` and by the current `PreviewCardTriggerProps` interface (`delay?: number`, `closeDelay?: number`, both with `@default` JSDoc). This differs from a component with a `Provider`-level shared delay: here, delay is inherently per-trigger, reinforcing the "no cross-card grouping" point in §5 — even if a maintainer wanted grouped-delay behavior, the API shape puts the knob at the individual trigger, not at a shared ancestor.
  Behavior on the numbers: opening after 600ms of continuous hover is roughly double the ~300ms industry-typical figure quoted for hover-card/tooltip patterns generally [I] — no Base UI citation states 600ms was chosen *because* it's roughly 2x atypical tooltip delay; this is a numeric observation, not a stated rationale [G] (searched git log/PR bodies around `be2bf142c`/#3182 and the introducing commit `00aebe315`/#469 for a rationale comment; found none beyond the prop move itself).
- [E] **Focus opens it too — with the same delay, and this is a real, deliberately-tested behavior, not an oversight.** `PreviewCardTrigger.tsx` wires `useFocus(floatingRootContext, { delay: delayWithDefault })` unconditionally alongside the hover interaction (`useHoverReferenceInteraction`). `PreviewCardRoot.test.tsx` has a dedicated test "should open when the trigger is focused" (and its counterpart "should close when the trigger is blurred") plus, for the multi-trigger/detached-trigger paths, "should open the preview card with any trigger on focus" and "should open again after escape when focusing another trigger". **This means a sighted keyboard user tabbing through links does get the preview card** (subject to the same 600ms delay) — it is only touch and screen-reader/AT users who are excluded by design (§2, §7). This is the single most important behavioral fact to state precisely per this brief's mandate, since it is easy to assume (wrongly) that a hover-only-sounding component has zero keyboard path.
- [E] **Escape closes it.** `PreviewCardRoot.test.tsx` "dismissal" describe block: `fireEvent.keyDown(document.body, { key: 'Escape' })` closes an open card; a subsequent fresh hover reopens it. `useDismiss(floatingRootContext)` in `PreviewCardRoot.tsx`'s `PreviewCardInteractions` wires this centrally (same `useDismiss` hook used by the rest of the popup family).
- [E] **Never opens on touch.** No touch-specific interaction hook is wired in `PreviewCardTrigger.tsx` (`useHoverReferenceInteraction(..., { mouseOnly: true, ... })` — the `mouseOnly: true` option is the direct code-level confirmation). Maintainer confirmation, twice: mui/base-ui#4778 (oliviertassinari): "with *touch* or *keyboard* modalities, it's as if the component doesn't exist" (touch specifically — keyboard focus does work per above, so this line should be read as being about touch primarily, given the focus tests) and colmtuite in the same thread: "Testing this on iOS 18, I'm not seeing it open when I tap, long press, or hover." A user-facing bug report asking for touch parity with Radix's hover-card was closed as expected behavior in both #4170 and #4778.
- [E] **Hoverable popup — the popup itself keeps the card open.** `PreviewCardPopup.tsx` wires `useHoverFloatingInteraction(floatingContext, { closeDelay: getCloseDelay })`, and `PreviewCardRoot.test.tsx` has an explicit test "closes after hovering out of a popup opened by its trigger" plus "does not close after hovering out of a popup opened externally" (i.e., a programmatically-`open`-forced card, opened without ever hovering the trigger, doesn't auto-close on popup-hover-out — it stays under full external control). This satisfies the WCAG 1.4.13 "hoverable" requirement (see §7).
- [E] **Handle-backed / detached triggers, imperative control.** `PreviewCard.createHandle()` creates a `PreviewCardHandle` with `open(triggerId)`, `close()`, and `isOpen` — usable to open/close/associate triggers programmatically, independent of hover. Calls made while no Root is attached are ignored with a `console.warn` ("no root using this handle is mounted") rather than throwing; `PreviewCardRoot.spec.tsx`/`.detached-triggers.test.tsx` pin this precisely, including the edge case of two Roots transiently sharing one handle during a remount.
- [E] **Controlled/uncontrolled.** Uncontrolled by default (`defaultOpen`, hover/focus/Escape drive it internally). Controlled via `open`+`onOpenChange`, with an additional `triggerId`/`defaultTriggerId` pair needed only for the multi-trigger controlled case (the `eventDetails.trigger` on the `onOpenChange` callback identifies which trigger fired the change instead of a separate `onTriggerIdChange` prop — confirmed by docs "Controlled mode with multiple triggers": "there is no separate `onTriggerIdChange` prop. Instead, the `onOpenChange` callback receives an additional argument, `eventDetails`…"). `eventDetails.cancel()` vetoes an open/close while remaining uncontrolled (`PreviewCardRoot.test.tsx` "onOpenChange cancel() prevents opening while uncontrolled").
- [E] **Nested preview cards work via `FloatingTree`.** `PreviewCardRoot.tsx` wraps a root-level `PreviewCardRoot` in `<FloatingTree>` unless already inside one, mirroring the rest of the popup family (Menu, Select). `PreviewCardRoot.test.tsx` "nested preview card interactions" describe block pins: hovering/clicking a nested child trigger keeps the parent open; a synchronized-closing test shows a parent auto-closes as soon as its child does when neither is hover-anchored anymore. There is also a dedicated race-condition test suite ("hover reopen race conditions across close transitions") reflecting a real bugfix (`4a37b4f0b [preview card] Fix hover reopen race conditions across close transitions (#4446)`).
- [E] **Animation hooks**: standard popup contract — `data-open/closed/starting-style/ending-style` on Popup/Backdrop, `transitionStatus` plumbed through, `--transform-origin`/`--positioner-width`/`--positioner-height`/`--available-width`/`--available-height`/`--anchor-width`/`--anchor-height` CSS vars on Positioner (`PreviewCardPositionerCssVars.ts`). `onOpenChangeComplete` fires after animations finish; `actionsRef.unmount()`/`.close()` for imperative JS-driven exits.
- [E] **SSR notes**: [G] no SSR-specific test file or docs callout found for Preview Card specifically (unlike Select's documented hidden-input `suppressHydrationWarning` case) — searched `*.test.tsx` for "hydrat" and "SSR"; no hits in the preview-card directory.
- [I] **Inline/anchored positioning against the exact hovered point** — `PreviewCardStore.ts`'s `setOpen` captures `clientX`/`clientY` from the triggering hover event and stores them via `updateInlineRectCoords`, and the Positioner uses `createInlineMiddleware(inlineRectCoordsRef)`. This lets the card anchor to the precise pixel position on a (possibly multi-line, wrapped) inline link the user hovered, rather than the link's full bounding box — relevant for links inside flowing prose (exactly the hero demo's use case: a single word inside a paragraph). Decision-log commits `eacfe8aa5 [preview card] Integrate inline positioning (#2074)` and `2b1d63543 [preview card] Keep inline preview anchored while open (#4836)` correspond to this.

## 7. Accessibility contract

- **No WAI-ARIA APG pattern applies, and this is stated honestly rather than invented.** Preview Card is a Base-UI-specific, purely visual enhancement pattern with no corresponding W3C ARIA Authoring Practices Guide entry (unlike Select→Combobox, Dialog→Dialog, Menu→Menu). [G] Searched https://www.w3.org/WAI/ARIA/apg/patterns/ pattern list mentally against "preview card"/"hover card"/"link preview" — no such APG pattern exists; this is a design pattern seen on real sites (Wikipedia, GitHub — both named directly by maintainers, §2/§4) rather than a codified ARIA role/pattern. The component's own accessibility posture, stated by a maintainer closing an issue that asked for one: it is intentionally excluded from assistive-technology interaction, on the theory that its content duplicates what's already reachable by following the link (mui/base-ui#4170, #4778).
- [E] **No ARIA relationship wiring between Trigger and Popup at all.** Verified by grepping the entire `packages/react/src/preview-card/` tree for `aria-describedby`, `aria-labelledby`, and `role=` — the only hits anywhere are `PreviewCardArrow.tsx`'s `'aria-hidden': true` (hides the decorative arrow) and `PreviewCardBackdrop.tsx`'s `role: 'presentation'` (marks the invisible overlay as non-content). **This is a meaningful difference from a component like Tooltip**, whose sibling brief documents `aria-describedby` linking the trigger to the tooltip text so screen readers announce it. Preview Card's Popup carries **no role at all** (not `role="tooltip"`, not `role="dialog"`, nothing) and the Trigger carries **no `aria-describedby`/`aria-expanded`/`aria-haspopup`** pointing at it — confirmed directly from `PreviewCardTrigger.tsx` (only `hoverProps`, `focusProps.reference`, `rootTriggerProps`, `inlineRectTriggerProps`, and an `id` are spread onto the rendered `<a>`; none of these inject ARIA relationship attributes) and `PreviewCardPopup.tsx` (its own prop list is `popupProps`, transition-disabling styles, and `elementProps` — no `role`). The practical consequence: a screen reader user tabbing to the trigger hears only "typography, link" (its own accessible name/href), never anything about the popup's existence or content — which is exactly the intended, maintainer-confirmed behavior (§2), not a bug.
- [E] **WCAG 1.4.13 (Content on Hover or Focus) — partially engaged, and the relevant sub-requirements are met by design, not stated in Base UI's own docs in SC-number terms** [I]:
  - *Dismissible*: Escape closes it (§6) — satisfied.
  - *Hoverable*: moving the pointer from Trigger into the Popup keeps it open (`useHoverFloatingInteraction` + `safePolygon()` for the path between them) — satisfied (§6).
  - *Persistent*: the card does not disappear on a timeout alone; it closes on hover-out/blur/Escape, all user-initiated — satisfied.
  - However, 1.4.13 is normally invoked to argue content *should* be reachable this way for all users; Base UI's stated position (§2) is that the criterion's spirit (don't hide essential content behind hover) is met not by making the hover content universally accessible, but by guaranteeing the hover content is a redundant, optional supplement to something already reachable another way (the link itself). [I] This framing is not spelled out in exactly these terms anywhere in the docs or issues found — it is this brief's synthesis of the "similar to Tooltip" + "content must also exist at destination" statements — marked [I] accordingly.
- [E] **Keyboard interaction table** (evidence: `PreviewCardTrigger.tsx` wiring; `PreviewCardRoot.test.tsx`/`.detached-triggers.test.tsx` as cited):

  | Key / action | Context | Result |
  |---|---|---|
  | Tab to focus the trigger | closed | Opens the preview card after `delay` (default 600ms) — same delay as hover, per `useFocus(..., { delay: delayWithDefault })` (test: "should open when the trigger is focused") |
  | Shift+Tab / Tab away (blur) | open | Closes the preview card (test: "should close when the trigger is blurred") |
  | Enter | trigger focused | Native `<a>` activation — navigates to `href`. Not intercepted by the component; there is no special "confirm/select" semantic the way there is for a Menu item |
  | Escape | open | Closes the preview card (reason: escape key); can reopen on a fresh hover/focus afterward (test: "reopens on hover after Escape closes it") |
  | (none — no dedicated open/toggle key) | closed | There is no keyboard shortcut to force-open the card independent of focusing the link; unlike Menu/Select, ArrowDown/Enter/Space do not "open a popup" here, because the trigger's own purpose (navigate) is preserved unmodified |
- [E] **Screen reader behavior**: nothing about the Popup is announced when the Trigger receives focus (no live region, no `aria-describedby`) — confirmed by the total absence of ARIA wiring noted above. The Trigger's accessible name is whatever the `<a>`'s content/`aria-label` naturally provides — no extra label is injected referencing the preview card.
- [E] **Touch**: never opens (§6); `mouseOnly: true` on the hover interaction is the direct code confirmation, corroborated twice in issues (#4778).
- **Open a11y-adjacent issues (honesty list, per GOV.UK-style disclosure)**:
  - [E] mui/base-ui#4170 ("Component has no a11y warning in the docs. Is it accessible?") — **closed**, resolved by adding a Usage guidelines section to the docs and underlining links in the hero demo (not by changing the component's accessibility model). The requester's stronger ask ("remove this component" or "add a separate control that behaves like a popover") was declined.
  - [E] mui/base-ui#4778 ("Touch support") — **closed** as expected/by-design; no code change, only clarifying discussion. A maintainer suggested Popover with `openOnHover` as the correct alternative when full-modality access is required.
  - [G] Searched issue titles/labels for `component: preview card` + `accessibility` beyond the two above (my 2-lookup budget for this component was spent entirely on #4170/#4778, both confirmed as the correct highest-signal picks — the per-component discussions.md index only lists these two for preview-card) — no further open a11y issues found within budget.

## 8. Prop-level guidance (decision-relevant props)

- [E] **`Trigger.delay` (default `600`)** and **`Trigger.closeDelay` (default `300`)** — `packages/react/src/preview-card/utils/constants.ts`: `OPEN_DELAY = 600`, `CLOSE_DELAY = 300`. Both props live on **Trigger**, not Root (moved there in `be2bf142c`/#3182) — set per-trigger when different links warrant different hover patience (e.g., a longer delay for a densely-linked paragraph to avoid accidental triggers while reading). Because there is no Provider (§5), these are the only delay knobs; there is no page-wide default to change except by passing the same `delay`/`closeDelay` to every Trigger, or wrapping your own default-prop-setting layer around `PreviewCard.Trigger`.
- [E] **`Trigger.handle` / `Root.handle`** — pass a `PreviewCard.createHandle()` value to both when the trigger needs to live outside its Root (e.g., a link deep in body text while the popup markup is defined once near the layout root), or when several separate links should share one popup instance. Use `Trigger.payload` (typed via `createHandle<PayloadType>()`) plus a function-as-children `Root` to vary content per trigger.
- [E] **`Trigger.href`** — not a required prop at the type level (the JSDoc/spec test shows `props.href: string | undefined`), but every test fixture, every docs demo, and every maintainer statement (§4, colmtuite: "The pattern is intended for links only") treats it as expected in practice; `Trigger` renders `<a>` regardless of whether `href` is supplied, and a `render` prop can override the element, but doing so contradicts the component's entire design intent (§2, §4) and should be actively discouraged in guidance copy: "Preview Card's trigger is a link — always give it an `href`." Type-level flexibility exists so `render` can attach custom props without fighting TypeScript, not to invite non-link usage.
- [E] **`Root.triggerId` / `Root.defaultTriggerId`** — required only for the controlled multi-trigger case, to tell a controlled Root which trigger is "active" (drives `payload` resolution and which trigger gets `data-popup-open`). Uncontrolled usage never needs this — the store tracks the implicit active trigger automatically (`useImplicitActiveTrigger`).
- [E] **`Root.onOpenChange` reason values** (`PreviewCardRootChangeEventReason`): `triggerHover | triggerFocus | triggerPress | outsidePress | escapeKey | imperativeAction | none` — note `triggerPress` exists (a click on the link itself is still a distinct reason value even though the component doesn't treat clicking as an "open" gesture the way Menu would) and `none` covers cases like a trigger unmounting out from under an active card.
- [E] **`Portal.keepMounted` (default `false`)** — set `true` to keep the popup subtree mounted while hidden; relevant specifically for the detached multi-trigger + Viewport animated-transition case, where `.detached-triggers.test.tsx`'s "should reposition to a different trigger when reopened with keepMounted=true" pins the exact interaction this unlocks (repositioning without remounting when switching which trigger is active).
- [E] **`Viewport`** — add only when one popup serves multiple triggers with materially different, animated content; per docs, pair it with `width: var(--positioner-width)` / `height: var(--positioner-height)` on Positioner so the box doesn't thrash while content transitions. Its data attributes for styling: `data-current` / `data-previous` (on the two content wrappers during a transition), `data-activation-direction` (space-separated `left|right` + `up|down` tokens, matched via `~=` selectors), `data-transitioning`, `data-instant`.
- [E] **Styling contract highlights** (from `*DataAttributes.ts` / `*CssVars.ts`):
  - Trigger — `data-popup-open` (`PreviewCardTriggerDataAttributes.popupOpen`) is the **only** trigger data attribute; no `data-pressed`/`data-placeholder`-style richness because the trigger is a plain link, not a form control.
  - Popup/Arrow/Backdrop/Positioner — all share the common popup contract: `data-open`/`data-closed`, `data-starting-style`/`data-ending-style`, `data-side`/`data-align` (Popup, Arrow, Positioner only — Backdrop doesn't carry side/align), `data-anchor-hidden` (Positioner only), `data-uncentered` (Arrow only, when the arrow can't center on a too-close-to-edge anchor).
  - Positioner CSS vars — `--available-width`/`--available-height`, `--anchor-width`/`--anchor-height`, `--transform-origin`, `--positioner-width`/`--positioner-height` (the latter two specifically documented for the Viewport-freezing use case).
  - Viewport CSS vars — `--popup-width`/`--popup-height` (placed on the "previous" content container, capturing the popup's size before a content swap, so exit/enter animations can be built against the pre-swap dimensions).

## 10. Pitfalls & FAQ

- **"Preview Card doesn't work with keyboard/touch/screen readers — is this a bug?"** → No: this is the component's stated design (mui/base-ui#4170, #4778). Correction: only use Preview Card when its content is a redundant convenience already reachable at the link's destination; if full-modality access is required, use Popover with `openOnHover` instead (mui/base-ui#4778, colmtuite). [E]
- **"I want the preview to open on tap, like Radix's hover-card on Android"** → Confirmed as a Radix/Android-specific bug, not something Base UI matches intentionally; Base UI's own touch behavior (never opens) matches iOS Safari, GitHub, and Wikipedia's own preview cards, which is the intended reference behavior (mui/base-ui#4778). [E]
- **"I rendered a `<button>` as the trigger and it doesn't behave right"** → The pattern is links-only, in both Base UI and Radix; shadcn/ui's own hover-card docs were called out as incorrectly demoing a `<button>` (mui/base-ui#4778, colmtuite: "The shadcn/ui docs incorrectly render `<button>` for these examples. The pattern is intended for links only"). Correction: always give `Trigger` an `href` and let it render its default `<a>`. [E]
- **"Why doesn't hovering a second link open the card instantly the way grouped tooltips do?"** → There is no `Provider`/shared-delay part for Preview Card (§5); each Trigger's `delay`/`closeDelay` (default 600ms/300ms, `utils/constants.ts`) is independent. If this UX is desired, it would need to be built in userland (e.g., a shared external timer/ref coordinating multiple `Root`s) — [G] no built-in escape hatch found in source, and no issue/PR requesting this was found within this component's 2-lookup budget (both lookups were spent on #4170/#4778, the only two issue numbers indexed for preview-card in `discussions.md`'s per-component index).
- **"My hover event opens the card, but I expected it after a shorter delay"** → 600ms is the documented default (`utils/constants.ts`), noticeably longer than short (~150–300ms) hover-hint delays elsewhere; pass an explicit `delay` on the `Trigger` to shorten it. No maintainer-stated rationale for the specific 600 number was found beyond the prop existing — marked [G] (searched #469, #3182 PR bodies/commit messages for a numeric rationale; none found).
- **Docs feedback on record** — both #4170 and #4778 doubled as "please improve the docs/guidance" threads and both prompted shipped fixes (a Usage guidelines section; underlined demo links) rather than remaining open feature asks, which is a positive signal that this Tier-2 component's documentation gaps are actively being closed by maintainers as they're reported. [E]
