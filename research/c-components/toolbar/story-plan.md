# Toolbar — story plan

Tier 3 floor for toolbar: 6–10 stories. 9 planned. Derived from `brief.md` §3 (when to use), §4 (when not to use — menus.md boundary), §5 (anatomy/composition direction), §6/§7 (behavior + a11y contract), §8 (props), §10 (pitfalls). Includes the **required** composite-keyboard story (arrow keys move within one tab stop).

| # | Name | Renders | Play function? | DoD section(s) served |
|---|---|---|---|---|
| 1 | `Hero` | The kept docs hero demo as-is — alignment ToggleGroup, currency-format Group, font Select, and a Link, all inside one `<Toolbar.Root>` | No | Hero example; identity strip; this single demo already proves the "mixed content in one bar" identity |
| 2 | `CompositeKeyboardNavigation` (required) | A toolbar with Button, Link, a Group of two Buttons, and an Input — the exact composition used in `ToolbarRoot.test.tsx`'s own parametrized suite | Yes — Tab in (single stop), Arrow keys move roving focus across *all* item types in document order (Button → Link → grouped Buttons → Input), confirm looping at the ends; explicitly assert Home/End do **not** move focus (brief §6, §7 — the source-confirmed absence) | Accessibility contract (full keyboard table, including the honest "Home/End not wired" row); the single required story per the task brief |
| 3 | `RTLKeyboardNavigation` | Same composition wired inside `DirectionProvider direction="rtl"`, both horizontal and vertical orientation variants | Yes — assert ArrowLeft/ArrowRight flip meaning under RTL for horizontal; assert vertical orientation is unaffected by direction (brief §6, `ToolbarRoot.test.tsx` parametrized suite) | Accessibility contract (RTL keyboard rows) |
| 4 | `DisabledCascadeExceptLinks` | `<Toolbar.Root disabled>` wrapping a Button, a Link, an Input, and a Group of Button+Link+Input | Yes — assert every Button/Input/Group gets `aria-disabled`/`data-disabled`; assert the Link(s) do **not** (brief §6, `ToolbarRoot.test.tsx` "disables all toolbar items except links") | Behavior (disabled cascade, link exemption); a documented, source-verified asymmetry worth its own story |
| 5 | `FocusableWhenDisabledDefaults` | Several disabled Buttons/Inputs with `focusableWhenDisabled` left at its default (`true`) vs explicitly set to `false` on one item | Yes — assert default-disabled items remain Tab/arrow-reachable and show `data-focusable`; assert the `focusableWhenDisabled={false}` item is skipped by roving focus entirely, including at the loop boundary (brief §6, §7, `ToolbarRoot.test.tsx` "prop: focusableWhenDisabled" suite) | Prop-level guidance (`focusableWhenDisabled` default differs from standalone Button); a11y contract (disabled-item discoverability) |
| 6 | `UsingWithMenuAndSelect` (composition recipe) | Recreates the docs "Using with Menu" and "Using with Select" recipes side by side inside one toolbar: `<Toolbar.Button render={<Menu.Trigger/>}/>` opening a menu, and the hero demo's Select composition, both driven through one shared roving-focus region | Yes — open the Menu via Enter/Space on its toolbar-hosted trigger, assert the menu opens and toolbar focus/tab-stop bookkeeping is preserved on close (brief §5, §6, docs "Using with Menu"/"Using with Select") | Anatomy & composition (the non-inverted `Toolbar.X render={<Y.Trigger/>}` composition direction); Examples (kept functionality demos per Decision 8) |
| 7 | `UsingWithTooltip` (composition recipe, inverted direction) | Recreates the docs "Using with Tooltip" recipe: `<Tooltip.Trigger render={<Toolbar.Button/>}/>`, the one documented *inverted* composition direction | Yes — hover/focus the toolbar-hosted button, assert the tooltip opens; assert the composition direction is exactly the inverse of story 6's Menu/Select pattern | Anatomy & composition (the explicitly-inverted Tooltip direction — the single easiest-to-get-wrong nuance in the brief, §5, §10) |
| 8 | `UsingWithNumberFieldInput` (composition recipe) | Recreates the docs "Using with NumberField" recipe: `<Toolbar.Input render={<NumberField.Input/>}/>` between Decrement/Increment buttons, plus the usage-guideline warning about placing the sole Input last in a horizontal toolbar | Yes — assert arrow-key behavior differs when focus is inside the Input (native text-cursor movement) vs on a Button/Link (roving toolbar focus) — directly demonstrates the one Usage-guidelines caveat on the whole docs page (brief §4, §6, §7) | Prop-level guidance (`Toolbar.Input` composition); Usage guidelines (the one documented behavioral tension) |
| 9 | `NotAMenubar` (boundary/anti-pattern illustration) | Side-by-side: a `Toolbar.Root` misused to host a File/Edit/View-style set of drop-down triggers, vs. the same content correctly built with `Menubar`, annotated to show the `role="toolbar"` vs `role="menubar"` distinction | No (static illustrative pair, cross-links `_clusters/menus.md`) | When not to use + alternatives (the toolbar-vs-menubar boundary, brief §4, linking `research/c-components/_clusters/menus.md`) |

Notes:
- Story 2 is the task-required composite-keyboard story; its honest inclusion of the "Home/End not wired" assertion is deliberate — it is the single most citable, source-verified behavioral fact in the whole toolbar brief and should not be softened into a vague "arrow keys navigate" story that omits it.
- Stories 6–8 collectively cover all four of the docs page's "Using with X" composition recipes (Menu/Select grouped as one story since they share the same non-inverted direction; Tooltip isolated because its direction is inverted; NumberField isolated because it's the only `Toolbar.Input` recipe and carries the arrow-key/text-cursor tension).
- No real-world recreation stories with Phase D interaction data planned (no corpus exists yet for toolbar — brief §11); story 9 is a plausible-boundary illustration, not a corpus recreation.
