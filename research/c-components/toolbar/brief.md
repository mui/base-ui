# Toolbar — component research brief

Tier 3 (lean brief). Mined 2026-07-07 from source, docs, tests, git history, and 1 targeted `gh pr view` lookup (introducing PR #1349). Evidence tags: [E] direct evidence, [I] inference, [G] gap.

## 1. Identity

- **Name / slug**: Toolbar — `@base-ui/react/toolbar`. Multi-part compound component, namespace export `Toolbar.*`. [E] `packages/react/src/toolbar/index.ts`, `index.parts.ts`.
- **Status**: stable since v1.0.0; no `[New]`/`[Preview]` tag. [E]
- **Purpose / IA statement**: "A container for grouping a set of buttons and controls" [E] (docs Subtitle); source JSDoc on `ToolbarRoot`: "A container for grouping a set of controls, such as buttons, toggle groups, or menus" [E]. Taxonomy: **actions** cluster (a command-bar container, sibling of Button/Toggle/ToggleGroup) and the composite-roving-focus sibling of ToggleGroup (both wrap `CompositeRoot`). See also the **toolbar-vs-menubar boundary** in §4, linking [`research/c-components/_clusters/menus.md`](../_clusters/menus.md).
- **6 parts** (from `packages/react/src/toolbar/index.parts.ts`; one-liners from part JSDoc):

  | Part | Renders | One-liner |
  |---|---|---|
  | Root | `<div>` | A container for grouping a set of controls, such as buttons, toggle groups, or menus |
  | Button | `<button>` | A button that can be used as-is or as a trigger for other components |
  | Link | `<a>` | A link component |
  | Input | `<input>` | A native input element that integrates with Toolbar keyboard navigation |
  | Group | `<div>` | Groups several toolbar items or toggles |
  | Separator | re-export | `Toolbar.Separator` re-exports the standalone `Separator` component, oriented perpendicular to the toolbar by default [E] `ToolbarSeparator.tsx` |
- No internal-only parts observed (unlike Select's `@internal` scroll-arrow wrapper).

## 2. Intention

- [E] **Introducing PR**: mui/base-ui#1349 "[Toolbar] Add Toolbar components", closing legacy issue #661. The PR body's own checklist frames the entire design surface at once: parts (`Root`, `Button`, `Input`, `Link`, `Group`, `Separator`); `focusableWhenDisabled` on Button & Input; `orientation`; roving focus with a `loop` prop; and — decisively — **component integrations** as a first-class deliverable, not an afterthought: "The three demos linked above show all of these: Dialog/AlertDialog, Menu, Menu with submenu, Popover, Select, Toggle/ToggleGroup, Tooltip, NumberField using `Toolbar.Input`, DirectionProvider, Switch." [E] `gh pr view 1349`.
- [E] **Design pole: Toolbar is an integration host, not a closed widget.** Unlike most Base UI composites, Toolbar's own docs page devotes three of its four "Examples" subsections entirely to composing *other* Base UI popup/input components into it ("Using with Menu", "Using with Tooltip", "Using with NumberField") rather than demonstrating Toolbar's own props. This matches the PR's stated scope directly — Toolbar was built to be the roving-focus/keyboard-navigation *chassis* that popup triggers, toggles, and inputs plug into, generalizing the same `useButton`/`CompositeItem` machinery ToggleGroup and Toggle also share (`research/c-components/toggle-group/brief.md` §2's `useToolbarRootContext`/`useToolbarGroupContext` detection is the same architectural move in reverse: ToggleGroup detects *whether* it's inside a Toolbar; Toolbar was built expecting to host it).
- [E] **A `hotkey` prop was scoped out during design.** The PR checklist shows a struck-through line: "~hotkey prop on Root~ https://github.com/mui/base-ui/issues/1454" — a considered feature was deliberately deferred to a separate, general-purpose issue rather than folded into Toolbar's initial scope. [E] `gh pr view 1349` body.
- [E] **A `cols` (grid layout) prop shipped, then was later removed** — `d419cfc26` (introducing commit) included `cols` in the PR checklist ("grid layout with `cols` prop"), but a later commit, `5daf2048d "[toolbar] Drop `cols` prop (#3133)"`, removed it — a rare case of a shipped-then-retracted API surface for this component (§9). [G] PR #3133 body not fetched (outside the 4-call `gh` budget spent elsewhere) — rationale inferred only from the commit subject, not confirmed maintainer reasoning.
- [I] Toolbar's usage-guidelines warning about mixing `Toolbar.Input` into a horizontal toolbar ("Left and right arrow keys are used to both move the text insertion cursor in an input, and to navigate among controls... use only one and place it as the last element") reveals that the roving-focus-vs-native-text-cursor conflict was an anticipated design tension from the start, not a discovered bug — this is the *only* Usage-guidelines bullet on the Toolbar docs page, suggesting the maintainers considered it the single most important thing to warn about. [E] `docs/.../toolbar/page.mdx` Usage guidelines, verbatim quoted in §4/§6.

## 3. When to use

- [E] A persistent command bar grouping buttons, toggle groups, links, an occasional single input, and/or popup triggers (Menu/Select/Dialog/Popover/Tooltip) that should share **one Tab stop** with arrow-key navigation among them (docs Anatomy + all four "Using with X" Examples subsections).
- [E] Whenever several related popup triggers or toggle controls need to behave as one coherent keyboard-navigable region rather than each occupying its own Tab stop — the entire PR #1349 integration checklist (Dialog/AlertDialog, Menu, Menu-with-submenu, Popover, Select, Toggle/ToggleGroup, Tooltip, NumberField, Switch) is direct evidence this is the intended general-purpose use case, not a corner case.
- [I] Toolbars in the classic desktop-app sense (formatting bars, editing command strips) — the `docs/src/app/(private)/experiments/toolbar/text-editor.tsx` experiment name strongly suggests this was an explicit manual-testing target, though its contents weren't read in this pass [G] (file listed, not opened).

## 4. When not to use + alternatives

- [E] **A menu bar hosting drop-down application menus (File/Edit/View) → Menubar, not Toolbar.** This is the toolbar-vs-menubar boundary the prompt calls out explicitly. Toolbar's items are *directly-actionable controls* (buttons, toggles, a lone input, popup triggers) sharing one composite tab stop; Menubar is a `role="menubar"` container of `Menu.Root`s whose triggers become `role="menuitem"` composite items with left/right roving focus and *hover-switching between open sibling menus* — a fundamentally different interaction model (opening drop-downs, not directly acting). Full comparative table: [`research/c-components/_clusters/menus.md`](../_clusters/menus.md) §"Decision table" (row: "App-chrome File/Edit/View bar | Menubar | `role='menubar'` container of Menu.Roots; arrow-key bar navigation"). [E] Both `menus.md` and this repo's source confirm Menubar is tracked as its own component (`mui/base-ui#1407 "[menu] Implement Menubar"`, shipped `#1684`) with distinct role semantics — it is not "Toolbar with menus in it."
- [E] **Plain unrelated action buttons that don't need shared keyboard navigation → no wrapper at all, or `role="group"` around plain `<button>`s.** mui/base-ui#4008 (closed, "Why is there no Button Group?"): "'button group' (non-toggles) is just `role='group'` around some normal buttons, so there isn't anything that a headless lib could add"; the same answer explicitly names **Toolbar** as the correct escalation only "when a command bar needs an exclusive/multi-toggle cluster" or roving focus is actually wanted — not for every row of buttons (mj12albert, cross-referenced in `research/c-components/button/brief.md` §4 and `discussions.md`'s `button / toolbar | 4008, ...` index entry). [E]
- [E] **A row of exclusive/multi-select toggle buttons with no other mixed content → ToggleGroup alone**, standalone, without a hosting Toolbar — Toolbar's value-add is coordinating *mixed* content (buttons + toggles + links + one input + popup triggers) in one bar; a single ToggleGroup doesn't need Toolbar's extra container semantics. [I] derived directly from the anatomy: ToggleGroup already ships its own complete `CompositeRoot` when standalone (`research/c-components/toggle-group/brief.md` §5) — Toolbar adds nothing in that case beyond `role="toolbar"` framing.
- [I] **A single popup trigger with no surrounding command bar → use the popup component's own Trigger directly**, not `Toolbar.Button render={<Menu.Trigger/>}`. Toolbar's composition recipes exist specifically to unify *multiple* triggers/controls under one roving-focus region; wrapping a lone trigger in Toolbar buys nothing.

## 5. Anatomy & composition

- [E] Canonical tree (docs Anatomy):
  `Root > (Button, Link, Separator, Group > (Button, Button), Input)`.
- [E] **No Portal/Positioner/Popup layering** — unlike overlay components, Toolbar itself never portals anything; it is a plain in-flow `<div role="toolbar">`. Popup components composed *into* it (Menu, Select, Dialog, Popover, Tooltip) bring their own Portal/Positioner/Popup grammar unchanged (B-P18) — Toolbar only supplies the trigger-hosting slot.
- [E] **The composition mechanism is `render`, always in one specific direction**: for Dialog/AlertDialog/Menu/Popover/Select, the pattern is `<Toolbar.Button render={<X.Trigger />} />` — the popup's Trigger is rendered *through* `Toolbar.Button`'s render prop, so `Toolbar.Button` remains the DOM node and composite-list item, while `X.Trigger` supplies its own ARIA (`aria-haspopup`, `aria-expanded`, etc.) merged in via `mergeProps`. **Tooltip is the sole documented exception, inverted**: `<Tooltip.Trigger render={<Toolbar.Button />} />` — Tooltip.Trigger is outermost because a tooltip must attach its hover/focus listeners to the actual interactive element rather than being rendered through it (docs "Using with Tooltip" section, explicit callout: "Unlike other popups, the toolbar item should be passed to the `render` prop of `<Tooltip.Trigger>`"). [E] This directional inversion is a precise, citable nuance a demo/story must get right.
- [E] **`Toolbar.Input` composition follows the same non-inverted direction as Button**: `<Toolbar.Input render={<NumberField.Input />} />` (docs "Using with NumberField") — `Toolbar.Input` stays the composite-registered DOM node, `NumberField.Input` supplies its own input semantics via `render`.
- [E] **`Toolbar.Group`** is a plain `role="group"` wrapper (`ToolbarGroup.tsx` L46) that additionally provides a `ToolbarGroupContext` (just `{ disabled }`) so nested `Toolbar.Button`/`Toolbar.Input` inherit a group-level disabled cascade without each needing it set individually (`ToolbarButton.tsx` L33-35, `ToolbarInput.tsx` L31-33) — mirroring the OR-chain pattern already seen in ToggleGroup (`research/c-components/toggle-group/brief.md` §2/§6).
- [E] **`Toolbar.Separator`** is a thin wrapper re-exporting the standalone `Separator` component, with its `orientation` **auto-flipped to the opposite of the toolbar's own orientation** by default (`ToolbarSeparator.tsx` L20-25: `{ vertical: 'horizontal', horizontal: 'vertical' }` lookup) — a horizontal toolbar gets vertical separator lines between groups, matching the hero demo's visual layout, without the consumer needing to compute this themselves.
- [E] **`ToggleGroup`-hosting is bidirectionally aware** (already detailed in `research/c-components/toggle-group/brief.md` §2/§5): Toolbar doesn't have to know about ToggleGroup specifically — ToggleGroup detects the Toolbar via context and adapts its own rendering strategy — but the practical anatomy consequence for Toolbar's own brief is that a Toolbar-hosted ToggleGroup's Home/End keys are governed by *Toolbar's* `CompositeRoot` settings, not ToggleGroup's own (§6).
- Text spec for a future anatomy diagram: (1) Root [`role="toolbar"`, the single composite region], (2) Button, (3) Link, (4) Separator [orientation auto-flipped], (5) Group [`role="group"`, disabled-cascade boundary], (5a)-(5b) nested Buttons, (6) Input [last position recommended when horizontal, §4/§6].

## 6. Behavior ("How it works")

- [E] **Not controlled/uncontrolled in the state-management sense** — like Button, Toolbar owns no persisted "value"; its only two props affecting shared behavior are `disabled` (cascades to all items) and the roving-focus configuration (`orientation`, `loopFocus`). State belongs to whatever is composed inside it (a hosted ToggleGroup's `value`, a hosted NumberField's `value`, etc.).
- [E] **Composite roving-focus engine**: `ToolbarRoot` wraps `CompositeRoot` and tracks every registered item's metadata (`disabled`, `focusableWhenDisabled`) in an internal `itemMap` state, computing `disabledIndices` — the set of items removed from the tab sequence entirely (items that are *both* `disabled` and *not* `focusableWhenDisabled`, `ToolbarRoot.tsx` L36-50). This is the single shared roving-focus contract every item type (Button, Link, Input, and a hosted ToggleGroup/Toggle) registers into via `CompositeItem`.
- [E] **`enableHomeAndEndKeys` is never passed to Toolbar's `CompositeRoot`** (confirmed by direct inspection of `ToolbarRoot.tsx` L70-81 — no such prop appears in the `<CompositeRoot>` invocation, unlike `ToggleGroup.tsx` L133 which passes it explicitly when standalone). **This means Home/End keys are not part of Toolbar's own keyboard contract** — only Tab (single stop) and orientation-appropriate arrow keys are wired. This is a genuine, source-verified behavioral difference from ToggleGroup and is directly responsible for the standalone-vs-Toolbar-hosted ToggleGroup Home/End asymmetry flagged in `research/c-components/toggle-group/brief.md` §5/§6/§7.
- [E] **Disabled cascade and link exemption**: `disabled` on `Toolbar.Root` propagates to every `Toolbar.Button`/`Toolbar.Input`/`Toolbar.Group` (each reads `useToolbarRootContext().disabled` and ORs it with its own prop) — but **not to `Toolbar.Link`**, whose metadata is hardcoded `{ disabled: false, focusableWhenDisabled: true }` with the inline comment "Links cannot be disabled, but they still occupy a focusable composite item slot" (`ToolbarLink.tsx` L8-12). Confirmed by `ToolbarRoot.test.tsx` "disables all toolbar items except links" — every button/input gets `aria-disabled="true"`/`data-disabled`, but link elements explicitly do not.
- [E] **`focusableWhenDisabled` defaults differ by item type**: `Toolbar.Button` and `Toolbar.Input` both default `focusableWhenDisabled` to **`true`** (`ToolbarButton.tsx` L24, `ToolbarInput.tsx` L22) — the opposite default from standalone `Button` (`focusableWhenDisabled: false`, `research/c-components/button/brief.md` §1) — because Toolbar items are composite/roving-focus members where discoverability of disabled controls matters more (matching the library-wide composite-vs-standalone rule, B-P29/#656). A disabled-but-`focusableWhenDisabled`-false first item correctly cedes the single roving tab stop to the next focusable item, and looping skips it (`ToolbarRoot.test.tsx` "moves the initial tab stop off a disabled, non-focusable first item").
- [E] **Render-prop-composed disabled forwarding is deliberately asymmetric**: `ToolbarButton.tsx` L54-73 forwards a React `disabled` prop to the rendered element **only when a custom `render` is supplied** (e.g. `render={<Menu.Trigger/>}`) — for the *default* toolbar button (no `render`), `disabled` is **not** forwarded as a literal prop, so that a focusable-when-disabled default button stays hoverable for things like tooltips. The inline `TODO` comment references an open follow-up: `https://github.com/mui/base-ui/issues/1976#issuecomment-2916905663` — this is an acknowledged, still-evolving edge case, not settled doctrine. [E]
- [E] **RTL awareness**: arrow-key direction flips correctly under `dir="rtl"` for horizontal orientation, verified across all four `[direction, orientation]` combinations in `ToolbarRoot.test.tsx`'s parametrized "keyboard navigation" suite (identical evidence shape to ToggleGroup's own RTL test, `research/c-components/toggle-group/brief.md` §6).
- [E] **Usage-guideline-documented tension: Input vs arrow-key text-cursor movement.** "Left and right arrow keys are used to both move the text insertion cursor in an input, and to navigate among controls in horizontal toolbars. When using an input in a horizontal toolbar, use only one and place it as the last element of the toolbar" (docs Usage guidelines, verbatim) — this is the single explicit behavioral caveat on the whole docs page, confirming Input's arrow-key conflict with native text-cursor movement was anticipated, not discovered after the fact.

## 7. Accessibility contract

- [E] **Pattern**: WAI-ARIA APG **Toolbar** pattern — `role="toolbar"` with single-tab-stop roving `tabindex` and arrow-key navigation among focusable members. Reference: https://www.w3.org/WAI/ARIA/apg/patterns/toolbar/ (not fetched in this pass — flagged [I]/general knowledge rather than [E], though the `role="toolbar"` + roving-focus implementation matches the pattern's mechanics precisely per source evidence below).
- [E] **ARIA managed by the component**: `role="toolbar"` and `aria-orientation={orientation}` on Root (`ToolbarRoot.tsx` L63-66, `defaultProps`); each hosted item supplies its own semantics (native `<button>`/`<a>`/`<input>`, or whatever ARIA a composed popup Trigger brings via `render`); `Toolbar.Group` adds `role="group"` (no `aria-orientation`, matching the same correctness fix already applied to ToggleGroup, §6 of that brief — though no dedicated commit citation for Toolbar.Group specifically was found in this pass [G]).
- [E] **Keyboard interaction table** (evidence: `ToolbarRoot.tsx`, `useCompositeRoot.ts`, `ToolbarRoot.test.tsx`, `ToolbarButton.test.tsx`):

  | Key | Context | Action |
  |---|---|---|
  | Tab | entering the toolbar | Moves focus to exactly one item — **single tab stop for the entire toolbar**, regardless of how many buttons/links/groups/inputs it contains |
  | ArrowRight / ArrowLeft | `orientation="horizontal"`, LTR | Move roving focus to next/previous item across *all* item types (Button → Link → grouped Buttons → Input, in document order); wraps at the ends if `loopFocus` (default `true`) |
  | ArrowLeft / ArrowRight | `orientation="horizontal"`, RTL (`dir="rtl"`) | Direction flipped, mirroring ToggleGroup's own RTL contract |
  | ArrowDown / ArrowUp | `orientation="vertical"` | Move roving focus to next/previous item; unaffected by RTL |
  | Home / End | any orientation | **Not wired** — `enableHomeAndEndKeys` is never passed to Toolbar's `CompositeRoot` (§6); only Tab + orientation-appropriate arrows move focus. This is a source-confirmed absence, not an oversight in this brief. |
  | Enter / Space | focused item (Button/Input/Link/composed Trigger) | Activates per that item's own contract (native `<button>`/`<a>`/`<input>` semantics, or the composed popup's own activation, e.g. opening a Menu) |
  | Left/Right arrow inside a hosted `Toolbar.Input` | horizontal toolbar | **Conflict, resolved by usage guideline, not by code**: arrow keys inside a focused text input move the text cursor (native input behavior) rather than roving toolbar focus — this is why the docs recommend using at most one Input, placed last (§6). No code-level disambiguation exists; it's a documented placement convention. |
- [E] **Disabled-item discoverability**: Toolbar's own item types (Button, Input) default to `focusableWhenDisabled={true}` — a deliberate a11y choice so disabled controls remain discoverable to keyboard/AT users navigating the bar (§6, matching the composite-vs-standalone rule, B-P29/#656), unlike standalone `Button`'s opposite default. **Links can never be disabled** — they always occupy a focusable slot (§6).
- [E] **Screen reader notes**: `aria-orientation` on the toolbar root tells AT the navigation axis; individual item accessible names are the consumer's responsibility (`aria-label` on icon-only Buttons — every worked demo in this pass sets one, e.g. `aria-label="Format as currency"`). A `Toolbar.Group`'s own accessible name (e.g. "Numerical format," "Alignment" in the hero demo) should be set via `aria-label` on the Group itself, since it carries `role="group"`.
- **Known issues / honesty list**: [E] the disabled-forwarding-for-composed-`render`-buttons behavior (§6) has an open, in-source-acknowledged follow-up referenced by inline comment (`https://github.com/mui/base-ui/issues/1976#issuecomment-2916905663`) — flagged honestly as unsettled rather than presented as finished design. [G] No Toolbar-specific accessibility bug beyond this was found in the mined discussions/issues corpus — the per-component index entry `button / toolbar | 4008, 4021, 5150, 3759` in `discussions.md` covers scope questions (#4008), and #4021/#5150/#3759 were not individually fetched (outside the 4-call `gh` budget) — not cited as claims here.

## 8. Prop-level guidance (decision-relevant props)

- [E] **`orientation` (Root, default `'horizontal'`)** — determines the active arrow-key axis and the default `aria-orientation`/perpendicular `Separator` flip (§5). Set to `'vertical'` for a sidebar-style command rail.
- [E] **`loopFocus` (Root, default `true`)** — same convention name/default as ToggleGroup's; set `false` to stop wrap-around arrow navigation at the toolbar's ends.
- [E] **`disabled` (Root)** — cascades to Button/Input/Group but **not Link** (§6) — use this when the entire command bar should go inert except for its navigational links.
- [E] **`focusableWhenDisabled` (Button/Input, default `true`)** — the opposite default from standalone Button; override to `false` only when a specific disabled item should be fully removed from the roving-focus sequence (e.g. the "skips a disabled Toolbar.Input with focusableWhenDisabled={false}" test scenario) rather than merely inert-but-discoverable.
- [E] **`render` composition direction matters and is asymmetric** (§5) — `Toolbar.Button render={<X.Trigger/>}` for Dialog/AlertDialog/Menu/Popover/Select (Toolbar.Button outermost); `Tooltip.Trigger render={<Toolbar.Button/>}` for Tooltip (inverted, Tooltip.Trigger outermost); `Toolbar.Input render={<NumberField.Input/>}` for NumberField (Toolbar.Input outermost, same direction as Button). Getting the direction wrong for Tooltip specifically will attach hover/focus listeners to the wrong node.
- [E] **`nativeButton` (Button, inherited)** — same convention as standalone Button (`research/c-components/button/brief.md` §8); required `false` whenever composing a non-`<button>`-rendering part (`Switch.Root`, a custom trigger) into `Toolbar.Button`, or a dev-mode `Base UI:` warning fires (verified exact warning string in `ToolbarButton.test.tsx` "renders a switch").
- [E] **Styling contract highlights** (from `*DataAttributes.ts` across all Toolbar parts): `data-disabled` (Root, Button, Group, Input — not Link), `data-orientation` (Root, Button, Group, Link, Input, Separator — universally present for orientation-aware styling), `data-focusable` (Button, Input — present specifically when the item remains focusable-when-disabled). No CSS variables anywhere in Toolbar's parts — no dynamic measurements to expose, matching Button's and ToggleGroup's equally lean styling contracts.

## 10. Pitfalls & FAQ

- **"Home/End keys don't move focus in my toolbar"** → not a bug: Toolbar never enables Home/End key navigation (`enableHomeAndEndKeys` is not passed to its `CompositeRoot`, §6/§7) — only Tab (single stop) and orientation-appropriate arrows are wired. If Home/End are needed, they are not currently available on Toolbar itself. [E]
- **"A ToggleGroup nested in my Toolbar loses Home/End support it has when used standalone"** → confirmed, source-verified asymmetry: a Toolbar-hosted ToggleGroup defers its composite behavior entirely to Toolbar's `CompositeRoot`, which doesn't enable Home/End; a standalone ToggleGroup enables it on its own `CompositeRoot` (`research/c-components/toggle-group/brief.md` §5/§6/§9, PR #3971). [E]
- **"Composing `Tooltip` into my toolbar the same way I compose `Menu`/`Select` breaks the tooltip"** → the composition direction is inverted for Tooltip specifically: use `<Tooltip.Trigger render={<Toolbar.Button />} />`, not `<Toolbar.Button render={<Tooltip.Trigger />} />` — the docs call this out explicitly as the one exception among the four popup-integration recipes (§5). [E]
- **"Arrow keys move the text cursor instead of toolbar focus when my input is focused"** → expected: Base UI does not disambiguate this in code; the documented convention is to use at most one `Toolbar.Input` in a horizontal toolbar and place it last (docs Usage guidelines, §4/§6). [E]
- **"My disabled `<Toolbar.Button render={<Menu.Trigger/>}>` doesn't look disabled on the rendered element"** → check whether `render` is supplied: `disabled` is only forwarded as a literal prop to the rendered element when a custom `render` is used; the *default* toolbar button (no `render`) intentionally does not receive a literal `disabled` prop so it stays hoverable while inert (§6) — this is an acknowledged, still-evolving area with an open follow-up referenced inline (`#1976`). [E]
- **"Why is there no Toolbar-native way to disable my `Toolbar.Link`?"** → intentional: links can never be marked disabled (`ToolbarLink.tsx` hardcodes `disabled: false`) because a disabled link has no accepted native/ARIA semantic — they always occupy a focusable roving-focus slot regardless of the toolbar's own `disabled` state (§6). [E]
- **"I upgraded and my `cols`/grid-layout toolbar broke"** → `cols` was removed (`#3133`, §9) after shipping in the introducing PR; grid-layout toolbars need a different composition approach (a plain CSS grid wrapper around toolbar content) — [G] no migration guidance found in the mined corpus for this specific removal.
