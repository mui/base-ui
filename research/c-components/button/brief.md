# Button — component research brief

Tier 3 (lean brief). Mined 2026-07-07 from source, docs, tests, git history, and 1 targeted `gh pr view` lookup (read-only). Evidence tags: [E] direct evidence, [I] inference, [G] gap.

## 1. Identity

- **Name / slug**: Button — `@base-ui/react/button`. Single-part component (no namespace of sub-parts); exported directly as `Button`. [E] `packages/react/src/button/index.ts`
- **Status**: stable since v1.0.0; no `[New]`/`[Preview]` tag. One of the youngest components by introduction date despite being conceptually primitive — merged 2025 (see §2), long after Toggle/ToggleGroup/Toolbar (2024). [E]
- **Purpose / IA statement**: "A button component that can be used to trigger actions" [E] (`Button.tsx` JSDoc); docs Subtitle: "A button component that can be rendered as another tag or focusable when disabled" [E] (`docs/src/app/(docs)/react/components/button/page.mdx`). Taxonomy: **actions** cluster (per `research/PROMPT.md` §12 Tier-3 taxonomy sketch: avatar/collapsible/fieldset/input/meter/progress/separator/switch/**toggle/toggle-group/toolbar**/button). Purpose: a polymorphic, accessibility-hardened wrapper around the native `<button>` — not a styling primitive (there is nothing to style; a plain `<button>` already works).
- **Parts**: none — single element. [E] `Button.tsx` renders one `<button>` via `useRenderElement('button', ...)`.
- **Props surface** (from `ButtonProps`, `Button.tsx`): `disabled`, `focusableWhenDisabled`, `nativeButton`, plus the inherited `NativeButtonProps` (`render`, `className`, `style`, standard `<button>` attributes). [E]

## 2. Intention

- [E] **Introducing PR**: mui/base-ui#2363 "[button] New `Button` component" (atomiks, merged; commit `ad0190c7e`). PR body verbatim: "Essentially this component allows two things: Render the button as another tag, like `div`, while remaining accessible[;] Add a `focusableWhenDisabled` prop which keeps the button focusable when it's disabled (for e.g. loading states)... It's a general-purpose button component, but is **not** meant to solve 'press events' (#1726), which is a separate issue." Closes #2225.
- [E] **The scope-philosophy reversal is the heart of this component's intention.** Per `research/b-library-principles/_mining/issues-prs.md` (#2138/#2225 → PR #2363): colmtuite had articulated the library's scope test for a headless primitive — ship one only where (a) HTML lacks the functionality, (b) the HTML element can't be styled, or (c) no good HTML primitive exists. A plain `<button>` fails all three tests, so Button was **initially rejected**. The decision was **reversed** specifically for accessibility reasons: `focusableWhenDisabled` (native `disabled` removes an element from the accessibility tree and the tab order entirely, which hides loading/pending buttons from assistive technology and breaks focus continuity), Enter/Space activation normalization on non-native tags (`render={<div/>}` needs Base UI to replicate what the browser gives `<button>` for free), and browser-bug patching. **Button exists purely for these a11y affordances — it is not a styling convenience.** [E] `research/b-library-principles/principles.md` B-P4 codifies the same episode as the library's canonical "scope test" case study.
- [I] Because the a11y affordances are the entire reason to reach for Button, a plain native `<button>` remains fully valid Base UI usage whenever `disabled`, tag-polymorphism, or `focusableWhenDisabled` semantics are not needed — Button doesn't replace `<button>`, it's what you reach for on top of it.
- [E] **Explicit non-goal**: press/pointer-event normalization is out of scope — tracked separately as #1726 (`usePress`-style polymorphic pointer normalization), not folded into Button (PR #2363 body).
- [E] Button's `useButton` internal hook is shared library-wide as the building block for every other "acts like a button" part — Toggle, Toolbar.Button, and (per `useButton.ts` JSDoc) any composite item — rather than duplicating disabled/keyboard logic per component (`packages/react/src/internals/use-button/useButton.ts`). This matches the "avoid duplicating logic" AGENTS.md rule.

## 3. When to use

- [E] Any actionable control that isn't a native `<button>` for a good reason — e.g. it must render as another tag while keeping button semantics (docs "Rendering as another tag" example: `render={<div />} nativeButton={false}`).
- [E] Buttons that enter a **loading/pending state** after being clicked and must not lose focus or drop out of tab order — the documented `focusableWhenDisabled` recipe with its own demo (`docs/.../button/demos/loading/`). [E] `Button.test.tsx` "prop: focusableWhenDisabled" suite pins exactly this: `aria-disabled="true"`, `tabindex="0"`, native `disabled` attribute absent, all interaction handlers no-op.
- [E] As the reusable building block Base UI itself composes into other parts (Toolbar.Button, Toggle both wrap `useButton`) — reach for it when hand-rolling a similar "acts as a button" custom part per the composition doctrine (`useRender`/`mergeProps`, B-P8–B-P12).
- [I] Submit buttons inside a `<Form>`/native `<form>` — the docs explicitly flag that `type="submit"` must be set manually (§8), implying Button is form-compatible but not form-opinionated.

## 4. When not to use + alternatives

- [E] **Never for links.** Docs Usage guidelines, verbatim: "**Links**: The Button component enforces button semantics (`role='button'`, keyboard interaction, disabled state). It should not be used for links." The "Rendering links as buttons" section is explicit: "If a link needs to look like a button visually, style the `<a>` element directly with CSS rather than using the Button component." [E] This is a stated, load-bearing boundary — `role="button"` on an anchor breaks its native link semantics (activation keys, screen-reader affordances, right-click/open-in-new-tab).
- [E] **A plain `<button>` HTML element** whenever no `disabled`/`focusableWhenDisabled`/tag-polymorphism is needed — Button adds zero value over the native element in that case ([I] derived directly from the scope test in §2; no dedicated maintainer statement contrasting "Button vs plain button" was found — [G] searched issues for "when to use Button component", no results beyond the #2138/#2225 arc already cited).
- [E] **A group of buttons → not `ButtonGroup`, none exists.** mui/base-ui#4008 (closed): "a 'button group' (non-toggles) is just `role='group'` around some normal buttons, so there isn't anything that a headless lib could add" (mj12albert); use **ToggleGroup** for pressed-state/roving-focus semantics or **Toolbar** for a mixed command bar (see `research/c-components/_clusters/binary-controls.md` and this component's §4 continuation below). Cross-referenced in `discussions.md` per-component index: `button / toolbar | 4008, 4021, 5150, 3759`.
- [E] **A two-state on/off control → Toggle**, not Button with manually-toggled `aria-pressed`. Toggle exists precisely to manage that state/ARIA pairing; see `research/c-components/toggle/brief.md`.
- [I] **Press/pointer-gesture normalization across element types** (the #1726 use case) is explicitly out of scope for Button — compose your own pointer handling; Button only normalizes keyboard activation and disabled semantics.

## 5. Anatomy & composition

- [E] Single element, no sub-parts: `Button` renders exactly one `<button>` (or the `render`-swapped tag). Anatomy code sample from docs is literally `<Button />` with nothing nested (`page.mdx` §Anatomy).
- [E] **Composition role**: Button is not typically nested as a "part" of another namespace the way Select.Trigger is — instead, other components' `Trigger` parts (`Menu.Trigger`, `Dialog.Trigger`, etc.) already embed the same `useButton` behavior directly, and Button itself gets composed the other direction: as the `render` target for `Toolbar.Button` (`docs/.../toolbar/page.mdx` "Using with Menu"/"Using with Tooltip" recipes use `Toolbar.Button render={<Menu.Trigger />}` — Button-shaped internals, not the `Button` component literally, but the identical `useButton` contract).
- [E] **`nativeButton` convention** (default `true`): the library-wide signal (B-P10) that must be set to `false` when `render` swaps the tag away from `<button>`, so ARIA (`role="button"`), keyboard handling, and dev-mode warnings adapt correctly (`useButton.ts` — mismatched `nativeButton` vs actual rendered tag triggers a `Base UI:` console error in both directions, verified in `ToolbarButton.test.tsx` "renders a switch"/"handles interactions" tests which show the exact warning string when `nativeButton` doesn't match the rendered element).
- Text spec for a future anatomy diagram: (1) the single `<button>` (or swapped element) — no numbered sub-parts exist for this component.

## 6. Behavior ("How it works")

- [E] **Not controlled/uncontrolled in the state-management sense** — Button has no persisted value; its only two-state-like behavior is `disabled`. There is no `onXChange` triplet (no `value`/`onValueChange`/`defaultValue`) because Button doesn't own state — this is a category difference from Toggle, whose state model is the state Button intentionally lacks [I].
- [E] **`disabled` semantics differ by rendered tag** (`Button.test.tsx`): on a real `<button>`, `disabled` sets the native `disabled` attribute and `data-disabled`, and the element is fully removed from the tab order (`user.keyboard('[Tab]')` skips it) and from all interaction handlers (click/mousedown/pointerdown/keydown all no-op). On a non-native element (`render={<span/>}` + `nativeButton={false}`), Base UI instead applies `aria-disabled="true"` + `tabindex="-1"` + `data-disabled` (since a `<span>` has no native `disabled` attribute to rely on) — same net effect (not focusable, all handlers no-op), different mechanism. [E]
- [E] **`focusableWhenDisabled` semantics** (`Button.test.tsx` "prop: focusableWhenDisabled" describe block): the button (native or custom-element) gets `aria-disabled="true"` + `data-disabled` but **keeps** `tabindex="0"` and remains reachable via Tab and hoverable (`onMouseMove` still fires per the Chromium-only test) — while click/keydown/pointerdown activation is still fully suppressed. This is the mechanism behind the "loading state" recipe in §3.
- [E] **Keyboard activation is inherited from the native `<button>` element when `nativeButton` is true** — no custom Enter/Space handling exists in `Button.tsx` itself; the shared `useButton` hook only intercepts `onKeyDown`/`onKeyUp` to (a) block activation entirely while `disabled`, and (b) — only for `!isNativeButton` elements — replicate Enter/Space→click normalization that a real `<button>` gets from the browser for free (`useButton.ts` L155-164, the `!isNativeButton && (isSpaceKey || isEnterKey)` branch). This confirms Button's core value proposition precisely: **Enter/Space activation and disabled semantics on non-native tags, and nothing else.**
- [E] **Composite-context awareness**: `useButton` reads an internal `CompositeRootContext` to decide whether Space fires on `keydown` (composite items — e.g. inside Toolbar/ToggleGroup roving-focus widgets) or `keyup` (standalone buttons) — matching the library-wide composite-vs-standalone rule (B-P29) even though this dispatch lives in the shared hook rather than in `Button.tsx` directly. [E] `useButton.ts` `isCompositeItem` branch.
- [E] **SSR/hydration**: no component-specific SSR notes found; Button has no hidden inputs, ids, or measurement state to desynchronize — [G] searched release notes for "button" + "SSR"/"hydration", nothing beyond the library-wide `nativeButton` rationale ("element identity must be known before hydration", B-P10) which explains *why* the prop exists but isn't a Button-specific fix.

## 7. Accessibility contract

- [E] **Pattern**: no dedicated WAI-ARIA APG pattern — Button implements native `<button>` semantics (`role="button"` when polymorphic) rather than a distinct composite widget pattern. Closest reference: https://www.w3.org/WAI/ARIA/apg/patterns/button/ (native-button case; not fetched — [G], general web knowledge, not mined from this repo — treat as [I] until verified against a docs citation).
- [E] **Keyboard interaction table** (evidence: `useButton.ts`, `Button.test.tsx`):

  | Key | Context | Action |
  |---|---|---|
  | Enter | native `<button>`, enabled | Native browser activation (no Base UI handler needed) |
  | Space | native `<button>`, enabled, standalone | Native browser activation on `keyup` (standard button behavior) |
  | Space | native `<button>`, enabled, **composite item** (e.g. inside Toolbar) | Activation normalized to `keydown` to match roving-focus/composite conventions (B-P29) |
  | Enter / Space | `render`-swapped non-`<button>` tag, `nativeButton={false}`, enabled | Base UI synthesizes click activation via `useButton`'s keydown handler (native elements don't do this for free) |
  | Any | `disabled` (no `focusableWhenDisabled`) | Element is unreachable by Tab; no handler fires |
  | Any | `disabled` + `focusableWhenDisabled` | Element remains Tab-reachable and hoverable; all activation handlers still no-op |
- [E] **ARIA attributes managed by the component**: `role="button"` only on non-native renders (a real `<button>` doesn't need it); `aria-disabled="true"` when `disabled` and (either non-native, or native+`focusableWhenDisabled`); native `disabled` attribute only for native `<button>` without `focusableWhenDisabled`. (`Button.test.tsx`, all four disabled/focusableWhenDisabled combinations pinned.)
- [E] **Dev-time correctness guard**: mismatched `nativeButton` vs actual rendered tag throws a console `Base UI:` error explaining the risk in both directions (native-expected-but-non-button-rendered, and non-native-expected-but-button-rendered) — verified via `ToolbarButton.test.tsx`'s exact error-string assertions when composing `Switch.Root` into `Toolbar.Button` (a stand-in for the same `useButton` code path Button itself uses).
- **Known issues / honesty list**: [G] searched `label:"component: button" label:accessibility` via local source/docs only (no live issue search available beyond the 4-call budget already spent on introducing-PR verification) — no Button-specific open a11y bug found in the mined `issues-prs.md`/`discussions.md` per-component index (`button / toolbar | 4008, 4021, 5150, 3759` — these are scope/API discussions, not a11y bugs). Treat the absence as unverified rather than confirmed-clean.

## 8. Prop-level guidance (decision-relevant props)

- [E] **`disabled` (default `false`)** — use for a genuinely inert action; native `<button disabled>` semantics apply when `nativeButton` is true (fully removed from AT and tab order) — this is the correct choice when the disabled state is not meant to be discoverable.
- [E] **`focusableWhenDisabled` (default `false`)** — "use `focusableWhenDisabled` when a button enters a loading state after being clicked, to ensure focus remains on the button... This prevents focus from being lost and maintains the tab order" (docs "Loading states" section, near-verbatim). Set this whenever a disabled state is temporary/predictable and losing focus would disorient the user — the canonical case Button was built to solve (§2).
- [E] **`nativeButton` (default `true`)** — set to `false` whenever `render` swaps the rendered tag away from `<button>` (e.g. `render={<div/>}`), so ARIA role and keyboard normalization apply correctly; leaving it mismatched against the actual rendered tag triggers a dev-mode `Base UI:` warning either direction (§6, §7).
- [E] **`render`** — the standard composition escape hatch (element or function form, per B-P9); docs' only Button-specific caveat is the link boundary (§4): don't `render={<a/>}` to fake a link-styled-as-button — style the anchor directly instead.
- [E] **Styling contract** (`ButtonDataAttributes.tsx`): exactly one data attribute, `data-disabled` — present whenever the button is disabled, regardless of `focusableWhenDisabled`. No CSS variables (Button has no dynamic measurements to expose). This is the leanest styling contract of any component in this batch, matching its single-part, single-state identity.

## 10. Pitfalls & FAQ

- **"Why is there no ButtonGroup?"** → there isn't one, by design: a non-toggle button group is just `role="group"` around plain buttons/Buttons, nothing a headless library can usefully add; reach for **ToggleGroup** (pressed-state semantics, roving focus) or **Toolbar** (mixed command bar, roving focus) instead → mui/base-ui#4008. [E]
- **"I rendered `<Button render={<a href=...>}>` to make a button-styled link and lost native link behavior"** → don't: Button enforces `role="button"` and button keyboard handling, which actively breaks anchor semantics (no more native right-click/open-in-new-tab expectations, Enter-only vs Space-that-scrolls conventions differ). Style the `<a>` directly instead → docs Usage guidelines, verbatim quoted in §4. [E]
- **"My loading-state button loses focus / drops from the tab order the moment I set `disabled`"** → set `focusableWhenDisabled` alongside `disabled`; native `disabled` unconditionally removes the element from the accessibility tree and tab order, which is the exact failure `focusableWhenDisabled` was built to prevent (§2, §8). [E]
- **"I get a console warning about `nativeButton` when composing another Base UI component into `Toolbar.Button` (e.g. `render={<Switch.Root/>}`)"** → the warning is intentional and load-bearing: it means the `nativeButton` prop (default `true`) doesn't match what's actually being rendered (a `<span role="switch">`, not a `<button>`); pass `nativeButton={false}` on the composite item, not on `Button` itself, matching whichever element the composed component actually renders → `ToolbarButton.test.tsx` "renders a switch" test reproduces the exact warning string. [E] (This is a Toolbar-composition pitfall that surfaces via the shared `useButton` contract Button also uses — cross-reference `research/c-components/toolbar/brief.md` §10.)
- **[G]** No Button-specific misuse issues surfaced in the mined discussions/issues corpus beyond the scope debate already covered in §2/§9 and the #4008 ButtonGroup thread in §4/§10 — searched `research/b-library-principles/_mining/discussions.md` per-component index (`button / toolbar | 4008, 4021, 5150, 3759`) and `issues-prs.md` decision catalog; #4021/#5150/#3759 were not individually fetched (outside the 4-call budget) and are not cited as claims here.
