# Alert Dialog — component research brief

Tier 2 (full brief, capped mining). Mined 2026-07-07 from source (`packages/react/src/alert-dialog/`, `packages/react/src/dialog/root/useRenderDialogRoot.tsx`, `useDialogRoot.ts`), docs (`docs/src/app/(docs)/react/components/alert-dialog/page.mdx`), the main test file, and 2 `gh pr view` lookups (#3237, #3127). ## 1. Identity

- **Name / subpath**: Alert Dialog — `@base-ui/react/alert-dialog`. Multi-part compound component, namespace export `AlertDialog.*`.
- **9 parts** (from `packages/react/src/alert-dialog/index.parts.ts`) — **only 2 are original** [E]:
  | Part | Source | One-liner |
  |---|---|---|
  | Root | own (`AlertDialogRoot.tsx`) | Groups all parts; calls `useRenderDialogRoot(props, 'alert-dialog')` |
  | Trigger | own (`AlertDialogTrigger.tsx`) | Thin re-export of `DialogTrigger` cast to a distinct type (`export const AlertDialogTrigger = DialogTrigger as AlertDialogTrigger`) — same runtime component, alert-dialog-flavored types only |
  | Backdrop | re-export | `DialogBackdrop as Backdrop` |
  | Portal | re-export | `DialogPortal as Portal` |
  | Viewport | re-export | `DialogViewport as Viewport` |
  | Popup | re-export | `DialogPopup as Popup` |
  | Title | re-export | `DialogTitle as Title` |
  | Description | re-export | `DialogDescription as Description` |
  | Close | re-export | `DialogClose as Close` |
  | Handle (non-element) | own (`handle.ts`) | `AlertDialogHandle` — a nominally-branded subclass of `DialogHandle` so alert-dialog and dialog handles can't be passed interchangeably (`createAlertDialogHandle()`) |
- [E] Root and Trigger are the *only* alert-dialog-authored runtime files; every other part is a verbatim import from `../dialog/*` (`index.parts.ts`). This is the identical re-export architecture the context-menu salvage documents for Menu (2-of-19 original), scaled down to a 2-of-9 ratio.
- **Status**: stable; one of the founding-era components — created together with Dialog on 2024-06-17 (PR #372) [E dialog brief §1]. No `[New]`/`[Preview]` tag.
- **Taxonomy**: overlays/popups cluster (see [`../_clusters/overlays.md`](../_clusters/overlays.md)). Purpose: a page-level overlay that **requires** a user decision before the page can continue — "A dialog that requires a user response to proceed" [E] (docs Subtitle). IA: same page-level-overlay-via-Portal-to-`<body>` shape as Dialog, but with three axes hard-coded so its every instance behaves the same way.
- **Architectural precision** [E] (`useRenderDialogRoot.tsx`): a single generic function `useRenderDialogRoot<Payload>(props, mode)` backs Dialog, AlertDialog, and Drawer, branching on `mode: 'dialog' | 'alert-dialog' | 'drawer'`. For `mode === 'alert-dialog'` it derives:
  ```ts
  const isAlertDialog = mode === 'alert-dialog';
  const modal = isAlertDialog ? true : modalProp;
  const disablePointerDismissal = isAlertDialog || disablePointerDismissalProp;
  const role: 'dialog' | 'alertdialog' = isAlertDialog ? 'alertdialog' : 'dialog';
  ```
  `AlertDialogRootProps` types this precisely by `Omit`ting `modal` and `disablePointerDismissal` from `DialogRoot.Props` entirely [E] (`AlertDialogRoot.tsx`) — the choice isn't merely defaulted, it is **removed from the public API surface**, so a consumer cannot even attempt to override it (a TypeScript-level guarantee, not just a runtime one).

## 2. Intention

- [E] **Origin**: PR #372 "[dialog] Create new component and hook" (michaldudak, 2024-06-17) created Dialog *and* Alert Dialog together, closing issue #215, itself opened against the W3C APG Dialog (Modal) pattern link [E dialog brief §2]. Alert Dialog was never a later addition bolted onto Dialog — it shipped in the same founding commit as a sibling, not a follow-up.
- [E] **Why a separate component instead of a `severity`/`type` prop on Dialog**: issue #2146 — michaldudak: "The main reason for separating them [from Dialog] is accessibility" — `role="alertdialog"` per MDN, versus Dialog's `role="dialog"`. The separation is an ARIA-semantics decision, not a styling or convenience one.
- [E] **Architectural consolidation, 15 months later**: for its first ~17 months alert-dialog had its own fully duplicated part implementations (own `AlertDialogBackdrop.tsx`, `AlertDialogPopup.tsx`, `AlertDialogClose.tsx`, etc., each with its own test file — verified via `git show 2d877d7ab --stat`, which shows the deletion of `AlertDialogBackdrop.{tsx,test.tsx}`, `AlertDialogPopup.{tsx,test.tsx,CssVars.ts,DataAttributes.ts}`, `AlertDialogClose.*`, `AlertDialogDescription.*`, `AlertDialogTitle.*`, `AlertDialogViewport.*`, `AlertDialogTrigger.*` — roughly 1,600 lines removed net of the `useRenderDialogRoot`/mode-branch addition). PR #3237 "[alert dialog] Re-export dialog parts" (michaldudak, merged 2025-11-17) collapsed this duplication down to the 2-original-part shape described in §1. [I] The consolidation is best read as the same "de-duplicate shared engines behind a mode parameter" instinct visible in Dialog's Drawer integration (`useRenderDialogRoot` already handled `'drawer'` before AlertDialog joined it) — one hardened root/dismissal/focus engine serving three public components rather than three parallel implementations.
- [E] **Explicit non-goal, confirmed by the type system**: `AlertDialogRoot.Props` omits `modal` and `disablePointerDismissal` — an alert dialog is never allowed to be non-modal or outside-dismissible. There is no escape hatch prop; the only way to get a dismissible/non-modal confirmation-style popup is to use plain Dialog. [E] (`AlertDialogRoot.tsx` `Omit<...,'modal' | 'disablePointerDismissal' | ...>`).
- [E] **Escape key is deliberately NOT part of the hard-coded lockdown**: `useDialogRoot.ts` gates outside-press dismissal on `isTopmost && !disablePointerDismissal` (line 98) but gates the Escape-key dismiss handler only on `isTopmost` (line 113, `escapeKey: isTopmost` — no `disablePointerDismissal` check at all). Confirmed by the test "calls onOpenChange with the reason for change when pressed Esc while the dialog is open" (`AlertDialogRoot.test.tsx`), which asserts `reason: escapeKey` fires normally on an open, undismissable-by-outside-press alert dialog. [I] This is a precise, easy-to-miss fact: "no light dismiss" for alert dialogs means specifically *no outside-press dismissal*, not *no keyboard dismissal at all* — Esc still closes it (the consumer decides in `onOpenChange` whether to actually let that state change go through, e.g. treating Esc as equivalent to Cancel).
- [E] **Docs gap acknowledged upstream**: issue #1687 "[dialog][docs] Document the difference(s) between `Dialog` and `AlertDialog`" (open) — the asker independently reverse-engineered the same three differences (role, modal, pointer dismissal) documented here and asked for them to be stated on the docs site; as of this writing they still aren't [E dialog brief §4]. This brief is effectively pre-filling that gap.
- [I] **Heart of the intention**: a deliberately narrowed, un-escapable variant of Dialog for the single case where the application must not let the user click past a decision — destructive confirmations, unsaved-changes warnings, required-acknowledgment prompts. Every axis Dialog exposes as configurable (modality, pointer dismissal) is fixed at its most conservative setting for AlertDialog, and the type system enforces it. [I] inferred from the `Omit` shape + #2146 rationale + APG alertdialog semantics; no single maintainer quote states the full synthesis in one place — but every piece of supporting evidence points the same direction.

## 3. When to use

- [E] "A dialog that requires a user response to proceed" (docs Subtitle) — destructive-action confirmations (discard/delete), unsaved-changes warnings, terms-of-service/required-acknowledgment prompts. The hero demo itself is exactly this: "Discard draft?" / "You can't undo this action." with Cancel/Discard actions (`demos/hero/css-modules/index.tsx`).
- [E] When the interruption must be impossible to dismiss by accident — no backdrop click, no outside click; only an explicit in-dialog choice (or Esc, which the app can still veto in `onOpenChange`) ends it.
- [E] Opening from inside a Menu item's `onClick`, controlling state externally (docs "Open from a menu" example) — the same detached-composition pattern Dialog documents, reused verbatim for AlertDialog.
- [E] As the **nested confirmation** inside an in-progress Dialog flow: the docs' own "Close confirmation" example reuses `../dialog/demos/close-confirmation` under the Alert Dialog page — "a nested confirmation dialog that opens if the text entered in the parent dialog is going to be discarded," triggered from the parent Dialog's `onOpenChange` when it receives a close request. This is the canonical dialog-vs-alert-dialog composition: a regular (dismissible) Dialog for the main flow, an AlertDialog for the one "are you sure?" decision point inside it.
- [E] Multiple/detached triggers and payload-driven content, identical to Dialog's mechanism (docs "Detached triggers"/"Multiple triggers"/"Controlled mode with multiple triggers" sections) — e.g. one confirmation alert dialog reused across a list of delete buttons via a shared `handle` and per-item `payload`.

## 4. When not to use + alternatives

Full cluster comparison: [`../_clusters/overlays.md`](../_clusters/overlays.md) (decision table row-by-row for Tooltip/Popover/Preview Card/Dialog/Alert Dialog/Drawer). Alert-dialog-specific boundary reasoning:

- **vs Dialog (the primary boundary)** — use plain Dialog for anything dismissible: settings panels, forms the user can back out of, notification centers. Use AlertDialog only for the one case Dialog can't express: a decision the user cannot escape by clicking outside. [E] The three concrete differences, precisely: `role="alertdialog"` vs `role="dialog"` (#2146, accessibility rationale); `modal` forced `true` (Dialog defaults `true` but can be relaxed) [E `useRenderDialogRoot.tsx`]; `disablePointerDismissal` forced `true` (Dialog defaults `false`) [E same file]. Reversing the decision (making a Dialog behave like an AlertDialog) is possible with `modal` + `disablePointerDismissal` set explicitly; the reverse (making an AlertDialog dismissible) is not possible — the props are removed from its type, not merely defaulted. [E] type-level `Omit`.
- **vs Toast** — passive, self-clearing notifications are Toast's job, not AlertDialog's; "Toasts are used to notify users without interrupting their workflow, and moving the focus to a toast is not desired. In scenarios when users' input is required, AlertDialog is better suited" (aarongarciah, #4533, rejecting a focus-on-open Toast option) [E, quoted verbatim in overlays cluster]. If the user's input is required, that is the signal to reach for AlertDialog over Toast.
- **Don't build a "confirm()" abstraction on top of a manager** — the same `dialogManager` rejection that applies to Dialog applies here (#2802): compose handles + `onOpenChange` reason-branching per confirmation instance rather than a single global imperative confirm function. [I] inferred by direct inheritance — AlertDialog shares Dialog's handle/store engine and #2802's rationale (dialogs "can range from simple to really complex", unlike toasts) applies without modification. [G] No AlertDialog-specific issue found asking for a `confirm()`/`alertDialogManager` helper — searched issues for "alert dialog manager"/"confirm helper", none found; the reasoning is carried over from Dialog's rejection, not independently re-litigated.
- **Don't reach for Dialog + manual "are you sure" state when AlertDialog says it for free** — [I] if the only reason a Dialog instance exists is to force a decision (no dismiss path, `role="alertdialog"`), using AlertDialog directly gets the correct ARIA semantics and removes an entire class of "user closed it by clicking outside, did we handle that?" bugs. Inferred from the type-level guarantee in §1; no maintainer statement phrases it as a "don't" directly. [G]

## 5. Anatomy & composition

```jsx
<AlertDialog.Root>                 // state owner; renders useRenderDialogRoot(props, 'alert-dialog')
  <AlertDialog.Trigger />          // optional if opened via handle/controlled state; = DialogTrigger at runtime
  <AlertDialog.Portal>             // REQUIRED wrapper for all floating parts (library-wide #1222)
    <AlertDialog.Backdrop />       // optional, presentation-only; = DialogBackdrop
    <AlertDialog.Viewport>         // optional positioning/scroll container; = DialogViewport
      <AlertDialog.Popup>          // role="alertdialog" surface; = DialogPopup
        <AlertDialog.Title />      // optional but strongly recommended (labels); = DialogTitle
        <AlertDialog.Description /> // optional (describes the consequence); = DialogDescription
        <AlertDialog.Close />      // the way OUT — required, since outside-press/backdrop click don't work
      </AlertDialog.Popup>
    </AlertDialog.Viewport>
  </AlertDialog.Portal>
</AlertDialog.Root>
```

- [E] `Portal` is mandatory, identical to Dialog (library-wide explicit-anatomy doctrine, PR #1222; `keepMounted` lives on Portal).
- [E] **`Close` carries more weight here than in Dialog**: because outside-press is always disabled and the backdrop never closes the dialog (`does not close when the backdrop is clicked` test, `AlertDialogRoot.test.tsx`), `AlertDialog.Close` (or an equivalent `onOpenChange`-driven button) is the *only* pointer-accessible way to leave the dialog besides Esc. [I] Every docs demo (hero, detached-triggers-simple, detached-triggers-controlled) renders at least one `Close` inside the Popup — no demo omits it, consistent with this being a near-mandatory part in practice even though the type system doesn't force it.
- [E] Handle pattern, identical shape to Dialog but **nominally typed apart**: `AlertDialog.createHandle()` returns an `AlertDialogHandle<Payload>` — a subclass of `DialogHandle<Payload>` carrying a private, type-only `__alertDialogBrand: never` field whose sole purpose is to make `AlertDialogHandle` and `DialogHandle` (and any other sibling handle, e.g. a future `DrawerHandle`) mutually non-interchangeable at the type level, even though they share 100% of their runtime behavior (`handle.ts` inline comment: "Nominal brand: makes this handle type distinct from `DialogHandle` and sibling handles so they can't be passed interchangeably. Type-only; has no runtime presence."). [I] This prevents a whole class of bugs where a `Dialog.Trigger handle={myAlertDialogHandle}` would silently compile and misbehave.
- [E] Root children can be a **function** `({ payload }) => …`, identical to Dialog (`PayloadChildRenderFunction`, docs "Multiple triggers" / "Detached triggers with payload").
- **Anatomy-diagram spec**: (1) Trigger button → (2) Backdrop covering page (never dismisses on click) → (3) Viewport (optional, dashed) → (4) Popup card, `role="alertdialog"` → (5) Title, (6) Description (states the consequence), (7) Close/Cancel + a destructive-action button in the card → (8) detached Trigger + handle wire to Root (optional) → (9) portal arrow from app root to `<body>`. Visually near-identical to Dialog's diagram; the callout that differs is "no outside-click arrow" on the Backdrop.

## 6. Behavior ("How it works")

Alert Dialog inherits essentially all of Dialog's behavior engine via `useRenderDialogRoot` — see [`../dialog/brief.md §6`](../dialog/brief.md) for the full mechanics (mount/unmount, animation hooks, nesting, SSR). This section states only what's fixed or alert-dialog-specific.

- [E] **State tiers**: identical three tiers to Dialog — uncontrolled (`defaultOpen`) → controlled (`open`/`onOpenChange`) → handle-imperative (`handle.open(triggerId)`/`openWithPayload`/`close`). `onOpenChange` signature and `eventDetails` (reason, cancel, trigger) are unchanged from Dialog, confirmed by test: clicking the trigger fires `reason: REASONS.triggerPress`, clicking Close fires `reason: REASONS.closePress`, pressing Esc fires `reason: REASONS.escapeKey` (`AlertDialogRoot.test.tsx`).
- [E] **`modal` is always `true`** — `data-modal="true"` is asserted directly in tests via a custom `<AlertDialogState>` test component reading `store.useState('modal')`. Full modal behavior applies unconditionally: focus trap, scroll lock, and the invisible pointer-blocking `InternalBackdrop` are always active — there is no `modal={false}`/`'trap-focus'` spectrum available (contrast with Dialog's three-value `modal` prop, [`../dialog/brief.md §8`](../dialog/brief.md)).
- [E] **`disablePointerDismissal` is always `true`** — `data-disable-pointer-dismissal="true"` similarly asserted directly in tests. Concretely: clicking the backdrop does nothing (`does not close when the backdrop is clicked` test); clicking outside the popup via any other route is likewise a no-op (same `outsidePress` gate in `useDialogRoot.ts` line 98, `isTopmost && !disablePointerDismissal`, evaluates to `false` whenever `disablePointerDismissal` is `true`).
- [E] **Escape is NOT blocked** — `escapeKey: isTopmost` in `useDialogRoot.ts` (line 113) has no `disablePointerDismissal` check; only the topmost open alert dialog responds, same as Dialog. Confirmed directly by test (Esc while `defaultOpen` fires `onOpenChange` with `reason: escapeKey`). [I] The app can still refuse the resulting close by not honoring it in `onOpenChange` (e.g. calling `eventDetails.cancel()`), but by default Esc will close an alert dialog exactly like it closes a Dialog. This is a genuinely easy misconception to correct: "no dismiss" describes pointer/outside-press behavior only.
- [E] **`role="alertdialog"`** is likewise a fixed, non-overridable derived value (`role: 'dialog' | 'alertdialog' = isAlertDialog ? 'alertdialog' : 'dialog'`), applied to the Popup element the same way Dialog applies `role="dialog"`.
- [E] **Nesting**: counts against the shared `nestedOpenDialogCount`/`--nested-dialogs`/`data-nested-dialog-open` machinery cross-type with Dialog and Drawer [E dialog brief §6, §9 — "counts cross dialog/alert-dialog types (tests)"]. This is the mechanism behind the docs' "Close confirmation" example: the nested AlertDialog and its parent Dialog are tracked by the same nesting counter, so the parent's styling hooks (`[data-nested-dialog-open]`, `--nested-dialogs`) fire correctly when the child AlertDialog opens.
- [E] **Handle timing semantics are identical to Dialog's, and separately hardened**: PR #4834 "[alert dialog] Fix handle defaults" (git log, not independently `gh`-verified body) and the store-ownership unification in #5109 ("[dialog][alert dialog][drawer] Make the Root part own the store") show AlertDialog's handle/store lifecycle receives the same maintenance attention as Dialog's, not a lagging fork. The handle's imperative methods only take effect while a Root using that handle is mounted; calls made before mount or after unmount are ignored (docs "Detached triggers" paragraph, verbatim for AlertDialog); each new mount starts from fresh state — no replay of calls made while unattached (docs, same section) — verified by test `attaches fresh root state when the root remounts after being unmounted while open`.
- [E] **Regression discipline**: PR #3127 "[alert dialog] Fix dismiss on outside press" (atomiks, merged 2025-11-05, body: "Fixes regression from dialog store refactor") shows the "backdrop click must never dismiss" invariant was treated as a correctness bug worth a dedicated fix when a shared refactor (the dialog store rework, #2834/#0dcd7ef00) threatened it — evidence this behavior is load-bearing, not incidental.
- [E] **SSR**: same unsupported-open-on-server caveat as Dialog (issue #3326, open) — inherited, not independently re-raised for AlertDialog. [I]

## 7. Accessibility contract

APG pattern: [Alert Dialog (Alert and Message Dialogs)](https://www.w3.org/WAI/ARIA/apg/patterns/alertdialog/) — the alertdialog-specific APG pattern, distinct from the plain Dialog (Modal) pattern Dialog cites. [I] inferred from `role="alertdialog"` being the component's entire raison d'être per #2146 (MDN alertdialog citation); the APG page itself is the authoritative reference for the role Base UI assigns — not separately quoted in-repo, so treat the link itself as [I]-supported rather than a direct maintainer citation.

**Keyboard** (same `FloatingFocusManager` engine as Dialog; behavior fixed rather than configurable where noted):

| Key | Behavior |
| --- | --- |
| `Tab` / `Shift+Tab` | Moves through tabbables inside the popup; **always loops** (trapped) — `modal` can never be `false`/`'trap-focus'`-relaxed for AlertDialog, so focus can never leave the popup via Tab |
| `Esc` | Closes the topmost open alert dialog (reason `escape-key`) — **not** blocked by the pointer-dismissal lockdown; propagation to parent popups stopped by default, same as Dialog |
| `Enter`/`Space` on Trigger/Close | Native button activation (real `<button>`s; `Trigger` is literally `DialogTrigger` at runtime) |
| Composite keys (arrows, etc.) inside popup | Stopped from propagating to wrapping composite widgets, same `COMPOSITE_KEYS` handler as Dialog (inherited `DialogPopup`) |
| Click/tap on Backdrop | **No-op** — the one keyboard-table row that most distinguishes AlertDialog from Dialog: there is no pointer equivalent of Esc for backdrop dismissal |

- [E] **ARIA wiring**, identical mechanism to Dialog, values differ only in role: Popup `role="alertdialog"`, `aria-labelledby` ← `AlertDialog.Title` id, `aria-describedby` ← `AlertDialog.Description` id (both auto-generated and auto-wired; test "ARIA attributes" confirms both linkages directly against a rendered Title/Description pair). Trigger: `aria-haspopup="dialog"` (verified in `popupConformanceTests` config: `expectedAriaHasPopupValue: 'dialog'` — note this stays `"dialog"`, not `"alertdialog"`; `aria-haspopup` has no `alertdialog` token in the ARIA spec), `aria-expanded`, `aria-controls` (per-trigger accurate with multiple/detached triggers — test "synchronizes ARIA attributes on the active trigger"). Backdrop/Viewport: `role="presentation"`.
- [E] **No `aria-modal`** — same deliberate omission as Dialog, for the same reason (outside elements are marked `aria-hidden`/inert instead, with a live-region exception; see [`../dialog/brief.md §7`](../dialog/brief.md), #4843). [I] Inherited, not independently re-litigated for AlertDialog.
- [E] **Modality test coverage**: `makes other interactive elements on the page inert when a modal dialog is open` (`AlertDialogRoot.test.tsx`, `describe.skipIf(isJSDOM)`) asserts the presentation/backdrop element renders — confirming the same inert-marking mechanism as Dialog applies unconditionally here (since `modal` can't be turned off).
- [E] **Touch screen readers**: because `modal` is always `true`, the JSDoc guidance on Dialog's `modal` prop — "render `<Dialog.Close>` inside `<Dialog.Popup>` so touch screen readers can escape the popup" — applies to AlertDialog **unconditionally**, with no opt-out. [I] This makes `AlertDialog.Close` closer to a hard requirement than Dialog's equivalent guidance, which only applies when a consumer has chosen `modal={true}`.
- **Open a11y issues (honesty list)**: [E] all of Dialog's open a11y issues are shared by AlertDialog since it's the same focus-management/ARIA-hiding engine — #4678 (background `aria-hidden` doesn't remove focusable elements from tab order), #4843 (`aria-modal` feature request), #4583 (Esc closes multiple sibling dialogs when 3+ open simultaneously — directly relevant here since nested AlertDialog-in-Dialog is a documented pattern), #3905 (CloseWatcher/Android back button) [E dialog brief §7]. [G] Searched for AlertDialog-specific a11y issues (`label:"component: alert dialog" label:accessibility`, "alertdialog role screen reader") — none found beyond the shared Dialog issues above; no evidence of independent AlertDialog a11y bug reports.
- [I] **Screen reader announcement of "alertdialog" vs "dialog"** differs by AT/browser combination per general ARIA alertdialog semantics (interrupting announcement expected), but no Base UI-specific verification (VoiceOver/NVDA/JAWS testing thread) was found for AlertDialog specifically — the `#4122`→`#4142` accessible-name fix cited in Dialog's principles evidence (B-P28) was verified for the general dialog role, not independently re-verified for `alertdialog`. [G]

## 8. Prop-level guidance (decision-relevant props)

Alert Dialog's prop surface is **Dialog's minus `modal` and `disablePointerDismissal`** (removed, not merely defaulted) plus alert-dialog-typed `handle`/`actionsRef`/`onOpenChange`. Every remaining prop behaves exactly as documented for Dialog ([`../dialog/brief.md §8`](../dialog/brief.md)); this section covers only what's distinctive.

- [E] **No `modal` prop** — cannot be set at all (`Omit<DialogRoot.Props<Payload>, 'modal' | ...>`). If a scenario needs `modal={false}` or `'trap-focus'`, it needs plain Dialog, not AlertDialog. This is the single most decision-relevant fact about AlertDialog's prop surface: the absence of a prop *is* the guidance.
- [E] **No `disablePointerDismissal` prop** — same removal; it is permanently forced `true`. There is no way to make an AlertDialog instance dismissible by clicking outside; that requirement signals "use Dialog instead."
- [E] **`onOpenChange(open, eventDetails)`** — same `eventDetails.reason`/`.cancel()`/`.trigger` contract as Dialog; the reason vocabulary that matters most here is distinguishing `escape-key` (the one dismissal path still open) from `close-press`/`trigger-press`. [I] A confirmation dialog that wants to treat Esc identically to clicking Cancel can branch on `reason === 'escape-key'` inside `onOpenChange` — no example of this exact pattern was found in docs/tests, so this is an inferred recipe, not a cited one. [G]
- [E] **`handle` + `AlertDialog.createHandle<Payload>()`** — nominally typed apart from `Dialog.createHandle()`/`DialogHandle` (see §5); otherwise identical semantics (per-trigger `payload`, `open(triggerId)`/`openWithPayload(payload)`/`close()`, ignored calls outside a mounted Root's lifetime). Use for a single confirmation dialog instance reused across many trigger points (e.g. delete buttons throughout a list) — the payload carries which row is being confirmed.
- [E] **`actionsRef`** — `{ unmount, close }`, identical shape and purpose to Dialog's (`AlertDialogRoot.Props` retains it, typed against `AlertDialogRoot.Actions = DialogRoot.Actions`). `unmount()` after an externally controlled exit animation; `close()` for imperative close without a handle. Test coverage confirms both (`prop: actionsRef` describe block).
- [E] **`Popup.initialFocus`/`finalFocus`, `Portal.keepMounted`/`container`, `Backdrop.forceRender`** — all inherited unchanged from Dialog; not independently re-documented or re-tested for AlertDialog beyond generic popup conformance. [I]
- [E] **Styling contract** — identical data attributes/CSS vars to Dialog since the Popup/Backdrop/Viewport/Trigger/Close parts are the *same components*: `data-open`/`data-closed`/`data-starting-style`/`data-ending-style`, `data-nested`/`data-nested-dialog-open`, `--nested-dialogs` CSS var, Trigger `data-popup-open`/`data-disabled`, Close `data-disabled` [E `*DataAttributes.ts` files under `dialog/`, re-exported]. No AlertDialog-specific data attribute exists beyond what Dialog already defines — styling an AlertDialog popup and a Dialog popup uses the same selector vocabulary.

## 10. Pitfalls & FAQ

- **"Why doesn't clicking the backdrop close my alert dialog?"** → by design — `disablePointerDismissal` is permanently `true` and cannot be overridden; use `AlertDialog.Close` (or a controlled `onOpenChange` call from your own button) as the exit path. [E] `does not close when the backdrop is clicked` test; `useDialogRoot.ts` line 98.
- **"Esc still closes it even though I can't click outside — is that a bug?"** → no, it's the documented (if only [I]-inferred from source, since docs don't state it explicitly per #1687) split: outside-press is blocked, Esc is not. If Esc-closing an unconfirmed destructive action is a concern, branch on `eventDetails.reason === 'escape-key'` inside `onOpenChange` and decide whether to actually apply the close. [E] source (`escapeKey: isTopmost`, no `disablePointerDismissal` gate) + test confirming Esc fires normally. [I] the mitigation recipe itself.
- **"Can I make my AlertDialog dismissible by clicking outside for one particular case?"** → no — the props are `Omit`ted from the type, this is enforced at compile time, not just a default. If you need that behavior, the answer is "you want Dialog, not AlertDialog" [E type-level `Omit` in `AlertDialogRoot.tsx`] — this is itself the "when not to use" signal from §4.
- **"Why is there both a `Dialog.createHandle()` and an `AlertDialog.createHandle()` — can I mix them?"** → no, by design: `AlertDialogHandle` carries a private nominal brand (`__alertDialogBrand: never`) specifically so a `DialogHandle` can't be passed to `<AlertDialog.Trigger handle={...}>` or vice versa, even though they behave identically at runtime. [E] `handle.ts` inline comment.
- **Docs gap on record**: issue #1687 (open) — a user asked for the Dialog-vs-AlertDialog differences to be documented; as of this brief they still aren't stated on the docs site itself, only discoverable from source/tests (which is exactly what this brief assembles). [E] dialog brief §4, confirmed still-open via the shared citation.
- **No AlertDialog-specific interop/z-index/stacking pitfalls found** beyond what's already documented for Dialog (nested-dialog counting, z-index-on-Positioner-not-Popup, Radix/vaul `pointer-events:none` interop) — since AlertDialog shares Dialog's Backdrop/Portal/Positioner-equivalent (Viewport) implementations verbatim, all of Dialog's pitfalls transfer unchanged [E dialog brief §10]. [G] Searched issues for "alert dialog z-index"/"alert dialog stacking" specifically — no independent reports found, consistent with the shared-implementation architecture producing shared (not duplicated) bug surface.
