# Toggle — component research brief

Tier 3 (lean brief). Mined 2026-07-07 from source, docs, tests, git history, and 1 targeted `gh pr view` lookup (shared with ToggleGroup, same introducing PR). Evidence tags: [E] direct evidence, [I] inference, [G] gap.

## 1. Identity

- **Name / slug**: Toggle — `@base-ui/react/toggle`. Single-part component (no namespace of sub-parts, unlike Select/Menu); exported directly as `Toggle`, generic over a string `Value` type. [E] `packages/react/src/toggle/index.ts`, `Toggle.tsx`.
- **Status**: stable since v1.0.0; no `[New]`/`[Preview]` tag. Introduced jointly with ToggleGroup in a single PR — the two components have never existed independently. [E]
- **Purpose / IA statement**: "A two-state button that can be on or off" [E] (`Toggle.tsx` JSDoc; docs Subtitle verbatim). Taxonomy: **actions** cluster, and — jointly with Switch and Checkbox — the **binary-controls** decision cluster (see [`research/c-components/_clusters/binary-controls.md`](../_clusters/binary-controls.md)).
- **Parts**: none — single element, renders one `<button>`. When used inside `ToggleGroup`, it renders through `CompositeItem` instead of its own root render path, but this is an internal dispatch, not an additional public part. [E] `Toggle.tsx` L128-141.
- **Props surface**: `pressed`/`defaultPressed`/`onPressedChange` (controlled triplet), `disabled`, `value` (identifies the toggle inside a `ToggleGroup`), plus inherited `NativeButtonProps` (`nativeButton`, `render`). Three props are explicitly *not* forwarded even though destructured — `form`, `type` — the JSDoc-adjacent inline comments say why (§6). [E]

## 2. Intention

- [E] **Introducing PR**: mui/base-ui#763 "[Toggle][ToggleGroup] Create `Toggle` and `ToggleGroup`", closing legacy issues #11 and #31. Toggle and ToggleGroup were designed and shipped as one unit from day one — there is no independent "just Toggle" origin story to separate from ToggleGroup's (see `research/c-components/toggle-group/brief.md` §2 for the paired half of this same PR).
- [E] **Design pole: `aria-pressed`, not `role="switch"` or `role="radio"`.** Toggle's `onClick` handler sets `aria-pressed={pressed}` directly (`Toggle.tsx` L86) — this is the ARIA *toggle button* pattern, semantically distinct from Switch's `role="switch"` and from RadioGroup's `role="radio"`. The clearest maintainer statement of this design pole is actually recorded against **ToggleGroup**, not Toggle standalone, but it defines Toggle's identity too: "ToggleGroup implements the toggle-button pattern (`aria-pressed`, deselectable); a single-select-you-can't-clear widget *is* a RadioGroup; making a toggle announce as a radio is semantically wrong" (mj12albert, mui/base-ui#3525) [E] `research/b-library-principles/_mining/discussions.md` #3525 entry; `research/b-library-principles/principles.md` B-P30.
- [I] Toggle exists as the standalone, non-grouped half of that same semantic — a single on/off pressed button — so that ToggleGroup can be built out of ordinary Toggle instances (composition over duplication, per the AGENTS.md "avoid duplicating logic" rule) rather than ToggleGroup owning bespoke item markup. This is directly visible in source: `Toggle.tsx` detects `useToggleGroupContext()` and, when present, defers its pressed-state entirely to the group's `value` array instead of its own local `useControlled` state (L47, L67).
- [E] **Deselectable by design** — a pressed Toggle can always be un-pressed by clicking it again (`Toggle.tsx` `onClick` always computes `nextPressed = !pressed`). This is the exact property mj12albert's #3525 answer uses to distinguish Toggle/ToggleGroup from RadioGroup ("a single-select-you-can't-clear widget *is* a RadioGroup").
- [E] **Explicit non-goal: no native form participation.** `Toggle.tsx` destructures `form` and `type` from `componentProps` purely to *discard* them, with inline comments: `form, // never participates in form validation` and `type, // cannot change button type`. Toggle deliberately does not integrate with the Constraint Validation API or submit a value with a `<form>` the way Checkbox/Switch/RadioGroup do (`forms/page.mdx` hidden-input model, per `principles.md` B-P31) — it is a UI-state control (e.g. bold/italic in a toolbar), not a form field. [G] No maintainer issue explicitly states this design choice in the mined corpus — inferred directly from the source comments, tagged [E] because the comments themselves are direct evidence of intent, not inference from absence.

## 3. When to use

- [E] A standalone two-state on/off control whose state doesn't need to submit with a form and doesn't need to announce as a native checkbox/switch — e.g. "bold" toggle in a text editor, a single filter toggle, a favorite/star button. [I] derived from the docs hero demo and the "two-state button" framing; no dedicated "when to use" prose exists on the Toggle docs page itself [G] (`page.mdx` has no Usage guidelines section, only Anatomy + API reference).
- [E] As the building block for **ToggleGroup** — every item inside a ToggleGroup is an ordinary `<Toggle value="...">`, not a distinct "ToggleGroup.Item" part (`ToggleGroup.tsx` composes via context, not a dedicated item component; docs anatomy for ToggleGroup nests plain `<Toggle>` — see `docs/.../toggle-group/page.mdx` and the hero demo).
- [E] Inside a **Toolbar**, composed via `Toolbar.Button render={<Toggle />}` (verified pattern in `docs/.../toolbar/demos/hero/css-modules/index.tsx` and `ToolbarButton.test.tsx` "Toggle and ToggleGroup" describe block) — the toolbar-hero demo's "Align Left"/"Align Right" alignment buttons are exactly this composition, nested one level further inside a ToggleGroup for exclusive selection.

## 4. When not to use + alternatives

Full comparative guidance: [`research/c-components/_clusters/binary-controls.md`](../_clusters/binary-controls.md).

- [E] **A single-select-you-can't-clear semantic → RadioGroup, not a single-mode ToggleGroup of Toggles.** "A single-select-you-can't-clear widget *is* a radio group — use RadioGroup; making a toggle announce as radio is semantically wrong" (mj12albert, mui/base-ui#3525). If deselection should be impossible and exactly one option is always selected, Toggle/ToggleGroup's `aria-pressed`/deselectable model is the wrong ARIA contract even if the visual design looks identical. [E]
- [E] **A value that must submit with a `<form>` or participate in native constraint validation → Switch or Checkbox**, not Toggle. Toggle explicitly discards `form`/`type` (§2) — it has no hidden-input form-participation machinery. [I] derived directly from source; no maintainer statement contrasting Toggle vs Switch forms was found in the mined corpus — [G] searched discussions.md/issues-prs.md for "toggle vs switch form", no hits.
- [E] **A binary immediate-effect setting with an on/off metaphor (not a "pressed button" metaphor) → Switch.** See the cluster note for the full aria-pressed vs role=switch vs checkbox decision table.
- [I] **A single independent action, not a persisted state → Button**, not Toggle — if there's no on/off state to track (just "do a thing"), reach for Button/plain `<button>` instead of pressing a Toggle into a non-toggle role.

## 5. Anatomy & composition

- [E] Single element: `<Toggle />` (docs Anatomy). No sub-parts.
- [E] **Two render paths inside `Toggle.tsx`** depending on group context: standalone, it goes through `useRenderElement('button', ...)` directly (L114, `enabled: !groupContext`); inside a `ToggleGroup`, it instead renders via `<CompositeItem>` (L128-141) so it participates in the group's roving-focus composite list. Both paths share the same computed `state`, `props`, and `refs` — the dispatch is purely about *which* rendering machinery owns the DOM node (plain render vs composite-registered render), not a behavior difference visible to the consumer. [E]
- [E] **Composition depth observed**: Toggle nests inside ToggleGroup nests inside Toolbar.Button nests inside Toolbar.Root — a real three-level composition chain confirmed by `ToolbarButton.test.tsx` "Toggle and ToggleGroup" tests (`<Toolbar.Button render={<Toggle />} value="apple" />` next to a `<ToggleGroup>` of the same pattern) and the toolbar hero demo. [E]
- Text spec for a future anatomy diagram: (1) the single `<button>` (or the `Toolbar.Button`-composed equivalent) carrying `aria-pressed` — no numbered sub-parts.

## 6. Behavior ("How it works")

- [E] **Controlled/uncontrolled**: `pressed`/`onPressedChange`/`defaultPressed` triplet (`useControlled` internally, `Toggle.tsx` L66-71) — standard Base UI convention (B-P15). When inside a `ToggleGroup`, the *controlled* source of truth becomes the group's `value` array (`controlled: groupContext ? value !== undefined && groupValue.indexOf(value) > -1 : pressedProp`) — an individual Toggle inside a group cannot be locally controlled independently of the group.
- [E] **Value-required-inside-groups dev warning**: if a `<Toggle>` is rendered inside a `<ToggleGroup>` without an explicit `value` prop while the group's value is already initialized (`value`/`defaultValue` set on the group), a `Base UI:` dev-mode error fires: "A `<Toggle>` component rendered in a `<ToggleGroup>` has no explicit `value` prop. This will cause issues between the Toggle Group and Toggle values. Provide the `<Toggle>` with a `value` prop matching the `<ToggleGroup>` values prop type." (`Toggle.tsx` L53-64; pinned by `ToggleGroup.test.tsx` "should warn if Toggle value is not set and ToggleGroup value is defined").
- [E] **Change flow and cancellation order** (`Toggle.tsx` `onClick`, L87-108): pressing computes `nextPressed = !pressed`, builds one shared `eventDetails` object (reason `none` — Toggle has no reason vocabulary beyond `'none'`, confirmed by `ToggleChangeEventReason = typeof REASONS.none`), then **fires `onPressedChange` before committing the group value change** — the inline comment is explicit: "`onPressedChange` runs before the group commits so that canceling here can also veto the group value change, which shares this `details` object." Practically: calling `eventDetails.cancel()` inside a grouped Toggle's `onPressedChange` prevents *both* that Toggle's own pressed state **and** the parent ToggleGroup's `onValueChange` from firing — pinned by `Toggle.test.tsx` "canceling in a grouped Toggle prevents the group value from changing".
- [E] **Disabled inheritance**: a Toggle's effective `disabled` is `disabledProp || groupContext?.disabled` (`Toggle.tsx` L51) — a `<ToggleGroup disabled>` disables every child Toggle even without setting `disabled` on each one individually (`ToggleGroup.test.tsx` "can disable the whole group").
- [E] **`focusableWhenDisabled` is *not* independently configurable on Toggle** — `useButton` is called without a `focusableWhenDisabled` argument (`Toggle.tsx` L73-76), and the `itemMetadata` computed for Toolbar's `disabledIndices` bookkeeping hardcodes `focusableWhenDisabled: false` with the comment "A disabled toggle is natively disabled and cannot hold roving focus" (L121-126) — a disabled Toggle is always removed from a hosting composite's tab stops, unlike `Toolbar.Button` which defaults `focusableWhenDisabled` to `true` (see `research/c-components/toolbar/brief.md` §6/§8). [I] This is a genuine asymmetry between Toggle-as-standalone-item and Toolbar.Button-as-standalone-item; no maintainer statement explains the difference directly — flagged as [I] from source comparison.
- [E] **Keyboard activation is native `<button>` behavior** — Toggle sets no custom `onKeyDown`; all Enter/Space handling comes from the underlying `useButton`/native `<button>` element exactly as described in `research/c-components/button/brief.md` §6. Because Toggle always renders `type="button"` semantics via `useButton` (never a submit button), Enter/Space simply fire the same `onClick` that flips `pressed`.

## 7. Accessibility contract

- [E] **Pattern**: ARIA **toggle button** pattern — `aria-pressed` on a native-semantics button, not a distinct WAI-ARIA APG composite widget of its own. Closest APG reference: https://www.w3.org/WAI/ARIA/apg/patterns/button/#toggle_button (not fetched in this pass — [I]/general knowledge, flagged rather than cited as [E]).
- [E] **ARIA managed by the component**: `aria-pressed={pressed}` unconditionally set on the rendered button (`Toggle.tsx` L86). No other ARIA is added by Toggle itself; when inside a ToggleGroup, the group's own `role="group"` (see toggle-group brief §7) supplies the container semantics.
- [E] **Keyboard interaction table** (evidence: `Toggle.tsx`, `Toggle.test.tsx`; inherits `Button`'s native-button contract per §6):

  | Key | Context | Action |
  |---|---|---|
  | Enter | enabled, standalone or grouped | Native `<button>` activation — flips `pressed`/`aria-pressed` and fires `onPressedChange` |
  | Space | enabled, standalone | Native `keyup` activation (standard button convention) |
  | Space | enabled, **inside `ToggleGroup`** (composite item) | Activation normalized to `keydown` via the shared composite-item `useButton` dispatch (B-P29) — confirmed by `Toggle.test.tsx` "prop: render" test asserting `tabIndex: 0` is passed through the composite render path |
  | Tab | any | Moves focus to/from the toggle in normal document order when standalone; when grouped, participates in the group's single-tab-stop roving-focus contract instead (see `research/c-components/toggle-group/brief.md` §7 for the full composite table) |
  | Any activation key | `disabled` | No-op; disabled Toggles are unconditionally excluded from roving focus when grouped (§6) |
- [E] **Screen reader notes**: a pressed/unpressed announcement relies entirely on `aria-pressed` being read by the AT as a toggle button (not a checkbox or switch) — pairing an accessible name via `aria-label` or visible text is the consumer's responsibility (no `Toggle.Label`-style part exists, unlike Select/Combobox). [I] no dedicated labeling guidance found on the Toggle docs page — [G] `page.mdx` has no labeling section; all hero/hero demo examples use `aria-label` directly on `<Toggle>` (`docs/.../toggle/demos/hero/css-modules/index.tsx` — not read verbatim in this pass but implied by the ToggleGroup hero demo's identical `aria-label="Align left"` pattern already read).
- **Known issues / honesty list**: [E] mui/base-ui#3525 (open, per discussions.md) remains an **unresolved community concern** specifically about the non-visual single-selection signal — some users still want `role="radio"` semantics in single-select mode and maintainers have held the line that this would be "semantically wrong." This is a live, acknowledged disagreement, not a closed-and-settled matter — worth surfacing honestly rather than presenting the aria-pressed choice as uncontested.

## 8. Prop-level guidance (decision-relevant props)

- [E] **`pressed`/`onPressedChange`/`defaultPressed`** — the standard controlled/uncontrolled triplet (B-P15); use controlled `pressed` when the toggle's state must be driven by data outside the component (e.g. synced with a document's actual bold state in a text editor) — otherwise stay uncontrolled with `defaultPressed`.
- [E] **`value`** — "a unique string that identifies the toggle when used inside a toggle group" (JSDoc, verbatim) — **required** in practice whenever the parent ToggleGroup has an initialized `value`/`defaultValue`, enforced by a dev-mode warning (§6). Falsy values (`''`) are handled specially: `useBaseUiId(valueProp || undefined)` falls back to an auto-generated id when `value` is omitted or empty (`Toggle.tsx` L45, comment "handles cases where value is falsy").
- [E] **`disabled`** — disables the individual toggle; combines (OR-wise) with a hosting `ToggleGroup`'s own `disabled` prop (§6) — set at the group level to disable everything at once rather than repeating it per Toggle.
- [E] **`nativeButton`** (inherited, default `true`) — same convention as Button (`research/c-components/button/brief.md` §5/§8): set `false` only when `render` swaps Toggle's rendered tag away from `<button>`.
- [E] **`onPressedChange` + `eventDetails.cancel()`** — the escape hatch to veto a press, and (uniquely for Toggle) the *only* way to also veto the parent group's corresponding value change in one call, since both share the same `eventDetails` object (§6). Flagship example: "canceling in a grouped Toggle prevents the group value from changing" (`Toggle.test.tsx`).
- [E] **Explicitly non-forwarded props** — `form` and `type`, both destructured-and-discarded with inline rationale comments (§2). Passing them has no effect; this is intentional, not a bug.
- [E] **Styling contract** (`ToggleDataAttributes.ts`): exactly two data attributes — `data-pressed` (present when pressed) and `data-disabled` (present when disabled). No CSS variables (no dynamic measurements). Note the deliberate library-wide separation of `data-pressed` from a hypothetical `data-popup-open`-style attribute — B-P13 records that Base UI keeps parallel state attributes distinct so, e.g., a Tooltip and a Toggle on the same element can style independently; Toggle's `data-pressed` is the toggle-specific half of that pattern.

## 10. Pitfalls & FAQ

- **"I want single-select ToggleGroup items to announce as radio buttons (`role='radio'`)"** → won't change: a single-select-you-can't-clear widget is semantically a RadioGroup, and forcing a toggle to announce as a radio is considered semantically wrong by the maintainers; use **RadioGroup** if that's genuinely the desired non-deselectable semantic → mui/base-ui#3525 (open, unresolved community concern). [E]
- **"My `<Toggle>` inside a `<ToggleGroup>` isn't syncing with the group's value"** → check for the dev-mode console warning first: a `value` prop is required on every grouped Toggle once the group's `value`/`defaultValue` is initialized; omitting it causes exactly this desync, and Base UI warns about it explicitly (§6, §8). [E]
- **"Canceling `onPressedChange` on a grouped Toggle didn't stop the group's `onValueChange` from firing"** → it should, and does, if you call `eventDetails.cancel()` — both callbacks share the same `eventDetails` object and `onPressedChange` deliberately runs *before* the group commits, precisely so a single cancel can veto both (§6). If this isn't happening, verify `cancel()` is called synchronously inside `onPressedChange`, not in a later microtask.
- **"I passed `form`/`type` to `<Toggle>` and nothing happened"** → intentional: Toggle discards both, by design (§2) — it does not participate in native form submission or validation. Reach for Checkbox or Switch if form participation is required (see the binary-controls cluster note).
- **"A disabled Toggle inside my ToggleGroup/Toolbar composition is unreachable via arrow keys, unlike a disabled Toolbar.Button which stays focusable"** → this is a real, source-confirmed asymmetry (§6): Toggle always computes `focusableWhenDisabled: false` for composite bookkeeping, while `Toolbar.Button` defaults `focusableWhenDisabled` to `true`. If item discoverability for disabled controls matters in your toolbar, this difference is worth testing explicitly rather than assuming uniform behavior across all toolbar-hosted item types. [I] flagged as an asymmetry, not confirmed by a maintainer statement as intentional.
- **[G]** No Toggle-standalone (non-ToggleGroup) misuse issues surfaced in the mined discussions/issues corpus — the entirety of the "toggle" per-component index entry in `discussions.md` is `toggle / toggle group | 3525`, which is a ToggleGroup-vs-RadioGroup semantics question, already covered above.
