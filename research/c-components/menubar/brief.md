# Menubar — component research brief

Tier 2 (full brief, capped mining). Mined 2026-07-07, built from [`_mining-salvage.md`](./_mining-salvage.md) (full source/docs/tests/history archaeology, preserved verbatim above) plus 1 `gh pr view` lookup (#1684, the introducing PR — the tracking-issue number #1407 the salvage only hypothesized is now confirmed) and a second lookup (#4922, vertical-orientation fix, spent because its full body contained more than the salvage anticipated). Evidence tags: [E] direct evidence, [I] inference, [G] gap.

## 1. Identity

- **Name / subpath**: Menubar — `@base-ui/react/menubar`. **Single-part component — no namespace, no `index.parts.ts`, no `Menubar.Root`/`Menubar.*` anything.** [E] direct source confirmation: `Menubar.tsx` exports only `{ Menubar }`; the file is the entire component (170 lines). This is architecturally unlike every other component in this brief batch — Menubar itself is a single host container; the actual menu content is composed from ordinary `Menu.Root` instances placed inside it.
- **One-liner** (JSDoc): "The container for menus." [E] `Menubar.tsx`.
- **Status**: stable; no `[New]`/`[Preview]` tag. Introduced by PR #1684, "[menubar] Create the Menubar component" (michaldudak, merged 2025-05-09T20:29:13Z), **confirmed via `gh pr view 1684`** to close **#1407** — resolving the salvage's flagged uncertainty ("brief hypothesized #1407 — CONFIRM number first").
- **Taxonomy**: menus cluster (see [`../_clusters/menus.md`](../_clusters/menus.md)). Purpose: a persistent, `role="menubar"` horizontal (or vertical) bar hosting several `Menu.Root` instances as roving-focus items — the desktop-application File/Edit/View mental model. IA: not a menu itself; a **composite container** that changes how the `Menu.Trigger`s placed inside it behave (roving `menuitem` semantics, sibling-switch-on-hover, shared close cascade) rather than introducing new menu-popup capability of its own.
- **Cross-component role, precisely stated**: Menu detects a Menubar ancestor via `useMenubarContext(true)` → branches on `parent.type === 'menubar'` across at least 5 Menu source files (`MenuTrigger.tsx`, `MenuRoot.tsx`, `MenuPositioner.tsx`, `MenuPopup.tsx`, `MenuStore.ts`) [E salvage, cross-verified by direct read of `MenuTrigger.tsx`]. Menubar owns zero popup rendering logic itself — everything the user perceives as "menubar behavior" (roving arrows, hover-switch, sibling close) is Menu code conditionally activated by menubar parentage.
- **Own props** (`MenubarProps`, direct source read): `orientation` (default `'horizontal'`), `loopFocus` (default `true`), `modal` (default `true`), `disabled` (default `false`). Renders a single `<div role="menubar">` via `CompositeRoot`. That is the component's entire public surface — four props and a container element.

## 2. Intention

- [E] **Tracked as a Menu deliverable from the start, confirmed**: `gh pr view 1684` — title "[menubar] Create the Menubar component", body: "Closes #1407," with an explicit to-do checklist covering modality, tests, docs and demos, vertical orientation, "implementation using `CompositeRoot`", `data-instant`, and "Use the released version of Floating UI." Every to-do item is checked in the merged PR — the PR shipped complete, not as a partial first cut. [E] This confirms the salvage's characterization precisely: "Menubar was never a bolt-on; 555-line test file from day one."
- [E] **The core design decision, confirmed by direct source inspection rather than PR prose** (the PR body itself does not narrate the "why context-detection instead of a parts namespace" reasoning — [G] no discussion of this specific architectural choice was found in the PR body): Menubar is a thin host. It never re-implements Menu's popup, item, submenu, or positioning logic. Instead, `Menu.Root` components check `useMenubarContext(true)` and change behavior when found:
  - `MenuTrigger.tsx` line 111: `const isInMenubar = parent.type === 'menubar';` — the single boolean that flips a Menu's Trigger from a standalone button into a roving `menuitem`.
  - Line 254 (direct read): `isInMenubar ? { role: 'menuitem' } : {}` — the literal role conversion.
  - Line 268: `if (isInMenubar) { return (…` — the trigger renders through a **different code path** (a `CompositeItem`-integrated render) than the standalone-button path, confirmed by direct source read (`useRenderElement('button', componentProps, { enabled: !isInMenubar, … })` on line 260-266 — the standalone button-rendering path is explicitly *disabled* when inside a menubar).
  [I] The design bet: reuse literally all of Menu's popup engine (submenus, checkbox/radio items, groups, positioning, dismissal, animation) for free, and add only the container/roving-focus/sibling-coordination layer as new code. The alternative (a `Menubar.Item`/`Menubar.Popup` parts namespace duplicating Menu's ~20 parts) would have meant maintaining two popup implementations in lockstep — the same "thin wrapper around Menu's engine" instinct visible in Context Menu (`../context-menu/brief.md` §1-2) and Alert Dialog (`../alert-dialog/brief.md` §1), scaled up to "the wrapped thing is a whole different component (Menu), not just re-exported parts."
- [E] **Cross-wiring is deliberately narrow and explicit, not a general "ambient container" mechanism**: the salvage documents `parentOrientation` handed to `useListNavigation`, `instantType='group'` gating for reasons `triggerFocus`/`focusOut`/`triggerHover`/`listNavigation`/`siblingOpen` (sibling-switch skips the open/close animation — visually "instant" menu-hopping, matching the native OS menubar feel where switching from File to Edit doesn't visibly fade/scale), and a `REASONS.siblingOpen` close reason defined once in `internals/reason-parts.ts:35` and shared by Context Menu's `[context menu][menubar]`-scoped commit (#2768, `findRootOwnerId` utility) [E salvage]. This narrow, named-reason approach — rather than a loose "is my parent special" boolean — is consistent with the library's `ChangeEventDetails`/reason-vocabulary convention (B-P16 in principles.md) applied to internal coordination, not just public API.
- [E] **Toolbar contrast is explicit and deliberately separate machinery** (salvage, cross-verified against the menus cluster note): Menubar (`role=menubar`, hosts `Menu.Root`s, converts triggers to roving `menuitem`s) and Toolbar (`ToolbarRoot.tsx` `role='toolbar'`, hosts arbitrary controls, may contain a menu-*button* but never becomes "a bar of persistent menus") are kept as **independent mechanisms in code** — `isInMenubar` and `insideToolbar` are two separate context checks in `MenuPopup.tsx`, not a shared "am I in some kind of bar" flag. [I] This is deliberate ARIA-pattern-separation: Menubar implements the APG Menubar pattern; a Toolbar containing a menu button implements the APG Toolbar pattern with an embedded Menu Button — they are different accessibility contracts even when visually similar, and the code keeps them from bleeding into each other.
- [I] **Heart of the intention**: give applications the desktop File/Edit/View menu-bar affordance (persistent triggers, roving single-tab-stop keyboard navigation, hover-switch-between-open-menus once the user has committed to browsing) entirely by composing ordinary `Menu.Root`s inside a coordinating container, rather than by inventing a parallel menu-bar-specific popup API. Inferred from the architecture (§1-2) and the completeness of the introducing PR; no single maintainer quote states this synthesis directly, but the to-do-checklist completeness and the narrow, named cross-wiring are strong converging evidence.

## 3. When to use

- [E] "A menu bar providing commands and options for your application" (docs Subtitle) — application chrome: File/Edit/View-style command bars for desktop-style web apps (the docs' own hero demo, confirmed by direct read: File[New, Open, Save, Export▸PDF/PNG/SVG, —, Print] / Edit[Cut, Copy, Paste] / View[Zoom In, Zoom Out, Layout▸Single Page/Two Pages/Continuous, —, Full Screen] / Help[disabled]).
- [E] When several *related, persistent* menus should behave as one coordinated unit — hovering between open menus switches without re-clicking (§6), a shared modal/disabled/orientation policy, and one roving Tab stop for the whole bar rather than one per menu trigger. [I] "several menus form one persistent bar" is the menus-cluster's own framing (`../_clusters/menus.md`: "Use Menubar when several menus form one persistent bar; a lone Menubar with one menu is just a Menu").
- [E] Vertical orientation for sidebar-style command bars — `orientation="vertical"` switches roving-focus axis and (since #4922) the default submenu-open side from `'bottom'` to `'inline-end'` so vertical menubars don't awkwardly overlay their own trigger column.
- [E] Building a keyboard-navigable command surface that should read to assistive technology as "menubar," not "toolbar" or "navigation" — the ARIA-semantics choice matters here the same way it matters for Context Menu vs Menu (see `../context-menu/brief.md` §2 for the parallel "the missing props/exact role are the guidance" pattern).

## 4. When not to use + alternatives

Full cluster comparison: [`../_clusters/menus.md`](../_clusters/menus.md). Menubar-specific boundary reasoning:

- **vs plain Menu** — [E/I] "Menubar is not a different menu — it's a *composite container* that changes Menu.Trigger into a roving `menuitem` and coordinates sibling menus… Use Menubar when several menus form one persistent bar; a lone Menubar with one menu is just a Menu" [E menus cluster, quoted verbatim]. If there's only ever one menu on screen, wrapping it in a `<Menubar>` buys nothing — skip it.
- **vs Toolbar** — [E] the two are deliberately separate ARIA patterns implemented by deliberately separate code paths (`isInMenubar` vs `insideToolbar`, §2). A Toolbar may contain a single menu-triggered button (e.g. a "More actions ⋯" button opening a Menu) without ever becoming a Menubar; if the bar's primary content is *persistent, always-visible menu triggers* rather than a mix of buttons/toggles/inputs with an occasional menu tucked in, that's the Menubar shape, not Toolbar-with-a-menu-button. [I] this line is drawn from the code's own separation, not an independent maintainer "don't confuse these" statement — the cluster note frames it the same way, without additional citation. [G]
- **vs Navigation Menu** — [I] Menubar = application commands (desktop-app mental model, actions that *do things*); Navigation Menu = website destinations (links that *navigate*) with no menu ARIA roles at all [E menus cluster §"Menubar ↔ Navigation Menu"]. If the bar's entries are hyperlinks rather than commands, Navigation Menu is the correct component regardless of visual similarity to a menu bar.
- **Don't reach for Menubar just to get roving-tabindex keyboard navigation across unrelated controls** — [I] Menubar's roving focus is specifically wired for hosting `Menu.Root`s with `role="menuitem"` triggers; a generic "arrow-key-navigable row of buttons that aren't menus" is a Toolbar or a hand-rolled composite, not a Menubar with non-menu children stuffed into it. Inferred from the component's narrow single-purpose design (§1-2: it has exactly 4 props, all menu/menubar-shaped); not independently argued by a maintainer. [G]

## 5. Anatomy & composition

```jsx
<Menubar>                              // single element: <div role="menubar" aria-orientation={orientation}>
  <Menu.Root>                          // ordinary Menu.Root — detects menubar parentage via context
    <Menu.Trigger>File</Menu.Trigger>  // converts to role="menuitem", roving CompositeItem, hover-switch-aware
    <Menu.Portal>
      <Menu.Positioner sideOffset={4}> {/* default side: 'bottom' (horizontal) / 'inline-end' (vertical, #4922) */}
        <Menu.Popup>
          <Menu.Item>New</Menu.Item>
          <Menu.Item>Open</Menu.Item>
          <Menu.SubmenuRoot>       {/* ordinary nested-menu composition, unchanged by menubar parentage */}
            <Menu.SubmenuTrigger>Export</Menu.SubmenuTrigger>
            {/* nested Popup tree */}
          </Menu.SubmenuRoot>
          <Menu.Separator />
          <Menu.Item>Print</Menu.Item>
        </Menu.Popup>
      </Menu.Positioner>
    </Menu.Portal>
  </Menu.Root>

  <Menu.Root>                          {/* Edit */}  …</Menu.Root>
  <Menu.Root>                          {/* View */}  …</Menu.Root>
  <Menu.Root disabled>                 {/* Help — a disabled top-level menu; trigger stays a focus stop but never opens */}
    <Menu.Trigger>Help</Menu.Trigger>
  </Menu.Root>
</Menubar>
```

(Anatomy JSX above matches both the docs page's own Anatomy code block and the hero demo's actual structure, direct-read-confirmed identical.)

- [E] **No mandatory Portal-at-the-Menubar-level** — Menubar itself renders no popup and needs no Portal; each child `Menu.Root` carries its own `Menu.Portal` exactly as it would standalone. The library-wide explicit-anatomy/mandatory-Portal doctrine (B-P18) still applies, just at the child-Menu level, not the Menubar level. [I]
- [E] **A `Menu.Root disabled>` with no popup content at all is a valid, demo-shown pattern** (the hero demo's "Help" entry: `<Menu.Root disabled><Menu.Trigger className={styles.MenuTrigger}>Help</Menu.Trigger></Menu.Root>`, no `Portal`/`Positioner`/`Popup` at all) — a disabled top-level trigger that occupies a roving-focus stop but never opens anything, useful for menus that are contextually unavailable (e.g. "Help" with no content yet, or a feature-gated menu). [I] this is inferred from the demo's own structure, not independently called out in prose anywhere.
- [E] **`FloatingTree` wraps the whole bar** (`Menubar.tsx` line 76: `<FloatingTree><MenubarContent>…`) — every child `Menu.Root`'s Floating UI node registers into this shared tree, which is the mechanism `MenubarContent`'s `menuopenchange` listener (§6) uses to hear about *any* descendant submenu opening/closing, not just its direct children.
- **Anatomy-diagram spec**: (1) Menubar container bar (dashed outline, `role="menubar"` label) → (2) File/Edit/View/Help trigger cells inside it, each labeled `role="menuitem"` → (3) one dashed arrow showing "one Tab stop, arrows move between 1/2/3/4" → (4) an open popup dropping from the File cell → (5) a submenu flyout from within that popup (Export▸) → (6) an annotation on the disabled Help cell: "focusable, never opens."

## 6. Behavior ("How it works")

- [E] **`hasSubmenuOpen` tracking, precisely sourced** (`Menubar.tsx` `MenubarContent`, direct read, lines 97-125): a `React.useEffect` subscribes to the shared `FloatingTree`'s `menuopenchange` bus, filtering to events where `details.parentNodeId === nodeId` (i.e., only direct children of this menubar). On `details.open === true`, it flips `hasSubmenuOpen` to `true`. On close, it flips back to `false` **except** when `details.reason === 'sibling-open'` or `details.reason === 'list-navigation'` — this is the exact mechanism that prevents a visible flicker of "no submenu open" for one tick while the user moves from File's popup to Edit's popup: the state genuinely never goes false during a File→Edit handoff, it just silently re-attaches to the new open submenu. [E] This resolves the salvage's description ("does NOT clear `hasSubmenuOpen` when close reason is `'sibling-open'`/`'list-navigation'` (no flicker across File→Edit handoff)") with the exact code path.
- [E] **Hover-switch only after the first click, precisely sourced** (`MenuTrigger.tsx` line 173-174, direct read): `const parentMenubarHasSubmenuOpen = isInMenubar && parent.context.hasSubmenuOpen; const openOnHover = openOnHoverProp ?? parentMenubarHasSubmenuOpen;` — a trigger's effective `openOnHover` is `false` until *some* sibling submenu is already open (`hasSubmenuOpen === true`), at which point every trigger in the bar becomes hover-openable. This is the exact mechanism behind the documented UX: hovering over File/Edit/View does nothing until the user clicks one open, after which hovering the others switches between them without another click. Confirmed by test names (`Menubar.test.tsx`, direct read): `should not open submenus on hover when no submenu is already open` and `should open submenus on hover when another submenu is already open`.
- [E] **Focus-follows-open for siblings**: `useFocus(floatingRootContext, { enabled: !disabled && parentMenubarHasSubmenuOpen })` (line 206-208) — once a sibling submenu is open, merely focusing a trigger (not just hovering it) also opens its menu, matching native OS menubar keyboard/pointer-mixed navigation.
- [E] **Click-event type flips based on state**: `event: isOpenedByThisTrigger && isInMenubar ? 'click' : 'mousedown'` (line 200) — a menubar trigger that is *already the open one* toggles closed on full `click` (press+release), while every other trigger (not yet open, or not in a menubar) opens on the earlier `mousedown`. [I] This avoids a menubar trigger's own mousedown-open racing against its own click-to-close on the same gesture.
- [E] **`CompositeRoot` roving-focus mechanics** (`Menubar.tsx` lines 78-90, direct read): `orientation`, `loopFocus`, `enableHomeAndEndKeys` (added by PR #4922 per its own body: "Enable Home and End navigation for Menubar triggers"), and `highlightItemOnHover={hasSubmenuOpen}` — the roving highlight only follows hover once a submenu is open, consistent with the hover-switch-after-first-click behavior above (no premature highlight-stealing while merely browsing the bar with the mouse before committing to a click).
- [E] **`ArrowRight`/`ArrowLeft` roving between top-level triggers**, `Home`/`End` to jump to first/last — standard `CompositeRoot` behavior, confirmed by test names (`moves focus to the first and last triggers with Home and End`) and the widely-used `{ArrowRight}` keyboard interaction throughout `Menubar.test.tsx`. [E] The salvage further notes "ArrowRight from deep submenu switches top-level menus in one step (jsdom-only)" and a flagged asymmetry: "left/right navigation with menus open (jsdom-only, 'Doesn't work in headless mode')" — [G] this jsdom-vs-Chromium test-environment split was reported by the salvage but not independently re-verified by direct test-run in this pass; treat as inherited evidence.
- [E] **Modality is bar-wide, not per-menu**: `modal` defaults `true` on `Menubar` itself and is threaded down — "popup `modal` derived from the MENUBAR's modal prop" [E salvage, `MenuPositioner.tsx`]. A child `Menu.Root` doesn't independently choose its own modality when hosted in a menubar; the bar sets the policy for all its menus at once.
- [E] **`disabled` cascades from the bar to every hosted menu**: `MenuStore.ts` "disabled selector ORs menubar disabled" [E salvage, #2736, hardened #2924] — setting `disabled` on `<Menubar>` disables every child `Menu.Root` without touching each one individually; PR #4922's own body adds a refinement here too: "Keep open menu items disabled when the Menubar is disabled" (i.e., items inside an already-open submenu correctly go inert too, not just the top-level triggers).
- [E] **Vertical orientation submenu placement, precisely sourced from PR #4922's own body** (`gh pr view 4922`, confirmed): "Open vertical Menubar popups on the inline-end side by default while keeping horizontal Menubars opening downward" — before this fix (merged 2026-05-28), vertical menubars still opened their popups on the `'bottom'` side (appropriate for horizontal bars, awkward for vertical ones, where the popup would visually collide with the bar's own next item below). The fix is in `MenuPositioner.tsx`'s default-side logic, gated on menubar orientation. The PR's full change list (direct quote): "Move the initial composite highlight to the first enabled item when the default item is disabled," "Enable Home and End navigation for Menubar triggers," the vertical-side fix above, and "Keep open menu items disabled when the Menubar is disabled" — a coordinated four-part fix, not a single narrow patch, plus "Add CompositeRoot, Menubar, and positioner regression coverage." [E] This is more comprehensive than the salvage's characterization ("vertical fix"), which understated the PR's scope — worth correcting here.
- [E] **Separate `insideToolbar` mechanism stops composite-key propagation only, doesn't change trigger semantics** — `MenuPopup.tsx`: "`insideToolbar = useToolbarRootContext(true) != null` only stops COMPOSITE_KEYS propagation out of an open popup into Toolbar's roving focus" [E salvage] — confirming §2's point that Toolbar integration and Menubar integration are two genuinely different mechanisms in the same file, not two branches of one shared flag.
- [G] The exact behavior of `closeOnClick={false}` on a nested item inside a menubar-hosted menu (issue #2092, cited by the salvage as a git-history reference) was not independently re-verified against current source in this pass — inherited from salvage evidence only.

## 7. Accessibility contract

APG pattern: **[Menubar](https://www.w3.org/WAI/ARIA/apg/patterns/menubar/)** — the dedicated APG pattern for persistent, always-visible application menu bars (distinct from Menu's Menu Button pattern and from Toolbar's Toolbar pattern). [I] inferred from the `role="menubar"`/`role="menuitem"` ARIA wiring and the component's stated purpose; not independently quoted from a maintainer citing this exact APG page — [G] the docs page itself does not link an APG pattern (checked directly: no APG reference anywhere in `page.mdx`), so this citation rests on the ARIA-role match, the same honesty caveat Context Menu's brief raises for its own (unlinked) pattern.

**Keyboard interaction table** (source: `CompositeRoot` wiring in `Menubar.tsx` + `MenuTrigger.tsx` branches + `Menubar.test.tsx` test names, direct-read-confirmed):

| Key | Context | Action |
|---|---|---|
| `Tab` | anywhere in the bar | Moves focus into/out of the bar as **one stop** — the bar as a whole is a single Tab stop; arrow keys move *within* it |
| `ArrowRight`/`ArrowLeft` (horizontal) or `ArrowDown`/`ArrowUp` (vertical) | bar focused | Rove focus between top-level triggers (`orientation` decides the active axis); loops when `loopFocus` (default `true`) |
| `Home`/`End` | bar focused | Jump to first/last trigger — added by PR #4922 ("Enable Home and End navigation for Menubar triggers") |
| `Enter`/`Space` | trigger focused, closed | Opens that trigger's submenu (native/composite activation) |
| Hover (mouse) | no submenu open yet | **No-op** — hover-open is disabled until the user has clicked one trigger open |
| Hover (mouse) | a sibling submenu is already open | Switches the open submenu to the hovered trigger without an extra click (`parentMenubarHasSubmenuOpen` gate) |
| `ArrowRight`/`ArrowLeft` | a submenu is open, focus inside it | Closes the current submenu and opens the adjacent top-level trigger's submenu in one step (the "browse across the whole bar without leaving keyboard-driven submenu mode" pattern the prompt's play scenario exercises) — [G] confirmed present as salvage-reported test behavior; jsdom-vs-Chromium environment split noted above, not independently re-verified here |
| Any arrow key | inside an open submenu, that submenu's items | Standard Menu item navigation (unchanged by menubar parentage) |
| Disabled top-level trigger (e.g. "Help") | roving sequence | Still receives a roving-focus stop (composite tab-stop edge case explicitly tested: "disabled-first-trigger tab-stop edge" per salvage) but activation is a no-op and it never opens |

Menu popups/items inside: unchanged `role="menu"`/`menuitem`/etc. from Menu itself.

## 8. Prop-level guidance (decision-relevant props)

Menubar's entire public prop surface is 4 props — a deliberately minimal API matching its "thin coordinating container" design (§1-2).

- [E] **`orientation`** (default `'horizontal'`) — switches the roving-focus axis (`ArrowLeft`/`ArrowRight` vs `ArrowUp`/`ArrowDown`) and, since PR #4922, the default submenu-open side (`'bottom'` for horizontal, `'inline-end'` for vertical). Use `'vertical'` for sidebar-style command bars where a horizontal bar's `'bottom'`-opening popups would visually collide with the bar's own next item. Before #4922, vertical menubars were stuck with the horizontal-appropriate `'bottom'` default regardless of this prop — a real historical footgun worth naming if a consumer is on an older version.
- [E] **`loopFocus`** (default `true`) — whether reaching the last (or first) top-level trigger with arrow keys wraps back around. Tested both ways (salvage: "loopFocus true/false"). Set `false` if wrap-around focus feels disorienting for a very long bar; `true` (default) matches most native OS menubar behavior.
- [E] **`modal`** (default `true`) — sets the modality policy for *every* hosted menu's popup at once; a child `Menu.Root` cannot independently opt out while hosted in the bar (§6, "modal derived from the MENUBAR's modal prop"). If some menus in the bar genuinely need different modality than others, that's a signal those menus don't belong in the same Menubar instance — [I] inferred from the derivation being bar-wide, not per-menu; not independently argued as a "don't" by a maintainer. [G]
- [E] **`disabled`** (default `false`) — cascades to every hosted `Menu.Root` (they OR their own `disabled` with the menubar's, per `MenuStore.ts`), including keeping already-open menu items disabled per #4922's fix. Use for a whole-bar disabled state (e.g. app-level "read-only mode") rather than disabling each `Menu.Root` individually — one prop does what would otherwise be N individual props kept in sync.
- [E] **No `Menubar`-level styling-state props exist beyond `data-has-submenu-open`/`data-modal`/`data-orientation`** — since Menubar itself renders one `<div>`, its entire styling contract is these three data attributes on that single element; all remaining styling surface (popup open/closed, item highlighted, checked states, etc.) lives on the hosted `Menu.*` parts exactly as it would standalone.
- [I] **A `Menu.Root disabled>` with no popup rendered at all** (the "Help" hero-demo pattern, §5) is a usable recipe for "menu that exists as a roving-focus placeholder but has no content yet" — not independently documented as a named pattern anywhere, inferred purely from the shipped demo. [G]

## 10. Pitfalls & FAQ

- **"Hovering over my menubar items does nothing until I click one"** → by design, not a bug: hover-switch only activates once a sibling submenu is already open (`parentMenubarHasSubmenuOpen` gate, §6). Click one trigger first; hovering the others then switches without another click. [E] `MenuTrigger.tsx` line 174, test names confirm both states.
- **"My vertical Menubar's submenus open on the wrong side / overlap the bar"** → fixed by PR #4922 (merged 2026-05-28); before that fix, vertical menubars still defaulted to `'bottom'`-side popups meant for horizontal bars. Upgrade if seeing this. [E]
- **"Setting `disabled` on one Menu.Root inside my Menubar doesn't fully disable it — items still respond"** → check whether the *Menubar's own* `disabled` prop is what's needed instead (it cascades and, since #4922, also correctly disables already-open items); per-`Menu.Root` `disabled` still works too, but the bar-level prop is the one-shot way to disable everything at once. [E] §6, §8.
- **"Why isn't there a `Menubar.Item` or `Menubar.Trigger` part?"** → there isn't one — Menubar hosts ordinary `Menu.Root` instances and changes their behavior via context detection; use `Menu.Trigger`/`Menu.Item`/etc. exactly as you would standalone. This is the component's core architectural choice (§1-2), not a missing-parts oversight.
- **"RTL arrow-key direction for a horizontal Menubar — does it flip correctly?"** → [G] no dedicated test exists for this; the salvage's full-file read confirms "NO RTL-specific tests" after reviewing all 1,343 lines of `Menubar.test.tsx`. Treat as an honest unknown rather than assuming correctness by inheritance from `DirectionProvider`/Composite internals — verify manually if RTL support for a horizontal application menubar is a requirement.
- **No menubar-specific z-index/stacking pitfalls found** beyond what Menu already documents for its own popups — since every popup a Menubar shows is an ordinary `Menu.Popup`, Menu's stacking guidance (B-M1 in principles.md) applies unchanged. [I] not independently re-tested for menubar. [G]
