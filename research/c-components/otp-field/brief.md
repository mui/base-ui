# OTP Field — component research brief

Tier 2 (lean-plus depth). Mined 2026-07-07 from source, docs, `.test.tsx` suites, git history, and 2 `gh` PR lookups (#4365, #5029; shared budget with slider: 3 of 4 lookups used across both components). Evidence tags: [E] direct evidence, [I] inference, [G] acknowledged gap.

## 1. Identity

- **Name / slug**: OTP Field — `@base-ui/react/otp-field`. Multi-part compound component, namespace export `OTPField.*`.
- **Preview → stable timeline (mandatory to state up front)**: [E] OTP Field shipped as a **preview** component in v1.4.0 via PR #4365, "[otp field] Add OTP Field component" (atomiks, merged 2026-04-10, commit `70eb4c45b`), exported as `OTPFieldPreview` — closing issue #75. It was graduated to stable in v1.6.0 via PR #5029, "[otp field] Unmark preview" (atomiks, merged 2026-06-17, commit `1ecbd50e9`), which renamed the namespace export `OTPFieldPreview` → `OTPField` as an explicit **breaking change** ("Follows the Drawer de-preview precedent," #5029 body, referencing `bb8140e1c`/#4293). The elapsed time from preview-launch to stable-graduation was **just over two months** (2026-04-10 → 2026-06-17) — a notably fast preview cycle by this library's own standard (contrast Drawer, which spent longer in `DrawerPreview` per `research/b-library-principles/principles.md` line 406).
- **The docs component index STILL shows `[New]`, and this is intentional, not stale** — a genuinely interesting finding worth calling out explicitly: `docs/src/app/(docs)/react/components/page.mdx` line 26 reads `- OTP Field [New]` (verified directly, contrast Slider's plain `- Slider` on line 34, no tag at all). This is not an oversight or a lagging badge system — **PR #5029's own body states the intended change verbatim**: "**Components list** (`components/page.mdx`): `OTP Field [Preview]` → `OTP Field [New]`." [E] In other words, the same PR that removed the Preview label *deliberately replaced it with a New label* rather than removing all badging outright — the maintainers chose to keep flagging OTP Field as recently-added-and-noteworthy even though it is no longer experimental/unstable. No component-level `tags:` frontmatter exists anywhere in any `page.mdx` in this repo (confirmed by grep across all `components/*/page.mdx` — the badge is computed by the `@mui/internal-docs-infra` `createSitemap` package, external to this repo, likely from git-history recency); [G] the exact automated heuristic (e.g., "tag anything whose founding commit is within N months") that decides *when* `[New]` eventually falls off was not traced further given this brief's Tier-2 budget, but the specific `[Preview]`→`[New]` *transition* is directly confirmed by the #5029 PR body, not inferred.

[E]
- **Status**: stable as of v1.6.0 (§ above); `[New]`-tagged in the docs nav (see above) — not `[Preview]` any longer, an important distinction since a reader skimming only the sidebar badge could still reasonably (and now correctly, per the maintainers' own framing) read `[New]` as "recently added, not yet battle-tested," even though the underlying API is explicitly declared settled ("It has been stable through several releases, all reported issues are resolved, and the API is settled," #5029 body). [E]
- **No `*CssVars.ts` file exists for this component** — confirmed directly: `find packages/react/src/otp-field -iname "*cssvars*"` returns nothing, and no `import.*CssVars`/`'--'`-prefixed custom property appears anywhere in `OTPFieldRoot.tsx`/`OTPFieldInput.tsx`. [E] This mirrors both Number Field and Slider's absence of a CSS-var contract (`research/c-components/number-field/brief.md` §8, `research/c-components/slider/brief.md` §1) — OTP Field has no popup/positioning surface and no computed-inline-style positioning either; every visible slot is a plain styled `<input>`, so there is simply nothing dynamic to expose as a variable.
- **Purpose / IA statement**: [E] "A one-time password input composed of individual character slots" (`docs/src/app/(docs)/react/components/otp-field/page.mdx` Subtitle) — a form control purpose-built for OTP/verification-code/2FA entry, where each character occupies its own visually distinct, individually-focusable input box, contrasted with a plain single `<input>` that happens to hold a short code. Taxonomy: **form inputs** category, "composed multi-slot text entry" — distinct from both Number Field (single formatted numeric value, `research/c-components/number-field/brief.md`) and Slider (continuous/glance value, `research/c-components/slider/brief.md`); no dedicated cluster note exists for OTP Field yet since it doesn't share an obvious overlap boundary with another Base UI component [G] — the closest analog outside this library is a generic "PIN input"/"segmented input" pattern, not one of Base UI's own components.

## 2. Intention

- [E] **Introducing PR**: mui/base-ui#4365, "[otp field] Add OTP Field component" (atomiks, merged 2026-04-10), closing #75. Body, quoted directly since it's the clearest founding-intent statement available: "This adds a preview OTP Field component with Root and Input parts, along with docs, hero demos, tests, and generated API reference." The PR's own **API note is the single most load-bearing design decision to document**: "`length` is required on `OTPField.Root` as the source of truth for the intended OTP size. The component uses it for validation, hidden-input form integration, completion state, and deterministic SSR or initial render behavior, while CompositeList is used only for slot order and indexing." This directly explains why `length` (not "count the rendered `Input`s") is the required, load-bearing prop — a design choice made explicitly to support SSR determinism (the server doesn't need to render every slot to know the field isn't yet complete) and to decouple "how many slots exist" from "which slot is at which index" (the latter handled by `CompositeList`, matching the same composite-list machinery used by Slider's thumbs, `research/c-components/slider/brief.md` §5).
- [E] **A dev-only runtime warning enforces the `length`-vs-rendered-count contract directly** (`useOTPFieldRootDevWarnings` in `OTPFieldRoot.tsx`): if the number of actually-rendered `<OTPField.Input>` parts doesn't match `length` (once at least one has mounted), a warning fires: "`<OTPField.Root>` `length` must match the number of rendered `<OTPField.Input />` parts. Received `length={N}` but rendered `M` input(s)." A second warning fires if `length` itself isn't a positive integer. This is strong, direct evidence that the founding PR's `length`-as-source-of-truth design is actively enforced, not merely a documented convention a consumer could silently violate.
- [I] **Design pole: a real, separately-focusable native `<input>` per character slot, wired together by one shared Root-level state machine** — inferred from the entire architecture: `OTPFieldRoot.tsx` owns the single string `value`, roving `tabIndex`/focus-index bookkeeping, and a hidden validation `<input>`; `OTPFieldInput.tsx` is a thin per-slot view that reads its own character (`value[index]`) out of the shared string and forwards native input events (change/keydown/paste/focus/blur) back up through context-provided callbacks (`setValue`, `queueFocusInput`, `focusInput`). This composition mirrors the founding PR's explicit separation of concerns (Root owns `length`/validation/completion, CompositeList owns slot order) rather than, say, a single hidden input with per-character visual overlays (a technique some other OTP-input libraries use) — Base UI's approach keeps every slot a genuinely independent, individually-labelable, individually-testable `<input>` element.
- [E] **A refactor immediately followed launch, signaling the initial architecture needed hardening quickly**: `7158d7b27 [otp] Refactor OTPFieldRoot and OTPFieldHiddenInput (#4609)` landed shortly after #4365 — the commit message alone ("Refactor... and OTPFieldHiddenInput") implies the hidden-input machinery was split out/reworked early, though the detailed diff/rationale wasn't pulled given this brief's `gh`-lookup budget (already spent on #4365/#5029) [G, acknowledged — the refactor's *existence* and rough shape is [E] from the commit message, but its detailed rationale is not independently verified here].
- [E] **Explicit non-goal, discoverable directly from the validation architecture**: OTP Field ships **three built-in `validationType` presets** (`'numeric'` default, `'alpha'`, `'alphanumeric'`) plus an escape hatch (`'none'` + a `normalizeValue` function prop for fully custom rules) — this is evidence the maintainers deliberately scoped the built-in validation surface to common OTP shapes (digits-only codes, letter-only codes, mixed alphanumeric recovery/invite codes) rather than trying to anticipate every possible custom format inline; anything more exotic is explicitly routed through `normalizeValue`, not a growing list of built-in `validationType` presets. [I, this scoping choice is derived directly from reading `packages/react/src/otp-field/utils/otp.ts`'s `OTP_VALIDATION_CONFIG` object and the `normalizeValue` JSDoc — no maintainer discussion thread explicitly stating "we will not add more built-in validationType presets" was found in the mined corpus, since OTP Field has no dedicated discussions-mining entry yet, §11]
- [E] **`autoSubmit` and `onValueComplete` are treated as two independent, separately-designed capabilities, not one feature with two names** — `onValueComplete(value, eventDetails)` fires whenever the OTP becomes complete (all `length` characters filled), regardless of whether `autoSubmit` is set; `autoSubmit` (default `false`) is a strictly additive behavior that, when enabled, calls `formElement.requestSubmit()` immediately *after* `onValueComplete` runs (`completeValue` function in `OTPFieldRoot.tsx`: `onValueCompleteProp?.(...)` then `if (autoSubmit) { requestSubmit(); }`). A consumer who wants to react to completion (e.g. show a "verifying…" spinner) without automatically triggering navigation-away-from-page form submission can use `onValueComplete` alone.
- [E] **A dense, still-active bugfix cluster around focus/validation edge cases signals the interaction surface is treated as continually-hardened, not "shipped and done"** (mirroring the same pattern documented for Number Field and Slider, `research/c-components/number-field/brief.md` §2, `research/c-components/slider/brief.md` §2): `9b128eb64 [otp field] Keep focus on the invalid field when autoSubmit is blocked (#5089)`, `4a339d027 [otp field] Prevent locked hidden autofill validation (#4827)`, `02240e7c8 [otp field] Ignore hidden input changes while readonly or disabled (#4806)`, `be57166ce [otp field] Avoid password manager bubbles after first input (#4868)`, `8535638e9 [otp field] Fix vertical arrow slot navigation (#4844)`, `8db1aa95e [otp field] Fix RTL arrow navigation (#4847)`, `b20d718e8 [otp field] Support ctrl/cmd keyboard shortcuts (#4606)`. All of these landed within roughly two months of the founding PR — evidence that browser-autofill interop, RTL, and boundary keyboard behaviors needed rapid, real-world-driven iteration immediately after the preview launch.

## 3. When to use

- [E] One-time password, SMS/email verification code, or 2FA code entry where each character should occupy its own visually distinct, individually-focusable box — the docs subtitle and every one of the six shipped demos (hero, alphanumeric, grouped, focused-placeholder, custom-sanitize, password) are built around this single use case; there is no "generic segmented text input" framing beyond OTP/verification codes in the current docs.
- [E] Numeric codes of a fixed, known length (`validationType="numeric"` default) — SMS/authenticator-app style codes; heavily represented in the test suite's default configuration (`OTPFieldRoot.test.tsx` uses numeric-default `<OTPField>` test helpers throughout).
- [E] Alphanumeric or alphabetic recovery/backup/invite codes (`validationType="alphanumeric"`/`"alpha"`) — explicitly named in the docs' own "Alphanumeric verification codes" section: "recovery, backup, or invite codes that mix letters and numbers."
- [E] Codes that need visual chunking for readability (e.g., `123-456`) without changing the underlying contiguous value — `<OTPField.Separator>` between grouped `<div>`-wrapped subsets of inputs, purely a visual/layout device that does not affect indexing or the value string (docs "Grouped layouts"; §5).
- [E] Forms where the code should be masked while typing (shared-screen/kiosk scenarios) — `mask` prop (docs "Masked entry": "Use `mask` when the code should be obscured while it is being typed"; the password demo's own copy: "obscure the code on shared screens").
- [E] Forms wanting to auto-submit the instant the code is complete, avoiding an explicit "Verify" button click — `autoSubmit` (docs "Form integration": "Pass `autoSubmit` to submit the owning form automatically when all slots are filled").
- [I] Custom/non-standard code shapes needing bespoke normalization (e.g., forcing uppercase, stripping dashes users might paste in) — evidenced directly by the shipped `normalizeValue` prop and its dedicated "Custom normalization" docs demo, though no specific real-world scenario beyond the docs' own uppercase-conversion example was independently corroborated [G — no Phase D corpus data exists yet for this component, §11].

## 4. When not to use + alternatives

- [E] **A short code with no need for individual-character visual segmentation → a plain text `Input`/`Field` is simpler.** OTP Field's entire machinery (per-slot roving focus, paste-splitting across slots, a hidden validation input, `CompositeList` slot-order tracking) is overhead when a single `<input>` box holding the whole code would serve the UI just as well — no maintainer statement directly comparing the two was found [G], but this follows directly from the component's stated purpose (individually-boxed character slots) versus a generic `Input`'s (a single text-entry primitive), the same reasoning Number Field's own brief applies to its "plain unformatted entry" boundary (`research/c-components/number-field/brief.md` §4).
- [E] **Long-form free text or multi-word entry → not a fit at all.** The entire validation/normalization pipeline (`validationType` presets, per-character `maxLength`-style slotting) is built around fixed-`length`, single-character-per-slot values; there is no accommodation for variable-length input or multi-character-per-slot entry.
- [I] **A masked *password* field with an unknown/variable length → a plain password `Input`, not OTP Field with `mask`.** OTP Field's `mask` prop obscures characters using the native `type="password"` rendering per slot (§6), but the component's fixed-`length` contract and per-slot navigation model are built for known-length codes, not open-ended password entry — using OTP Field for a general password field would be fighting the component's core assumption rather than leveraging it. No maintainer statement directly makes this comparison [G]; derived from the fixed-`length`-as-source-of-truth design (§2).
- [E] **Math-expression, formula, or free-form numeric entry → Number Field, not OTP Field.** Trivially out of scope; OTP Field validates single characters against a fixed pattern per slot, with no numeric parsing/formatting pipeline at all (contrast Number Field's `Intl.NumberFormat`-based pipeline, `research/c-components/number-field/brief.md` §6).
- [G] No dedicated `_clusters/` comparative note exists for OTP Field against any other Base UI component, and none of the eight mandatory overlap clusters listed in `research/PROMPT.md` §8.2(4) name OTP Field — this component does not have an obvious sibling/overlap boundary within Base UI's own component set the way Select/Combobox/Autocomplete or Toggle/Switch/Checkbox do. This absence is itself worth recording rather than manufacturing a false comparison.

## 5. Anatomy & composition

- [E] Canonical tree (docs Anatomy, `page.mdx`):
  `Root > (Input, Separator)` — repeated: one `Input` per character slot, with `Separator` used only where visual grouping is wanted.
- [E] **Minimal composition is `Root > Input × length`** — no `Group`/`Label`/`Portal`/`Positioner` grammar exists at all (unlike the popup-family components); the entire anatomy is flat. `Separator` is purely optional and purely visual.
- [E] **Arbitrary DOM nesting between Root and its Inputs is explicitly supported and evidenced**, not just theoretically possible: the "Grouped layouts" docs demo wraps two subsets of three `<OTPField.Input>`s each inside plain `<div className={styles.Group}>` wrappers, with an `<OTPField.Separator>` between the two groups (`demos/grouped/css-modules/index.tsx`, read directly) — `CompositeList` tracks slot order via actual DOM/render order (`IndexGuessBehavior.GuessFromOrder` in `useCompositeListItem`), so wrapping inputs in intermediate `<div>`s for layout purposes does not break indexing or the contiguous value string.
- [E] **Labeling requires deliberate authoring, not automatic multi-slot propagation** — confirmed directly from source and docs: only the *first* slot (`index === 0`) is intended to carry the field's overall accessible name (via an external `<label htmlFor>` or `<Field.Label>`); every subsequent slot needs its own distinguishing `aria-label` (e.g. `"Character 2 of 6"`) so a screen-reader user tabbing through slots hears which position they're on (docs "Labeling an OTP field": "Let the first input use the field label, and add `aria-label` to the remaining inputs"). **A dev-only runtime warning actively enforces the first-slot rule**: if `aria-label` is set on the first slot without an associated `<label>`/`Field.Label`, a warning fires: "`<OTPField.Input>` ignores `aria-label` on the first input. Use a `<label>` or `<Field.Label>` to label the OTP field." (verified directly in `OTPFieldInput.tsx`; the `ariaLabel` computed value is literally `index === 0 ? undefined : slotAriaLabel` — the first slot's `aria-label` prop is silently dropped at the ARIA-attribute level even before the warning fires, confirmed by the test "does not warn for a first-slot aria-label when a native label is associated" alongside "warns when aria-label is provided on the first slot without an associated label"). **This is exactly the kind of a11y contract detail this task's brief asked to verify rather than assume** — a naive assumption might be "each Input gets its own label the way Select.Item might," but the actual contract is asymmetric: slot 0 inherits the shared field label, slots 1..N-1 need individually-authored `aria-label`s.
- [E] **A hidden, `aria-hidden` validation `<input type="text">` is rendered as a sibling of the visible slots** (only when `length` is a valid positive integer, `hasValidLength` check in `OTPFieldRoot.tsx`) — carries the full OTP value, native `pattern`/`minLength`/`maxLength`/`required`/`name`/`form`/`autoComplete`/`inputMode`, and is the element that native browser autofill (password managers, SMS-code autofill) actually targets and fills; its own `onFocus` redirects real focus to the first visible slot, and its own `onChange` handler re-splits any autofilled/pasted-in-bulk value back out across the visible per-slot state via the same `setValue`/normalization pipeline the visible slots use. This is architecturally the same "hidden input carries native form semantics, visible UI is a separate layer" pattern documented for Number Field's hidden `<input type="number">` (`research/c-components/number-field/brief.md` §5) and Select's hidden `<input>` (`research/c-components/select/brief.md` §6) — a recurring library-wide convention, not novel to OTP Field.
- [E] **`Separator` is a verbatim re-export of the library-wide standalone `Separator` component** (`export { Separator } from '../separator/Separator';`, `index.parts.ts`) — identical to how Select/Menu/Combobox/Autocomplete/ContextMenu re-export the same component unchanged, per the cross-component re-export map already verified in `research/c-components/_clusters/status.md` (progress/meter/avatar/separator brief batch). No OTP-Field-specific separator behavior exists; it carries no orientation-inversion logic the way `Toolbar.Separator` does.
- Text spec for a future anatomy diagram: (1) Root [invisible `<div role="group">` wrapper, plus a hidden `aria-hidden` validation `<input>` rendered as its sibling — not shown visually], (2) Input × `length` [each a real, separately-focusable `<input>` box; slot 0 optionally carries the field's shared accessible name via external labeling, slots 1..N-1 need their own `aria-label`], (3) Separator [optional, purely visual, placed between groups of Inputs]. A dotted annotation should mark the hidden validation input as the true native-form/autofill target, analogous to Number Field's and Select's hidden inputs.

## 6. Behavior ("How it works")

- [E] **Controlled/uncontrolled**: uncontrolled by default via `defaultValue` (a string); controlled via `value`/`onValueChange` (`useControlled` in `OTPFieldRoot.tsx`). The single source of truth is one string value (e.g. `"123456"`), sliced per-index for each visible slot (`slotValue = value[index] ?? ''`) — there is no per-slot controlled-value API; consumers only ever control the whole string.
- [E] **`onValueChange` reason vocabulary** (`OTPFieldRootChangeEventReason`): `'input-change'` (typing/autofill into a slot or the hidden input), `'input-clear'` (a character removed via native text-input clearing — distinct from an explicit Backspace/Delete keypress, which is its own `'keyboard'` reason), `'input-paste'`, `'keyboard'` (Backspace/Delete/Ctrl+Backspace handled explicitly in `onKeyDown`, §7). Note the asymmetry: clearing a character by selecting-and-deleting text natively (browser-level, e.g. via an IME or selecting-then-typing-nothing) reports `'input-clear'`, while a literal Backspace/Delete keydown handled by the component's own logic reports `'keyboard'` — these are mechanically different code paths (`onChange` handler vs `onKeyDown` handler in `OTPFieldInput.tsx`) even though both remove a character.
- [E] **`onValueInvalid(value, eventDetails)` — a distinct callback for rejected-characters, fired independently of whether the value actually changes**: whenever typed, pasted, or autofilled text contains characters that get stripped by `validationType` filtering or a custom `normalizeValue` (e.g. typing a letter into a `validationType="numeric"` field), `onValueInvalid` fires with the raw, pre-normalization attempted string and a reason (`'input-change'` or `'input-paste'`) — this fires even in cases where the *net* value doesn't change (e.g. typing an invalid character into an already-filled slot leaves that slot's committed value unchanged, but `onValueInvalid` still reports the rejected attempt). This is the mechanism the "Custom normalization" docs demo pairs with a `useInvalidFeedback` hook to show inline rejected-character feedback.
- [E] **`onValueComplete(value, eventDetails)` fires once the OTP reaches full `length`**, including the specific case of "a complete value is pasted while the OTP is already complete" (JSDoc explicitly calls this out) — pasting a *different* complete 6-digit code over an already-complete field fires `onValueComplete` again with the new value (confirmed by test: "fires `input-paste` when a complete paste replaces a complete OTP" — asserts 3 total calls across a sequence of paste operations, correctly tracking each transition). A stale/canceled controlled-completion attempt correctly does *not* fire (tested: "does not fire later for a stale controlled completion attempt," "does not fire again when a controlled value changes from one complete value to another" — i.e., no-op re-application of the same complete value is correctly suppressed).
- [E] **Paste behavior is normalize-then-distribute, starting from the pasted-into slot, not necessarily slot 0**: pasting text into slot index `i` calls `replaceOTPValue(value, i, nextDigits, length, validationType, normalizeValue)`, which normalizes the pasted text, splices it into the existing value string starting at index `i`, then re-normalizes/re-clamps the whole result to `length` — so pasting into a middle slot correctly overwrites-forward from that position rather than always starting at slot 0 (tested: "replaces values from the middle when pasting into a later slot," and separately at the Root level: "fires `input-paste` when pasting into a middle slot completes the OTP" using a `defaultValue="12"` starting state). Multi-byte/Unicode-aware slicing (`Array.from(normalizedValue)`, `otp.ts`) ensures multi-byte characters never split across slot boundaries. After a successful paste, focus queues to `Math.min(index + nextDigits.length, length - 1)` — the slot immediately after the last pasted character, clamped to the final slot.
- [E] **Backspace navigation has a real, tested state machine, not a single uniform rule**: pressing Backspace on a **filled** slot clears that slot and keeps focus there ("deletes the current character with Delete without moving focus" — Delete-specific; contrast Backspace) — actually, precisely: Backspace's behavior branches on whether the *current* slot is already empty. If the current slot has a character, Backspace removes that slot's character and focus stays put conceptually but the *target* index for the next backspace becomes the current slot (`deleteIndex = slotValue === '' ? targetIndex : index`); if the current slot is already empty, Backspace instead deletes the *previous* slot's character and moves focus there (tested: "deletes the current character and moves focus to the previous slot on backspace" for a filled slot, and "deletes the previous filled slot when backspacing on an empty non-first slot" for an empty one) — i.e., **Backspace on an empty slot cascades backward to delete the nearest filled slot to its left**, a deliberately forgiving UX (matching the common OTP-input convention where users expect Backspace to "eat backward" through empty slots) rather than a strict "delete only this exact slot" rule. **Ctrl/Cmd+Backspace (or the `hasBoundaryModifier` check, which also covers Meta) clears the entire value in one keystroke** and returns focus to the first slot (tested: "clears all slots with %s + Backspace" for each modifier key) — added specifically in `b20d718e8 [otp field] Support ctrl/cmd keyboard shortcuts (#4606)`.
- [E] **Delete (forward-delete) behaves differently from Backspace**: Delete always removes the *current* slot's character (if any) and does **not** move focus (tested: "deletes the current character with Delete without moving focus") — a deliberate asymmetry from Backspace's cascading/focus-moving behavior, matching how native text-field Backspace-vs-Delete conventionally differ.
- [E] **Typing a character that exactly matches the already-filled current slot's value advances focus without changing the value** (tested: "moves focus to the next slot when typing the same character into a filled slot," and at the Input level: "does not reselect the final slot when typing the same character") — a small but real UX nicety: a user who "re-types" the digit that's already there (e.g. from muscle memory or a slow autofill race) still advances, rather than the input silently rejecting the keystroke and leaving them stuck.
- [E] **Typing into an already-filled slot with a *different* character replaces it and, if more than one character is somehow entered in one native `change` event (e.g. rapid typing captured as a batch, or certain IME/mobile-keyboard behaviors), fills consecutive slots forward from the current position** — both the single-slot-typing path and the multi-character-in-one-event path are separately tested ("fills consecutive slots when typing multiple characters into the first input," "replaces consecutive slots when typing multiple characters into a later input") — this is mechanically the same `replaceOTPValue`-based distribution logic paste uses, just triggered from a native `onChange` rather than `onPaste`.
- [E] **RTL and vertical-orientation navigation are both explicitly handled and were both the subject of dedicated post-launch fixes**: `direction === 'rtl'` swaps which physical arrow key (`ArrowLeft`/`ArrowRight`) means "previous slot" vs "next slot" (`isRtl` check in `OTPFieldInput.tsx`'s `onKeyDown`, confirmed directly; test: "moves focus with arrow keys in RTL" asserts the swap). `Home`/`ArrowUp` always jump to the first slot; `End`/`ArrowDown` always jump to the "end target" slot — defined as `Math.min(value.length, lastIndex)`, i.e., the first *empty* slot if the value isn't yet complete, or the last slot if it is (tests: "moves focus to the empty end slot with ArrowDown," "keeps focus on the final slot with ArrowDown when the value is complete," "stops propagation when ArrowDown moves focus to the empty end slot"). Both RTL arrow handling (`8db1aa95e Fix RTL arrow navigation (#4847)`) and vertical `ArrowUp`/`ArrowDown` slot-jump behavior (`8535638e9 Fix vertical arrow slot navigation (#4844)`) needed dedicated post-launch bugfix commits, evidence this navigation matrix wasn't fully correct on first ship.
- [E] **Focus-jump redirect for out-of-sync clicks**: clicking directly into a slot *beyond* the current value's length plus one (i.e., skipping ahead past unfilled slots) redirects focus back to the correct next-empty-or-last slot instead of letting the user focus an arbitrarily-far-ahead empty slot (`handleInputFocus` in `OTPFieldRoot.tsx`: `if (index > valueRef.current.length) { focusInput(...); return; }`) — this keeps the roving-focus model consistent (users can't tab/click their way into slot 5 while slots 2-4 remain empty).
- [E] **`readOnly` blocks value changes but explicitly preserves arrow/Home/End navigation** — confirmed by a test written specifically to pin this distinction: "keeps arrow and home/end navigation working in readonly mode" alongside "blocks Delete and Backspace from changing the value in readonly mode" and "blocks paste from changing the value in readonly mode." This mirrors the general library convention that `readOnly` disables *mutation* but not *navigation/inspection* (the same distinction Number Field draws for its own `readOnly` handling, `research/c-components/number-field/brief.md` §8).
- [E] **`disabled` mode leaves vertical arrow navigation unhandled entirely** (test: "leaves vertical arrow navigation unhandled in disabled mode") — the `onKeyDown` handler's very first check is `if (event.defaultPrevented || disabled) { return; }`, so a fully `disabled` field doesn't intercept *any* keyboard handling, deferring completely to native input behavior (which itself won't respond meaningfully since the input is natively disabled).
- [E] **Masking is native `type="password"` per slot, not a custom bullet-character overlay**: `mask` (Root prop, default `false`) sets every slot's `type` to `"password"` (`type: mask ? 'password' : 'text'`, `OTPFieldInput.tsx`) — this means masked OTP Field slots get whatever native password-dot rendering the browser/OS provides, and individual slots can still be overridden with their own explicit `type` prop if a consumer wants a different masking character or behavior on specific slots (docs "Masked entry" section header note: "Pass `type` directly to individual `<OTPField.Input>` parts to use a custom input type" — directly from the Root's `mask` JSDoc).
- [E] **SSR**: the hidden validation input renders with the correct `pattern`/`minLength`/`maxLength` derived from `length` even before any visible slot has hydrated (test: "renders a hidden validation input with the provided length" under a `describe('server-side rendering')` block) — this is the concrete payoff of `length` being a required, standalone prop rather than inferred from rendered children (§2): the server can produce fully-correct native-validation markup with zero visible-slot information.
- [I] No dedicated "SSR mismatch" source comment (the kind Number Field carries for locale-formatting hydration, `research/c-components/number-field/brief.md` §6) was found for OTP Field — plausibly because the value is a plain string with no locale-dependent formatting step, so there's nothing analogous to reformat between server and client [I, absence-based inference, not an explicit maintainer statement].

## 7. Accessibility contract

- [E] **Root renders `role="group"`** (confirmed directly, `OTPFieldRoot.tsx` line 378: `role: 'group'`) — the same pattern Slider's Root uses (`research/c-components/slider/brief.md` §7) and Number Field's `Group` part uses (`research/c-components/number-field/brief.md` §7): a semantic ARIA grouping, not a composite-widget role like `listbox`/`radiogroup`. **This is a "verify, don't assume" finding worth stating plainly**: OTP Field does *not* implement a dedicated ARIA composite-widget pattern (there is no W3C APG pattern named "OTP input" or "PIN input" at all — the closest APG patterns are generic; none apply directly) — it is architecturally **N genuinely separate native `<input>` textboxes**, roving-`tabIndex`-coordinated by the Root, inside a `role="group"` container, not a single combined accessible object the way a `listbox`/`combobox`/`slider` widget is.
- [E] **Each slot is `role="textbox"` (the native implicit role of `<input type="text">`)** — or, when `mask` is enabled, the native role a `type="password"` input exposes (verified by test: `screen.getByRole('textbox')` is used pervasively for unmasked slots throughout `OTPFieldInput.test.tsx`; the mask-specific test instead asserts `toHaveAttribute('type', 'password')` directly rather than a role query, since password inputs are typically excluded from the standard accessible-textbox role mapping in most browsers — a genuine, if minor, a11y nuance: **masked OTP slots may not be discoverable via `getByRole('textbox')` in assistive tech the same way unmasked slots are**, mirroring the same accessible-name/role ambiguity native password fields always have. [I, this specific role-exposure caveat for `type="password"` is derived from general ARIA/HTML role-mapping knowledge plus the test file's own choice to query by attribute rather than role for the masked case, not from an explicit maintainer statement — flagged as a nuance worth independent verification with a real screen reader before asserting confidently on the docs page].
- [E] **Roving `tabIndex`, not full Tab-stops-on-every-slot**: only the slot at `activeIndex` gets `tabIndex={0}`; every other slot gets `tabIndex={-1}` (`OTPFieldInput.tsx`: `tabIndex: activeIndex === index ? 0 : -1`) — a keyboard user Tabbing into the field lands on exactly one slot (the active one, computed from current value length/focus state in `OTPFieldRoot.tsx`'s `activeIndex` memo), then must use ArrowLeft/ArrowRight (or type/paste) to move between slots rather than Tab — Tab/Shift+Tab move focus *out* of the OTP field entirely to the next/previous focusable element on the page (tested: "allows tabbing out of the field from the active slot," which asserts focus lands on an external "Next" button after Tab). This is architecturally identical to the roving-tabindex convention used by composite widgets throughout the library (e.g. Select's items, Slider's multi-thumb Tab order is native-input-based rather than roving, so OTP Field's roving-tabIndex model is closer to a listbox/menu's internal navigation convention than to Slider's).
- [E] **Labeling is asymmetric across slots and requires deliberate authoring — verified, not assumed** (full mechanics in §5): slot 0 inherits the field's overall accessible name via an external `<label htmlFor>` or `<Field.Label>`; slots 1..N-1 need individually-authored `aria-label`s (e.g. `"Character 2 of 6"`) to be distinguishable to assistive technology, and a dev-only warning actively catches the common mistake of putting `aria-label` on slot 0 instead of relying on an associated label. **This directly answers this task's explicit instruction to verify "single input semantics? per-slot labeling?" rather than assume**: the answer is neither pure single-input semantics nor uniform per-slot labeling — it's a hybrid where slot 0 carries the shared name and the rest carry positional labels.
- [E] **`aria-describedby` composition**: the Root's own `aria-describedby` prop is merged (via a dedupe-and-join `mergeAriaIds` helper) with any description id supplied by an ancestor `Field.Root`/`Field.Description`, and the merged result is placed on the `role="group"` container itself (not on individual slots) — confirmed directly (`OTPFieldRoot.tsx`'s `ariaDescribedBy` computation and `getDescriptionProps` from `useLabelableContext`); tested: "forwards root `aria-describedby` to the group," "applies the Field description to the group."
- [E] **`aria-invalid` is applied per-slot, not just at the group level**: each `<OTPField.Input>` receives its own `aria-invalid={!disabled && invalid ? true : undefined}` (`OTPFieldInput.tsx`) — so a screen-reader user landing on *any* slot (not just slot 0) hears the invalid state, which matters since focus can land on any slot depending on where the user last was.
- [E] **Keyboard interaction table** (evidence: `OTPFieldInput.tsx` `onKeyDown`/`onPaste`/`onChange`, `OTPFieldInput.test.tsx`/`OTPFieldRoot.test.tsx` `describe('keyboard')`/`describe('pasting')` blocks):
  | Key / gesture | Context | Action |
  |---|---|---|
  | Any character matching `validationType` (or passing `normalizeValue`) | Slot focused | Fills the current slot; if a native `change` event somehow carries multiple characters, fills consecutive slots forward; focus advances to the next slot (or stays on the final slot if already there) |
  | ArrowLeft (ArrowRight in RTL) | Slot focused | Moves focus to the previous slot (`Math.max(0, index - 1)`); Ctrl/Cmd+ArrowLeft jumps straight to the first slot |
  | ArrowRight (ArrowLeft in RTL) | Slot focused | Moves focus to the next slot (or the "end target" slot, clamped); Ctrl/Cmd+ArrowRight jumps straight to the end-target slot |
  | Home / ArrowUp | Slot focused | Jumps focus to the first slot (index 0) |
  | End / ArrowDown | Slot focused | Jumps focus to the "end target" slot — the first empty slot if the value is incomplete, or the last slot if complete |
  | Backspace | Slot focused, slot has a character | Clears the current slot's character; the *next* Backspace's delete-target becomes this now-empty slot |
  | Backspace | Slot focused, slot is already empty (not the first slot) | Cascades backward: clears the nearest filled slot to the left and moves focus there |
  | Ctrl/Cmd+Backspace | Slot focused | Clears the entire OTP value in one keystroke and returns focus to the first slot |
  | Delete | Slot focused | Clears the current slot's character only; focus does **not** move |
  | Same character as already in the slot | Slot focused, slot filled, full text selected | No value change; focus simply advances to the next slot (unless already on the final slot) |
  | Ctrl/Cmd+V (native paste) | Slot focused | Reads clipboard text, normalizes it, splices into the value starting at the current slot's index, re-clamps to `length`; focus queues to just past the last pasted character (clamped to the final slot) |
  | Tab / Shift+Tab | Slot focused | Moves focus out of the OTP field entirely (native Tab order, since only the active slot is a tab stop) |
- [E] **`autoComplete` targeting is deliberately asymmetric across slots**: only slot 0 (and the hidden validation input) receives the real `autoComplete` value (default `"one-time-code"`, the standard browser/OS hint for SMS-autofill-style OTP suggestions); every other visible slot gets `autoComplete="off"` explicitly (`OTPFieldInput.tsx`: `autoComplete: index === 0 ? autoComplete : 'off'`) — this is a deliberate anti-autofill-noise measure, confirmed by its own dedicated bugfix commit (`be57166ce [otp field] Avoid password manager bubbles after first input (#4868)`): without this, browsers/password managers could show an autofill suggestion bubble on *every* slot, which would be visually noisy and confusing for a multi-box layout. The same reasoning applies to `maxLength`, which is likewise only set on slot 0 (`index === 0 ? length : undefined`) — explicitly to avoid triggering password-manager bubble UI on later slots, per an inline source comment: "Only the first slot has a max length to avoid password manager bubbles appearing after later inputs."
- **Open a11y-adjacent issues / honesty notes**: [G] A targeted search for open upstream a11y-labeled issues specific to OTP Field was not performed within this brief's Tier-2 lean-plus budget (already spent on the founding/unmark-preview PRs) — unlike Select/Number Field/Slider's briefs, no explicit "searched `label:accessibility` for otp-field, found none" sweep was logged, so absence of known open a11y bugs here should be read as "not yet searched," not "confirmed clean." [E] The one directly-verified, genuine a11y nuance worth flagging prominently on a future docs page is the asymmetric-labeling contract (§5, this section) — it is easy to misapply (labeling only slot 0, or labeling every slot identically) without reading source/tests closely, and the dev warning that catches *one* misuse (aria-label on slot 0) does not catch the inverse mistake (forgetting aria-label on slots 1..N-1 entirely — no warning fires for that omission, confirmed by the absence of any such warning in `OTPFieldInput.tsx`).

## 8. Prop-level guidance (decision-relevant props)

- [E] **`length` (Root, required, no default)** — the single most load-bearing prop in this component (§2); must exactly match the number of rendered `<OTPField.Input>` parts, enforced by a dev-only warning. Drives validation pattern generation, hidden-input form integration, completion detection, and SSR determinism.
- [E] **`validationType`** (default `'numeric'`; `'alpha'` | `'alphanumeric'` | `'none'`) — selects a built-in character-filtering regexp and a matching `inputMode` hint (`'numeric'`→`inputMode="numeric"`, `'alpha'`/`'alphanumeric'`→`inputMode="text"`); `'none'` disables built-in filtering entirely, deferring fully to `normalizeValue` (or accepting anything) — use `'none'` specifically when a custom `normalizeValue` should be the sole source of validation truth rather than being layered on top of a built-in preset (docs "Custom normalization": "Use `validationType='none'` when the normalizer should provide the full validation rule").
- [E] **`normalizeValue`** (function, no default) — runs *after* built-in `validationType` filtering, and its output is filtered by `validationType` again afterward (so a custom normalizer that, say, uppercases input still gets re-checked against the alphanumeric character set if `validationType="alphanumeric"` is also set); must be idempotent, since the pipeline may apply it more than once across different code paths (typing, pasting, autofill, controlled-value normalization) — the JSDoc explicitly warns: "Non-idempotent normalizers can compound across those normalization passes."
- [E] **`autoSubmit`** (default `false`) — submits the owning form via `requestSubmit()` immediately after `onValueComplete` fires, once all slots are filled; independent of `onValueComplete` itself (§2). Interacts carefully with Base UI's own `Form`/`Field` validation: if a *different* field in the same `Form` is currently invalid, auto-submit is blocked and focus moves to the first invalid field instead (tested: "keeps focus on the first invalid field when auto-submit is blocked"), and the blocking check is implemented without calling `flushSync` inside a layout effect (two dedicated tests confirm this specifically: "does not call flushSync inside a layout effect when auto-submitting a Base UI Form," "blocks invalid Base UI Form auto-submit without calling flushSync inside a layout effect" — evidence of a specific React-concurrency correctness concern the maintainers took care to avoid).
- [E] **`mask`** (Root, default `false`) — switches every slot's native `type` to `"password"`; can be overridden per-slot via an explicit `type` prop directly on individual `<OTPField.Input>` parts (§6).
- [E] **`inputMode`** (optional override) — built-in `validationType` presets provide a sensible default (`'numeric'` or `'text'`), but this prop lets a consumer override it when needed (e.g. forcing a full keyboard even for a numeric code, or vice versa); applied to slot inputs and the hidden validation input alike.
- [E] **`autoComplete`** (default `'one-time-code'`) — applied only to the first slot and the hidden validation input, deliberately excluded from later slots to prevent redundant/noisy autofill-suggestion bubbles (§7); overriding it (e.g. to `'off'`) suppresses the SMS/password-manager-style OTP autofill hint entirely.
- [E] **`onValueChange` / `onValueInvalid` / `onValueComplete`** — three genuinely distinct callbacks, not overlapping names for the same event (§6): `onValueChange` for any committed value change, `onValueInvalid` for rejected/filtered characters (fires independent of whether the value net-changes), `onValueComplete` for reaching full length (fires even for a same-length replacement paste over an already-complete value). Document all three together since their triggers are easy to conflate.
- [E] **`disabled` / `readOnly` / `required`** — `disabled` merges with Field's own disabled state (mirroring Number Field/Slider's identical `fieldDisabled || disabledProp` pattern); `readOnly` blocks value mutation (typing/paste/Backspace/Delete) while explicitly preserving arrow/Home/End navigation (§6) and hidden-input changes are ignored entirely while `readonly`/`disabled` (`02240e7c8`, §2); `required` feeds native constraint validation via the hidden input's `required` attribute.
- [E] **`name` / `form`** — standard hidden-input form-participation props, targeting an external `<form>` by id when the component isn't rendered inside one, matching the same convention documented for Number Field and Slider.
- [E] **Data attributes that matter for styling** — two separate `*DataAttributes.ts` enums exist (Root and Input), sharing most entries but each with one part-specific addition:
  - **Root** (`OTPFieldRootDataAttributes`): `data-complete` (all slots filled), `data-disabled`, `data-readonly`, `data-required`, plus the Field-integration set `data-valid`/`data-invalid`/`data-touched`/`data-dirty`, `data-filled` (at least one character anywhere in the value), `data-focused` (any slot currently focused).
  - **Input** (`OTPFieldInputDataAttributes`): the same `data-complete` (reflecting the *whole field's* completion state, not just this slot — confirmed directly, `getOTPFieldInputState` in `OTPFieldRootContext.ts` spreads the shared Root state onto every slot's per-input state) plus a **slot-specific** `data-filled` meaning (present when *this individual slot* contains a character — distinct from Root's `data-filled`, which means "at least one character anywhere in the whole value"). This is a genuine, easy-to-miss naming collision worth flagging explicitly: `data-filled` means two different things depending on whether it's read on Root vs. an individual Input, and a consumer writing a single shared CSS selector for `[data-filled]` across both levels could get confused results if they don't realize the semantic differs by part.
  - No CSS custom properties exist in the styling contract at all (§1).
- [E] **Form wiring — the hidden validation input's role**: carries `pattern` (derived from `validationType`+`length`, e.g. `\d{6}` for a 6-digit numeric code), `minLength`/`maxLength` (both set to `length`), `required`, `name`, `form`, `autoComplete`, `inputMode` — this is the element real password-manager/SMS autofill targets, and its own `onChange` handler re-splits any bulk-filled value back across the visible per-slot state.

## 10. Pitfalls & FAQ

- **"Why does my sidebar still show `[New]` if OTP Field is out of preview?"** → intentional, not a stale badge: the same PR that removed the `[Preview]` tag (#5029) explicitly replaced it with `[New]`, per the PR's own stated change list. `[New]` here means "recently added to the library," not "still experimental" — the two badges are semantically distinct even though both can look similarly attention-grabbing in a sidebar. (mui/base-ui#5029). [E]
- **Import breaks after upgrading past v1.6.0** → the namespace export was renamed `OTPFieldPreview` → `OTPField` as an explicit breaking change in the same de-preview PR; update `import { OTPFieldPreview as OTPField } from '@base-ui/react/otp-field'` to `import { OTPField } from '@base-ui/react/otp-field'` (mui/base-ui#5029). [E]
- **Dev warning: "`length` must match the number of rendered `<OTPField.Input />` parts"** → the `length` prop and the actual count of rendered `Input` children have drifted (e.g. conditionally rendering fewer/more slots than `length` specifies); fix by making the two consistent, since `length` — not rendered-child-count — is the authoritative source of truth for validation/forms/SSR (mui/base-ui#4365 body). [E]
- **First slot's `aria-label` seems to be silently ignored** → intentional and warned-about: slot 0 is meant to inherit the field's shared accessible name from an external `<label>`/`Field.Label`, not carry its own `aria-label`; the dev-only warning "`<OTPField.Input>` ignores `aria-label` on the first input" fires specifically to catch this. Use `aria-label` only on slots 1..N-1. [E]
- **Every slot shows an autofill/password-manager suggestion bubble** → this was a real, fixed bug (`be57166ce`), not an unavoidable browser quirk; the fix restricts both `autoComplete` and `maxLength` to slot 0 only, with every later slot explicitly set to `autoComplete="off"` and no `maxLength`. If a custom build somehow re-adds `autoComplete`/`maxLength` to later slots directly via `render`/prop overrides, the bubble-spam behavior could reappear — don't override these on non-first slots. [E]
- **Backspace on an empty (non-first) slot unexpectedly deletes a *different*, earlier slot** → intentional cascading-delete UX, not a bug: Backspace on an empty slot deletes the nearest filled slot to its left and moves focus there, matching the common OTP-input convention of "eating backward" through empty slots. If a stricter "only delete this exact slot" behavior is wanted, it isn't currently configurable — this is fixed component behavior. [E]
- **`onValueComplete` doesn't seem to fire when I expected, or fires "again" for what looks like the same value** → check the specific trigger: it fires on reaching full length via typing OR pasting, and it *does* fire again for a different complete value pasted over an already-complete one, but correctly does *not* re-fire for a no-op re-application of the identical complete value (e.g. certain controlled-value update patterns) — the distinction is same-value-no-op (suppressed) vs different-value-replacement (fires), not complete-vs-incomplete. [E]
- **`autoSubmit` doesn't submit even though all slots look filled** → check for another invalid field in the same `Form`/`Field.Root` tree; auto-submit is blocked library-wide when any sibling field fails validation, and focus is redirected to that invalid field instead — this isn't a bug in OTP Field specifically, it's the Form-level validation gate applying. [E]
- **Masked slots don't behave like unmasked ones when queried by testing tools** → `mask` sets native `type="password"` per slot, which may not expose the same accessible role (`textbox`) that unmasked slots do in all browsers/AT combinations — if writing tests or accessibility audits against a masked OTP Field, verify role expectations directly rather than assuming `getByRole('textbox')` will find masked slots the same way it finds unmasked ones. [I, flagged as a nuance to verify independently, §7]
- **No dedicated "OTP Field vs X" comparison exists** → unlike Select/Combobox/Autocomplete or Toggle/Switch/Checkbox, OTP Field doesn't have an established sibling/overlap boundary within Base UI's own component set; there's no cluster note to consult because none of the mandatory overlap clusters name this component (§4).
