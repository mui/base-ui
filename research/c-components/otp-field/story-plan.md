# OTP Field — story plan

10 stories. Derived from `brief.md` evidence (source, tests, docs demos, git history, #4365/#5029 PR bodies). Styling: follow the docs hero demo CSS (CSS Modules, raw oklch values); import from `@base-ui/react/otp-field` as `{ OTPField }` (post-#5029 stable export name, never `OTPFieldPreview`).

## 2. One story per use case

Columns: name — renders — controlled? — play? — doc section served.

1. **Basic** — hero recreation: 6 numeric slots, native `<label htmlFor>` on slot 0, `aria-label` on slots 2-6, `aria-describedby` pointing at a description paragraph — uncontrolled — no — Hero / Identity / Anatomy (§1, §5).
2. **FullCodeEntry** *(required interaction story — full code-entry play)* — Basic + play: type six digits one at a time into the first slot area (`user.keyboard` targeting the currently-focused slot after each keystroke, since focus auto-advances) → assert each slot fills in turn, focus lands on the final slot after the sixth digit, and `onValueComplete` fires exactly once with the full value and reason `input-change` — uncontrolled — **play** — Behavior §6 (typing/auto-advance) / API contract (`onValueChange`/`onValueComplete`).
3. **PasteCompletesCode** *(required interaction story — paste play)* — Basic (or a fresh empty instance) + play: dispatch a paste event with a full 6-character clipboard payload onto the first slot → assert all six slots fill in one operation, `onValueComplete` fires with reason `input-paste`, and focus lands on the final slot. A second play step pastes a *different* complete code over the now-complete field and asserts `onValueComplete` fires again with the new value (brief §6's "replaces a complete OTP" contract) — uncontrolled — **play** — Behavior (paste distribution) / Pitfalls (§10's onValueComplete re-fire nuance).
4. **BackspaceAcrossSlots** *(required interaction story — backspace-across-slots play)* — a partially-filled instance (e.g. `defaultValue="123"` in a 6-slot field, focus on slot 4, the first empty one) + play: press Backspace once — assert it cascades to clear slot 3 (the nearest filled slot to the left) and moves focus there, per the brief §6 cascading-delete contract; a second play step demonstrates Delete on a filled slot clearing in place *without* moving focus, for contrast; a third step demonstrates Ctrl/Cmd+Backspace clearing the entire value and returning focus to slot 1 — uncontrolled — **play** — Behavior (Backspace/Delete state machine, the single most distinctive and easy-to-misread interaction in this component) / A11y keyboard table.
5. **FormSubmitWithAutoSubmit** — a `<form>` containing `<OTPField autoSubmit name="otp" length={6} />` plus a second, deliberately-invalid sibling `Field` (mirroring the brief §8/§9 auto-submit-blocked-by-sibling-invalid-field contract) and a submit-log readout — uncontrolled — **play**: (a) fill the OTP in a form with no other invalid fields, assert `requestSubmit()` fires automatically; (b) repeat with the invalid sibling field present, assert submission is blocked and focus moves to the invalid sibling instead of submitting — Behavior (forms) / Prop guidance (`autoSubmit`) / Pitfalls.
6. **InvalidCharacterFeedback** — `validationType="numeric"` field with an inline rejected-character message driven by `onValueInvalid`, recreating the docs' `useInvalidFeedback` pattern from `demos/custom-sanitize` — uncontrolled — **play**: type a letter into a numeric-only field, assert the letter is rejected (slot stays empty/unchanged) and the invalid-feedback message appears, driven by `onValueInvalid` firing with the raw rejected string — Behavior (`onValueInvalid`) / Prop guidance.
7. **MaskedVariant** — password demo recreation (`mask` prop, 6 slots) alongside an unmasked sibling for contrast — uncontrolled — play (type a code; assert masked slots render `type="password"` while the unmasked sibling shows `type="text"`; include an honest inline note that masked-slot accessible-role exposure should be independently verified with a real screen reader per brief §7's flagged nuance) — Anatomy / Prop guidance (`mask`) / A11y (documented uncertainty, not overclaimed).
8. **GroupedWithSeparator** — grouped-layout recreation: two 3-slot groups wrapped in plain `<div>`s with an `<OTPField.Separator>` between them — uncontrolled — no — Anatomy (arbitrary DOM nesting / CompositeList order-tracking, §5).
9. **LabelingContract** — a didactic side-by-side: one correctly-labeled instance (native label on slot 0 only, `aria-label` on slots 2-N) and, in a code comment (not live-rendered, to avoid triggering the real dev warning noisily in Storybook's own console), a description of the incorrect pattern the dev warning catches (`aria-label` mistakenly placed on slot 0) — uncontrolled — play (inspect via `preview_inspect`-equivalent assertions that slot 0 has no `aria-label` attribute while later slots do) — A11y contract (the asymmetric-labeling finding, brief §5/§7 — the most surprising a11y fact in this brief) / Pitfalls.
10. **DisabledAndReadOnly** — side-by-side `disabled` and `readOnly` 6-slot instances, both pre-filled with a partial value — uncontrolled — play: assert `disabled` blocks all keyboard handling including vertical-arrow navigation (brief §6's "leaves vertical arrow navigation unhandled" contract) and removes native focusability; assert `readOnly` blocks Backspace/Delete/paste from changing the value while explicitly preserving ArrowLeft/ArrowRight/Home/End navigation (brief §6's readOnly-preserves-navigation contract) — A11y contract / Prop guidance / Behavior.

## Honesty notes carried from the brief

- Story 1's title bar/nav badge context is worth a one-line MDX callout, not a story itself: the docs sidebar shows `[New]` for this component even though it is fully stable (unmarked from Preview in v1.6.0, #5029) — this is a deliberate maintainer choice recorded directly in the #5029 PR body, not a stale badge, and should be stated plainly rather than silently reproduced without comment.
- Story 7 (masked variant) must not assert a specific accessible-role query (`getByRole('textbox')`) will reliably find masked slots across all browser/AT combinations — the brief flags this as an [I] nuance requiring independent screen-reader verification, not a confirmed fact; the story's play function should verify `type="password"` directly via attribute assertion rather than implying a settled a11y claim.
- No story should invent a "PIN input" or "segmented input" framing beyond OTP/verification codes — the current docs and all six existing demos are scoped specifically to one-time-password/verification-code entry (brief §3), and no maintainer statement broadens this to a generic segmented-text-input use case.
- The current docs directory has exactly six demos (hero, alphanumeric, grouped, focused-placeholder, custom-sanitize, password), confirmed by listing `docs/src/app/(docs)/react/components/otp-field/demos/`. Stories 2-5, 9, and 10 are evidence-backed net-new Storybook coverage derived from source/tests/git history, not ports of existing docs content, and should be labeled as such.
- Real paste-event dispatch in jsdom/Testing Library requires the same manual `clipboardData`-property-definition technique the project's own tests use (`OTPFieldInput.test.tsx`'s `pasteText`/`pasteWithError` helpers) — Playwright/Testing Library's higher-level paste helpers may not be sufficient; story 3's play function should follow the project's own tested technique rather than assuming a simpler API works.
