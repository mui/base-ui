# Toggle Group — component research brief

Tier 3 (lean brief). Mined 2026-07-07 from source, docs, tests, git history, and 2 targeted `gh pr view` lookups (introducing PR #763, shared with Toggle; Home/End PR #3971). Evidence tags: [E] direct evidence, [I] inference, [G] gap.

## 1. Identity

- **Name / slug**: Toggle Group — `@base-ui/react/toggle-group`. Single-part component at the export surface (no namespace of sub-parts — `ToggleGroup` is used directly, its "items" are plain `Toggle` components from the sibling package); generic over a string `Value` type. [E] `packages/react/src/toggle-group/index.ts`, `ToggleGroup.tsx`.
- **Status**: stable since v1.0.0; no `[New]`/`[Preview]` tag. Introduced jointly with Toggle in a single PR (§2). [E]
- **Purpose / IA statement**: "Provides a shared state to a series of toggle buttons" [E] (`ToggleGroup.tsx` JSDoc; docs Subtitle verbatim). Taxonomy: **actions** cluster, and the composite-roving-focus sibling of Toolbar (both wrap `CompositeRoot`). Jointly with Toggle, part of the **binary-controls** decision cluster (see [`research/c-components/_clusters/binary-controls.md`](../_clusters/binary-controls.md)) for the single-select-mode boundary against RadioGroup.
- **Parts**: none exported separately — the "items" of a ToggleGroup are ordinary `Toggle` components, not a `ToggleGroup.Item`. This is a deliberate anatomy choice: composition over a bespoke item part (see §5).
- **Props surface**: `value`/`defaultValue`/`onValueChange` (array-valued controlled triplet), `disabled`, `orientation`, `loopFocus`, `multiple`, plus inherited `render`/`className`/`style`. [E] `ToggleGroupProps`, `ToggleGroup.tsx`.

## 2. Intention

- [E] **Introducing PR**: mui/base-ui#763 "[Toggle][ToggleGroup] Create `Toggle` and `ToggleGroup`", closing legacy issues #11 and #31 — the same PR that introduced Toggle (`research/c-components/toggle/brief.md` §2). ToggleGroup was never designed as a bolt-on to a pre-existing Toggle; the two shipped as one coherent feature.
- [E] **Design pole: the ARIA *toggle-button group* pattern (`aria-pressed` + deselectable), explicitly not RadioGroup's exclusive-choice pattern.** The clearest maintainer articulation is mj12albert's answer in mui/base-ui#3525 (open, per `discussions.md`): "ToggleGroup implements the toggle-button pattern (`aria-pressed`, deselectable); a single-select-you-can't-clear widget *is* a RadioGroup; making a toggle announce as a radio is semantically wrong." [E] Also `research/b-library-principles/principles.md` B-P30, which frames this as one of the library's clearest examples of "ARIA semantics draw the component boundaries."
- [E] **Toolbar-detection is a first-class architectural feature, not an afterthought.** `ToggleGroup.tsx` calls `useToolbarRootContext(true)` and `useToolbarGroupContext(true)` (both optional) at the top of the component, and branches its entire rendering strategy on whether a `ToolbarRootContext` is present: inside a Toolbar, ToggleGroup renders as a plain `useRenderElement('div', ...)` (`enabled: Boolean(toolbarContext)`) and *defers* composite/roving-focus/keyboard behavior entirely to the hosting Toolbar's own `CompositeRoot`; standalone, it wraps itself in its own `<CompositeRoot>` with `enableHomeAndEndKeys` and its own `loopFocus`/`orientation` (`ToggleGroup.tsx` L111-137). This directly matches the toolbar docs' documented composition pattern (`docs/.../toolbar/demos/hero/` nests `<ToggleGroup>` directly inside `<Toolbar.Root>` for the alignment buttons) and is exactly the kind of context-sharing the AGENTS.md rule prescribes ("if two components can share logic... define the logic/handlers in the parent and share it through a context"). [E]
- [E] **Explicit disabled-inheritance chain**: a ToggleGroup's effective `disabled` state is the OR of its own `disabled` prop, a hosting `ToolbarRootContext.disabled`, and a hosting `ToolbarGroupContext.disabled` (`ToggleGroup.tsx` L56-57) — ToggleGroup is designed to compose transparently into a Toolbar's disabled-cascade rather than needing its disabled state wired manually at every level.
- [I] `multiple` (default `false`) exists so ToggleGroup can serve both "exclusive choice you can still clear entirely" (single mode — e.g. text alignment, where no alignment can also be valid) and "independent multi-press" (multiple mode — e.g. bold/italic/underline, all independently toggleable) from one component, following the library-wide convention of one `multiple` prop rather than a separate component per cardinality (`principles.md` B-P40, same convention family as Accordion/Select). [E] the docs' only worked "Examples" subsection is literally titled "Multiple" (`docs/.../toggle-group/page.mdx`).

## 3. When to use

- [E] A row/column of related, independently-styleable toggle buttons that need **shared pressed-state bookkeeping** — e.g. text alignment (left/center/right, single mode — the hero demo), or text formatting (bold/italic/underline, multiple mode — the docs "Multiple" example and the `docs/src/app/(private)/experiments/toggle-group.tsx` RTL demo).
- [E] Whenever the "segmented control" visual pattern is wanted with correct ARIA (`aria-pressed` per item, `role="group"` container) and roving keyboard focus for free — the docs metadata keywords explicitly include "Segmented Control" [E] (`page.mdx` metadata export).
- [E] As a hostable child of **Toolbar**, when a command bar needs an exclusive/multi-toggle cluster alongside plain buttons, links, menus, and inputs (`docs/.../toolbar/demos/hero/` — Alignment ToggleGroup sits directly next to a currency-format ToolbarGroup and a font Select in the same toolbar).
- [I] Any case where RadioGroup's exclusive-and-you-can't-clear-it semantics would be wrong but a segmented-control look is desired — i.e., when "none selected" must remain a reachable, valid state even in single-selection mode.

## 4. When not to use + alternatives

Full comparative guidance: [`research/c-components/_clusters/binary-controls.md`](../_clusters/binary-controls.md).

- [E] **Exclusive choice that must always have exactly one selection and can't be cleared → RadioGroup**, not single-mode ToggleGroup. This is the maintainer-drawn boundary from mui/base-ui#3525 (§2): if deselecting to "nothing selected" should be impossible, the widget's correct ARIA identity is a radio group, not a toggle-button group — "making a toggle announce as a radio is semantically wrong" runs in both directions (don't fake a radio's exclusivity contract with ToggleGroup, and don't fake ToggleGroup's deselectability with a radio group either). [E]
- [E] **A single independent action bar with no shared pressed-state → Toolbar**, or if the items are truly unrelated plain buttons, no wrapper at all (per Button's brief §4, `role="group"` around plain `<button>`s suffices — mui/base-ui#4008). ToggleGroup specifically earns its keep when items share *pressed-state* bookkeeping; if nothing needs to track "which ones are pressed," it's overhead.
- [I] **A binary immediate-effect on/off setting, singular (not a row of related toggles) → Switch**, not a one-item ToggleGroup. ToggleGroup's value is inherently an array — using it for a single boolean is the wrong cardinality; reach for `Toggle` alone (no group) or `Switch` (see the binary-controls cluster).
- [I] **Values that must submit with a native `<form>` → CheckboxGroup**, not ToggleGroup — like standalone Toggle (`research/c-components/toggle/brief.md` §4), ToggleGroup inherits Toggle's lack of hidden-input form participation; there is no evidence in source of a hidden input anywhere in `ToggleGroup.tsx`. [G] no maintainer statement found contrasting ToggleGroup vs CheckboxGroup directly — inferred from the absence of form-integration code, same caveat as Toggle's brief §4.

## 5. Anatomy & composition

- [E] **No dedicated item part** — the docs anatomy is literally `<ToggleGroup />` with no children shown (`page.mdx` §Anatomy), but every worked example nests plain `<Toggle value="...">` elements directly (hero demo, "Multiple" demo). This is a deliberate anatomy minimalism: ToggleGroup is a *context provider* over ordinary Toggles, not a compound namespace like Select/Menu with an `Item` sub-part. [I] inferred from the consistent absence of a `ToggleGroup.Item` export across `index.ts`/`index.parts.ts`-equivalent (ToggleGroup has no `index.parts.ts` at all — it's flat, single-export, unlike Toolbar).
- [E] **Dual rendering strategy, keyed off Toolbar presence** (§2): standalone, `ToggleGroup` wraps its own `<CompositeRoot>` (with `enableHomeAndEndKeys`, its own `orientation`/`loopFocus`); inside a `Toolbar.Root`, it renders a plain `<div role="group">` and relies entirely on the *hosting* Toolbar's single `CompositeRoot` for roving focus — meaning a Toolbar-hosted ToggleGroup's keyboard navigation is governed by Toolbar's own `enableHomeAndEndKeys` setting (which, per `research/c-components/toolbar/brief.md` §6, is **not** enabled — Toolbar never passes `enableHomeAndEndKeys` to its `CompositeRoot`). **This means Home/End behave differently for a ToggleGroup depending on whether it's used standalone (works) or nested inside a Toolbar (does not, because the composite root that actually owns the keys is Toolbar's, not ToggleGroup's).** [E] This is a genuine, source-verified behavioral asymmetry worth flagging prominently rather than glossing over.
- [E] **`ToggleGroupContext`** (`ToggleGroupContext.ts`) is the mechanism `Toggle` reads to detect group membership: `value` (readonly array), `setGroupValue`, `disabled`, `orientation`, `isValueInitialized` (used only for the dev-mode warning described in Toggle's brief §6). Calling any `ToggleGroup`-specific hook outside a `<ToggleGroup>` throws a `Base UI:` error naming the requirement (`useToggleGroupContext` with `optional=false`) — though in practice `Toggle` always calls it with the default optional signature, since a standalone Toggle is fully valid.
- Text spec for a future anatomy diagram: (1) the ToggleGroup container (`role="group"`, no visible chrome of its own), (2)-(n) plain Toggle children in a row/column, each independently styled — no numbered internal sub-parts beyond the children themselves.

## 6. Behavior ("How it works")

- [E] **Controlled/uncontrolled**: `value`/`onValueChange`/`defaultValue` triplet, all array-valued (`readonly Value[]`) regardless of `multiple` — even single-select mode represents its pressed state as a 0-or-1-length array (`ToggleGroup.tsx` `useControlled` default `EMPTY_ARRAY`). [E]
- [E] **Value-commit logic** (`setGroupValue`, `ToggleGroup.tsx` L66-92): in `multiple` mode, pressing a Toggle pushes/splices its value into the existing array; in single mode (`multiple=false`), pressing replaces the entire array with either `[newValue]` (pressed) or `[]` (unpressed) — meaning single mode *is* deselectable to empty, the exact property that distinguishes it from RadioGroup per §2/§4.
- [E] **Cancellation runs through the same shared `eventDetails` as the pressing Toggle** — `onValueChange` is called from inside `setGroupValue`, which is itself invoked from `Toggle`'s own `onClick` *before* `Toggle` commits its own local pressed state; canceling in either `Toggle.onPressedChange` or `ToggleGroup.onValueChange` prevents the whole cascade from committing (cross-referenced fully in `research/c-components/toggle/brief.md` §6, pinned by `Toggle.test.tsx` "canceling in a grouped Toggle prevents the group value from changing").
- [E] **Keyboard navigation, orientation, and RTL** (`CompositeRoot`/`useCompositeRoot`, exercised via `ToggleGroup.test.tsx` "keyboard interactions" parametrized suite over `[direction, orientation, nextKey, prevKey, ...]`): arrow keys move roving focus among items; the *active* arrow pair flips with `orientation` (`horizontal` → ArrowRight/ArrowLeft; `vertical` → ArrowDown/ArrowUp) and with `dir="rtl"` (horizontal RTL flips ArrowRight/ArrowLeft's meaning); the *inactive* axis's arrow keys are explicitly ignored (test parametrizes `ignoredNextKey`/`ignoredPrevKey` too, confirming cross-axis keys are no-ops, not accidental focus movers) — this exact cross-axis guard was itself a bug fix, see §9 (#4488).
- [E] **`loopFocus` (default `true`)**: arrow-key navigation wraps from the last item back to the first (and vice versa) when reaching an end; disable by setting `loopFocus={false}` if wrap-around navigation is undesired. [E] prop JSDoc, `ToggleGroupProps`.
- [E] **Home/End key navigation**: `enableHomeAndEndKeys` is passed unconditionally to `CompositeRoot` when ToggleGroup renders standalone (`ToggleGroup.tsx` L133) — Home moves focus to the first item, End to the last, confirmed by `ToggleGroup.test.tsx` "Home key moves focus to the first item"/"End key moves focus to the last item." **This was not part of the original 2024 design** — it shipped later as a dedicated PR (§9), meaning any ToggleGroup usage recorded/screenshotted before that PR would not have had Home/End support.
- [E] **Disabled cascade**: a `disabled` ToggleGroup disables every child Toggle without needing `disabled` on each one (§2, confirmed by `ToggleGroup.test.tsx` "can disable the whole group"); individual Toggles can still be disabled independently (`ToggleGroup.test.tsx` "can disable individual items").
- [E] **Falsy/omitted `value` tolerance**: Toggles with `value=""` or no `value` at all still function correctly for click-based interaction (`ToggleGroup.test.tsx` "when Toggles omit value" — both uncontrolled and multiple variants), thanks to Toggle's own `useBaseUiId(valueProp || undefined)` fallback (`research/c-components/toggle/brief.md` §8) — though the dev-mode warning (§2) still fires if the group's value was explicitly initialized.
- [E] **`role="group"` carries no `aria-orientation`** — a dedicated fix (`ToggleGroup.test.tsx` "does not render aria-orientation on role='group'"; git commit `01ba3f566` "[toggle group] Remove invalid `aria-orientation` from `role='group'` element (#4628)") removed an invalid ARIA attribute; `data-orientation` (a styling hook, not ARIA) is still present.

## 7. Accessibility contract

- [E] **Pattern**: closest to the WAI-ARIA APG **Toolbar** pattern's roving-tabindex mechanics (single tab stop, arrow-key movement) layered under a `role="group"` container rather than `role="toolbar"` — ToggleGroup is not itself a toolbar (no `aria-orientation` even, per §6), but shares its composite roving-focus engine (`CompositeRoot`) with Toolbar. No single, distinct "toggle-group" APG pattern exists; the closest named pattern for the item-level semantics is the **toggle button** (https://www.w3.org/WAI/ARIA/apg/patterns/button/#toggle_button, same reference as Toggle's brief §7 — not fetched, flagged [I]/general knowledge).
- [E] **Keyboard interaction table** (evidence: `ToggleGroup.tsx`, `CompositeRoot`/`useCompositeRoot.ts`, `ToggleGroup.test.tsx`):

  | Key | Context | Action |
  |---|---|---|
  | Tab | entering the group | Moves focus to exactly one item (the roving tab stop) — **single tab stop for the whole group**, not one per Toggle |
  | ArrowRight / ArrowLeft | `orientation="horizontal"`, LTR | Move roving focus to next/previous item; wraps at the ends if `loopFocus` (default `true`) |
  | ArrowLeft / ArrowRight | `orientation="horizontal"`, RTL (`dir="rtl"`) | Direction flipped — Left moves to next item, Right to previous |
  | ArrowDown / ArrowUp | `orientation="vertical"` (either direction) | Move roving focus to next/previous item; direction is unaffected by RTL |
  | *(cross-axis arrows)* | either orientation | No-op — explicitly ignored, confirmed by test parametrization and by the dedicated cross-axis bug fix (#4488, §9) |
  | Home | standalone ToggleGroup | Moves focus to the first item (`enableHomeAndEndKeys`, shipped in #3971 — see §9; **not guaranteed when ToggleGroup is nested inside a Toolbar**, §5) |
  | End | standalone ToggleGroup | Moves focus to the last item (same caveat) |
  | Enter / Space | focused item | Toggles that item's pressed state (inherits Toggle's own keyboard contract, `research/c-components/toggle/brief.md` §7, including the composite Space-on-keydown dispatch) |
- [E] **ARIA managed by the component**: `role="group"` on the container (no `aria-orientation` — deliberately removed, §6); each child Toggle supplies its own `aria-pressed` (Toggle's contract). A group-level accessible name (e.g. "Text alignment") must come from the consumer via `aria-label`/`aria-labelledby` on `<ToggleGroup>` — every worked demo in this pass sets `aria-label` explicitly (hero demo: `aria-label="Text alignment"`).
- [E] **Disabled items and discoverability**: disabled Toggles are unconditionally excluded from the roving-focus tab sequence when grouped (Toggle's brief §6 — `focusableWhenDisabled: false` is hardcoded for Toggle's composite metadata), which differs from Toolbar's own default of keeping disabled composite items focusable-when-disabled — flagged as an asymmetry in Toggle's brief §6/§10, relevant here too since ToggleGroup is the composite host in the standalone case.
- **Known issues / honesty list**: [E] mui/base-ui#3525 remains open and unresolved — a live community concern that single-selection ToggleGroup has no non-visual signal distinguishing it from a "you can also select nothing" toggle group, and some users want `role="radio"` semantics that maintainers have declined to add (§2, §4). [E] The Home/End behavior gap when ToggleGroup is Toolbar-hosted (§5, §6) has no dedicated maintainer discussion found in the mined corpus — flagged here as a documentation-worthy nuance, not confirmed as a known/tracked bug — [G] searched `discussions.md`/`issues-prs.md` per-component indices for "toggle group toolbar home end", no hits.

## 8. Prop-level guidance (decision-relevant props)

- [E] **`multiple` (default `false`)** — set `true` when items are independently toggleable (bold/italic/underline); leave `false` for exclusive-with-clear (text alignment, view mode). This is the single library-wide convention prop for cardinality (B-P40) rather than a separate "MultiToggleGroup" component.
- [E] **`value`/`defaultValue`/`onValueChange`** — always array-shaped, even in single mode. Use controlled `value` when the selection must be driven or synced from outside (e.g. persisting alignment choice); prefer `defaultValue` otherwise.
- [E] **`orientation` (default `'horizontal'`)** — determines which arrow-key pair drives roving focus (§6/§7); set to `'vertical'` for a stacked segmented control. Always pair with a `data-orientation`-aware styling rule (styling contract below) rather than assuming a fixed layout direction.
- [E] **`loopFocus` (default `true`)** — set `false` if wrap-around arrow navigation at the group's ends is undesired (e.g. to signal "this is the end of the list" more strongly). Same convention name and default as Toolbar's `loopFocus`.
- [E] **`disabled`** — cascades to every child Toggle (§6); prefer this over disabling each Toggle individually when the whole group should be inert, and combine with Toolbar's own cascading `disabled` when nested (§2's OR-chain).
- [E] **Styling contract** (`ToggleGroupDataAttributes.ts`): `data-disabled` (group disabled), `data-orientation` (`horizontal`/`vertical`), `data-multiple` (present only when `multiple` is `true`). No CSS variables — no dynamic measurements to expose. Each child Toggle carries its own independent `data-pressed`/`data-disabled` (Toggle's own contract) — style selection state on the Toggle, not the group.

## 10. Pitfalls & FAQ

- **"Home/End don't move focus in my ToggleGroup, even though the docs/tests show they should"** → check whether the ToggleGroup is nested inside a `Toolbar.Root`. When hosted by a Toolbar, ToggleGroup defers its composite/roving-focus behavior entirely to the Toolbar's own `CompositeRoot`, which does not enable Home/End (see `research/c-components/toolbar/brief.md` §6) — this is a source-confirmed asymmetry, not user error. [E]/[I] (asymmetry confirmed by source; the "why" is inferred, not stated by a maintainer).
- **"Arrow keys move focus in the wrong axis / do nothing on the cross axis"** → this exact bug existed and was fixed (#4488); if still reproducible on a current version, it's a regression worth reporting, not expected behavior — the shipped contract explicitly ignores cross-axis arrow keys (§6/§7). [E]
- **"Can I make single-select mode announce as a radio group for screen readers?"** → not supported by design; maintainers consider `role="radio"` on a deselectable toggle group semantically wrong. If exclusive-and-non-clearable is truly the desired contract, use **RadioGroup** instead of trying to retrofit ToggleGroup → mui/base-ui#3525 (open). [E]
- **"My ToggleGroup's `value` prop change from an external source didn't take effect"** → verify the value is a **readonly array** even in single-select mode (`['align-left']`, not `'align-left'`) — this is a common cardinality mismatch when migrating from a scalar-value mental model (e.g. from RadioGroup or a plain `useState<string>`). [I] inferred from the array-only prop type; not a documented pitfall in the mined corpus — [G] no issue found stating this exact confusion.
- **"A disabled Toggle inside my group is completely unreachable, even with `focusableWhenDisabled`-style expectations from other composites"** → intentional and hardcoded: Toggle's composite metadata always reports `focusableWhenDisabled: false` (cross-referenced from Toggle's brief §6/§10) — unlike `Toolbar.Button`, a disabled Toggle inside a ToggleGroup cannot be made focusable-when-disabled. [E]
