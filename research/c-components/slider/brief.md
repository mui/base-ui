# Slider — component research brief

Tier 2 (lean-plus depth). Mined 2026-07-07 from source, docs, `.test.tsx` suites, git history, and 1 `gh` PR lookup (#373, shared budget with otp-field: 3 of 4 lookups used across both components). Evidence tags: [E] direct evidence, [I] inference, [G] acknowledged gap.

## 1. Identity

- **Name / slug**: Slider — `@base-ui/react/slider`. Multi-part compound component, namespace export `Slider.*`.
- **Status**: stable, no `[New]`/`[Preview]` tag on the docs page or components index [E] (`docs/src/app/(docs)/react/components/page.mdx` lists plain "Slider", contrast OTP Field's `[New]`). One of the library's oldest lineages: legacy `@mui/base` file history traces back to 2023 (`git log --follow`, earliest `a9f731874`, mui/material-ui#36873); the current component is a ground-up rewrite shipped as PR #373, "[Slider] New Slider components and hook" (mj12albert, merged 2024-07-02, commit `55ba1a807`). [E]
- **7 parts** (from `packages/react/src/slider/index.parts.ts`; one-liners from part JSDoc):
  | Part | Renders | One-liner |
  |---|---|---|
  | Root | `<div>`, `role="group"` | Groups all parts of the slider [E] |
  | Label | `<div>` | An accessible label that is automatically associated with the slider thumbs [E] |
  | Value | `<output>` | Displays the current value of the slider as text [E] |
  | Control | `<div>` | The clickable, interactive part of the slider [E] |
  | Track | `<div>` | Contains the slider indicator and represents the entire range of the slider [E] |
  | Indicator | `<div>` | Visualizes the current value of the slider [E] |
  | Thumb | `<div>` + nested `<input type="range">` | The draggable part of the slider at the tip of the indicator [E] |
- **Purpose / IA statement**: [E] "An easily stylable range input" (`docs/src/app/(docs)/react/components/slider/page.mdx` Subtitle) — a headless wrapper that behaves like a native `<input type="range">` (each Thumb nests a real one) while exposing every visual layer (Track/Indicator/Thumb) as a separately stylable part. Taxonomy: **selection & input** category, glance-and-drag numeric/range selection, distinct from Number Field's typed-and-formatted numeric entry — see `research/c-components/number-field/brief.md` §4 for the from-Number-Field-side framing of this same boundary; no dedicated `_clusters/` note exists for this pair yet [G].
- **No `*CssVars.ts` file exists for any Slider part** — confirmed directly: `find packages/react/src/slider -iname "*cssvars*"` returns nothing, and no `import.*CssVars` appears anywhere in the component tree. [E] Positioning is instead driven by **inline styles carrying ad hoc custom properties** set directly by JS (not a documented/enumerated styling contract): `SliderThumb.tsx` sets `--position` (a percentage) on the thumb wrapper when `thumbAlignment !== 'center'`; `SliderIndicator.tsx` sets `--start-position` and, for range sliders, `--relative-size` inside its computed `style` object. These are consumed internally by the same inline `style` object (e.g. `[startEdge]: 'var(--position)'`) rather than exposed for consumer overriding — a consumer cannot usefully re-theme via these variables the way Select/Popover consumers hook `--anchor-width`. This is a genuine gap relative to sibling popup components' documented CSS-var contracts (B-P14) and is called out explicitly in `research/c-components/number-field/brief.md` §8 for the analogous number-field absence; worth flagging on the future docs page as "no CSS variables — position values are computed inline."

## 2. Intention

- [E] **Introducing PR** for the current API: mui/base-ui#373, "[Slider] New Slider components and hook" (mj12albert, merged 2024-07-02). Body closes #216, #76, #77 and is an explicit, itemized break from the legacy MUI Slider: thumbs can no longer swap/be dragged past each other (the `disableSwap` prop dropped, later reintroduced differently as `thumbCollisionBehavior`, §9); a new `Control` part represents the click/touch area (previously fused with Root); `rail` → `Track` (full length) and `track` (filled bar) → `Indicator` (naming clarified so "track" unambiguously means the whole range, matching the current source); the thumb's `input` is no longer exposed as swappable-out API — it's automatically wired to the thumb `<div>`, enabling direct `:focus-visible` styling on the wrapper; **`Mark`/`marks` dropped outright** ("Marks will be reintroduced with a new API in #462" — still open, no built-in marks exist as of this mining date, §4); **`ValueLabel` dropped outright** ("This can be implemented using `Tooltip`", with a linked experiment demo) — floating value-bubble-on-drag is explicitly userland composition, not a built-in slot; `step` defaults to `1` and can no longer be `null`; `isRtl` boolean replaced by the library-wide `DirectionProvider`/`direction` convention; a new `minStepsBetweenValues` prop (default `0`) prevents thumb overlap.
- [E] **Explicit non-goals stated in the PR body**: marks/visual tick rendering (deferred to #462, open); floating value-label bubble (deferred to userland `Tooltip` composition, demoed in `experiments/slider-tooltip`); restricted/discrete-value snapping decoupled from visual marks (deferred to #412). These are first-class "when not built-in" boundaries, not oversights.
- [I] **Design pole: behave like a native range input, but with every visual layer exposed as a stylable part.** Inferred from convergent decisions: each `Slider.Thumb` nests a real `<input type="range">` (not a synthetic `div role="slider"`) so assistive technologies that only understand native range-input semantics (see §7) work automatically; `Slider.Value` renders a native `<output>` element with `htmlFor` wired to every thumb's input id (`SliderValue.tsx`), matching the native `<output for="...">` labeling pattern rather than inventing a new one; `Slider.Control` isolates the drag/click hit-area from the visual `Track`/`Indicator`/`Thumb` layering so padding-driven layout tricks (inset thumbs, larger tap targets) don't desync pointer tracking from visual position — directly the subject of #981 (§10).
- [E] **Field/Form integration was folded in as a dedicated, separately-tracked effort**, not bolted on at launch: `44f7acaa0 [slider][form][field] Integrate range sliders with Form and Field (#1929)` — multi-thumb sliders needed their own Field/Form wiring pass distinct from the single-thumb case, evidence that range-slider forms integration was a nontrivial, deliberately-scoped problem.
- [E] **TalkBack/Android accessibility was treated as a first-class, recurring investment, not a one-off fix**: a dense cluster of commits targets this specifically — `40308ba37 [slider] Fix interacting with TalkBack (#2578)` (moved thumb `:focus-visible` styling from the div to `.Thumb:has(:focus-visible)` since focus genuinely lives on the native input now), `ce22c16f6 [slider] Remove extra TalkBack focus targets (#4242)`. The test suite carries an explicit design-rationale comment (`SliderThumb.test.tsx` lines 119-125): "AT (e.g. Android Talkback) may use increase/decrease actions to interact with the slider which works on `input type="range"` via change events, but not pure ARIA implementations using `div role="slider"`. The `input` element(s) must be the only focusable element(s)." — this is the strongest direct evidence for why the component is built around real native range inputs rather than a custom ARIA widget (see §7).
- [E] Performance/correctness on pointer interaction has been an ongoing, actively-maintained concern, not settled legacy code: `8563cd37c [Slider] Position thumb based on value instead of pointer location when dragging (#1750)`, `5c11ddfb7 Fix overlapping slider thumbs stuck at min or max (#1732)`, `f9c7f24d0 Prevent range slider thumbs from being dragged past each other (#1612)`, `a32d7bb02 Correct thumb positioning when control has padding (#1661)`, and a 2026 cluster (`79a0755d4 Fix assorted issues (#5097)`, `4c33d7427 Fix touchend listener accumulation leak (#5070)`, `ec358d2b8 Fix interaction edge cases (#4937)`, `481ddf74a Fix keyboard floating-point rounding (#4548)`) — pointer/touch/keyboard interaction correctness is treated as continually-invested surface area, mirroring Number Field's parsing/formatting correctness cluster (`research/c-components/number-field/brief.md` §2).

## 3. When to use

- [E] A continuous or discretely-stepped numeric range where the current position **relative to its bounds** is the primary thing being communicated and adjusted — volume, brightness, price-range filtering, playback scrubbing (docs Subtitle "an easily stylable range input"; the entire Control/Track/Indicator/Thumb visual grammar exists to show fill-relative-to-range at a glance).
- [E] Single-value selection from a bounded range via drag, click-on-track, or keyboard, with an optional visible numeric readout (`Slider.Value`) — the default single-thumb case (docs Anatomy; hero demo).
- [E] Range (two-or-more-thumb) selection where each thumb represents one edge of a selected span — filtering UIs, price ranges — supported natively via an array `value`/`defaultValue` (docs "Range slider" example; `thumbCollisionBehavior` controls collision semantics, §8).
- [I] Contexts where a native-range-input-compatible accessibility tree matters more than exact typed precision — inferred from the TalkBack-driven native-`input[type=range]` design (§2, §7); contrast with Number Field, whose entire pipeline (Intl.NumberFormat formatting, typed entry, dual hidden+visible inputs) exists specifically for typed numeric precision and locale-aware display (`research/c-components/number-field/brief.md` §2-§3) rather than glanceable position.
- [E] Forms needing a numeric value with native constraint participation — `name`/`form` props wire directly into native form submission via the nested range input(s) (docs "Form integration" example; §6).
- [E] Vertical orientation for stacked layouts (EQ-style controls, sidebars) — `orientation="vertical"` is a first-class, tested prop, not an afterthought (docs "Vertical" example; dedicated RTL+orientation test matrix, §7).

## 4. When not to use + alternatives

- [I] **Typed, formatted, or locale-sensitive numeric precision → Number Field.** From the Slider side of the boundary (the from-Number-Field-side framing lives in `research/c-components/number-field/brief.md` §4, written by a parallel agent covering that component): Slider has no `Intl.NumberFormat`-based formatting pipeline, no locale-aware parsing, and its `Slider.Value` output is a plain joined-string readout (`formatNumber`/`valueToPercent`), not an editable typed field — a user cannot type an exact value into a Slider; they can only drag, click-the-track, or use keyboard step/Home/End/PageUp/PageDown. Where an exact number (a price, a quantity, a currency amount) must be typed, edited digit-by-digit, or displayed with currency/unit/percent formatting, Number Field's dedicated format/parse pipeline is the fit; Slider's entire visual grammar (Track/Indicator/relative-fill) is built for **glance-and-drag positioning on a continuum**, not precise value entry [I, unless a maintainer statement is found stating this explicitly — none was in the mined corpus, this is derived from the two components' divergent design poles per §2].
- [E] **Built-in tick marks or restricted/snap-to-marks values → not currently available.** The legacy `Mark`/`marks` API was deliberately dropped in the #373 rewrite and explicitly deferred to a future, still-open issue (#462, "slider marks" — cited as actively-desired new scope in `research/b-library-principles/_mining/issues-prs.md` line 219's upvote-driven backlog). Today, visual marks must be hand-composed as plain `<span>` children positioned inside `Slider.Track` (demonstrated in `docs/src/app/(private)/experiments/slider/slider.tsx`'s `DelayUntilRepeat` function, which renders six manually-placed `.Mark` spans) — this is real, evidenced userland composition, not speculation, but it means "Slider with marks" is not an out-of-the-box capability.
- [E] **A floating value-label bubble that tracks the thumb while dragging → compose with Tooltip, not built in.** The legacy `ValueLabel` slot was dropped in #373 with an explicit pointer to a Tooltip-based experiment (`experiments/slider-tooltip`) as the intended replacement pattern.
- [I] **Binary or tiny (2-4) fixed option sets → Toggle Group / Radio Group.** No maintainer statement found comparing Slider to these directly [G]; derived purely from API-affordance overhead (a drag gesture and a continuous Track/Indicator visual are unnecessary machinery for a handful of discrete named options).
- [E] **A rating widget out of discrete unlabeled steps (e.g., 1-5 stars) is technically buildable on Slider** (`step`/`largeStep`/`min`/`max` support small integer ranges) but no maintainer guidance endorses this pattern over a purpose-built radio-based rating control [G] — flagged as an open question rather than asserted either way.

## 5. Anatomy & composition

- [E] Canonical tree (docs Anatomy):
  `Root > (Label, Value, Control > Track > (Indicator, Thumb))`.
- [E] **Minimal composition is `Root > Control > Track > (Indicator, Thumb)`** — Label and Value are optional. A single unlabeled `Root` renders without them, but then the Thumb must carry its own `aria-label` (docs "Labeling a slider" — the unlabeled-volume-control example composes only `Control > Track > (Indicator, Thumb aria-label="Volume")`).
- [E] **Control vs Track is a semantics-vs-visuals split, confirmed by direct maintainer answer** (mj12albert/colmtuite, mui/base-ui#981, `research/b-library-principles/_mining/discussions.md` lines 83-87): `Control` is the tracked, clickable/draggable hit area (pointer/touch listeners live here, `SliderControl.tsx`); `Track` is purely visual/styling — it can be given `padding` to create an inset hitbox-larger-than-visual-bar effect without breaking pointer tracking, because pointer math is computed against `Control`'s bounding rect (`getControlOffset` in `SliderControl.tsx` explicitly reads the *Control* element's computed border/padding, not Track's). Confusing the two (e.g. adding padding to Control expecting Track's visual bounds to shrink accordingly) is the exact misuse pattern #981 documents (§10).
- [E] **Indicator's positioning math is orientation- and mode-aware**: for a single-thumb slider it renders from the start edge to the current value's percent (`getCenteredStyles`); for a range slider it renders between the lowest and highest active-value percentages, using the CSS custom properties described in §1. When `thumbAlignment` is `'edge'`/`'edge-client-only'` (inset mode), positioning switches to `getInsetStyles`, which additionally hides the indicator (`visibility: hidden`) until both edge positions are known post-layout, avoiding a visible jump.
- [E] **Thumb wraps a real native `<input type="range">` as a child**, visually hidden (`visuallyHidden` style, sized to fill the thumb's box so VoiceOver's focus indicator matches the thumb's visual dimensions) but functionally primary — all ARIA state (`aria-valuenow`, `aria-valuetext`, `aria-orientation`) and all keyboard handling live on this nested input, not on the outer `<div>` (`SliderThumb.tsx`; confirmed directly, not inferred — see §7). The outer `<div>` only carries pointer-down bookkeeping and visual positioning styles.
- [E] **Multi-thumb range sliders require an explicit `index` prop per thumb for SSR determinism**: "This prop is required to support server-side rendering for range sliders with multiple thumbs" (`SliderThumb.tsx` JSDoc); without it, index is inferred client-side from composite-list render order (`useCompositeListItem`), which is fine for client-only rendering but under-determined during SSR with multiple thumbs.
- [E] **`Value` is a native `<output>` element**, its `htmlFor` attribute built by concatenating every registered thumb's resolved input id (`SliderValue.tsx` — iterates `thumbMap` populated by each Thumb's `CompositeList` registration), so a single `<Slider.Value>` labels-for every thumb simultaneously, matching the native `<output for="id1 id2">` multi-target pattern. Its default rendered text joins all current values with an en-dash separator (`values.map(...).join(' – ')`); a `children` render function receives `(formattedValues, values)` for full custom formatting.
- [I] Text spec for a future anatomy diagram: (1) Label [optional, outside Control], (2) Value [optional `<output>`, outside Control], (3) Control [invisible hit-area box], (4) Track [visual full-range bar, inside Control], (5) Indicator [visual filled-portion bar, inside Track], (6) Thumb(s) [visual draggable handle(s), inside Track, each wrapping a visually-hidden native range input]. A dotted annotation should mark the hidden `<input type="range">` nested in each Thumb — the true focus/ARIA target — as distinct from the visible Thumb `<div>` box.

## 6. Behavior ("How it works")

- [E] **Controlled/uncontrolled**: uncontrolled by default via `defaultValue` (defaulting further to `min` if omitted entirely); controlled via `value`/`onValueChange` (`useControlled` in `SliderRoot.tsx`). `value`/`defaultValue` accept either a single `number` or a `readonly number[]` — passing an array (even of length 1) puts the slider into "range" rendering mode internally (`range = Array.isArray(valueUnwrapped)`), though a true multi-thumb range slider needs one `<Slider.Thumb>` per array entry.
- [E] **Two change callbacks, following the library-wide split also used by Number Field** (`research/c-components/number-field/brief.md` §6): `onValueChange(value, eventDetails)` fires on every applied change (drag move, track press, keyboard step, or hidden-input `change` event); a separate `onValueCommitted(value, eventDetails)` fires once when an interaction finalizes — on `pointerup`/`touchend` after a drag, immediately after a keyboard step (keyboard changes commit immediately, unlike drag), or after a track press. Its JSDoc carries the same generic-event warning as Number Field's: "This is a generic event, not a change event" — it cannot cancel anything; only `onValueChange`'s implicit `eventDetails.cancel()` path (via the internal `setValue` returning `false`) vetoes a change while keeping the component uncontrolled.
- [E] **Reason vocabulary** (`SliderRootChangeEventReason`/`CommitEventReason`, `SliderRoot.tsx`): `'input-change'` (the nested native range input emits a raw DOM `change` event — e.g. programmatic dispatch or certain AT interactions), `'track-press'` (clicking/tapping the Control/Track directly, not on an existing thumb), `'drag'` (continuous pointer/touch movement while a thumb is pressed), `'keyboard'`, `'none'` (a value set without a specific interaction, e.g. some internal resets). This vocabulary was deliberately kept in sync with Number Field's via `e693fd9df [number field] Align value changes with Slider (#2726)` and `c2420fbc0 [slider][number field] Granular change reasons (#3132)` — shared `eventDetails` shape infrastructure, not a claim of interchangeable components.
- [E] **Track-press vs drag vs thumb-click are distinguished mechanically** in `SliderControl.tsx`: on `pointerdown`, the closest thumb to the pointer position is identified (`startPressing`); if the press lands on an *already-focused* thumb, the browser's default click-to-focus is prevented (avoiding an unwanted focus-visible outline from a mouse click) — otherwise the newly-identified thumb is focused via `requestAnimationFrame`. A press directly on the control/track (not on any thumb) immediately jumps the nearest thumb's value to the pressed position (`REASONS.trackPress`) unless that would violate `minStepsBetweenValues`. Subsequent `pointermove` while still pressed accumulates into `REASONS.drag`, gated by a same `dragging` state that flips true only after a small movement-count threshold (`INTENTIONAL_DRAG_COUNT_THRESHOLD = 2`) so a plain click/tap isn't misclassified as a drag.
- [E] **Range-slider collision handling — `thumbCollisionBehavior`** (default `'push'`, added `c2271e45d [slider] Add thumbCollisionBehavior prop (#2856)`, replacing the legacy `disableSwap` boolean dropped in #373): `'push'` — thumbs push each other forward without restoring earlier positions if dragged back past a previous collision point; `'swap'` — thumbs exchange identities/positions when dragged past one another (with `focusThumb` re-targeting the now-differently-indexed thumb so focus follows the value, not the DOM position, per `resolveThumbCollision.ts`); `'none'` — thumbs are hard-blocked from passing each other, excess drag movement is simply ignored. `minStepsBetweenValues` (default `0`) works in conjunction with any collision mode to enforce a minimum gap between adjacent thumb values.
- [E] **Keyboard step arithmetic is decimal-precision-aware**: `getNewValue` in `SliderThumb.tsx` rounds the result of an arrow-key step to the maximum decimal precision among the current value, the step amount, and `min` (`getDecimalPrecision`/`.toFixed(...)`), then clamps to `[min, max]` — this avoids floating-point drift (e.g. repeatedly adding `0.1`) accumulating visible error, mirroring Number Field's `removeFloatingPointErrors` concern (`research/c-components/number-field/brief.md` §6) though implemented differently (precision-matched rounding vs. epsilon-bounded cleanup).
- [E] **`thumbAlignment`** (Root prop, default `'center'`): `'center'` — the thumb's visual center sits exactly on the value's percentage position (can visually overflow the Control's edges at `min`/`max`, the classic native-range-input look); `'edge'` — the thumb is inset so its outer edge, not its center, aligns with the Control's edge at `min`/`max`, achieved via a `ResizeObserver`-driven inset-position calculation (`getInsetPosition` in `SliderThumb.tsx`) that measures thumb and Control rects and derives a percentage inset; `'edge-client-only'` — same visual result as `'edge'` but the inset calculation only runs after React hydration (a `renderBeforeHydration` flag flips false), trading a delayed-but-correct render for smaller SSR bundle size — the mechanism for this deferred bundle-size trade includes a literal inline `<script>` (`PrehydrationScript`, sourced from a `prehydrationScript.min.ts` minified script) injected alongside the last thumb specifically for the *non*-client-only `'edge'` mode, to compute inset position before hydration completes and avoid a flash of un-inset thumbs.
- [E] **RTL**: handled via the library-wide `DirectionProvider`/`useDirection()` convention (not a `isRtl` boolean, per the #373 rewrite, §2); direction affects both the visual start-edge calculation (`insetInlineStart` vs `bottom` per orientation) and the sign of keyboard ArrowLeft/ArrowRight stepping (flipped in RTL so ArrowRight always visually "advances" — confirmed directly in `SliderThumb.tsx`'s `getNewValue` direction argument: `rtl ? -1 : 1` for ArrowRight, `rtl ? 1 : -1` for ArrowLeft) and pointer-drag value calculation in `SliderControl.tsx` (`direction === 'rtl' ? right - fingerX : fingerX - left`). Vertical orientation is RTL-agnostic (no horizontal flip applies).
- [E] **Forms**: each `Slider.Thumb`'s nested native `<input type="range">` carries `name`/`form`/`min`/`max`/`step` directly, so a single-thumb slider participates in native form submission with zero extra machinery; a range slider submits **one form value per thumb** under the same `name` (confirmed by test: `formData.getAll('slider')` returns `['25', '50']` for a two-thumb slider) — i.e., range Sliders serialize as a same-named multi-value form field, not a single delimited string. Field integration mirrors the standard `data-valid/invalid/touched/dirty/focused` set on every part (duplicated per the library-wide doctrine, §8).
- [E] **SSR**: the "does not link Slider.Label before hydration" test (`SliderRoot.test.tsx`) confirms the same pattern documented on Select (`research/c-components/select/brief.md` §6) — label-to-control ARIA association is deferred until after hydration, avoiding a mismatch when ids are generated client-side.

## 7. Accessibility contract

- [E] **The rendered role is the native, implicit `role="slider"` of `<input type="range">` — not a custom `div role="slider"` ARIA widget.** Verified directly: `screen.getByRole('slider')` resolves to an `<input>` element throughout the entire test suite (`expect(slider.tagName).toBe('INPUT')`, `SliderRoot.test.tsx`); `Root` itself renders `role="group"` (`SliderRoot.tsx` line 402), not `role="slider"`. This is a considered, explicitly documented design decision, not a default fallen into by accident: the test suite carries an inline rationale comment (`SliderThumb.test.tsx` lines 119-125): "AT (e.g. Android Talkback) may use increase/decrease actions to interact with the slider which works on `input type="range"` via change events, but not pure ARIA implementations using `div role="slider"`. The `input` element(s) must be the only focusable element(s)." Two upstream references are cited directly in the comment: a Chromium issue (issues.chromium.org/issues/40816094) and a mui/material-ui issue (#23506) documenting TalkBack's failure to simulate increment/decrement keyboard events on synthetic ARIA sliders. **This is the single most important a11y-contract fact for this component and should be foregrounded on the docs page** — a consumer inspecting the DOM expecting a styled `div[role=slider]` (as many other headless-slider libraries implement) will instead find a visually-hidden native `<input type="range">` doing the real work, with the visible `<div>` Thumb purely a positioned box around it.
- [E] **ARIA actually rendered** (verified directly, `SliderThumb.tsx`/`SliderRoot.tsx`/`SliderValue.tsx`):
  - Root: `role="group"`, `aria-labelledby` (auto-resolved from `Slider.Label` or an explicit prop via `resolveAriaLabelledBy`), plus the Field-integration validity attributes.
  - Thumb's nested `<input type="range">`: implicit `role="slider"`, `aria-valuenow` (the raw numeric value), `aria-valuetext` (see below), `aria-orientation` (`"horizontal"`/`"vertical"`), `aria-labelledby` (falls back to the Root's label id when no direct `aria-label`/`getAriaLabel` is given), `aria-label` (direct string or computed via `getAriaLabel(index)`), `aria-describedby`, native `min`/`max`/`step`/`disabled`/`form`/`name`, plus Field validation props (`validation.getValidationProps`).
  - Value: rendered as a native `<output>` with `htmlFor` pointing at every thumb's input id, `aria-live="off"` by default — deliberately off, with an inline source comment: "off by default because it will keep announcing when the slider is being dragged and also when the value is changing (but not yet committed)" (`SliderValue.tsx`) — i.e., turning `aria-live` on is a documented anti-pattern for continuous drag interactions, a genuine, evidenced pitfall (§10).
- [E] **Default `aria-valuetext` generation is range-aware**: for a two-value range slider, the two thumbs automatically receive `"<value> start range"` / `"<value> end range"` text (`getDefaultAriaValueText` in `SliderThumb.tsx`; confirmed by test "should set default aria-valuetext on range slider thumbs" → `'44 start range'` / `'50 end range'`); for a single-thumb slider or 3+-thumb slider, no automatic text is generated unless a `format` option is supplied (in which case the formatted number itself becomes the text) — a `getAriaValueText` function prop takes precedence over everything, receiving `(formattedValue, value, index)` for full custom control (docs recommend this for meaningful announcements, e.g. the "Delay until repeat" experiment demo's inline comment: "`getAriaValueText` could be used so more meaningful values are announced").
- [E] **Labeling multi-thumb sliders**: because a single `Slider.Label` labels the whole Root/group (not any individual thumb), **each thumb in a multi-thumb range slider needs its own distinguishing `aria-label`** to be announced individually — the docs' own multi-thumb labeling example sets `aria-label="Minimum price"` / `aria-label="Maximum price"` per thumb alongside a shared visible `Slider.Label` ("Price range"), and the library-wide forms handbook explicitly prescribes this: "per-thumb `aria-label` on multi-thumb sliders" (`research/b-library-principles/principles.md` line 346, forms page). For a `Fieldset`-grouped multi-thumb slider (docs' "Using Fieldset with a multi-thumb slider" example), `Fieldset.Legend` supplies the shared group label while each thumb still keeps its own `aria-label`.
- [E] **Keyboard interaction table** (evidence: `SliderThumb.tsx` `onKeyDown`, `../../internals/composite/composite` key constants, `SliderRoot.test.tsx`'s `describe('keyboard interactions')` matrix crossing every arrow-key direction × orientation):
  | Key | Context | Action |
  |---|---|---|
  | ArrowUp / ArrowRight (ArrowLeft in RTL) | Thumb (input) focused, horizontal orientation | Increments by `step`; commits immediately (`onValueCommitted` fires on the same keypress, unlike drag which commits only on release) |
  | ArrowDown / ArrowLeft (ArrowRight in RTL) | Thumb focused, horizontal orientation | Decrements by `step`, symmetric to above |
  | ArrowUp / ArrowRight | Thumb focused, vertical orientation | Same increment semantics; ArrowUp/ArrowDown govern vertical regardless of orientation per the composite key set — the maintained matrix explicitly tests both orientations against both arrow-key pairs |
  | Shift + any arrow key | Thumb focused | Steps by `largeStep` (default `10`) instead of `step`; still clamped to `min`/`max` (tested: "stops at min/max when decrementing/incrementing while Shift is pressed") |
  | Home | Thumb focused | Single-value slider: jumps to `min`. Range slider: jumps to `min`, or to the immediately-lower thumb's value plus `step * minStepsBetweenValues` if a lower thumb exists (prevents overlap) |
  | End | Thumb focused | Symmetric to Home: jumps to `max`, or to the next-higher thumb's value minus the same offset |
  | PageUp | Thumb focused | Increments by `largeStep` — a genuine, tested, and **library-uniquely-implemented** large-step key (contrast Number Field, which deliberately does NOT implement PageUp/PageDown, `research/c-components/number-field/brief.md` §7) |
  | PageDown | Thumb focused | Decrements by `largeStep`, symmetric to PageUp |
  | Tab / Shift+Tab | Thumb focused | Moves focus to the next/previous thumb (multi-thumb) or out of the slider entirely (single-thumb) — since only the nested `<input>`s are tabbable, the roving-focus complexity other composite widgets need is unnecessary; native Tab order handles it |
  | Any key not in the handled set (e.g. Enter) | Thumb focused | Forwarded unmodified to a user's own `onKeyDown` handler (tested: "forwards key events that the slider does not handle") |
- [E] **`event.preventDefault()` in a user `onKeyDown` handler suppresses the internal key handling** for handled keys too (tested: "allows preventing the internal key handling" — calling `preventDefault()` on an ArrowRight keydown leaves `aria-valuenow` unchanged) — a documented escape hatch for consumers wanting fully custom keyboard behavior.
- [E] **Focus-visible restoration on keyboard step**: when a keyboard arrow/Home/End/PageUp/PageDown step occurs on an input that isn't currently in `:focus-visible` state (e.g. it was focused by a prior mouse click), the component synthetically blurs and refocuses the input with `focusVisible: true` so the keyboard-driven change gets a visible focus ring even though the interaction started with a mouse (`restoringFocusVisibleRef` machinery in `SliderThumb.tsx`) — the synthetic blur/focus pair is explicitly suppressed from reaching the user's own `onFocus`/`onBlur` handlers (tested: "does not emit extra blur and focus events when restoring focus-visible").
- [E] **Disabled state removes the thumb from the a11y tree entirely and returns focus if needed**: `disabled` sets native `disabled` on every nested range input (removing it from Tab order and the accessibility tree, standard native-input behavior — no `aria-disabled` workaround is needed since these are real `<input>` elements, unlike the button-based components that must use `aria-disabled` to stay focusable); an `useIsoLayoutEffect` in `SliderRoot.tsx` explicitly blurs the active element if the slider becomes disabled while an internal element retains focus, with an inline comment citing a Firefox/Safari quirk where disabled elements otherwise incorrectly retain focus.
- **Open a11y-adjacent issues / honesty notes**: [G] A targeted search for `label:"component: slider" label:accessibility` and "slider voiceover/nvda/jaws" in the time budget available for this lean-plus brief did not surface additional open a11y-specific issues beyond the TalkBack work already merged (§2, this section) — this is a narrower search than Select/Number Field's briefs performed (their higher tier allowed a deeper issue sweep) and should be treated as a gap, not a confirmed clean bill of health. [E] The one explicitly-flagged, evidenced honesty item is the `aria-live="off"`-by-default choice on `Slider.Value` (documented above) — turning it on for a drag-updated value is a real, source-commented anti-pattern worth surfacing prominently rather than leaving implicit.

## 8. Prop-level guidance (decision-relevant props)

- [E] **`value` / `defaultValue`** — a single `number` renders a one-thumb slider; a `readonly number[]` renders a range slider (one thumb per array entry, values kept sorted ascending internally via `asc`). Passing `[10]` (a one-element array) still triggers range-mode internals even with only one thumb rendered — a subtle typing/behavior detail worth documenting explicitly since it affects `range`/`values.length` branch logic throughout the component.
- [E] **`min` / `max`** (defaults `0`/`100`) — a dev-only warning fires if `min >= max` ("Slider `max` must be greater than `min`.", `b0defb4df` #1475); serve as both the clamping bounds and, for range sliders, the origin/end for Home/End jump-to-adjacent-thumb behavior (§7).
- [E] **`step`** (default `1`) — governs both pointer-drag rounding (`roundValueToStep`) and keyboard single-step amount; unlike Number Field, there is no `'any'` escape hatch — `step` is always a concrete number here, and decimal steps are fully supported with precision-matched rounding (§6).
- [E] **`largeStep`** (default `10`) — the Shift-modified keyboard step amount, and also the sole amount used by PageUp/PageDown (§7) — notably, PageUp/PageDown always use `largeStep` regardless of Shift, so Shift+Arrow and bare PageUp/PageDown are two equivalent paths to the same large-step behavior.
- [E] **`minStepsBetweenValues`** (default `0`) — enforces a minimum gap (expressed in units of `step`, not raw value) between adjacent range-slider thumb values; validated on every pointer/keyboard change via `validateMinimumDistance` before the change is even attempted, so an out-of-bounds attempted move is simply dropped rather than clamped to the nearest valid position.
- [E] **`thumbCollisionBehavior`** (default `'push'`) — see §6 for full mechanics of `'push'`/`'swap'`/`'none'`; this is the direct functional replacement for the legacy `disableSwap` boolean dropped in the #373 rewrite (`'none'` is the closest equivalent to the old `disableSwap={true}`).
- [E] **`thumbAlignment`** (default `'center'`) — `'center'`/`'edge'`/`'edge-client-only'`; see §6 for the SSR/bundle-size trade-off between `'edge'` and `'edge-client-only'`. Choose `'edge'` when the visual design wants the thumb fully contained within the track's bounds at the extremes (no overflow past `min`/`max`); choose `'edge-client-only'` specifically to shave the pre-hydration script's bundle-size cost when a brief post-hydration layout shift at the initial render is acceptable.
- [E] **`orientation`** (default `'horizontal'`) — flips the geometry calculations (`insetInlineStart`/`bottom` edges, `width`/`height` sizing) throughout Control/Track/Indicator/Thumb; vertical sliders need explicit height set via CSS on Control/Track since there's no default intrinsic vertical size the way horizontal sliders get from block-level width.
- [E] **`format` / `locale`** (`Intl.NumberFormatOptions` / `Intl.LocalesArgument`) — used by `Slider.Value`'s default text rendering and by the default `aria-valuetext` generation when present (§7); shares the same `formatNumber` utility as Number Field (`packages/react/src/utils/formatNumber.ts`), so currency/percent/unit formatting work identically on both components, but Slider has no corresponding typed-entry/parse pipeline — formatting here is purely for display/announcement, never for input parsing.
- [E] **`getAriaLabel` / `getAriaValueText`** (Thumb props, index-aware functions) — the recommended mechanism for meaningful multi-thumb or context-specific announcements beyond the automatic start/end-range text; `getAriaValueText` takes precedence over a direct `aria-valuetext` prop (tested: "prefers getAriaValueText over a direct aria-valuetext prop").
- [E] **`index` (Thumb, required for multi-thumb SSR)** — see §5; omit safely in client-only-rendered range sliders (order is inferred from render position), but always set explicitly when server-rendering more than one thumb.
- [E] **Form wiring** — `name` (submission key, shared across all thumbs in a range slider — submits as same-named multi-value, §6), `form` (owning form id when rendered outside it), `disabled` (also merges with Field's own disabled state, mirroring Number Field's `fieldDisabled || disabledProp` pattern).
- [E] **Styling contract — data attributes** (identical duplicated set across every part's `*DataAttributes.ts`, confirmed by reading all six files: Root/Thumb/Control/Track/Indicator/Value): `data-dragging` (present while actively dragging — explicitly NOT present for a simple click/track-press, tested: "does not apply data-dragging for click modality" / "applies data-dragging for dragging modality"), `data-orientation` (`"horizontal"`/`"vertical"`), `data-disabled`, plus the Field-integration set `data-valid`/`data-invalid`/`data-touched`/`data-dirty`/`data-focused` (populated only when wrapped in `Field.Root`). Thumb additionally carries `data-index` (its position in the value array) — the one part-unique data attribute in this component, needed for per-thumb styling in range sliders (e.g. `[data-index="0"]` for the "minimum" handle). **No `--`-prefixed CSS custom property is part of the documented/enumerated styling contract** (§1) — this is the single biggest styling-contract difference from popup-family components and should be stated plainly on the docs page rather than left to be discovered by inspecting computed styles.
- [E] **`.Thumb:focus-visible` vs `.Thumb:has(:focus-visible)`** — a breaking styling-contract change from `40308ba37` (beta.3, #2578, TalkBack fix): because real focus lives on the nested `<input>`, not the visible `<div>` Thumb wrapper, consumers must style the *wrapper's* focus ring using `:has(:focus-visible)` (matching a focus-visible descendant) rather than `:focus-visible` directly on the div (which never itself receives focus). This is exactly the kind of subtle, easy-to-miss styling-contract fact GOV.UK-style honest documentation should foreground with a code example, since a consumer copying `:focus-visible` from another component's stylesheet will silently get no focus ring at all.

## 10. Pitfalls & FAQ

- **"Adding padding to `Slider.Control` makes the thumb poke outside the track" / pointer tracking feels off after styling** → this is the exact, evidenced #981 misuse pattern: `Control` is the tracked hit-area (pointer math reads *its* padding/border), `Track` is purely visual. Style Track for the visual inset-bar look; if you need a larger tap target than the visual bar, add the padding to Control specifically, understanding that pointer math already accounts for it — don't expect Track's padding to affect hit-testing (mj12albert/colmtuite, mui/base-ui#981). [E]
- **Expecting `div[role="slider"]` in the DOM** → it isn't there; the real accessible node is a visually-hidden native `<input type="range">` nested inside each visible Thumb `<div>`. Querying `[role="slider"]` in tests/automation resolves to the `<input>`, not the styled wrapper (confirmed throughout `SliderRoot.test.tsx`/`SliderThumb.test.tsx`, `screen.getByRole('slider')` always returns an `INPUT` element). [E]
- **Styling `:focus-visible` directly on a custom Thumb class does nothing** → focus never lands on the visible Thumb `<div>` itself; use `.Thumb:has(:focus-visible)` (the hero demo's own CSS uses exactly this selector) — a breaking change from `#2578` that predates any given consumer's first contact with the component and is easy to get wrong by copying focus-ring CSS from a button-based component. [E]
- **Turning on `aria-live` on `Slider.Value` to "make sure it's announced"** → it's `"off"` by default specifically because a drag interaction would otherwise cause continuous, unhelpful announcements mid-gesture; the value is naturally discoverable via the thumb's own `aria-valuenow`/`aria-valuetext` when focused. Source-commented as a deliberate choice, not an oversight (`SliderValue.tsx`). [E]
- **"Where are the tick marks / restricted values?"** → not a built-in feature; the legacy `Mark`/`marks` API was intentionally dropped in the #373 rewrite pending a still-open redesign (#462). Compose marks manually as positioned `<span>` children of `Track` (see `docs/src/app/(private)/experiments/slider/slider.tsx`'s `DelayUntilRepeat` for a working recipe) until #462 lands. [E]
- **"Where's the floating value bubble that follows the thumb while dragging?"** → also intentionally dropped (the legacy `ValueLabel`); compose it yourself with `Tooltip`, per the `experiments/slider-tooltip` demo referenced directly in the #373 PR body. [E]
- **Range slider submits multiple form values under one name, not a single string** → by design; `formData.getAll(name)` returns one entry per thumb (e.g. `['25', '50']`), not `"25,50"` or similar — server-side form handling must anticipate an array/multi-value field for the same key, not a delimited string (confirmed by `SliderRoot.test.tsx` Form describe block). [E]
- **`value={[10]}` (single-element array) behaves like a range slider internally, not a single-value slider** → `range` mode is derived purely from `Array.isArray(valueUnwrapped)`, so a one-element array still takes the range code path even though only one thumb is rendered; pass a bare `number` for genuinely single-value sliders. [I, derived directly from `SliderRoot.tsx`'s `range = Array.isArray(valueUnwrapped)` line — not an explicitly documented gotcha in any issue found, but a direct reading of the source that a consumer could easily trip over].
- **No CSS variables to hook for custom positioning math** → unlike Select/Popover's `--anchor-width`/`--available-height` contract, Slider's `--position`/`--start-position`/`--relative-size` custom properties are internal implementation plumbing inside inline `style` objects, not a documented, consumer-facing theming API (§1) — don't expect to read or override them from external CSS the way you would a documented CSS variable. [E]
