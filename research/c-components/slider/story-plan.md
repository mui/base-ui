14 stories. Derived from `brief.md` evidence (source, tests, docs demos, git history, #373 PR body). Styling: follow the docs hero demo CSS (CSS Modules, raw oklch values, `:has(:focus-visible)` selector per brief §8).

## 2. One story per use case

Columns: name — renders — controlled? — play? — doc section served.

1. **Basic** — hero recreation: Root+Control+Track+Indicator+Thumb, `defaultValue`, hero CSS incl. `.Thumb:has(:focus-visible)` — uncontrolled — no — Hero / Anatomy / styling contract (§1, §8).
2. **KeyboardStepping** *(required interaction story — the reliable keyboard-set play)* — Basic + play: focus thumb via Tab → `fireEvent.keyDown`/`user.keyboard` for ArrowRight (increments by `step`), ArrowLeft (decrements), Shift+ArrowRight (`largeStep`), PageUp (`largeStep`, no Shift needed), PageDown, Home (jumps to `min`), End (jumps to `max`) → assert `aria-valuenow` after each, and that `onValueCommitted` fires once per keypress (unlike drag, which only commits on release) — uncontrolled — **play** — Behavior §6 / A11y keyboard table §7.
3. **RangeTwoThumb** *(required interaction story — range play)* — range-slider recreation (`defaultValue={[25, 45]}`, `index={0}`/`index={1}`, per-thumb `aria-label`) + play: Tab to first thumb, ArrowRight to move it, Tab to second thumb, ArrowLeft to move it, assert both `aria-valuenow`s update independently and `getAllByRole('slider')` returns two inputs with correct `aria-valuetext` (`"<n> start range"` / `"<n> end range"` per the default range-aria-valuetext generation, brief §7) — uncontrolled — **play** — Anatomy / A11y (multi-thumb labeling) / Behavior (range mode).
4. **ThumbCollisionBehavior** — three side-by-side range sliders demonstrating `thumbCollisionBehavior="push"` (default), `"swap"`, and `"none"` — same starting `defaultValue={[30, 70]}` in each — uncontrolled — play (drag or keyboard-step one thumb past the other in each variant; assert push/swap/blocked behavior differs per variant per brief §6/§8) — Prop guidance / Decision log (#2856).
5. **MinStepsBetweenValues** — range slider with `minStepsBetweenValues={5}` and `step={1}` — uncontrolled — play (attempt to step one thumb within 5 units of its neighbor via Home/End or repeated ArrowRight; assert it stops short rather than colliding) — Prop guidance / Behavior §6.
6. **Vertical** — vertical-orientation recreation, explicit `height` on Control/Track per orientation's CSS requirement (brief §8) — uncontrolled — play (focus thumb, ArrowUp/ArrowDown; assert value changes match horizontal ArrowRight/Left semantics per the orientation-crossed test matrix, brief §7) — Anatomy / Behavior (orientation).
7. **RTL** — `DirectionProvider direction="rtl"` wrapping a horizontal slider — uncontrolled — play (ArrowRight/ArrowLeft; assert the physical-direction flip described in brief §6 — ArrowRight still visually "advances" despite the sign flip in `getNewValue`) — Behavior (RTL) / A11y.
8. **ThumbAlignmentEdge** — edge-alignment recreation (`thumbAlignment="edge"`) alongside a `"center"` (default) sibling for visual contrast — uncontrolled — play (open at `min`/`max` and assert the thumb stays fully inside the Control's bounds in edge mode vs. overflowing in center mode, using `getBoundingClientRect` comparisons; **Chromium-only** since layout measurement is required, `describe.skipIf(isJSDOM)` per brief §6's `ResizeObserver`-driven inset calculation) — Prop guidance / Anatomy.
9. **ControlledValueWithCommit** — external `value` state + a live log distinguishing `onValueChange` (fires on every applied change) from `onValueCommitted` (fires once per finalized interaction) — controlled — **play**: drag or keyboard-step, assert `onValueChange` call count exceeds `onValueCommitted`'s during a drag but the two match 1:1 for keyboard steps (brief §6's two-callback split) — Behavior / API contract (mirrors Number Field story-plan's analogous item 14, `research/c-components/number-field/story-plan.md`).
10. **TrackPressVsDrag** — single-thumb slider with a live `data-dragging`-state readout — uncontrolled — play: (a) a single click on the track (not on the thumb) — assert the value jumps immediately with reason `track-press` and `data-dragging` stays absent; (b) a press-and-move sequence on the thumb — assert `data-dragging="true"` appears only after real movement (brief §6's `INTENTIONAL_DRAG_COUNT_THRESHOLD`) — **Chromium-only** (real pointer events needed) — uncontrolled — play — Behavior / styling contract (`data-dragging`).
11. **CustomMarks** — recreation of the `experiments/slider/slider.tsx` `DelayUntilRepeat` pattern: manually-positioned `<span>` marks inside `Slider.Track`, small integer range (`min={0}` `max={5}` `step={1}`) — uncontrolled — no — When-not-to-use boundary / Pitfalls (#462, "marks aren't built in, here's the recipe").
12. **FormattedValueWithGetAriaValueText** — `Slider.Value` with `format={{ style: 'unit', unit: 'percent' }}` (or similar) alongside `Thumb getAriaValueText` for a more meaningful announcement than the raw number — uncontrolled — no — Prop guidance (`format`/`locale`/`getAriaValueText`).
13. **FormIntegration** — `<form>` + `name="volume"` + submit button reading `FormData.getAll(name)`, single-thumb and range variants side by side to make the "range submits multiple same-named values, not a delimited string" contract visible (brief §6/§10) — uncontrolled — **play**: adjust value(s), submit, assert the readout shows the correct array-of-strings payload — Behavior (forms) / Pitfalls.
14. **DisabledState** — a `disabled` slider next to an enabled one, both pre-set to a mid-range value — uncontrolled — play (assert `data-disabled` present on every part per brief §8's duplicated-data-attribute doctrine; assert the disabled thumb's native input is unfocusable/excluded from Tab order, requiring zero `aria-disabled` workaround since it's a real `<input>`) — A11y contract / Prop guidance.

## Honesty notes carried from the brief

- Story 8 (thumb alignment) and story 10 (track-press vs. drag) require real layout measurement (`getBoundingClientRect`) or real pointer-event sequences and must be gated `Chromium`-only (`describe.skipIf(isJSDOM)`), mirroring the project's own test suite's use of `it.skipIf(isJSDOM || isWebKit)` for equivalent drag/touch/RTL tests (`SliderRoot.test.tsx`, `SliderThumb.test.tsx`).
- No story should claim a built-in "marks" or "value-label bubble" feature exists — both were explicitly dropped in the #373 rewrite (brief §2/§4/§10). Story 11 is a documented workaround recipe (recreating an existing experiment file), not a port of an existing docs demo — it is net-new Storybook coverage.
- Story 3's default `aria-valuetext` ("start range"/"end range") only applies to exactly-two-thumb range sliders per `getDefaultAriaValueText` (brief §7) — do not generalize this to 3+-thumb sliders without a `getAriaValueText` override; if a 3+-thumb story is added later, it must use an explicit `getAriaValueText` since no automatic text is generated for that case.
- The current docs directory has exactly four demos (hero, range-slider, edge-alignment, vertical) confirmed by listing `docs/src/app/(docs)/react/components/slider/demos/` — stories 4-13 are evidence-backed net-new Storybook coverage derived from source/tests/experiments, not ports of existing docs content, and should be labeled as such.
