# Composite keyboard navigation

[Toolbar](?path=/docs/actions-toolbar--docs), [Menubar](?path=/docs/navigation-menubar--docs),
[Toggle Group](?path=/docs/actions-toggle-group--docs),
[Radio (Group)](?path=/docs/form-inputs-radio--docs), and
[Tabs](?path=/docs/navigation-tabs--docs) all wrap the same `CompositeRoot` roving-tabindex
engine: **one Tab stop for the whole widget**, arrow keys move focus among its members. This is
a defended design choice, not an accident — "fewer tab stops is more accessible" is the library's
own framing for why keyboard users shouldn't have to Tab through every toolbar button or menu
item individually. This page is the single place that compares the model
across all five composites, including where each one deliberately deviates from it.

## The shared model

| | Tab stops | Arrow keys move | Wraps at ends | Home/End |
|---|---|---|---|---|
| [Toolbar](?path=/docs/actions-toolbar--docs) | 1 for the whole bar | orientation-appropriate | `loopFocus` (default `true`) | **not wired** |
| [Menubar](?path=/docs/navigation-menubar--docs) | 1 for the whole bar | left/right between menu triggers | yes | — |
| [Toggle Group](?path=/docs/actions-toggle-group--docs) (standalone) | 1 for the whole group | orientation-appropriate | `loopFocus` (default `true`) | enabled since [#3971](https://github.com/mui/base-ui/pull/3971) |
| [Radio Group](?path=/docs/form-inputs-radio--docs) | 1 for the whole group | vertical + horizontal (RTL-aware) | yes | **disabled**, `enableHomeAndEndKeys={false}` |
| [Tabs](?path=/docs/navigation-tabs--docs) | 1 for the whole tablist | orientation-appropriate | yes | — |

Arrow direction always flips correctly under `dir="rtl"` for horizontal orientation across every
one of these — tested identically in Toolbar's and Toggle Group's suites (parametrized
`[direction, orientation]` matrices).

## Toolbar never enables Home/End

Toolbar's `CompositeRoot` never receives `enableHomeAndEndKeys` — confirmed directly by source
inspection, unlike Toggle Group's own standalone `CompositeRoot`, which passes it explicitly. This has a real, source-verified consequence: **a Toggle Group nested
inside a Toolbar loses the Home/End support it has when used standalone**, because a
Toolbar-hosted Toggle Group defers its entire composite behavior to the *hosting* Toolbar's
`CompositeRoot`, not its own. No maintainer discussion resolves this
asymmetry as either intentional or a gap — it is simply what today's source does; treat it as a
documented nuance, not a bug to route around. See
[Toolbar hero](?path=/docs/actions-toolbar--docs) and
[Toggle Group: composite keyboard navigation](?path=/story/actions-toggle-group--composite-keyboard-navigation).

## Deliberate deviations from the roving-focus model

**Accordion removed roving focus entirely** ([mui/base-ui#4965](https://github.com/mui/base-ui/pull/4965)),
aligning with a change to the W3C APG pattern itself: "The Accordion APG pattern no longer
recommends roving focus, as documented in
[w3c/aria-practices#3434](https://github.com/w3c/aria-practices/pull/3434)… removing roving focus
from Accordion to align with the latest APG guidance" (PR body, verbatim). Every
`Accordion.Trigger` is now its own ordinary Tab stop — Tab/Shift+Tab moves between headers the
plain way, and arrow keys do nothing. The companion PR
([mui/base-ui#4961](https://github.com/mui/base-ui/pull/4961)) removed `role="region"` from
`Accordion.Root` the same day, since that role belongs on `Panel`, which already had it. The
`orientation`/`loopFocus` Root props are now `@deprecated` — they still drive the
`data-orientation` styling hook, but "no longer affect keyboard focus behavior"
(`AccordionRoot.tsx` JSDoc). Accordion is filed under Disclosure & structure, not this cluster,
precisely because it isn't a composite anymore — see
[Disclosure: Accordion, Collapsible, Tabs](?path=/docs/overview-choosing-components--docs) for the
full three-way comparison.

**Tabs switched from focus-activation to click-activation** —
[mui/base-ui#3176](https://github.com/mui/base-ui/pull/3176) flipped `activateOnFocus`'s default
to `false` after a WCAG 2.5.2 (pointer cancellation) argument beat the "feels more responsive"
counter-case. Arrow-key movement still roves focus across the tablist, but
selecting a tab now requires a distinct activation (Enter/Space, or focus if you explicitly
opt back in) rather than firing on every arrow press. See
[Tabs: keyboard focus does not activate](?path=/story/navigation-tabs--keyboard-focus-does-not-activate)
and [Tabs: activate on focus](?path=/story/navigation-tabs--activate-on-focus) to compare both
modes directly.

**Radio Group selects on navigation — the opposite deviation.** Unlike every other composite
here, arrow-key movement in a Radio Group both moves focus **and commits the selection**,
matching native `<input type="radio">` group behavior: focusing an untouched radio via keyboard
synthetically clicks its own hidden input (pinned by the
"should automatically select radio upon navigation" test). This is a deliberate parity choice
with the native element, not an inconsistency — Radio Group is the one composite in this table
where "move focus" and "select" are the same action. Holding **Shift** while arrowing suppresses
the auto-select and moves focus only. See
[Radio: arrow-down selects on navigation](?path=/story/form-inputs-radio--arrow-down-selects-on-navigation).

## Toolbar-hosted Toggle Group: the Home/End asymmetry in full

To restate the sharpest cross-component nuance on this page: the Home/End gap
traces to its exact origin — Home/End shipped for standalone Toggle Group in
[#3971](https://github.com/mui/base-ui/pull/3971), closing a reported gap
([#3970](https://github.com/mui/base-ui/issues/3970)), by passing `enableHomeAndEndKeys` into
*Toggle Group's own* `CompositeRoot`. That PR could not — and structurally did not — touch
Toolbar's separate `CompositeRoot` the same way, so a Toggle Group loses Home/End the moment it's
hosted inside a Toolbar. If your toolbar-hosted segmented control needs Home/End, it isn't
available today; file it as a Toolbar-side gap rather than assuming user error.
