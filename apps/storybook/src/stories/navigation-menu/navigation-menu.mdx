# Navigation Menu

A collection of links and menus for website navigation — a `<nav>` bar whose disclosure buttons reveal rich link panels in one shared, morphing popup.

- **Import:** `import { NavigationMenu } from '@base-ui/react/navigation-menu';`
- **Parts:** 13 — see Anatomy.
- **Status:** Stable since v1.0.0. Introduced in [#1741](https://github.com/mui/base-ui/pull/1741) (beta.0, 2025-05-29).
- **ARIA pattern:** [Disclosure Navigation](https://www.w3.org/WAI/ARIA/apg/patterns/disclosure/examples/disclosure-navigation/) — **not** the [Menu Button](https://www.w3.org/WAI/ARIA/apg/patterns/menu-button/) pattern. There is no `role="menu"`/`menuitem`/`menubar` anywhere in the source.
- **Docs verified against:** `@base-ui/react` workspace source (v1.6.0 line).

<Canvas of={NavigationMenuStories.Hero} />

## Features

- **One shared, morphing popup** — every trigger feeds a single `Positioner`/`Popup`; switching triggers re-anchors and resizes it via `--popup-width/height` + `--positioner-width/height` while `Content` cross-fades directionally, with no close/reopen flicker (the founding [#1741](https://github.com/mui/base-ui/pull/1741) requirement: "full interruptibility of transitions").
- **Semantic HTML, not ARIA menu semantics** — `Root` renders `<nav>`, `List` a real `<ul>`, `Item` a real `<li>` — deliberately reworked away from menu roles in [#2526](https://github.com/mui/base-ui/pull/2526) ("I was intending to change this as I agree it should default to rendering a semantic list structure"), and a later axe-audit request to add `role="menubar"` was declined in favor of removing the invalid `aria-orientation` instead ([#4349](https://github.com/mui/base-ui/issues/4349) → [#4355](https://github.com/mui/base-ui/pull/4355)).
- **Content authored inline, teleported on demand** — each `Content` lives next to its `Trigger` for colocation but portals into the shared `Viewport` only while its item is active, so outgoing and incoming content can be tagged with `data-activation-direction` and cross-fade.
- **Two composition modes, no extra API** — the top-level `Portal → Positioner → Popup → Viewport` chain is one mode; a nested `Root` rendering only `List` + `Viewport` (no Portal at all) swaps second-level content in place — "for inlined content, the inline viewport never gets unmounted like with popups" ([#2269](https://github.com/mui/base-ui/pull/2269)).
- **Nested popups share one dismissal tree** — a full nested `Root orientation="vertical"` with its own `Portal`/`Positioner` opens a second flyout beside the first, sharing a `FloatingTree` so outside-press/hover-close propagate correctly ([#2978](https://github.com/mui/base-ui/pull/2978)).
- **Crawlable by default when you opt in** — `Content keepMounted` server-renders the panel as hidden inline HTML "for SEO purposes … the content is crawlable" ([#3794](https://github.com/mui/base-ui/pull/3794)), then it moves into the popup permanently on first open.
- **Every `Link` composes your router** — `render={<NextLink href={…}/>}` (or any client-router Link) is the documented "Custom links" recipe, with `active` → `aria-current="page"` for the current destination.

## When to use

- Site headers where top-level entries open **rich link panels** — the hero renders "Overview"/"Handbook" triggers whose panels are link cards (title + description), plus a plain "GitHub" link with no panel: triggers and plain links mix freely in one `List`.
- Mega menus / flyout menus / multi-level navbars — nested submenus (a vertical sub-`Root` opening its own side popup) and nested inline submenus (second-level content swapping inside the same panel via `defaultValue`) are both documented first-class examples.
- Framework-routed navigation — every link part accepts `render`, so the whole nav participates in Next.js/React Router soft navigation.
- Crawlable site nav — `Content keepMounted` exists specifically so panels are indexable before any interaction.
- Persistent-layout navs where links should **not** close the menu by default — `closeOnClick` is opt-in, matching "Stripe and Apple [who] leave theirs open as they act as external links" ([#2740](https://github.com/mui/base-ui/pull/2740)).
- Both horizontal header bars (default) and vertical side rails (`orientation="vertical"`, mirrored open key).

## When not to use

Comparative guidance for the four menu-shaped components lives in the cluster note (`research/c-components/_clusters/menus.md`); the short version: _button-anchored actions → Menu; right-click/long-press → Context Menu; persistent app-chrome bar → Menubar; site navigation links → Navigation Menu._

- **Action lists that do things, then forget** (commands, not destinations) → [Menu](?path=/docs/overlays-menu--docs). Navigation Menu content is links and free markup, not `menuitem` actions.
- **App-chrome command bars (File/Edit/View)** → [Menubar](?path=/docs/navigation-menubar--docs) — "menubar = application commands (desktop-app mental model); navigation menu = website destinations. If the entries are links, it's a navigation menu regardless of looking like a bar."
- **Mobile hamburger navigation** → [Drawer](?path=/docs/overlays-drawer--docs), which ships a dedicated "Mobile navigation" demo (a full-screen sheet with flick-to-dismiss). Navigation Menu has no small-screen presentation mode; on touch it degrades to tap-to-open only. [G] no maintainer statement comparing the two directly was found.
- **Switching between in-page views** → Tabs (`role="tablist"`, selected state) — Navigation Menu links navigate documents, they don't toggle panels-as-state.
- **A single dropdown of arbitrary interactive content** (one trigger, no cross-trigger morphing) → [Popover](?path=/docs/overlays-popover--docs). Navigation Menu's extra machinery (shared viewport, composite list, morph sizing) only pays off with ≥2 panels.
- **A simple row of links with no panels** → plain `<nav><ul><li><a>` — HTML already does this.

## Anatomy

```tsx
import { NavigationMenu } from '@base-ui/react/navigation-menu';

<NavigationMenu.Root>
  <NavigationMenu.List>
    <NavigationMenu.Item>
      <NavigationMenu.Trigger>
        <NavigationMenu.Icon />
      </NavigationMenu.Trigger>
      <NavigationMenu.Content>
        <NavigationMenu.Link />
        {/* or a nested Root for submenus */}
      </NavigationMenu.Content>
    </NavigationMenu.Item>
  </NavigationMenu.List>

  <NavigationMenu.Portal>
    <NavigationMenu.Backdrop />
    <NavigationMenu.Positioner>
      <NavigationMenu.Popup>
        <NavigationMenu.Arrow />
        <NavigationMenu.Viewport />
      </NavigationMenu.Popup>
    </NavigationMenu.Positioner>
  </NavigationMenu.Portal>
</NavigationMenu.Root>;
```

| Part                        | Renders                                | Purpose                                                                                                                                                    | Optional?                     |
| --------------------------- | -------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------- |
| `NavigationMenu.Root`       | `<nav>` (top level) / `<div>` (nested) | Groups all parts; owns the scalar `value` — exactly one panel open at a time                                                                               | required                      |
| `NavigationMenu.List`       | `<ul>`                                 | Roving arrow-key composite over top-level items (no `aria-orientation`, removed in [#4355](https://github.com/mui/base-ui/pull/4355))                      | required                      |
| `NavigationMenu.Item`       | `<li>`                                 | One entry; provides its `value` (auto-generated if omitted)                                                                                                | required                      |
| `NavigationMenu.Trigger`    | `<button aria-expanded aria-controls>` | Opens the item's panel on hover/click/keyboard                                                                                                             | with Content                  |
| `NavigationMenu.Icon`       | `<span aria-hidden>`                   | Indicates the trigger opens a panel (default child `▼`)                                                                                                    | yes                           |
| `NavigationMenu.Content`    | `<div>`                                | Authored inline next to its Trigger; **teleports into the Viewport** when its item is active                                                               | with Trigger                  |
| `NavigationMenu.Link`       | `<a>`                                  | Navigates; `active` → `aria-current="page"` + `data-active`                                                                                                | expected                      |
| `NavigationMenu.Portal`     | moves subtree to `<body>`              | The classic anchored-flyout mode — **omit it entirely** for inline nested content ([#2269](https://github.com/mui/base-ui/pull/2269))                      | optional, unlike Menu/Popover |
| `NavigationMenu.Backdrop`   | `<div role="presentation">`            | Purely visual — the component is never modal                                                                                                               | yes                           |
| `NavigationMenu.Positioner` | `<div>`                                | Anchors the popup against the **currently active trigger**; re-targets as it changes                                                                       | with Portal                   |
| `NavigationMenu.Popup`      | `<nav>`                                | Container for the content — itself a navigation landmark, stitched back into reading order via a visually-hidden `aria-owns` span after the active trigger | with Positioner               |
| `NavigationMenu.Viewport`   | `<div>`                                | The clipping viewport where `Content` mounts; hosts old+new content during cross-fade                                                                      | with Popup                    |
| `NavigationMenu.Arrow`      | `<div aria-hidden>`                    | Visual pointer toward the anchor                                                                                                                           | yes                           |

**Two composition modes**: (1) **Popup mode** — the full `Portal → Positioner → Popup → Viewport` chain, anchored to the active trigger (the hero). (2) **Inline mode** — a nested `Root` rendering only `List` + `Viewport` with a `defaultValue`; content swaps inside the parent's panel with no new popup at all.

<Canvas of={NavigationMenuStories.NestedInlineSubmenu} />

**Nested popup mode** — a full nested `Root orientation="vertical"` with its own `Portal`/`Positioner` inside a parent `Content` opens a second flyout beside the first; nested Roots render `<div>` (not `<nav>`) and share a `FloatingTree` so dismissal propagates correctly.

## How it works

### The shared, morphing popup

Unlike Menu (one popup per trigger), **all** triggers feed one `Positioner`/`Popup`: the popup travels and resizes between triggers while `Content` cross-fades inside the `Viewport`, tagged with `data-activation-direction` (which way the new trigger lies relative to the previous one). Trigger measures the incoming `Content`, writes `--popup-width/height` (Popup) and `--positioner-width/height` (Positioner) as fixed pixel values so CSS can transition them, then resets both to `auto` after the animation settles so content can reflow while open ([#2070](https://github.com/mui/base-ui/pull/2070)).

<Canvas of={NavigationMenuStories.FlyoutViewportMorph} />

Because triggers and the shared popup coordinate on click as well as hover, clicking a _different_ trigger switches panels without a flicker-close — presses on other nav triggers are exempted from outside-press dismissal via an internal marker attribute.

<Canvas of={NavigationMenuStories.ClickToggleActivation} />

### Open interactions and the patient click

Hover opens after `delay` (default 50ms) using Floating UI's `safePolygon` hover intent, so diagonal travel from trigger to popup doesn't close it early. Click toggles the active item. Keyboard has its own open key (see Keyboard & screen reader). Within 500ms of a hover-open, a click will **not** toggle the popup shut — the "patient click" threshold, preventing the accidental open-then-instantly-close double-take when a user hovers and clicks near-simultaneously.

<Canvas of={NavigationMenuStories.PatientClickThreshold} />

There is no per-trigger click-only prop yet ([#2254](https://github.com/mui/base-ui/issues/2254), open — "I don't think it should support hover-only, because keyboard users need a way to access the content"); the documented approximation is a very large `delay` so hover effectively never opens while click and keyboard still work.

<Canvas of={NavigationMenuStories.DelayTuningClickOnlyApprox} />

### Touch

Hover never opens on `pointerType === 'touch'` — a tap opens, a tap outside closes. There is no separate mobile presentation mode.

<Canvas of={NavigationMenuStories.TouchTapToOpen} />

### Focus model — no trap, no focus-move on open

Opening does not move focus into the popup on pointer or click; focus stays on the trigger and `Tab` walks _into_ the panel via focus guards, then out to the next tabbable element, closing the menu when it exits entirely. The component is never modal: no scroll lock, no `inert` page marking — `Backdrop` is purely presentational.

<Canvas of={NavigationMenuStories.KeyboardNavigation} />

### Controlled value and close reasons

Open state is value-driven, not boolean: `value` names _which_ item is open (`open === value != null`). Control it with `value` + `onValueChange(value, eventDetails)` — every close path carries a typed `reason`: `trigger-press`, `trigger-hover`, `outside-press`, `escape-key`, `focus-out`, `link-press`, `list-navigation`, `none`.

<Canvas of={NavigationMenuStories.ControlledValueWithEventDetails} />

### Links, routers, and closeOnClick

Every link part accepts `render` to compose a client-side router Link, and `active` marks the current page with `aria-current="page"` + `data-active` for styling.

<Canvas of={NavigationMenuStories.LinkWithRenderClientRouter} />

`Link closeOnClick` defaults to `false` after a deliberate reversal ([#2535](https://github.com/mui/base-ui/pull/2535) → [#2740](https://github.com/mui/base-ui/pull/2740)): "Stripe and Apple leave theirs open as they act as external links". Opt in per link for client-side navigations within a persistent layout.

<Canvas of={NavigationMenuStories.CloseOnClickLinks} />

A top-level item that is both a link and a trigger breaks keyboard flow if you overload one element — the recommended pattern ([#4186](https://github.com/mui/base-ui/issues/4186)) is Apple.com's split: a plain `Link` label plus a separate keyboard-visible chevron `Trigger`.

<Canvas of={NavigationMenuStories.TriggerAndLinkSplitPattern} />

### Orientation

`orientation="vertical"` turns the bar into a side rail: `ArrowDown`/`ArrowUp` rove focus along the rail, and the open key mirrors to `ArrowRight` (`ArrowLeft` in RTL) — pair it with `Positioner side="right"`.

<Canvas of={NavigationMenuStories.VerticalOrientation} />

### Positioning

The Positioner exposes the standard anchored-positioning set (`side`/`align`/`sideOffset`/collision config); Popup and Arrow expose the resolved placement via `data-side`/`data-align`.

<Canvas of={NavigationMenuStories.PositioningSideAlign} />

### SSR/SEO with keepMounted

`Content keepMounted` renders the panel inline as `<div hidden>` in server HTML, "this ensures the content is crawlable," then moves it into the popup permanently on first open. `Portal keepMounted` is **not** needed for SEO alone — `Content keepMounted` suffices.

<Canvas of={NavigationMenuStories.KeepMountedSEOContent} />

### Animation

Popup/Positioner/Backdrop/Content expose `data-open`/`data-closed`/`data-starting-style`/`data-ending-style`; Content adds `data-activation-direction`; Positioner adds `data-instant` (initial open frame + window resize). `onOpenChangeComplete(false)` fires only after the exit transition settles.

<Canvas of={NavigationMenuStories.AnimatedMorphTransition} />

## Keyboard & screen reader

Pattern: [APG Disclosure Navigation](https://www.w3.org/WAI/ARIA/apg/patterns/disclosure/examples/disclosure-navigation/) — a `<nav>` landmark, a plain list, and `<button aria-expanded aria-controls>` disclosure triggers over link panels. This is deliberately **not** the ARIA menu pattern: there is no `role="menu"`, `menuitem`, or `menubar` anywhere, a choice pinned twice — once by reworking `List`/`Item` into real `<ul>/<li>` with `Link active` → `aria-current` ([#2526](https://github.com/mui/base-ui/pull/2526)), and again when an axe-audit request to add `role="menubar"` was declined in favor of removing the invalid `aria-orientation` instead ([#4349](https://github.com/mui/base-ui/issues/4349) → [#4355](https://github.com/mui/base-ui/pull/4355)).

| Key                                    | Context                      | Result                                                                                                                   |
| -------------------------------------- | ---------------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| `Tab` / `Shift+Tab`                    | anywhere                     | Sequential focus: triggers → (into an open panel's links) → next trigger; leaving the whole menu closes it (`focus-out`) |
| `ArrowRight` / `ArrowLeft`             | trigger focused (horizontal) | Move focus between top-level items (roving composite, no loop); does **not** open or switch content                      |
| `ArrowDown`                            | trigger focused (horizontal) | Open that item's panel, focus moves into it (reason `list-navigation`)                                                   |
| `ArrowRight` (LTR) / `ArrowLeft` (RTL) | trigger focused (vertical)   | Open (mirrored per direction)                                                                                            |
| `Enter` / `Space`                      | trigger focused              | Toggle the panel                                                                                                         |
| `Escape`                               | menu open                    | Close; focus returns to the trigger                                                                                      |
| Arrow keys                             | inside nested inline lists   | Reach nested submenu triggers through the parent composite (3+ levels supported)                                         |

[G] Home/End inside the List: no navigation-menu test pins this behavior; not claimed.

- **ARIA managed for you**: Root — `<nav>` at top level. List — `<ul>`, no `aria-orientation`. Trigger — `<button aria-expanded aria-controls={popupId}>` + `data-popup-open`. Popup — `<nav id>`. A visually-hidden `<span aria-owns={viewportId}>` after the active trigger keeps the portalled panel adjacent in the accessibility tree. Link — `aria-current="page"` when `active`. Icon — `aria-hidden="true"`. Backdrop — `role="presentation"`.
- **Screen reader**: browse-mode users read `nav > ul > li` structure with plain links; panel content is reachable adjacent to its trigger via `aria-owns`. No open-on-focus — tabbing across the bar never pops panels open (WCAG 3.2.1 "On Focus").
- **You must provide**: focus-visible styling, sufficient contrast, and an `aria-label` on `Root` when a page has multiple `<nav>` landmarks (`Popup` itself renders `<nav>`).
- **Known issues**: [#2254](https://github.com/mui/base-ui/issues/2254) — no per-trigger click-only mode (open); [#3570](https://github.com/mui/base-ui/issues/3570) — pre-hydration hover lost, needs a second hover pass (open); [#5103](https://github.com/mui/base-ui/issues/5103) — a suspending child inside `Content` crashes on first open, put the Suspense boundary inside `Content` (open); [#4496](https://github.com/mui/base-ui/issues/4496) — `keepMounted` preserves stale submenu state across reopens, by design (open). Fixed & shipped: Shift+Tab submenu re-entry ([#4464](https://github.com/mui/base-ui/pull/4464)); arrow keys reaching submenu triggers ([#3344](https://github.com/mui/base-ui/pull/3344)); duplicate/invalid `aria-orientation` ([#4309](https://github.com/mui/base-ui/pull/4309), [#4355](https://github.com/mui/base-ui/pull/4355)).

## Styling hooks

| Part                                    | Attribute / variable                                                      | Present when                                                                                                    |
| --------------------------------------- | ------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------- |
| Trigger                                 | `data-popup-open` / `data-pressed`                                        | the trigger's panel is open / pressed                                                                           |
| Icon                                    | `data-popup-open`                                                         | rotate the chevron when the panel is open                                                                       |
| Link                                    | `data-active`                                                             | `active` prop is set (pairs with `aria-current`)                                                                |
| Popup / Positioner / Backdrop / Content | `data-open` / `data-closed` / `data-starting-style` / `data-ending-style` | open/close transition lifecycle                                                                                 |
| Content                                 | `data-activation-direction` (`left`\|`right`\|`up`\|`down`)               | which way the new trigger lies relative to the previous one                                                     |
| Positioner                              | `data-instant`                                                            | initial open frame + window resize — disables transitions                                                       |
| Popup / Positioner / Arrow              | `data-side` / `data-align` / `data-anchor-hidden`                         | resolved anchored-positioning result                                                                            |
| Arrow                                   | `data-uncentered`                                                         | the arrow can't stay centered on the anchor                                                                     |
| Positioner (CSS vars)                   | `--available-width/height`, `--anchor-width/height`, `--transform-origin` | while positioned                                                                                                |
| Positioner (CSS var)                    | `--positioner-width/height`                                               | consume unconditionally — required for the morph to animate ([#2269](https://github.com/mui/base-ui/pull/2269)) |
| Popup (CSS var)                         | `--popup-width/height`                                                    | same — becomes `auto` once idle so content can reflow ([#2070](https://github.com/mui/base-ui/pull/2070))       |

Popup/Positioner **must** consume the size variables for morphing to work at all — `.Positioner { width: var(--positioner-width); height: var(--positioner-height) }`, `.Popup { width: var(--popup-width); height: var(--popup-height) }`, plus `[data-instant] { transition: none }`. Constrain scrollable panels with `max-height: var(--available-height)` and `box-sizing: border-box` on `Content`/`Popup` ("really wish it was CSS's default" — maintainer, [#2240](https://github.com/mui/base-ui/pull/2240)).

## Choosing the right props

- **`value` / `defaultValue` / `onValueChange` (Root)** — uncontrolled + `Item` auto-values for plain headers; add explicit `Item value`s + controlled `value` when syncing to router state or building disclosure analytics. Type it with `NavigationMenu.Root<MyId>` ([#4328](https://github.com/mui/base-ui/pull/4328)).
- **`delay` / `closeDelay` (Root, both default 50ms)** — raise `delay` to suppress accidental opens on hover-past ("Large dropdowns that shouldn't open on accidental hover", [#2254](https://github.com/mui/base-ui/issues/2254)); a very large `delay` is the documented click-only approximation.
- **`orientation` (Root, default `'horizontal'`)** — flips the composite arrow axis and the open key; used for vertical submenu rails and side navs.
- **`actionsRef` (Root)** — `{ unmount }`; providing it opts into manual unmount control for JS animation libraries.
- **`Item value`** — omit for static navs; set it when controlling the menu programmatically. Falsy values (`0`, `''`, `false`) are valid ([#4942](https://github.com/mui/base-ui/pull/4942)).
- **`Trigger disabled` / `nativeButton`** — disabled triggers never open (hover/click/keyboard/touch) but stay focusable; declare `nativeButton={false}` if `render` swaps in a non-button element.
- **`Content keepMounted` (default `false`)** — SEO/SSR switch: server-renders the panel as hidden inline HTML, then it lives in the popup permanently. Trade-off: kept content preserves internal state (scroll, nested submenu selection) across closes, by design.
- **`Link active`** — render-time flag for the current page; pair with your router's pathname.
- **`Link closeOnClick` (default `false`)** — set it on links that soft-navigate within a persistent layout; a nested `closeOnClick` link closes every ancestor menu too ([#4276](https://github.com/mui/base-ui/pull/4276)).
- **`Portal` (optional!) + `container`** — include it for the classic anchored flyout; omit `Portal`/`Positioner`/`Popup` entirely in nested Roots to get inline content swapping.
- **`Positioner` props** — `side` (default `bottom`), `align` (default `center`), `sideOffset`, `collisionPadding` (default 5), `sticky`, `disableAnchorTracking`. The hero sets `collisionAvoidance={{ side: 'none' }}` so a shared header popup never flips above the bar mid-morph.

## Do / Don't

**Do**

- Consume `--popup-width/height` and `--positioner-width/height` unconditionally in your CSS — the old media-query pattern broke when inline nesting shipped ([#2269](https://github.com/mui/base-ui/pull/2269)).
- Add `list-style: none` when upgrading from pre-beta.3 — `List`/`Item` became real `<ul>/<li>` and the PR was marked breaking for exactly this ([#2526](https://github.com/mui/base-ui/pull/2526)).
- Split an item that is both a destination and a disclosure into a plain `Link` label plus a separate chevron `Trigger` (Apple.com pattern) — never overload one element with both roles ([#4186](https://github.com/mui/base-ui/issues/4186)).
- Give `Root` an `aria-label` if the page has multiple `<nav>` landmarks — `Popup` itself renders `<nav>`.
- React to `onValueChange` and its typed `reason`, not `Trigger onClick` — open/close also flows through hover, keyboard, and dismissal paths.

**Don't**

- Don't set `--popup-width/height` to your own fixed values or expect them while idle — they intentionally reset to `auto` after each transition so content can reflow ([#2070](https://github.com/mui/base-ui/pull/2070)).
- Don't use `Portal keepMounted` for SEO alone — `Content keepMounted` is what makes the panel crawlable ([#4496](https://github.com/mui/base-ui/issues/4496)).
- Don't expect a first-class "link that is also a trigger" part — none exists; the split-chevron pattern is the documented recommendation, not a workaround for a missing feature ([#4186](https://github.com/mui/base-ui/issues/4186)).
- Don't reach for Navigation Menu for a plain row of links with no panels — HTML already does that.
- Don't port Radix idioms directly: `forceMount` → `Content keepMounted`; no `asChild` — use `render`; no `data-state` — use `data-open`/`data-popup-open`; there is no `NavigationMenu.Indicator` part (`Arrow` covers the pointer). Positioning is anchored per-trigger via Floating UI, not a CSS-Grid full-width strip — "we also investigated CSS Grid … but decided not to use it" ([#2253](https://github.com/mui/base-ui/issues/2253)).

- **2025-05 — [#1741](https://github.com/mui/base-ui/pull/1741)** — component ships (beta.0). Founding goals: nested menus and full interruptibility of transitions.
- **2025-06 — [#2070](https://github.com/mui/base-ui/pull/2070)** — layout resize while open: sizes reset to `auto` post-animation; documented follow-up in [#2240](https://github.com/mui/base-ui/pull/2240) (`max-height: var(--available-height)` + `box-sizing: border-box`).

## Known issues & open questions

- [#2254](https://github.com/mui/base-ui/issues/2254) — no per-trigger click-only mode; the documented workaround is a very large `delay` (open).
- [#3570](https://github.com/mui/base-ui/issues/3570) — hover events fired before hydration aren't replayed; a too-eager first hover needs a second pass (open).
- [#5103](https://github.com/mui/base-ui/issues/5103) — a suspending child (`React.lazy`/suspense query) inside `Content` crashes on first open; put the Suspense boundary inside `Content` (open).
- [#4496](https://github.com/mui/base-ui/issues/4496) — `keepMounted` preserves stale submenu state/scroll across reopens — by design, not a bug, but easy to be surprised by (open).
- [#4186](https://github.com/mui/base-ui/issues/4186) — no first-class "link that is also a trigger" part; the split-chevron pattern is the recommended composition, and can leave `pointer-events: none` stuck on `<body>` in rare route-change races (open).
