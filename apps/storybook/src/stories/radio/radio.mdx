# Radio

A mutually-exclusive, always-visible option picker, treated as one research unit with **Radio Group** — the two are never meaningfully used apart.

- **Import:** `import { Radio } from '@base-ui/react/radio';` + `import { RadioGroup } from '@base-ui/react/radio-group';`
- **Parts:** 3 — `Radio.Root`, `Radio.Indicator`, plus the single-part `RadioGroup`.
- **Status:** Stable. Radio Group's introducing issue is `#26` (2024), implementing PR `#487` (merged 2024-07-24).
- **ARIA pattern:** [Radio Group](https://www.w3.org/WAI/ARIA/apg/patterns/radio/)
- **Docs verified against:** `@base-ui/react` workspace source (v1.6.0 line).
- **⚠️ Packaging asymmetry:** `radio-group` is a fully public, independently exported subpath (own `index.ts`, own tests, own git history) — but it has **no standalone docs page**. All of its user-facing documentation lives inside `radio/page.mdx` under a `### RadioGroup` heading, and it's absent from the docs nav entirely. A user browsing the component list would never discover it exists as its own thing. This page (and this Storybook entry) documents both together for that reason.

## Features

- **Exclusive, always-one-selected, form-native** — `RadioGroup` wraps a roving-tabindex `CompositeRoot`; every option is visible at once and reachable by a single Tab stop.
- **Arrow keys both move focus *and* select** — the single most important behavioral fact for this component: pressing `ArrowDown`/`ArrowUp` moves focus to the next radio **and immediately selects it**, matching native `<input type="radio">` groups, unlike listbox-style composites (Select) where arrow keys only move a highlight.
- **Native `<input type="radio">` semantics are the contract** — every radio ships a hidden native input that participates in real form submission, `FormData`, and native validation.
- **Unselected state submits `null`**, not an empty string and not an omitted field — matching native radio-group behavior exactly (`#2473`).

## When to use

- A small, always-visible set of mutually exclusive options where the user picks exactly one (or leaves the group at its unselected `null` state).
- Inside forms where the selected value should submit natively.
- When every option should be simultaneously visible and comparable without an extra open/close interaction — RadioGroup's structural difference from [Select](?path=/docs/form-inputs-select--docs), a popup-based picker for the same "choose one" task.

## When not to use

- **Many options, or options that should stay collapsed until interacted with** → prefer [Select](?path=/docs/form-inputs-select--docs). Once the option count grows past a handful, the always-visible layout becomes a liability.
- **A single-select-you-can't-clear-back-to-nothing widget disguised as a toggle group** → use RadioGroup, not Toggle Group. A maintainer drew this line directly: "a single-select-you-can't-clear widget *is* a RadioGroup; making a toggle announce as a radio is semantically wrong" (`mui/base-ui#3525`).
- **A single on/off value, or a submit-time bulk boolean set** → use [Checkbox](?path=/docs/form-inputs-checkbox--docs) / [Checkbox Group](?path=/docs/form-inputs-checkbox-group--docs) instead — cardinality differs: Radio always resolves to "exactly one of N, or none," Checkbox resolves to "any independent subset."

## Anatomy

```tsx
import { Radio } from '@base-ui/react/radio';
import { RadioGroup } from '@base-ui/react/radio-group';

<RadioGroup>
  <Radio.Root>
    <Radio.Indicator />
  </Radio.Root>
  {/* …repeated per option */}
</RadioGroup>;
```

| Part | Renders | Purpose | Optional? |
|---|---|---|---|
| `RadioGroup` | `<div role="radiogroup">` | Shared state + roving-tabindex composite for a series of radios | required |
| `Radio.Root` | `<span role="radio">` (or `<button>` via `render` + `nativeButton`) | One option: state, interaction, hidden native input | required |
| `Radio.Indicator` | `<span>` | The selected-state visual; conditionally rendered, transition-aware | yes — purely visual |

## How it works

### Uncontrolled vs controlled

`defaultValue` for uncontrolled use; `value` + `onValueChange(value, eventDetails)` when external state drives selection. `eventDetails.cancel()` vetoes the change while staying uncontrolled.

<Canvas of={RadioStories.ControlledValue} />

`null` is a real, externally-reachable app state (matching the native-parity contract from `#2473`), not merely "nothing selected yet."

### Labeling

Form controls must have an accessible name. Wrap each option in a `<label>` (hero above), or label the whole group with `Fieldset.Legend` + per-option `Field.Item` for managed wiring (see [Field](?path=/docs/form-inputs-field--docs)).

<Canvas of={RadioStories.WithFieldAndFieldset} />

### Required / validation

<Canvas of={RadioStories.RequiredInvalidState} />

`required` on the `Field.Root`/`RadioGroup` pair flags `valueMissing` when nothing is selected on submit; selecting any option clears the error.

### Read-only groups

<Canvas of={RadioStories.ReadOnlyGroup} />

`readOnly` blocks every selection path (click, arrow-key auto-select, Space) while keeping the group focusable and its current value visible — distinct from `disabled`, which also removes it from the tab sequence.

### Rendering a native button

The default element is a `<span>` — required so an enclosing `<label>` doesn't produce invalid HTML wrapping two form controls (a `<button>` + hidden `<input>`). Pass both `render` and `nativeButton` to render an actual `<button>` for the sibling-label pattern.

### Disabled items

<Canvas of={RadioStories.DisabledItem} />

A `disabled` radio stays keyboard-discoverable (roving-tabindex composites keep disabled items focusable) but cannot be selected by click or keyboard.

### Form integration

<Canvas of={RadioStories.FormIntegration} />

With `name` on `RadioGroup`, the selected value submits via each radio's hidden native input. With **nothing selected**, the group submits `null` — matching native `<input type="radio">` group behavior, not an empty string.

## Keyboard & screen reader

<Canvas of={RadioStories.ArrowDownSelectsOnNavigation} />

<Canvas of={RadioStories.SpaceSelectsOnKeyUp} />

<Canvas of={RadioStories.HomeEndHaveNoEffect} />

| Key | Context | Action |
|---|---|---|
| `Tab` | anywhere before/in the group | Moves focus to the group as one stop — lands on the checked radio if one exists, otherwise the first enabled radio |
| `ArrowDown` / `ArrowRight` (LTR) | focus in the group | Moves focus to the next radio **and selects it**, wraps last→first |
| `ArrowUp` / `ArrowLeft` (LTR) | focus in the group | Moves focus to the previous radio **and selects it**, wraps first→last |
| `Shift` + arrow keys | focus in the group | Moves focus **only** — suppresses the auto-select-on-focus behavior |
| `Home` / `End` | focus in the group | **No effect, by design** — `enableHomeAndEndKeys={false}` is passed explicitly, a deliberate difference from listbox-style composites like Select, which does support Home/End |
| `Space` | a focused, unchecked radio | Selects on **keyup**, not keydown |
| `Enter` | a focused radio | No effect — radios never activate on Enter |
| Click | the visible radio or its hidden input | Selects (unless `disabled`/`readOnly`) |

- **ARIA**: `role="radiogroup"` on the container, `role="radio"` + `aria-checked` on each option; `aria-labelledby` auto-resolved from `Field.Label`/`Fieldset.Legend` (`#4142`).
- **You must provide**: an accessible name for the group (label element, `Field`/`Fieldset`, or `aria-labelledby`) and for each option, plus visible `:focus-visible` styling.
- **Known issues**: `.radio:hover` does not fire when hovering an enclosing `<label>` (post-`#3205`'s `<span>` shape) — an acknowledged CSS-authoring surprise, not a bug (`#3263`).

## Styling hooks

State attributes present on **both** `Root` and `Indicator`:

| Attribute | Present when |
|---|---|
| `data-checked` / `data-unchecked` | the radio is selected / not selected |
| `data-disabled` | disabled |
| `data-readonly` | read-only |
| `data-required` | required in a Field |
| `data-valid` / `data-invalid` | Field validation state |
| `data-touched` / `data-dirty` / `data-filled` | Field interaction state |
| `data-focused` | visibly focused |

## Choosing the right props

- **`nativeButton` + `render`** — use for the sibling-label pattern (`htmlFor`/`id`); the default `<span>` supports enclosing labels without the invalid-HTML problem.
- **`name`/`form` (RadioGroup)** — `name` (or the enclosing `Field.Root`'s `name`, which wins) identifies the submitted value; `form` targets an owning `<form>` when the group renders outside it.
- **`inputRef` (RadioGroup)** — unusually, points to *the checked radio's* native input, re-pointed dynamically as selection changes.

## Do / Don't

**Do**

- Give the group and every radio an accessible name — the consumer half of the a11y contract.

**Don't**

- Don't expect Home/End to jump to the first/last option — that's deliberately unsupported (`enableHomeAndEndKeys={false}`), not a bug.

## Decision log

- **2024-07-24** — Radio Group ships (`#487`, closing `#26`), a ground-up answer to accumulated pain documented in `mui/material-ui#6218`.
- **2025-11 (`#3205`, breaking)** — Root renders `<span role="radio">` instead of `<button>`, driven by the invalid-HTML-when-wrapped-in-`<label>` constraint; introduces `nativeButton`.
- **2025-12 (`#4142`, closing `#4122`)** — `aria-labelledby` auto-linking fix, resolving a WCAG failure verified across NVDA/Narrator/JAWS.
- **2026 (`#4930`)** — "Fix selection on space key press": pins Space-selects-on-keyup as a fixed regression, not incidental.
- **2026 (`#4926`)** — "Fix disabled selected form submission": hardens the disabled-radio-excluded-from-form-data contract.

## Related

- For a mutually-**independent** set of boolean values, use [Checkbox Group](?path=/docs/form-inputs-checkbox-group--docs).
- For a single boolean setting with **immediate** effect, use [Switch](?path=/docs/form-inputs-switch--docs).
- To add labels, descriptions, and validation, wrap in [Field](?path=/docs/form-inputs-field--docs).
